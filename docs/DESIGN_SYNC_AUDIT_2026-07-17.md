## 1. 알림 설정 (A-3)

### 1.1 자가 검수 체크리스트

- [x] `BackHeader`·`SettingsRow`·세그먼트 버튼 그룹을 새로 그리지 않고 재사용 — **부분 재사용**.
      헤더는 기존 `BackHeader`와 동등(‹ + 타이틀). 행은 `SettingsRow`(경계선 목록)가 아니라
      **카드형 섹션**(rounded-14, 흰 배경, 섹션당 개별 카드)이다 — 토글 컨트롤을 담기 위한
      정당한 변형으로 판단, 결함 아님(신규 컴포넌트 도입이므로 새 레이아웃 필요).
- [x] 토글 표현 (A)/(B) 명시 — **목업이 명확히 (A) 신규 pill 스위치를 채택**(주석: "코드베이스에
      기존 토글이 없어 새로 정의"). 세그먼트 버튼 재사용(B)이 아니다 — 계획 수정 필요(§1.3).
- [x] "승인 요청 즉시 알림" 비활성 + 사유 캡션 — 반투명 파랑 트랙(`rgba(0,102,255,.35)`) +
      잠금 아이콘 노브 + "성급한 승인을 막기 위해 항상 켜져 있으며, 끌 수 없습니다."
- [x] "채널 우선순위" 읽기 전용 구분 — `›` 대신 "읽기 전용" 배지. 통과.
- [x] 색상 전부 토큰 안 — `#0066FF`/`#171719`/`rgba(55,56,60,.61/.88)`/`rgba(112,115,124,.08/.22)`/
      `#F7F7F8`/`#FFFFFF`/`#D47800`+`#FEF4E6`(HIGH 페어) 전부 일치. **`rgba(55,56,60,.43)`**
      1건 사용(시각 힌트 아이콘·읽기전용 배지 텍스트) — 이는 **기존 알려진 결함**
      (`rules/design.md` §2 각주1, disable을 .43으로 쓰는 디자인 프로젝트 관행)의 재발이지
      새 결함 아님. **잠금 트랙 색 `rgba(0,102,255,.35)`**는 토큰표에 없는 새 값이나 primary의
      반투명 파생이라 임의 hex로 보지 않는다(신규 "locked" 상태 표현이므로 허용).
- [x] 단정형·이모지·느낌표 — 0건.

### 1.2 추가 발견 (체크리스트 밖)

- 화면 타이틀은 **"알림 설정"**이다(브리프/계획 초안의 "알림"이 아님) — 구현 시 헤더 타이틀 정정.
- "아침 브리핑 시각" 행이 **값+슈브런(`›`) 표시**로 그려져 있다 — 현재 `SettingsHubPage`의
  인라인 `<input type="time">`과 다른 시각 처리. 목업 자체엔 이 행에 대한 `onClick` 배선이
  없어(다른 토글 행과 달리 상호작용 미구현) 실제 의도(네이티브 피커를 여는 스타일링된
  `<input type="time">`인지, 별도 시트로 여는 것인지)가 목업만으론 확정 불가.
  → **감사 판정**: 신규 시트를 만들지 않는다(스코프 확장 방지) — 기존 `<input type="time">`을
  이 행의 시각(값 오른쪽 정렬 + 슈브런 아이콘)에 맞게 스타일링해서 재사용한다(네이티브 input을
  커스텀 스타일로 감싸는 흔한 패턴 — 기능은 그대로, 겉모습만 목업에 맞춘다).
- **"승인 요청 즉시 알림" 노출 범위가 브리프와 다르다.** 목업 스크립트: `showApprovalRow: role
  === '대표'`(owner만). 브리프: "승인 권한자(owner/manager)에게만 노출". 도메인 모델
  (`APPROVER_ROLES = ('owner','manager')`, `DEFAULT_POLICY = 'manager_allowed'`로 매니저가
  일상적 승인자)과 대조하면 매니저에게 이 사실을 안 보여주는 건 실사용상 공백이다.
  → **판정: 디자인 내부 결함으로 기록, 구현은 브리프대로 owner+manager 둘 다에게 노출**
  (기존 감사 관례 — 목업이 코드/도메인 규칙과 어긋나면 임의 보정 없이 기록만 남기고
  브리프·규칙 기준으로 구현). 1b 정적 화면의 "담당자 — 승인 요청 알림 행 비노출" 비교
  프레임은 이 결함을 그대로 보여주는 참고 자료로만 남긴다.

### 1.2b ui-matcher 대조 결과 (구현 완료 후, 2026-07-17)

- [수정 완료] 캡션 박스 2곳(승인 요청 잠금 안내, CRITICAL 야간 OFF 경고)에 목업에 있던 장식
  아이콘(자물쇠·시계)이 구현에 빠져 있었다 — `IconLock`/`IconClock` 추가로 반영.
- [수용, 미수정] 잠금 노브 안 `IconLock`의 stroke-width가 목업(2.4)과 다르다(공용 아이콘
  기본값 1.8) — 이 화면 하나를 위해 다른 화면도 쓰는 공용 아이콘의 stroke-width를 바꾸는 건
  과도해 그대로 둔다. 10px 크기에서 시각적 차이는 미미하다.
- [수용, 미수정] 카드 섹션 라운드가 목업(14px)과 다르다(`rounded-in`=10px 재사용) —
  `SettingsHubPage.tsx`의 기존 카드 패턴을 그대로 따른 결과이자 기존 부채(정확한 14px 카드
  토큰이 `tokens.css`에 없음, 가장 가까운 기존 값은 `rounded-card` 12px). 이 화면만 새 토큰을
  추가하지 않는다.
- **[판정 확정] "승인 요청 즉시 알림" 부제 카피에서 "(대표)"를 뺐다** — 목업 원문
  "승인 권한자(대표)에게만 표시됩니다"는 owner-only 노출 전제의 카피다. §1.2에서 이미
  owner+manager 노출로 결정했으므로 "(대표)"를 남기면 매니저에게 잘못된 정보를 주게 된다 —
  카피 삭제는 노출 범위 결정의 필연적 파생이지 별도 창작이 아니다.
- **[판정 확정] viewer 차단 화면은 목업의 신규 아이콘 배지를 재현하지 않는다** — 목업은 이
  상태를 원형 배지+제목+부제 2줄로 새로 그렸으나, 구현은 같은 설정 화면군의 기존 3화면
  (`MembersPage`·`DelegationPage`·`SettingsHubPage`)과 동일한 텍스트 1줄+버튼 패턴을 그대로
  썼다. 이 화면 하나만 새 패턴을 쓰면 설정 화면군 안에서 viewer 차단 표현이 갈라진다 — 개별
  화면의 목업 재현보다 화면군 내부 일관성을 우선한다(GOTCHAS §4). 4화면 전체를 배지 패턴으로
  통일하는 건 이번 스코프 밖(후속 과제로 남김, 아래 §5 참고).

카드는 삭제하고, 같은 자리에 "스캔 업로드 시작" 버튼(`nav.toCasesScan()`)으로 교체 —
`nav.toCasesImport()` 버튼과 동일한 패턴.

### 2.3 확인 대기 반영 게이트 — 브리프 가정과 실제 로직이 다르다

**브리프 가정**: CSV의 "정상 M명만 등록"과 동일하게, 미매칭·확인 필요 행은 자동 제외하고
정상 매칭 건만 확인 대기로 올린다.
**목업 실제 로직**(`confirm()` 함수): 미매칭 행(이 데모에선 id=6)에 근로자가 **수동 지정되기
전까지 "확인 대기로 올리기" 버튼 자체가 비활성**이다(`canConfirm = !!(corrections[6]?.worker)`).
확인 필요(저신뢰) 행은 수정 없이도 그대로 포함된다. 확정 힌트 문구: 미해결 시 "미매칭 건에
근로자를 지정해야 진행할 수 있습니다", 해결 시 "정상 매칭·확인 필요 건도 함께 확인 대기로
올라갑니다". 완료 메시지도 전체 건수("6건이 확인 대기로 올라갔습니다")를 말한다 — 일부 제외
언급이 없다.

**판정: 목업 로직 채택.** 미매칭 건이 하나라도 남아있으면 confirm 버튼 비활성 — CSV처럼
"나쁜 행은 자동으로 빼고 진행"이 아니라 "아무 것도 빠뜨리지 않고 사람이 전부 확인·지정한 뒤
전량 반영"이 이 화면의 원칙이다(서류 확보 여부는 근로자 개개인에게 중요해 자동 누락이 CSV
등록 실패보다 리스크가 크다는 판단으로 이해). `docScan.eligibleForPending()` 같은 "제외" 함수는
만들지 않는다 — 대신 `hasUnresolvedRows(rows)` 게이트 함수로 confirm 버튼 활성화 여부만 계산.

### 2.4 보정 UI — `<select>`가 아니라 인라인 버튼 그룹

행 클릭 → 행 아래로 패널이 펼쳐지며 "근로자 지정"(버튼 3개: 이 데모는 미매칭 근로자 후보
Oyunaa T/Batbayar E/Pham Duc M) + "서류 유형 지정"(버튼 3개: 여권/재직확인서/외국인등록증) +
"적용" 버튼. 네이티브 `<select>` 두 개가 아니라 이 버튼 그룹 패턴으로 구현한다(선택된 버튼은
`inset 0 0 0 1px rgba(0,102,255,.43)` + 파란 텍스트, 미선택은 회색 아웃라인).

1. **라우트 신설**: `ROUTES.casesScan`/`ROUTE_PATHS.casesScan` = `/cases/scan`,
   `nav.toCasesScan()`, `CaseWorkbenchPage`류 패턴처럼 `CasesScanPage`(PC 전용 게이트, 모바일은
   `PcOnlyNotice`) + `DocScanWorkbench.tsx`(실제 4단계 플로우) 이중 구조.
2. **좌측 스텝 트래커**: 4단계(업로드/분류 중/결과 확인/확인 대기 반영), done=초록 체크 원,
   active=파랑 펄스 링(`StepTimeline`과 동일 모션 어휘), pending=회색 아웃라인 원.
3. **중앙 플로우**: idle(드롭존) → classifying(shimmer skeleton, 수치 없음) → review(필터 칩
   4개 + 결과 테이블, 행별 인라인 보정 패널) → done(완료 아이콘 + 재업로드 버튼).
4. **확인 대기 반영**: `caseStore.applyInterpretationUpdates` 재사용(기존 `docUpdates` 계약,
   `doc.name` 키). confirm 게이트는 미매칭 행 전부 해결 여부만 검사(§2.3). statusLabel은
   **`'스캔 확인 대기'`**로 — 활동 타임라인의 "담당자 확인 대기"(해석 확인 대기 의미) 및 기존
   `'확인 필요'`(이 화면 자체의 확인-필요 칩 라벨)와의 표기 충돌을 피한다.
5. **Evidence**: `'tool_executed'` 재사용(신규 EvidenceType 발급 없음 — CSV 확정이
   `'plan_created'`를 재사용하는 것과 동일 관례). summary는 건수만("서류 스캔 분류 — N건 확인
   대기 반영"), 파일명은 절대 포함하지 않는다(파일명에 근로자 이름이 들어있을 수 있어 PII
   인접 정보).
6. **썸네일**: 항상 장식용 placeholder(회색 라인 + 검은 마스킹 바 아이콘) — 실제 업로드
   파일을 렌더링하지 않는다.
7. `WorkerDataWorkbench.tsx` 우측 레일의 "준비 중" 카드 → "스캔 업로드 시작" 버튼으로 교체.

## 3. 근로자 응답 링크 (A-1)

### 3.1 자가 검수 체크리스트

- [~] 만료/링크없음 상태가 `ExpertLinkPage` 레이아웃 그대로 재사용 — **아니다, 개선된 패턴**.
      목업은 원형 아이콘 배지(만료=주황 시계, 무효=회색 물음표, 이미응답=파랑 체크)+타이틀+
      설명으로 구성 — `ExpertLinkPage`의 단순 텍스트 박스보다 더 정교하다. 결함이 아니라
      **목업이 기존 화면보다 나은 패턴을 제시**한 경우 — `ExpertLinkPage`를 이 수준으로
      맞추는 건 이번 스코프 밖(후속 과제로만 기록), 이 화면 자체는 목업대로 구현한다.
- [x] inset box-shadow — 프리셋 선택 `inset 0 0 0 2px #0066FF`, 미선택
      `inset 0 0 0 1px rgba(112,115,124,.22)`. 통과.
- [x] 응답 원문 재노출 없음(이미 응답함) — S4는 안내문 2줄뿐, 이전 응답 내용 전혀 없음. 통과.
- [x] "담당자 확인 후 반영" 안내 — 제출 전("답변은 담당자 확인 후 반영됩니다")·제출 후
      ("답변은 담당자 확인 후 반영됩니다", 잠금 안내 별도) 둘 다 존재.
- [x] PII 필드 없음 — 이름·서류 목록만, 식별번호류 전무.
- [x] 단정형·이모지·느낌표 — 0건.
- [x] PC 캔버스 없음, 모바일 320~402px 프레임만.

### 3.2 추가 발견 (브리프가 열어뒀던 결정들이 확정됨)

- **언어 토글은 실제 인터랙티브 요소로 확정** — vi/ko 세그먼트 버튼 쌍(우상단). 브리프의
  "vi 기준 1벌 + 병기 여부는 디자인 결정"이 이제 "인터랙티브 토글"로 확정됐다.
- **프리셋은 단일선택 라디오** — 3개 중 정확히 1개만 선택 가능(다중선택 아님).
- **제출 게이트는 프리셋 선택 여부만 본다** — `canSubmit = preset !== null`. 자유 텍스트만
  입력하고 프리셋을 고르지 않으면 제출 불가. (이전 계획의 "선택도 입력도 없으면 비활성" 테스트
  문구는 부정확 — "프리셋 미선택 시 항상 비활성, 자유 텍스트는 게이트에 영향 없음"으로 정정.)
- **자유 텍스트는 실제 textarea여야 한다** — 목업의 `toggleFree`(버튼 클릭 시 placeholder↔
  샘플 텍스트 전환)는 `.dc.html` 인터랙션 시뮬레이션의 한계(실제 키보드 입력 불가)로 대체한
  장치다. 실제 구현은 `min-height:88px`, `inset` 박스 스타일의 진짜 `<textarea>`.
- **정확한 카피 사전 확보** — 스크립트에 vi/ko 전체 문자열 딕셔너리가 그대로 있다(아래 §3.3에
  옮김) — 창작 없이 그대로 재사용한다.
- **데모 매핑**: 발신자 "김민수 담당자" / 회사 "그린푸드 제조"(기존 `DEFAULT_PROFILE.name`과
  일치) / 근로자 "Nguyen Van A" / 메시지 내용이 체류기간 연장 서류 요청 — 이 조합은 기존
  `mocks/threads.ts`의 **`nguyen` 스레드**와 자연스럽게 대응한다(Bayar가 아님 — 이전 계획의
  "bayar 데모 토큰" 가정을 정정).
- 화면 타이틀/서비스명 표기: ko "외고반장", vi "Ngoại Cao Ban Trưởng"(음역).

### 3.3 카피 사전 (그대로 재사용 — 창작 금지 원칙 충족)

```
vi: 확인했습니다=Tôi đã xác nhận · 다시 보내주세요=Vui lòng gửi lại · 질문이 있습니다=Tôi có câu hỏi
    답변 선택=Chọn câu trả lời · 추가 내용=Nội dung thêm · 선택 사항=Không bắt buộc
    답변은 담당자 확인 후 반영됩니다=Câu trả lời sẽ được phản ánh sau khi người quản lý xác nhận.
    답변 보내기=Gửi câu trả lời · 답변을 보냈습니다=Đã gửi câu trả lời
    답변을 다시 보낼 수 없습니다=Không thể gửi lại câu trả lời.
ko: (위 각 항목의 한국어 원문 — 브리핑 상세는 목업 스크립트 strings().ko 참고)
```

### 3.4 구현 계획 수정 사항

- 언어 토글 인터랙티브 구현(vi 기본 진입 — 스레드의 `workerRef`/메시지 `lang` 필드 기준,
  ko는 확인용 병기).
- 프리셋 단일선택 라디오, 제출 게이트는 프리셋 유무만.
- 실제 `<textarea>`(자유 입력, 제출 게이트 영향 없음).
- 만료/무효/이미응답 상태는 목업의 아이콘 배지 패턴으로 신규 구현(3종 원형 배지 색:
  주황 `#FEF4E6`/`#D47800`, 회색 `rgba(112,115,124,.08)`/`#70737C`, 파랑 `#EAF2FE`/`#0066FF`).
- 데모 fixture는 `nguyen` 스레드 매핑, 발신자/회사/메시지 카피는 목업 문자열 그대로.
- `threadStore.receiveInbound` 신설안(기존 계획 유지 — 이번 목업 확인으로 안전성 재검증
  완료, `ThreadPage`/`MessagesWorkbench`의 `interpretation` 가드 재확인함).
