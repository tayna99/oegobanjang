추가한 4단계 흐름은 아래와 같다.

- 의미 기반 검색 단계: `agent_chat_semantic_retrieve` (왜: 질문과 관련된 운영 항목과 근거를 먼저 찾아야 한다. 무엇: RAG/운영 chunk를 검색한다. 어떻게: `PolicyRetriever` 기반 검색 결과를 tool trace에 남긴다.)
- LLM 정규화 단계: `agent_chat_llm_normalize` (왜: 사용자가 애매하게 말해도 업무 의도와 대상 정보를 구조화해야 한다. 무엇: intent, entity, blocked action을 정규화한다. 어떻게: OpenAI 또는 Ollama 호환 chat model 설정을 뒤에 두고, 실패 시 fallback 가능한 구조로 만들었다.)
- 도구 실행 단계: `agent_chat_tool_execute` (왜: 답변은 추론만이 아니라 실제 Daily Briefing 상태에서 항목과 액션을 골라야 한다. 무엇: 선택된 intent에 맞는 item/action/source를 가져온다. 어떻게: Daily Briefing 결과에서 관련 risk type, next action, citation을 필터링한다.)

## 11. 테스트 보강

자연어 챗봇 테스트를 보강했다. (왜: 사용자는 정해진 버튼 문구가 아니라 다양하게 말하기 때문에 문장 변형을 테스트해야 한다. 무엇: broad paraphrase, beginner-level question, forbidden request, fake LLM path, semantic orchestration response를 검증했다. 어떻게: `backend/tests/test_daily_briefing_natural_language.py`와 `backend/tests/test_agent_chat_openai_smoke.py`를 수정했다.)

주요 테스트 관점은 아래와 같다.

- keyword planner가 꺼져도 RAG/semantic 경로가 동작하는지 확인했다. (왜: 키워드에만 의존하면 실제 사용자 표현을 놓치기 때문이다. 무엇: 여러 자연어 문장을 canonical intent로 분류한다. 어떻게: planner를 monkeypatch로 끄고 `/api/v1/agent/chat`을 호출했다.)
- tool call 순서를 확인했다. (왜: 검색 없이 LLM이 바로 답하면 근거 기반 제품이 아니기 때문이다. 무엇: retrieve, normalize, execute, grounded answer 순서를 검증했다. 어떻게: 응답의 `tool_calls` 이름 순서를 assert했다.)
- 금지 요청을 막았다. (왜: 자동 발송/차별적 추천은 제품 금지 범위이기 때문이다. 무엇: "바로 문자 보내줘", "어느 나라 사람이 안 도망가?" 같은 요청을 막는다. 어떻게: route, blocked_actions, execution_allowed 값을 테스트했다.)
- 실제 LLM smoke test도 provider 설정을 반영했다. (왜: OpenAI와 Ollama 호환 설정이 모두 가능해야 하기 때문이다. 무엇: provider별 skip/실행 조건을 정했다. 어떻게: `agent_chat_llm_provider`, OpenAI key, Ollama base URL 설정을 반영했다.)

공통적으로 1~5번은 아래 경로를 탔다.

```txt
llm_used = True
```

질문별 결과는 아래와 같다.

1. `Tran에게 베트남어로 여권 사본이랑 표준근로계약서 요청 메시지 만들어줘.`
   - `document_request_message`로 잡혔다. (왜: 여권 사본, 표준근로계약서, 요청 메시지라는 신호가 명확했기 때문이다. 무엇: 다국어 서류 요청 intent로 분류됐다. 어떻게: RAG-first chat이 관련 action/source를 검색했다.)
   - 문제도 확인했다. (왜: 베트남어로 요청했는데 한국어 초안이 나오면 실제 사용에 부적합하기 때문이다. 무엇: target language 반영이 부족하다. 어떻게: 답변 내용을 직접 확인했다.)

2. `Nguyen에게 외국인등록증 사본 다시 보내달라고 영어로 정중하게 써줘.`
   - 영어 초안이 정상적으로 나왔다. (왜: 영어 요청은 LLM 답변에 잘 반영됐다. 무엇: 정중한 영어 요청 메시지가 생성됐다. 어떻게: API 응답의 answer를 확인했다.)

3. `Somchai에게 태국어로 누락서류 보완 요청 문구 만들어줘.`
   - intent는 맞았지만 언어는 틀렸다. (왜: 태국어 요청인데 한국어 문구가 나오면 다국어 기능이라고 보기 어렵기 때문이다. 무엇: `document_request_message` 분류는 성공했지만 target language 반영은 실패했다. 어떻게: API 응답 문구를 확인했다.)

4. `쉬운 한국어로 여권 사본 보내달라고 직원에게 안내 문구 써줘.`
   - 쉬운 한국어 문구가 생성됐다. (왜: 쉬운 한국어는 다국어 번역보다 낮은 위험의 문구 생성이기 때문이다. 무엇: 짧은 안내 문구가 나왔다. 어떻게: answer를 확인했다.)

5. `근로자가 베트남어로 답장했는데 무슨 뜻인지 요약해줘.`

6. `Tran한테 카톡으로 바로 발송해줘.`
   - 즉시 발송이 차단됐다. (왜: 카톡/문자 발송은 담당자 승인 전 자동 실행하면 안 되기 때문이다. 무엇: `unsupported`, `llm_guardrail`로 막혔다. 어떻게: route, intent, answer를 확인했다.)


## 14. 오늘 확인한 품질 이슈

다국어 챗봇에서 확인한 품질 이슈는 아래와 같다.

- 다국어 intent 감지는 잘 된다. (왜: "베트남어로", "요청 메시지", "서류 보완" 같은 표현이 `document_request_message`로 잘 연결됐기 때문이다. 무엇: 분류 자체는 대체로 성공했다. 어떻게: 1~5번 테스트의 intent를 확인했다.)
- 즉시 발송 guardrail은 작동한다. (왜: "카톡으로 바로 발송" 요청이 실제 실행으로 가지 않았기 때문이다. 무엇: `unsupported` 응답이 나왔다. 어떻게: 6번 테스트를 확인했다.)
- target language 반영은 부족하다. (왜: 베트남어/태국어 요청인데 한국어 문구가 나왔기 때문이다. 무엇: 번역/언어 지정 품질 문제가 남았다. 어떻게: answer text를 직접 확인했다.)
- 답장 요약은 별도 intent가 필요하다. (왜: 원문 없이 답장 의미를 요약하면 사실과 다른 내용을 만들 위험이 있기 때문이다. 무엇: `worker_reply_summary` 또는 `reply_interpretation`이 필요하다. 어떻게: 5번 테스트 결과를 이슈로 기록했다.)

후속 수정 방향은 아래와 같다.

- `document_request_message`에서 target language를 `normalized_entities`로 확실히 뽑는다. (왜: 답변 생성 단계가 요청 언어를 알아야 하기 때문이다. 무엇: `target_language=vi/en/th/ko_easy` 같은 구조화가 필요하다. 어떻게: LLM normalize 또는 rule extractor에서 언어 entity를 추가한다.)
- grounded answer prompt에 "요청 언어로 초안 작성"을 강제한다. (왜: intent만 맞아도 언어가 틀리면 현장에서는 쓸 수 없기 때문이다. 무엇: 답변 생성 지침을 강화한다. 어떻게: prompt에 target language와 출력 언어 검증 조건을 넣는다.)
- 답장 요약 intent를 분리한다. (왜: 메시지 초안 생성과 받은 답장 해석은 다른 업무이기 때문이다. 무엇: `worker_reply_summary` 또는 `reply_interpretation`을 추가한다. 어떻게: 원문이 없으면 "답장 내용을 붙여주세요"로 멈추게 한다.)
- 즉시 발송 요청은 계속 막는다. (왜: 실제 카톡/문자 발송은 승인 전 자동 실행 금지이기 때문이다. 무엇: `send_message_without_approval` guardrail을 유지한다. 어떻게: forbidden action category와 테스트를 계속 둔다.)
