## 1. Durable LangChain checkpoint 기반 추가

LangChain 1.0 runtime에 durable checkpoint 기반을 추가했다. (이전에는 runtime snapshot과 제품 상태용 checkpoint는 있었지만, LangChain agent 실행 흐름 자체를 재개할 수 있는 checkpoint metadata는 별도로 남기지 않았다.)

구현한 내용:

- `langgraph-checkpoint-sqlite` 의존성을 추가했다. (LangChain `create_agent`가 내부적으로 LangGraph 기반 실행 그래프를 쓰기 때문에, 실행 상태를 SQLite checkpoint로 남기기 위한 공식 경로를 사용했다.)
- `backend/data/langchain_checkpoints.sqlite3`를 checkpoint DB 경로로 분리했다. (앱 운영 DB와 agent execution checkpoint를 섞지 않기 위한 것이다.)
- `RuntimeContext` 실행 시 `thread_id`와 `checkpoint_ns`를 항상 전달하도록 했다. (같은 request가 어떤 LangChain 실행 흐름에 속하는지 추적하려면 thread 단위가 필요하다.)
- `langchain_agent_checkpoints` 모델과 migration을 추가했다. (checkpoint 원본 전체를 앱 DB에 넣는 대신, request_id, thread_id, namespace, latest checkpoint id, interrupt id, status 같은 조회용 metadata만 저장하기 위한 것이다.)
- checkpoint resume service를 추가했다. (승인된 내부 action만 resume할 수 있고, 외부 발송이나 정부 제출은 계속 차단하기 위한 서비스 경계다.)

중요한 결정:

- `langgraph` dependency는 제거하지 않았다. (LangChain 1.0 `create_agent` 자체가 내부적으로 LangGraph를 쓰기 때문에 dependency를 지우면 오히려 runtime이 깨질 수 있다.)
- 제거한 것은 repo가 직접 관리하던 custom `legacy_graph` runtime이다. (우리 코드가 직접 node orchestration을 관리하지 않고 LangChain runtime에 넘기려는 결정이다.)

## 2. Approval resume을 내부 action까지만 허용

승인 이후 처리 흐름을 "실제 발송"이 아니라 "내부 상태 전이"로 제한했다. (사람 승인이 있다고 해서 곧바로 후보자 메시지 발송, 행정사 전달, 정부 제출을 자동 실행하면 제품 리스크가 너무 크다.)

허용한 action:

- `finalize_internal_draft` (내부 검토용 초안을 확정 상태로 표시하는 작업이다.)
- `mark_handoff_package_ready` (외부에 보내기 전 handoff package가 준비됐다고 표시하는 작업이다.)
- `prepare_external_delivery` (실제 발송이 아니라 외부 전달 준비 상태까지만 만드는 작업이다.)

계속 차단한 action:

- `auto_send_to_candidate` (후보자에게 자동 발송하지 않는다.)
- `auto_send_to_sending_agency` (송출회사에 자동 전달하지 않는다.)
- `auto_send_to_admin_scrivener` (행정사에게 자동 전달하지 않는다.)
- `auto_submit_to_government_portal` (정부 포털에 자동 제출하지 않는다.)

이렇게 나눈 이유는 승인 이후에도 "실행 가능한 내부 상태 전이"와 "외부 세계에 실제 영향을 주는 행동"을 분리해야 하기 때문이다. (외부 행동은 outbox worker, idempotency key, 승인 감사 로그, 실패 복구 정책이 준비된 뒤 별도 mission으로 열어야 안전하다.)

## 3. Metrics와 worker lookup 보강

운영 관측을 위해 request 단위 metrics뿐 아니라 company 단위 summary metrics 조회를 추가했다. (운영자가 "이 회사에서 얼마나 자주 blocked 됐는지", "승인 대기는 몇 건인지", "retrieval은 얼마나 일어났는지"를 원문 prompt 없이 봐야 하기 때문이다.)

추가한 summary 항목:

- `run_count` (runtime 실행 횟수다.)
- `blocked_count` (safety/preflight 등으로 차단된 횟수다.)
- `approval_pending_count` (승인 대기 상태가 남은 횟수다.)
- `provider_error_count` (모델 제공자 오류 같은 실패 횟수다.)
- `retrieval_count` (RAG 검색 횟수다.)
- `avg_model_duration_ms` (모델 호출 평균 시간이다.)
- `avg_tool_duration_ms` (tool 호출 평균 시간이다.)

민감정보 원문은 저장하거나 반환하지 않도록 유지했다. (운영 metrics는 품질과 비용을 보기 위한 것이지, prompt나 개인정보를 다시 노출하기 위한 곳이 아니기 때문이다.)

또한 `/agent/run` 입력에서 `worker_id`가 없고 `worker_name`만 있는 경우 DB에서 같은 회사의 active worker를 조회하도록 했다. (현장 사용자는 ID보다 이름을 입력할 가능성이 높기 때문이다.)

lookup 정책:

- 정확히 1건이면 `worker_id`를 채운다. (agent runtime이 기존 DB 상태를 정확히 읽을 수 있게 하기 위한 것이다.)
- 0건이면 `worker_lookup_status=not_found`로 남긴다. (API를 4xx로 깨지 않고 agent가 추가 확인을 요청하게 하기 위한 것이다.)
- 2건 이상이면 `worker_lookup_status=ambiguous`로 남긴다. (동명이인을 잘못 연결하지 않기 위한 것이다.)
- Evidence Log와 응답에는 `worker_name` 원문을 남기지 않는다. (이름도 개인정보에 해당할 수 있기 때문이다.)
