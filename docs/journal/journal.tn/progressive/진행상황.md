## history.md에서 진행상황에 반영할 핵심 결정

`history.md`에서 유의미한 내용은 단순 과거 기록이 아니라, 지금 진행상황을 해석하는 기준이다. (왜 지금 구조가 이렇게 됐는지 설명하는 결정들)

| history 결정 | 진행상황에 반영할 의미 | 현재 반영 상태 |
|---|---|---|
| 인력 확보 agent는 후보 추천기가 아니다 | 채용 매칭이 아니라 절차, 서류, 근거, 리스크 정리 agent로 봐야 한다 | guardrail, hiring_agent scope, Non-goals에 반영 |
| RAG는 thin slice로 먼저 끝까지 돌린다 | 큰 문서셋보다 작은 검색/eval을 먼저 검증한 뒤 확장한다 | 5-case에서 20-case retrieval eval로 확장 |
| `case_type` 4종을 고정한다 | 신규고용, 성실근로자 재입국, 사업장 변경, 동일 근로자 재고용을 기본 업무 유형으로 본다 | schema/router/eval 방향에 반영 |
| `chunk_type`과 `evidence_grade`를 enum처럼 다룬다 | 검색 근거의 종류와 신뢰도를 코드가 판별할 수 있어야 한다 | chunk metadata, evidence package, source manifest에 반영 |
| agent 간 직접 호출 금지 | agent는 shared state, aggregator, approval gate를 통해 만난다 | workflow runtime과 approval/evidence 구조에 반영 |
| PII masking은 P0 | 외국인등록번호, 여권번호, 전화번호는 router/LLM/RAG 전에 마스킹해야 한다 | PII 테스트와 evidence 마스킹 경로에 반영 |
| Chroma persistent index는 Windows 이슈로 우회 | 운영 Vector DB보다 먼저 cache/numpy/deterministic 검색으로 검증한다 | MVP 검색은 local/deterministic 중심, Chroma는 adapter/artifact 경로로 분리 |
| silent fallback 금지 | 근거나 metadata가 없으면 조용히 대체 답변을 만들지 않는다 | `insufficient_evidence`, blocked/error 처리에 반영 |
| phase에서 mission으로 전환 | 현재 실행 기준은 `phases/*`가 아니라 `missions/*`다 | active는 비어 있고 completed 001~013으로 정리 |
| Human approval gate 통일 | 메시지 발송, 정부 제출, 외부 전달은 자동 실행하지 않는다 | approval_required/PENDING 테스트에 반영 |

따라서 이 진행상황 문서는 단순히 "얼마나 했나"가 아니라, Notion의 주차별 체크리스트와 `history.md`의 설계 결정을 현재 mission 완료 상태에 맞춰 재해석한 문서다. (과거 계획과 현재 구현 상태를 한 장에서 이어주는 문서)

## 전체 결론

2주차 RAG 기본 구축은 MVP 기준으로 완료됐다. (문서를 모으고, 쪼개고, 검색되게 만들고, 검색 품질까지 확인한 상태) 처음에는 5문서/5 case thin slice였고, 이후 20개 source, raw 파일 수집, manifest collected 전환, metadata 정규화, semantic chunking, 20-case retrieval eval까지 확장됐다. (작은 검증판에서 시작해 실제 근거 문서와 평가 케이스를 늘린 흐름)

3주차 판단 체인은 “실제 LangChain/OpenAI provider 운영 호출”까지 간 것은 아니지만, WorkBridge 기준 판단 체인 핵심 계약은 완료됐다. (외부 유료 LLM을 실제로 부르기 전, 입력/출력/안전 규칙을 먼저 고정한 상태) RAG Evidence Package, prompt 구조, 판단 JSON schema, parser, fake LLM client, post-LLM guardrail, risk/report 생성, optional `runtime_mode="langchain_judgment"`, LangChain-compatible safe adapter와 index builder까지 구현됐다. (근거 검색 결과를 판단 JSON과 리포트로 바꾸는 기본 체인이 테스트 가능하게 만들어짐)

남은 것은 production-grade 단계다. (MVP 검증을 넘어서 실제 운영 환경에서 안정적으로 쓰기 위한 단계) 실제 OpenAI/LangChain SDK 호출, Approval/Evidence/Document state의 DB 영속화, HiKorea/KOSHA 자료의 deep crawling, 실제 Next.js build는 별도 후속 과제로 남아 있다. (외부 API, 데이터베이스 저장, 더 깊은 공식문서 수집, 실제 프론트 빌드는 아직 운영화 과제)

### 2주차 상세 흐름

1. 초기 방향 설정 (인력 확보 에이전트가 무엇을 하고 무엇을 하지 않을지 정한 단계)

- `docs/journal/history.md`에서 인력 확보 agent는 후보 추천기가 아니라 절차, 서류, 증거 정리 agent로 정의됐다. (사람을 추천하거나 평가하는 AI가 아니라 업무 누락을 줄이는 도우미로 범위를 제한)
- RAG는 크게 만들기보다 5문서/12 eval thin slice로 끝까지 돌린 뒤 확장하는 방향으로 결정됐다. (작은 범위를 먼저 완전히 검증한 뒤 문서와 평가를 늘리는 방식)
- evidence grade A~F, chunk type, case type 같은 계약을 먼저 고정하는 원칙이 잡혔다. (근거 신뢰도, 문서 조각 종류, 업무 유형을 코드가 이해할 수 있게 분류)

2. 새 구조로 RAG runtime 이식 (예전 RAG 코드를 현재 backend/data-pipeline 구조로 옮긴 단계)

- `docs/journal/codex/2026-05-04.md`에 따르면 `backend/app/agent_runtime/rag/` 아래에 chunking, embeddings, retriever, citation, vector_store, evaluate 코드가 추가됐다. (문서 쪼개기, 검색용 숫자화, 검색, 출처 표시, 저장, 평가 코드가 들어감)
- `data-pipeline/seed`, `data-pipeline/processed/chunks`, `evals/datasets`, `evals/reports`로 데이터 준비, 검색 산출물, 평가가 분리됐다. (원본/처리결과/평가데이터/평가결과를 폴더별로 나눠 관리)
- 이 시점에는 예전 `data/raw/`를 바로 섞지 않고 `data-pipeline/raw/`로 별도 migration하기로 했다. (기존 원본 문서를 새 구조에 안전하게 옮기기 위해 분리)

3. RAG thin slice 검증 (작은 검색 시스템이 실제로 답을 찾는지 확인한 단계)

- `docs/journal/codex/2026-05-05.md` 기준으로 RAG thin slice는 Hit@3 5/5 = 100%까지 확인됐다. (질문 5개 모두 정답 근거가 상위 3개 검색 결과 안에 들어옴)
- 당시 한계도 명확했다. eval case가 5개였고, 실제 LLM 판단 체인과는 아직 결합되지 않은 상태였다. (검색은 됐지만 판단 리포트 생성까지 연결된 것은 아니었음)

4. RAG source hardening (검색 근거 문서의 출처와 메타데이터를 더 단단하게 만든 단계)

- `docs/journal/codex/2026-05-06.md`에서 source manifest가 20개 이상으로 확장됐다. (검색에 쓸 공식/템플릿 문서 목록을 늘림)
- 정부24 체류기간연장, HiKorea, KOSHA, 출입국관리법 계열, 외국인근로자법 시행령/시행규칙 등 planned source가 raw HTML로 확보됐다. (예정 상태였던 공식 문서들을 실제 원문 파일로 저장)

5. 검색 품질 회복 (문서가 늘어난 뒤 떨어진 검색 성능을 다시 맞춘 단계)

- 최종 actual RAG retrieval은 Hit@3 20/20 = 100%로 회복됐다. (20개 평가 질문 모두 기대 근거가 상위 3개 안에 다시 들어옴)

### 3주차 상세 흐름

1. Mission 008: RAG Evidence Package (검색 결과를 LLM이 읽을 수 있는 표준 입력 JSON으로 만든 단계)

- `backend/app/agent_runtime/rag/evidence_package.py`가 추가됐다. (검색 결과 패키지를 만드는 전용 코드 파일)
- 검색 결과를 LLM 입력용 Evidence Package로 표준화했다. (LLM이 근거를 일정한 형식으로 받도록 고정)
- `request_id`, `query`, `case_type`, `retrieved_chunks`, `citations`, `missing_evidence`, `evidence_policy`를 포함하는 계약으로 정리했다. (요청 ID, 질문, 업무 유형, 검색 조각, 출처, 부족한 근거, 근거 사용 규칙을 함께 담음)
- A/B/E evidence만 답변 근거로 허용하고, F grade synthetic/demo data는 공식 판단 근거에서 제외했다. (신뢰 가능한 근거만 판단에 쓰고 데모 데이터는 공식 근거로 쓰지 않음)

2. Mission 009: LLM Judgment JSON Chain (LLM 판단 결과를 자유문이 아니라 검증 가능한 JSON으로 받는 단계)

- `backend/app/agent_runtime/llm/` 패키지가 추가됐다. (LLM 판단 체인 관련 코드 묶음)
- `prompts.py`에서 prompt 구간을 분리했다. (system/developer/user/evidence 역할을 나눠 LLM 입력을 안정화)
- `parser.py`에서 `WorkBridgeJudgmentReport`와 JSON-only parser를 구현했다. (정해진 JSON 형식만 통과시키고 자유문은 거절)
- `client.py`에서 fake client와 provider protocol을 정의했다. (실제 API 없이 테스트할 가짜 LLM과 나중에 실제 provider를 붙일 인터페이스)
- `judgment_chain.py`에서 fake client 기반 judgment chain과 post-LLM guardrail 검사를 구현했다. (LLM 판단 뒤에도 금지 문구/행동을 다시 검사)

3. Mission 010: Risk Report Generation (판단 JSON을 사람이 읽는 위험도/리포트로 바꾸는 단계)

- `backend/app/agent_runtime/reporting/` 패키지가 추가됐다. (리포트 생성 관련 코드 묶음)
- 판단 JSON에서 risk flag type/level을 enum 기반으로 정규화했다. (위험 유형과 수준을 정해진 목록 안에서만 쓰게 함)
- deterministic 기본 리포트를 생성했다. (LLM이 매번 다르게 쓰지 않도록 규칙 기반으로 리포트 작성)
- 리포트에는 Evidence source_id와 approval_required/PENDING 상태가 유지된다. (어떤 근거를 봤고 승인이 필요한지 리포트에 남김)

4. Mission 011: Judgment Runtime Mode (기존 실행 흐름을 유지하면서 선택형 판단 모드를 붙인 단계)

- `AgentRunRequest`에 `runtime_mode: Literal["deterministic", "langchain_judgment"] = "deterministic"`가 추가됐다. (기본은 기존 방식이고 요청할 때만 판단 모드를 켤 수 있음)
- 기존 workflow는 유지하고 approval gate 이후 final response 전 optional judgment node를 붙였다. (기존 안전 흐름을 깨지 않고 중간에 판단 노드만 추가)
- blocked/unsupported request에서는 judgment node가 호출되지 않는다. (금지/미지원 요청은 LLM 판단 전에 차단)
- evidence event로 `rag_retrieved`, `llm_judgment_generated`, `risk_flagged`가 남도록 했다. (검색, 판단 생성, 위험 플래그 기록이 감사 로그에 남음)

5. Mission 012: LangChain Judgment Adapter (LangChain을 기존 시스템 밖의 보조 adapter로 붙인 단계)

- 기존 WorkBridge runtime을 대체하지 않는 adapter 계층으로 구현됐다. (LangChain이 라우팅/차단/승인 최종권을 갖지 않게 제한)
- 추가된 경로: (LangChain adapter를 구성하기 위해 새로 생긴 코드/스크립트 위치)
  - `backend/app/agent_runtime/langchain_runtime/schemas.py` (LangChain 판단 리포트 schema 정의)
  - `backend/app/agent_runtime/langchain_runtime/documents.py` (chunk JSONL을 LangChain Document로 바꾸는 코드)
  - `backend/app/agent_runtime/langchain_runtime/vectorstore.py` (LangChain/Chroma용 벡터 저장소 연결 코드)
  - `backend/app/agent_runtime/langchain_runtime/tools.py` (LangChain agent가 호출할 수 있는 안전한 tool 정의)
  - `backend/app/agent_runtime/langchain_runtime/judgment_agent.py` (safe tool과 structured output을 묶은 판단 agent)
  - `scripts/build_langchain_rag_index.py` (기존 chunk 파일로 LangChain용 RAG index를 만드는 스크립트)
- safe tool만 등록했다. (읽기/점검 중심의 안전한 기능만 LLM에 제공)
  - `retrieve_policy_context` (정책 근거 검색)
  - `assess_readiness` (준비상태/누락 입력 확인)
- 금지 tool은 등록하지 않았다. (LLM이 실제 외부 행동을 직접 실행하지 못하게 함)
  - 메시지 발송 (외국인 근로자나 외부인에게 실제 메시지를 보내는 기능)
  - 정부 포털 제출 (정부 사이트에 신청/신고를 자동 제출하는 기능)
  - 케이스 완료 처리 (업무 상태를 완료로 확정 변경하는 기능)
  - 외부 export (대외 제출용 파일을 자동으로 내보내는 기능)
  - destructive DB update (데이터베이스를 삭제/파괴적으로 수정하는 기능)

6. Mission 013: Real LLM Provider Integration (실제 LLM provider를 안전하게 붙이기 위한 준비 단계)

- 실제 provider 호출은 기본값으로 꺼진 상태다. (실수로 OpenAI 같은 외부 API가 호출되지 않게 함)
- feature flag와 error/PII 차단 경로를 추가했다. (명시적으로 켤 때만 실제 호출하고 개인정보 노출을 막음)
- fake client가 기본이며, real provider는 env flag 없이는 호출되지 않는다. (테스트와 기본 실행은 네트워크/API 키 없이 동작)
- provider error, non-JSON output, missing key 상황은 blocked/error로 드러나게 했다. (외부 API 오류나 잘못된 응답을 조용히 성공처럼 처리하지 않음)
