## 디자인 시스템 (반드시 준수)

- **Montage(Wanted) v2** 토큰만. PC 워크벤치(§3b)와 같은 크롬 재사용: 52px 상단 nav, 좌측
  패널·우측 레일 구조, 1560px 캔버스.
- PC 밀도 타입램프(`--fs-pc-*`, 10.5~13.5px) 사용 가능. 아웃라인 inset box-shadow.
- 분류 신뢰도/검증 상태는 Chip 색표 재사용: 정상 매칭=positive(초록 `#009632`/`#D9FFE6`),
  저신뢰·확인 필요=high(오렌지 `#D47800`/`#FEF4E6`), 미매칭=critical(빨강
  `#E52222`/`#FEECEC`).
- 이미 만든 컴포넌트: Chip, Button, 워크벤치 nav/좌측패널/우측레일, 드롭존·검증 테이블 —
  **CSV 업로드 브리프**(`CSV_업로드_브리프.md`)의 "드롭존 → 검증 미리보기 → 확정" 3단계
  구조와 테이블 행 관용구를 그대로 승계한다(같은 회사 화면 크롬, 같은 상태 Chip 색표 —
  새로 발명하지 않는다).

## 화면 구성 (PC 단일 화면, `/cases/workers` 우측 레일 내부 플로우 — 별도 라우트 불필요)

### 진입점 (교체 대상)
- 현재 정적 카드 위치(우측 레일) 그대로 — "서류 스캔 업로드" 제목 + 안내 문구는 유지하되
  "준비 중" 캡션을 제거하고 실제 드롭존으로 전환.

### 1. 드롭존 (업로드 전)
- 점선 박스: "여권·계약서·증명서 파일을 여기에 끌어다 놓거나 클릭해 선택" + 파일 아이콘.
- 지원 형식(이미지/PDF) 안내.

### 2. 분류 중 (로딩)
- Skeleton — **수치는 `--`**(rules/design.md §7 상태 표현 원칙), 파일별 진행 표시.

### 3. 분류 결과 미리보기 (신규 핵심 화면)
- 상단 요약 스탯 로우: **전체 N건 · 정상 매칭 M · 확인 필요 K · 미매칭 J**
  (CSV 브리프의 "전체 N행·정상 M·오류 K" 스탯 로우와 동일 패턴).
- 테이블(워크벤치 스타일): 컬럼 = 파일명(썸네일) · 매칭된 근로자 · 서류 유형 · **분류
  상태 칩**(정상/확인 필요/미매칭).
  - 정상 매칭 행: positive 칩.
  - 확인 필요 행: high 칩 + 사유("신뢰도 낮음"류).
  - 미매칭 행: critical 칩 + "근로자 매칭 실패".
- **문서 썸네일에 외국인등록번호 등 식별정보가 원문으로 보이면 안 된다** — 마스킹 처리
  또는 해당 필드가 보이지 않는 크롭/블러 처리 중 하나를 디자인이 결정.

### 4. 수동 보정 (확인 필요·미매칭 행 대상)
- 행 클릭 → 근로자 드롭다운(케이스 목록에서 선택) + 서류 유형 드롭다운으로 교정.
- 교정 후 상태 칩이 positive로 전환.

### 5. 확인 대기 반영 (하단 고정 바)
- "확인 대기로 올리기"(미매칭 행이 남아있으면 그 행은 제외하고 진행하거나 버튼 disabled —
  CSV 브리프의 "정상 M명만 등록" 관례와 동일하게 **오류/미매칭 행은 자동 포함하지
  않는다**) + "취소".
- 하단 안내: "개인정보는 DB에만 저장되고, LLM/RAG에 전달되지 않습니다"(기존 정적 카드
  문구 그대로 승계) + "자동 분류 결과는 확인 후 반영됩니다"류 1줄 추가(신규 — 자동
  확정이 아님을 명시).

## 가드레일 (디자인에 반드시 반영)

- 자동 분류 결과를 사람 확인 없이 "제출/완료" 상태로 바로 처리하지 않는다 — 종착은 항상
  "확인 대기"까지.
- 스캔 원본 파일 자체는 LLM/RAG에 전달되지 않는다는 문구 유지.
- 문서 썸네일·미리보기에 외국인등록번호 등 PII 원문 노출 금지.
- 미매칭·확인 필요 행에 체크박스로 "일괄 확정" 같은 UI를 만들지 않는다(사람이 행 단위로
  확인해야 함 — 일괄 승인 금지 원칙과 같은 결의 가드레일).
- 단정형·이모지·느낌표 금지.
