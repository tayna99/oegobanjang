## 0. 공통 규칙

### 0.0 디자인 시스템 기준 (v1.1 개정 — DESIGN.md 채택)

프론트 디자인 시스템은 사용자 제공 `DESIGN.md`(SOCAR 스타일)를 기준으로 한다. 상세 토큰은 4단계 문서 참조. 본 스펙에 미치는 개정 사항:

- **무채색 크롬**: 텍스트 `#354153`(검정 아님), 보더 `#e5e8ef`, 입력 필 `#f2f3f8`, placeholder `#b4bbcb`, 배경 흰색. 브랜드 블루 `#0066FF`는 **화면당 primary CTA 1개와 활성 탭에만** 사용
- **severity/승인 상태색은 유지** — 단, 배지·D-day·상태 표시 같은 기능적 요소에만. 장식(아이콘 타일 배경, 그라데이션)에는 금지. M1 SummaryStatCard의 7색 아이콘 타일은 무채색 타일 + 텍스트 카운트로 변경
- **radius 사다리 12/14/16**: 입력·버튼 12, 칩·배지 14, 카드 16, 바텀시트 상단 20. 필(9999) 금지
- **스켈레톤은 `#e5e8ef` 블록**, 카드 기하 유지. 금액·D-day 수치는 스켈레톤 대신 `--` 표시
- **마스코트·일러스트 빈 상태 금지** — 기존 기획의 "AI 반장 캐릭터 제한적 사용"을 폐지. 빈 상태는 한 문장 + 행동 1개
- **이모지·느낌표 금지** (승인·발송·기록 등 트랜잭션 화면 전체)
- **승인 완료는 전용 화면, 토스트 금지** — M5가 이미 부합. 소액 액션만 하단 3초 다크 토스트
- **모션 토큰**: instant 0 / fast 150ms / standard 240ms(바텀시트 rise 기본) / slow 360ms(M5 승인 완료 전용). `prefers-reduced-motion` 시 전부 instant

### 0.1 화면 상태 5종 (모든 화면 공통 정의)

| 상태 | 정의 | 공통 원칙 |
|---|---|---|
| `default` | 데이터 로드 완료, 정상 표시 | — |
| `empty` | 로드 성공했으나 표시할 항목 0건 | 절대 빈 화면으로 두지 않음. "다음에 할 수 있는 일" 1개를 항상 제안 |
| `loading` | 데이터 요청 중 | 스피너 금지, **스켈레톤**(`#e5e8ef`) 사용. 카드 기하 그대로 회색 블록. 0.3초 미만이면 스켈레톤 생략. 수치는 `--` 표시 |
| `error` | 요청 실패 (서버 오류, 4xx/5xx) | 원인 1줄 + 행동 1개([다시 시도]). 기술 용어(500, timeout)와 범용 문구("문제가 발생했습니다") 금지 — 무엇이 왜 안 됐는지 서술 |
| `offline` | 네트워크 없음/불안정 | 마지막 동기화 데이터를 캐시로 표시 + 상단 배너 "오프라인 · 마지막 업데이트 HH:MM". **승인 버튼은 오프라인에서 비활성** (승인은 반드시 서버 확정) |

### 0.2 배지 색 규칙 (전 화면 공통)

**위험도 배지** (severity — Daily Briefing PRD §9 준용)

| 값 | 색 | 표기 |
|---|---|---|
| CRITICAL | 빨강 `#EF4444` | 즉시 확인 |
| HIGH | 주황 `#F97316` | 우선 확인 |
| MEDIUM | 파랑 `#3B82F6` | 확인 필요 |
| LOW | 회색 `#6B7280` | 참고 |

**D-day 배지**

| 조건 | 표기 | 색 |
|---|---|---|
| 만료 경과 | `D+N` | 빨강 |
| D-0 | `D-day` | 빨강 |
| D-1~30 | `D-N` | 주황 |
| D-31~90 | `D-N` | 파랑 |
| D-91+ | `D-N` | 회색 |

**승인 상태 배지** (Approval.status)

| 값 | 색 | 라벨 |
|---|---|---|
| pending | 노랑 `#F59E0B` | 승인 필요 |
| approved | 초록 `#00BF40` | 승인 완료 |
| rejected | 회색 | 반려됨 |
| locked | 노랑+자물쇠 아이콘 | 승인 전 차단 |

**케이스 상태 배지** (Case.state)

| 값 | 라벨 | 색 |
|---|---|---|
| risk_review | 확인 필요 | 주황 |
| approval_pending | 승인 대기 | 노랑 |
| human_approved | 승인 완료 | 초록 |
| completed | 처리 완료 | 회색 |
| blocked | 차단됨 | 빨강 |
| scheduled | 예정 | 파랑 |

### 0.4 공통 데이터 타입

```ts
type Severity = 'CRITICAL'|'HIGH'|'MEDIUM'|'LOW'
type CaseState = 'draft'|'risk_review'|'approval_pending'|'human_approved'|'completed'|'blocked'
type Role = 'manager'   /* 담당자 */ | 'owner'   /* 대표 */

interface CaseCard {
  caseId: string
  title: string               // "Nguyen V. 체류기간 연장 서류 요청" (업무 단위 — D2)
  workerRef?: { displayName: string; nationality: string; maskLevel: 'masked' }
  severity: Severity
  dDay?: number               // 음수=경과
  missingDocCount?: number
  state: CaseState
  approvalRequired: boolean
  primaryAction: NextActionRef   // CTA 2개 고정 원칙
  secondaryAction: NextActionRef
  preparedBy: 'agent' | 'rule'   // v1.2: 프로액티브 런이 준비물을 만든 카드는 'agent'
  preparedRunRef?: string        // v1.2: "AI가 준비를 마쳤습니다 · 런 #4791 보기" 링크 (M9 재생 뷰)
}

interface NextActionRef {
  actionId: string
  label: string               // "초안 보기" | "보내기 승인" | "응답 요약 보기" ...
  state: 'ready'|'locked'|'scheduled'|'waiting'
  requiresApproval: boolean
}

interface Citation { grade: 'A'|'B'|'C'|'E'; title: string; source: string; updatedAt: string }
```

# 승인 플로우 (M1 → M2 → M3 → M4 → M5)

## M1. 오늘 브리핑 (홈)

**목적**: 입력 없이 오늘 승인·확인할 일 Top 1~3을 처리한다.
**진입**: 앱 실행 기본 화면 / 푸시 "오늘 승인 필요 N건" 딥링크 / M5·M9 완료 후 복귀.

### 레이아웃 (위→아래)

```
[OfflineBanner?]
[BriefingHeader]  사업장명 · 날짜 · 알림 아이콘 · 프로필(역할 전환)
[GreetingBanner]  "{호칭}, 오늘 확인이 필요한 외국인 고용 업무가 N건 있습니다."
[ApprovalCardList] 승인 카드 1~3장 (최대 3, 초과분은 "케이스 탭에서 N건 더 보기" 링크)
[SummaryStatRow]  요약 통계 카드 (서류 보완 N건 / 승인 대기 N건 / 응답 도착 N건)
[SafetyNotice]
[CommandBar]      ← 하단 탭 바로 위 고정
[TabBar active=briefing]
```

### 컴포넌트 props

```ts
BriefingHeader {
  companyName: string
  date: string                   // "2026.07.04"
  unreadNotifications: number
  role: Role                     // owner면 GreetingBanner 호칭 "대표님", manager면 "담당자님"
}

ApprovalCard {                   // = CaseCard 렌더러
  data: CaseCard
  layout: 'hero' | 'compact'     // 1번 카드는 hero(추천 이유 1줄 포함), 2~3번은 compact
  recommendReason?: string       // hero만. "D-30이고 누락 서류 2건이 있어 오늘 요청이 필요합니다"
}
// v1.2: preparedBy==='agent'면 카드 하단에 준비 상태 행 노출:
//   "AI가 준비를 마쳤습니다 · 런 #4791 보기" (탭 → M9 재생 뷰, 읽기 전용)
//   프로액티브 런이 QuestionCard로 멈춘 경우: "AI가 확인을 기다립니다" 행 + 질문 칩
// 배지 렌더 순서 고정: [D-day] [누락 N건] [승인 필요] [severity]
// CTA: primaryAction(파랑 filled) + secondaryAction(아웃라인). 3개 이상 금지.

SummaryStatCard { icon, label, count, unit: '건'|'명', deepLink }
// v1.1: 무채색 카드(#fff, 보더 #e5e8ef, radius 14). 아이콘·라벨 #354153,
// 카운트만 severity 연동 텍스트 컬러 허용. 색 타일 배경 금지.
// 탭하면 해당 필터가 걸린 M7 케이스 목록으로 이동
```

### 역할 분기 (D1)

| | owner(대표) | manager(담당자) |
|---|---|---|
| ApprovalCardList | 승인 필요 카드만 | 승인 + 확인 필요(risk_review) 카드 |
| SummaryStatRow | 숨김 (단순화) | 표시 |
| CommandBar | 표시하되 suggestions 승인 관련만 | 전체 |

### 상태 5종

| 상태 | 스펙 |
|---|---|
| default | 위 레이아웃. 카드 정렬: severity → D-day → 유형 우선순위 (deterministic) |
| empty | GreetingBanner → "오늘 승인할 업무가 없습니다". 아래에 (a) 다음 예정 이벤트 1줄 "다음: 7/12 Tran 계약종료 D-30 진입 예정" (b) [케이스 전체 보기] 버튼. 근로자 0명이면 온보딩 유도: "근로자를 등록하면 매일 브리핑이 시작됩니다" + [근로자 등록] |
| loading | GreetingBanner 자리 + 카드 3장 스켈레톤. 헤더·탭바는 즉시 렌더 |
| error | "브리핑을 불러오지 못했습니다" + [다시 시도]. 이전 브리핑 캐시가 있으면 캐시 표시 + 상단 "어제 데이터입니다" 배너 |
| offline | 캐시된 브리핑 표시 + OfflineBanner. 모든 승인 CTA disabled + 탭 시 토스트 "오프라인에서는 승인할 수 없습니다". [초안 보기] 등 읽기 액션은 캐시 범위 내 허용 |

### 전이

- 카드 탭(CTA 외 영역) → M2 바텀시트
- [초안 보기] → M3 / [보내기 승인] → M4 / [응답 요약 보기] → M6
- CommandBar 제출 → M9

## M2. 케이스 바텀시트 (상세)

**목적**: 승인 전에 필요한 판단 정보만 짧게. 페이지 이동 없이 half→full 시트로 끝낸다.
**진입**: M1 카드 탭 / M7 목록 탭 / 푸시 개별 케이스 딥링크.

### 섹션 구조 (full 기준, 5블록 고정)

```
1. CaseSummaryBlock   제목 + 배지 행 + 요약 1문장
2. AICheckedBlock     "AI가 확인한 내용" 3~5줄 (체류만료일, 누락 서류, 이전 요청 이력)
3. MissingDocChecklist 누락 서류 체크리스트 (있는 경우만)
4. CitationBlock      근거 1~2개 (등급 배지 + 제목 + 출처 + 갱신일)
5. AgentActivityBlock (v1.2) "이 케이스의 에이전트 활동" — 오케스트레이터 런 체인 타임라인
6. ActionBar          [secondaryAction] [primaryAction]  ← 시트 하단 고정
```

```ts
// v1.2 신규
AgentActivityBlock {
  runs: { runRef: string; startedBy: 'event'|'user'; label: string;   // "서류요청 준비 (D-30 감지)"
          at: string; outcome: 'approved'|'pending'|'question'|'replanned' }[]
  nextWake?: { condition: string }    // "다음: 응답 없으면 D-28에 리마인드 판단"
}
// 런 #1~#N이 하나의 이야기로 보이게 시간순. 항목 탭 → 해당 M9 재생 뷰
```

### 컴포넌트 props

```ts
CaseSummaryBlock {
  title: string; badges: Badge[]
  summary: string        // 1문장 강제. "체류만료가 30일 남았고 서류 2건이 누락되어 요청이 필요합니다."
}
AICheckedBlock {
  items: { icon, label, value, link? }[]   // 최대 5. 항목 탭 시 근거 상세로
}
MissingDocChecklist {
  docs: { name: string; status: 'missing'|'requested'|'received'|'company_check' }[]
  readinessPercent?: number                 // "제출 준비율 60%" 프로그레스 바
}
CitationBlock {
  citations: Citation[]                     // 0건이면 블록 대신 경고 렌더 (아래 error 참조)
}
```

### 상태 5종

| 상태 | 스펙 |
|---|---|
| default | half로 열림(1+5블록만) → 위로 드래그 시 full(전체) |
| empty | 해당 없음 — 케이스 없이 열리지 않음. 단 **citation 0건**이면 CitationBlock 자리에 노랑 경고 "공식 근거가 연결되지 않았습니다. 승인 전 확인이 필요합니다" + primaryAction을 locked로 강등 (근거 품질 게이트) |
| loading | 블록별 스켈레톤. 제목/배지는 M1 카드에서 받은 데이터로 즉시 표시 (optimistic) |
| error | 시트 내부에 "상세를 불러오지 못했습니다" + [다시 시도] + [닫기] |
| offline | 캐시 표시. ActionBar의 승인성 버튼 disabled, 읽기 버튼은 활성 |

## M3. 초안 상세 (VN/KR)

**목적**: 다국어 초안을 원문·번역 동시 검토. 발송은 구조적으로 불가, 승인 요청까지만.
**진입**: M1/M2의 [초안 보기].

### 레이아웃

```
[DraftHeader]        "Nguyen V. 서류 요청 메시지" + 배지(D-30/누락 2건) + 채널 표시(Zalo)
[SafetyNotice]
[LangToggle]         [한국어] [베트남어] 세그먼트 — 기본값: 한국어
[DraftBody]          메시지 본문 (스크롤)
[ComplianceChecks]   ✓ 개인정보 사용 목적 포함  ✓ 제출 기한 포함  (자동 검사 결과)
[ExpectedResponses]  예상 응답 시나리오 3종 카드 (긍정/추가 질문/지연)
[ActionBar]          [수정 요청] [보내기 승인]
```

### 컴포넌트 props

```ts
LangToggle { languages: ('ko'|'vi'|'id')[]; active: string }
DraftBody { text: string; lang: string; editable: false }   // 직접 편집은 v1 범위 밖 → 수정 요청으로
ComplianceChecks {
  checks: { label: string; passed: boolean }[]
  // 하나라도 false → [보내기 승인] locked + 사유 표시 "개인정보 사용 목적 문구가 없습니다"
}
ExpectedResponseCard { type: 'positive'|'question'|'delayed'; label; description }
// 지연 카드에는 "2일 뒤 리마인드 제안" — scheduled 배지(파랑)
RevisionSheet {      // [수정 요청] 탭 시 바텀시트
```

## M9. 에이전트 런 (Agent Run) — D3 신규

**목적**: 자연어 요청 → LLM 툴콜링 루프(생각→도구 호출→관찰) → 결과(Next Action)까지를 하나의 스레드로 보여준다. M4는 이 화면의 `mode='pre_approval'` 특수 케이스.
**진입**: M1 CommandBar 제출 / M3 수정 요청 제출 / M3 empty의 [초안 생성하기].

### 레이아웃

```
[AgentRunHeader]     라우팅된 에이전트명 · 자율성 · 판단 기록 # (run 시작 시 발급)
[UserGoalBubble]     사용자 요청 원문 "이번 달 급한 직원만 정리해줘"
[RunStream]          단계 스트림 (아래 RunStep 타입)
[QuestionCard?]      에이전트가 되묻는 경우 (interrupt)
[ResultBlock]        완료 시: 요약 + NextAction 카드들 (승인 필요 배지 포함)
[FollowUpBar]        후속 질문 입력 (이 런 스레드에 한정된 대화)
```

### 컴포넌트 props

```ts
RunStep =
  | { kind: 'thinking';  summary: string }                       // "케이스 조회가 필요합니다"
  | { kind: 'tool_call'; toolName: string; toolLabel: string;    // "근로자 상태 조회"
      args_summary: string; status: 'running'|'done'|'failed'|'blocked'
      resultSummary?: string }                                   // "E-9 근로자 12명 중 D-30 이내 2명"
  | { kind: 'guardrail'; message: string }                       // "외부 발송은 승인 전 차단되어 승인 요청으로 전환했습니다"
  | { kind: 'handoff';   from: string; to: string; reason: string }   // v1.2: "Visa → Contact Agent · 메시지 생성 필요"
                                                                       //   헤더 에이전트명이 전환되는 연출
  | { kind: 'replan';    reason: string; from: string; to: string }   // v1.2: "RAG 타임아웃 → 캐시된 근거로 대체"
                                                                       //   조용한 경로 변경 금지 — 반드시 스텝으로 노출

// 도구 호출 표시 규칙:
//  - toolLabel은 한국어 업무 언어 ("search_regulation" → "관련 규정 검색")
//  - blocked(가드레일 차단)는 숨기지 않고 guardrail 스텝으로 명시 — 신뢰 자산
//  - 스텝 탭 → 근거/결과 상세 펼침

QuestionCard {                    // human-in-the-loop interrupt
  question: string                // "어느 사업장 기준으로 정리할까요?"
  options?: string[]              // 선택지 칩 (있으면 칩, 없으면 자유 입력)
  blocking: true                  // 답변 전 루프 일시정지
}

ResultBlock {
  summary: string
  actions: NextActionRef[]        // 각 카드에서 M2/M3/M4로 연결
  evidenceRef: string
}
```

### 상태 5종

| 상태 | 스펙 |
|---|---|
| default(실행 중) | RunStream 실시간 스트리밍. 상단에 [일시정지] [취소] 상시 노출. 백그라운드 전환 허용 — M1 상단에 "에이전트 실행 중 1건" 칩 표시, 탭하면 복귀 |
| empty | CommandBar 제출 직후: UserGoalBubble + "요청을 분석하고 있습니다" (Intent Router 단계) |
| loading | empty와 동일 개념 — 단계 간 대기는 active 스텝의 점멸로 표현, 전체 화면 로딩 없음 |
| error | (a) 라우팅 실패: "요청을 이해하지 못했습니다" + 재입력 유도 + 예시 칩 3개 (b) 도구 실패: 해당 스텝 failed 표시 + 에이전트가 대안 시도 or "여기서 멈췄습니다" + [다시 시도] (c) 범위 밖 요청(비자 판정 등): 거절 카드 "이 판단은 행정사 검토가 필요합니다" + [행정사 검토 요청 만들기] |
| offline | CommandBar 자체가 disabled이므로 신규 런 불가. 실행 중 끊김: "연결이 끊겨 일시정지되었습니다. 재연결 시 이어서 진행합니다" (서버 측 런 상태 보존 전제) |

### 런 수명 규칙 (v1.2 개정)

- 런 1건 = 판단 기록 # 1건. 모든 tool_call·guardrail·handoff·replan·approval이 EvidenceEvent로 append
- **동시 런**: 사용자 포그라운드 런 1개 + 백그라운드 프로액티브 런 N개(우선순위 큐: severity → D-day). 사용자 런이 항상 큐 최우선
- **런 트리거 2종**: `startedBy:'user'`(커맨드 바) / `startedBy:'event'`(프로액티브 — D-30 진입, 응답 도착, 케이스 생성). 프로액티브 런의 도구 화이트리스트는 읽기+초안 생성만, 종착점은 항상 승인 요청 생성
- **런 체이닝**: 같은 케이스의 런은 `parentCaseId`로 연결 — M2 AgentActivityBlock이 이 체인을 렌더. 오케스트레이터의 다음 기상 조건(`nextWake`)도 케이스에 저장
- 런 히스토리는 M8 기록 탭에서 event_type 필터로 조회 (별도 채팅 히스토리 탭 없음 — Zero-Input 원칙)
- **재생 뷰(replay)**: 완료된 런은 M9 화면을 읽기 전용으로 재생 (M1 "런 #보기", M2 활동 타임라인에서 진입)
