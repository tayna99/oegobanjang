# 3단계 — 온보딩 플로우 (v1)

> 작성일: 2026-07-04
> 목표: **가입 후 5분 안에, 근로자 1명만 등록해도 첫 브리핑 카드를 본다.**
> 전제: D6(수기+OCR / CSV / 대행 3경로), 1단계 M1 empty 스펙, 2단계 N21 온보딩 리마인드, DESIGN.md 디자인 시스템(무채색, 일러스트 없는 빈 상태, 단문 카피)

---

## 1. 설계 원칙

1. **첫 브리핑이 Aha Moment다.** 온보딩의 종착점은 "설정 완료"가 아니라 M1에 첫 카드가 뜨는 순간이다. 모든 단계는 그 순간까지의 거리로 평가한다.
2. **최소 입력으로 D-day부터.** 근로자 1명의 필수 필드는 4개뿐(이름, 국적, 체류자격, 체류만료일). 이 4개만 있으면 Rule Engine이 즉시 D-day를 계산해 카드를 만든다. 서류 체크리스트는 나중에 채워도 된다.
3. **역할에 따라 다른 문. 같은 목적지.** 담당자는 직접 등록, 대표는 담당자 초대가 1순위 경로.
4. **온보딩 중에도 에이전트가 말한다.** 등록 직후 "체류만료가 62일 남았습니다. 지금 확인할까요?"처럼 시스템이 먼저 반응하는 경험을 온보딩 안에서 보여준다.

## 2. 전체 여정 맵

```
O1 가입 ─ O2 역할 선택 ─ O3 사업장 등록 ─ O4 근로자 입력(3경로) ─ O5 첫 브리핑 생성 ─ M1
 30초        10초           60초              90초~                 즉시(자동)
                                                                  누적 목표: 5분 이내
```

각 단계 상단에 진행 표시(4단계 도트, 무채색). "나중에 하기"는 O4에서만 허용 — 단, 누르면 M1 empty(온보딩 유도 상태)로 떨어지고 N21 주간 리마인드가 걸린다.

## 3. 단계별 스펙

### O1. 가입 / 로그인

| 항목 | 스펙 |
|---|---|
| 방식 | 휴대폰 번호 + 인증번호 (이메일은 선택 보강). 소셜 로그인 v1 제외 |
| 필수 입력 | 휴대폰, 이름, 이용약관·개인정보 동의 (개인정보 처리 고지에 "근로자 정보의 행정 목적 사용" 명시 — 기존 가드레일 준용) |
| 보안 | 가입 후 생체 인증(Face/지문) 등록 제안 — 승인 행위 본인확인용. 건너뛰기 가능하되 첫 승인 시 재제안 |
| 상태 | error: 인증번호 불일치 시 필드 인라인 1줄 "인증번호가 일치하지 않습니다" (범용 문구 금지) |

### O2. 역할 선택

```
"어떤 일을 하시나요?"
[ 외국인 직원 관리를 직접 합니다 ]   → manager 플로우 (O3 직접 진행)
[ 승인과 보고만 받으면 됩니다 ]      → owner 플로우
```

- owner 선택 시: "실무 담당자를 초대하면 등록을 맡길 수 있습니다" → [담당자 초대(문자 링크)] / [직접 입력할게요]. 초대를 보낸 owner는 **대기 화면이 아니라 샘플 브리핑 프리뷰**(아래 §5)로 이동 — 기다리는 동안 제품 가치를 먼저 본다.
- 역할은 나중에 설정에서 변경 가능. 초대받은 담당자는 O1→O3부터 진행하고 사업장은 자동 연결.

### O3. 사업장 등록

| 필드 | 필수 | 비고 |
|---|---|---|
| 사업장명 | ✓ | |
| 업종 / 지역 | ✓ | 셀렉트. E-9 허용 업종 목록 기반 |
| 외국인 근로자 수(대략) | ✓ | "5명 미만 / 5~20 / 20~50 / 50+" — O4 경로 추천에 사용 |
| 사업자등록번호 | 선택 | 행정사 패키지 생성 시점에 재요청 |
| 브리핑 시각 | 기본 08:30 | 변경 가능 (2단계 N10 연동) |

한 화면, 스크롤 없이. 완료 버튼 라벨은 "다음" (SOCAR 보이스: 명령형 동사, 수식 없음).

### O4. 근로자 입력 — 3경로 분기

진입 화면:

```
"근로자를 등록해볼까요?
 1명만 등록해도 오늘 브리핑이 시작됩니다."

[ 사진으로 등록 ]   여권·외국인등록증 촬영 (추천 배지)
[ 직접 입력 ]       4개 필드만
[ 파일로 한 번에 ]  CSV — PC 링크 안내
─────────────────
운영팀이 대신 세팅해드릴 수도 있습니다 → [대행 요청]
```

경로 추천 로직: O3의 근로자 수가 20+ 이면 [파일로 한 번에]를 첫 번째로, 5명 미만이면 [사진으로 등록]을 첫 번째로 정렬.

#### O4-A. 사진 OCR 경로 (모바일 기본)

```
1. 촬영 가이드: 외국인등록증 앞면 프레임 오버레이. "빛 반사가 없게 찍어주세요"
2. 업로드 → OCR 추출 (loading: "정보를 읽고 있습니다" + 스켈레톤 필드)
3. 확인 화면: 추출된 필드가 편집 가능한 폼으로 — 이름 / 국적 / 체류자격 / 체류만료일
   각 필드에 신뢰도 낮으면 노랑 밑줄 + "확인해주세요"
4. [등록] → O5
```

- 안전: 촬영 원본은 서류 보관함에 저장(암호화), 외국인등록번호는 **마스킹 저장** — 화면에는 `***-*******`만. OCR 실패 시 수기 폼으로 자연 전환(입력값 유지).
- OCR은 기존 기획에서 후순위(P1)였으나 온보딩 한정으로 P0 승격. 범위는 외국인등록증·여권 2종만.

#### O4-B. 수기 입력 경로

| 필드 | 필수 | 이유 |
|---|---|---|
| 이름(표시명) | ✓ | 카드 표시용 |
| 국적 | ✓ | 컨택 언어 기본값 결정 |
| 체류자격 (E-9 등) | ✓ | 케이스 규칙 선택 |
| 체류만료일 | ✓ | **D-day 계산의 유일한 필수 재료** |
| 계약종료일 | 선택 | 있으면 충돌 감지 즉시 활성 |
| 연락 채널/언어 | 선택 | 컨택 에이전트용, 나중에 가능 |

- 4필드 1화면. 날짜는 휠 피커. 완료 버튼: "등록하고 브리핑 보기" — 목적지를 버튼에 씀.
- 등록 직후 "1명 더 등록" / "브리핑 보기" 선택지. 반복 등록 시 직전 국적·자격 기본값 유지.

#### O4-C. CSV 경로 (PC)

- 모바일에서 선택 시: "PC에서 이어서 하기" — 링크를 본인에게 문자/이메일 발송 + "그동안 1명만 먼저 사진으로 등록해두면 브리핑이 시작됩니다" (모바일 이탈 방지 훅)
- PC 화면(반응형 웹, D4): 템플릿 다운로드 → 업로드 → 컬럼 매핑 확인 → 오류 행 인라인 표시(오류 행만 수정 재업로드) → 일괄 등록. 기존 PRD P0 "CSV Source Import" 스펙 준용.

#### O4-D. 대행 세팅 (파일럿/행정사 화이트라벨)

- [대행 요청] → 연락처 확인 + 서류 전달 방법 안내 (보안 업로드 링크). 운영팀 세팅 완료 시 N10 아침 브리핑 알림으로 복귀 유도: "사업장 세팅이 완료되었습니다. 첫 브리핑이 준비되어 있습니다."

### O5. 첫 브리핑 생성 (자동 — 이 단계가 Aha Moment)

등록 완료 즉시 (스케줄러를 기다리지 않고) 브리핑을 **동기 생성**한다.

```
[생성 연출 — M9 에이전트 런 축약판, 2~3초]
 1. 근로자 정보 확인          Nguyen Van A · 베트남 · E-9
 2. 체류만료 D-day 계산       D-62
 3. 확인 항목 정리            서류 체크리스트 준비
      ↓ (motion-standard)
[M1 오늘 브리핑]
 "담당자님, 확인이 필요한 업무가 1건 있습니다."
 ┌ Nguyen V. 체류만료 D-62 · 확인 필요 (MEDIUM) ┐
 │ 서류 체크리스트를 채우면 누락 여부를           │
 │ 자동으로 점검합니다                            │
 │ [서류 상태 입력]  [케이스 보기]                │
 └───────────────────────────┘
```

- **D-90 밖이어도 카드는 반드시 뜬다.** 위험이 없으면 "온보딩 케이스"(서류 체크리스트 채우기, 계약종료일 입력)를 첫 카드로 생성 — 첫 브리핑이 비는 경우를 구조적으로 제거.
- 첫 브리핑 생성도 EvidenceEvent로 기록 (`briefing_emitted`, 첫 판단 기록 #1 — M8에 첫 줄이 생김).
- 이 연출은 M9 컴포넌트(StepTimeline) 재사용. 스킵 버튼 제공.

## 4. 5분 타임라인 검증 (수기 경로 기준)

| 단계 | 목표 시간 | 누적 |
|---|---|---|
| O1 가입(휴대폰 인증) | 40초 | 0:40 |
| O2 역할 선택 | 10초 | 0:50 |
| O3 사업장 등록 | 60초 | 1:50 |
| O4-B 근로자 1명 수기 입력 | 60초 | 2:50 |
| O5 브리핑 생성 연출 | 5초 | 2:55 |
| M1 첫 카드 확인 | — | **3분 이내 달성** |

OCR 경로는 촬영 재시도 포함 +60~90초 버퍼 → 여전히 5분 이내.

## 5. owner(대표) 특수 경로 — 샘플 브리핑 프리뷰

담당자 초대 후 대기 상태의 대표에게는 **샘플 데이터로 만든 읽기 전용 브리핑**을 보여준다.

- 상단 고정 배너: "예시 화면입니다 · 담당자 등록이 끝나면 실제 데이터로 바뀝니다" (기존 '데모 데이터 배지' 패턴 재사용)
- 샘플은 화면구성 v2 §14 Nguyen 시나리오. 승인 버튼은 눌리지만 "예시에서는 저장되지 않습니다" 토스트.
- 담당자가 등록을 마치면 N01/N10 알림으로 실데이터 브리핑 진입.

## 6. 상태·엣지 케이스

| 케이스 | 처리 |
|---|---|
| O4에서 이탈(나중에 하기) | M1 empty 온보딩 유도 상태 + N21 주간 리마인드. 앱 재진입 시 O4부터 재개(진행 상태 저장) |
| OCR 3회 연속 실패 | 수기 폼 자동 전환 + "사진 인식이 어렵습니다. 직접 입력해주세요" |
| 체류만료일이 이미 지난 날짜 입력 | 등록은 허용하되 즉시 CRITICAL 케이스 생성 + "행정사 검토가 필요한 상태입니다" — 판정 표현 금지 원칙 준수 |
| 중복 등록(동일 이름+만료일) | "이미 등록된 근로자와 같습니다" 확인 시트 (idempotency) |
| 오프라인 중 O4 | 입력값 로컬 보존, 재연결 시 자동 제출. O5 생성은 서버 필수라 대기 표시 |
| 초대 링크 만료(7일) | 재발송 버튼, owner에게 상태 표시 |

## 7. 성공 지표 (온보딩 전용)

| 지표 | 목표 |
|---|---|
| 가입 → 첫 브리핑 도달률 | 70% 이상 |
| 가입 → 첫 브리핑 중위 시간 | 5분 이내 |
| 첫 카드에서 액션(서류 입력/케이스 보기) 수행률 | 50% 이상 |
| O4 이탈 후 7일 내 복귀율 (N21 효과) | 30% 이상 |
