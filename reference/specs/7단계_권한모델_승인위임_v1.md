# 7단계 — 권한 모델 + 승인 위임 규칙 (v1)

> 작성일: 2026-07-04
> 전제: D1(담당자 겸용 단일 앱, 역할별 홈 분기), 기존 PRD의 Tenant Scope·Evidence 원칙, 2단계 N01 승인 알림 정책
> 범위: MVP용 최소 권한 모델. 운영급 SSO/RBAC(기존 PRD P2)의 선행 설계로, 그 축소판이 아니라 **그 뼈대가 될 모델**을 정의한다.

---

## 1. 역할 정의 (MVP 4역할)

| 역할 | 코드 | 설명 | 앱 사용 |
|---|---|---|---|
| 대표 | `owner` | 최종 승인 권한자. 승인/반려/수정요청 중심 | 모바일 (승인 카드 모드 홈) |
| 담당자 | `manager` | 운영 전반: 케이스·근로자·초안·패키지 준비 + 승인 | 모바일+PC (운영 모드 홈) |
| 열람자 | `viewer` | 감사·내부 관리자. 읽기 전용 | PC 위주 |
| 행정사 | `expert` | 외부 수신자. 계정이 아니라 **패키지 링크 수신** — 로그인 없는 만료형 보안 링크 | 앱 미사용 |

원칙: 역할은 회사(tenant) 단위로 부여된다. 한 사용자가 여러 회사에 속하면 회사별로 다른 역할을 가질 수 있다 (행정사 화이트라벨 대비). 모든 조회·행동은 tenant scope 검사를 먼저 통과한다 — scope 밖은 404로 처리(존재 여부 자체 비노출).

## 2. 권한 매트릭스

C=생성, R=읽기, U=수정, A=승인/반려, ―=불가

| 리소스/행동 | owner | manager | viewer | expert |
|---|---|---|---|---|
| 오늘 브리핑 (M1) | R (승인 카드만) | R (전체) | R | ― |
| 케이스 (M2/M7) | R | C R U | R | ― |
| 근로자 등록/수정 (O4) | ― | C R U | R | ― |
| 메시지 초안 (M3) | R | C R U(수정요청) | R | ― |
| **승인/반려 (M4)** | **A** | **A**¹ | ― | ― |
| 발송 실행(승인 후) | ― | 실행 확인² | ― | ― |
| 행정사 패키지 | R, A(전달 승인) | C R U | R | R (링크 범위만) |
| 판단 기록 (M8) | R | R | R | R (자기 패키지 관련만) |
| 에이전트 런 (M9) | 실행³ | 실행 | ― | ― |
| 알림 설정 | 자기 것 | 자기 것 + 브리핑 시각(회사) | ― | ― |
| 구성원 초대/역할 변경 | C U | 초대만(manager·viewer) | ― | ― |
| 자율성 L3 승급 동의 (9단계 사다리) | **A** (owner 전용) | 요청만 가능 | ― | ― |
| 회사 프로필/데이터 삭제 | U | U(삭제 제외) | ― | ― |

¹ **승인 정책(Policy)으로 분기**: 회사 설정 `approval_policy`
- `owner_only` — 모든 외부 발송성 액션은 owner 승인 필수 (기본값: 근로자 20인 미만)
- `manager_allowed` — low risk(단순 안내·리마인드)는 manager 승인 가능, mid 이상은 owner (기본값: 20인 이상)
- high risk는 정책과 무관하게 **승인이 아니라 행정사 전달로 강제** (기존 5.7 에이전트 원칙 유지)

² 승인과 실행의 분리: owner 승인 = 상태를 `human_approved`로 변경. 실제 발송 트리거는 manager의 실행 확인(1탭)으로 분리 — 기존 문서의 "실행은 PC에서" 흐름을 역할로 재정의. 발송 연동 전(MVP)에는 mock 경계까지만.

³ owner의 M9는 읽기성 요청만 (정리·요약·질문). 케이스 생성·초안 생성 도구는 manager 런에서만 활성 — 도구 레벨에서 역할 검사 (D3 "경계는 도구에" 원칙의 권한 버전).

## 3. 승인 위임 (Delegation)

### 3.1 명시적 위임

owner가 설정에서 지정: "내가 승인하지 못할 때 김대리가 대신 승인" (`delegate_to: user_id, 기간 옵션`)

- 위임받은 manager의 승인은 배지로 구분: **"대리 승인 · 위임: 김대표"**
- 위임 중에도 owner에게 N01 알림은 병행 발송 (몰래 진행 방지)
- 위임 설정/해제 자체가 EvidenceEvent (`delegation_granted` / `delegation_revoked`)

### 3.2 자동 에스컬레이션 (미응답)

```
N01 승인 요청 발송
→ 48h 미응답: 재알림 1회 (2단계 룰)
→ 72h 미응답 + 위임자 있음: 위임자에게 승인 요청 이관 알림
→ 72h 미응답 + 위임자 없음: manager에게 "승인 지연" 알림 + 케이스 blocked 표시
→ D-day 임박(해당 케이스 D-7 이내) 시: 위 타임라인을 24h/48h로 단축
```

에스컬레이션은 승인 권한을 자동으로 넘기지 **않는다** — 위임자가 미리 지정된 경우에만 이관. 지정이 없으면 사람에게 알리고 멈춘다(silent failure 금지, blocked 표면화 원칙).

### 3.3 공동대표 (2단계 미해결 항목 해소)

- `owner` 복수 허용. N01은 전원 발송, **먼저 결정한 1인으로 확정** (first-decision-wins)
- 다른 owner에게는 즉시 "김대표가 승인했습니다" 알림 — 중복 승인은 idempotency key로 차단
- 반려도 동일. 승인·반려가 동시 도착하면 서버 타임스탬프 선착순, 후착은 "이미 처리됨" 응답

## 4. 인증·본인확인

| 시점 | 요구 |
|---|---|
| 로그인 | 휴대폰 인증 (O1) |
| 승인 버튼(M4 최종 승인) | 생체 인증 재확인 (등록자) 또는 PIN. 세션만으로 승인 불가 — 승인은 본인확인 이력이 남는 행위 |
| 위임자 대리 승인 | 동일 + 위임 유효성 서버 검증 |
| expert 패키지 링크 | 만료형(기본 7일) + 열람 로그. 재발급은 manager |

## 5. Evidence 기록 (권한 이벤트)

모든 권한성 행위는 기존 EvidenceEvent 스트림에 append:

```
role_granted / role_changed / member_invited / member_removed
delegation_granted / delegation_revoked
approval_granted { decided_by, on_behalf_of? }   ← 대리 승인 시 위임자 명시
approval_escalated { from, to, reason: 'timeout_72h' }
package_link_issued / package_link_viewed { expert_ref }
```

M8 판단 기록에서 승인 이력 블록은 "누가(역할) · 언제 · 본인/대리 여부"를 항상 표시. 기존 JudgmentLog 우측 패널의 승인 이력 UI를 그대로 확장한다 (6단계 분석의 재사용 사례).

## 7. 미해결 → 후속

- viewer의 M8 PII 마스킹 수준 차등(감사자용 상세 열람 권한) — 파일럿 피드백 후
- 행정사에게 계정을 주는 화이트라벨 모드(expert가 여러 회사를 보는 뷰) — BM 검증 후
- 승인 정책의 케이스 유형별 세분화(체류=owner, 리마인드=manager 자동) — 운영 데이터 축적 후
