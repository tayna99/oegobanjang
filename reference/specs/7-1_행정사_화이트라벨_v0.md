# 7-1 — 행정사 화이트라벨 (v0 설계)

> 작성일: 2026-07-14
> 상위: `7단계_권한모델_승인위임_v1.md` §7 "미해결 → 후속" 중 **"행정사에게 계정을 주는
> 화이트라벨 모드(expert가 여러 회사를 보는 뷰) — BM 검증 후"**의 선행 설계.
> 결정(2026-07-14, 사용자): 인증 = **영속 매직링크 + 개인 대시보드**, 범위 = **여러 회사 통합
> 뷰 + 행정사 자체 브랜딩**, 산출물 = **설계 문서 + 동작 목업**.

---

## 0. 결론 요약

- 화이트라벨은 새 축이 아니다. 스펙 §1이 이미 심어둔 **"역할은 tenant 단위, 한 사용자가
  여러 회사에 회사별 역할(행정사 화이트라벨 대비)"** 씨앗의 실체화다.
- 갭은 두 겹: **① 데이터 모델**(코드에 tenant 엔티티·expert 신원 없음) + **② 인증**(로그인
  체계 자체가 없음, expert 접근은 URL 토큰뿐).
- **BM 결정 전에도 "설계 + 동작 목업"까지는 가능**하다(이 문서 + 아래 구현). 실 매직링크
  발송·검증, tenant scope 서버 강제, 실 계정만 백엔드로 남는다.

## 1. 현재 vs 목표

| | 현재(링크 모델) | 화이트라벨(멀티테넌트) |
|---|---|---|
| expert 신원 | 없음. `recipient` 문자열 | `ExpertAccount`(영속) |
| 접근 | 패키지별 만료 링크 `/link/:packageId` | 영속 개인 링크 `/expert/:expertId` → 대시보드 |
| 회사 | 암묵적 단일 | `Tenant` 다수, `ExpertMembership`로 연결 |
| 브랜드 | 외고반장 | 행정사무소 브랜드 + "외고반장 제공"(powered-by) |
| 뷰 | 링크 1개 = 패키지 1개 | 여러 회사 검토 대기를 회사별로 통합 |

## 2. 데이터 모델 (구현: `src/types.ts` + `src/mocks/expert.ts`)

```ts
interface Tenant { id; name }                        // 회사 — 지금까지 암묵적 단일
interface ExpertAccount { id; officeName; brandInitial; brandColor }  // 영속 신원 + 브랜드
interface ExpertMembership { expertId; tenantId }     // §1 "회사별 역할"의 expert 고정판
// HandoffPackage 에 tenantId? 추가 — 회사별 그룹핑 키
```

- `ExpertAccount.id`가 **영속 매직링크 토큰** 역할(mock). 실서비스는 서명 토큰 + 이메일 OTP로 교체.
- `brandColor`는 **행정사 제공 브랜드 데이터**(업로드 로고와 동급) — 디자인 토큰이 아니라
  테넌트 커스텀 값이라 인라인 style로만 적용한다(Montage cyan-30 값으로 시드, 앱 primary와 구분).
- 시드: `TENANTS`(그린푸드·한빛) / `EXPERT_ACCOUNTS`(김앤리) / `EXPERT_MEMBERSHIPS`(김앤리↔둘).
  패키지는 `batbayar`(그린푸드) + `levan`(한빛) — 한 행정사가 두 회사를 통합해 보는 데모.
- 조회 헬퍼: `packagesForExpert(expertId)` → tenant별 그룹. `tenant.id` + `recipient===officeName`
  둘 다 일치하는 패키지만(= scope 검사의 데이터층).

## 3. 인증 흐름 (지금 mock, 실서비스 경계 명시)

```
[담당자] 행정사 초대/패키지 발송
   → [행정사] 이메일로 영속 개인 링크 수신 (mock: /expert/:expertId)
   → 개인 대시보드: 연결된 전 회사 검토 대기 통합
   → 패키지 뷰 → 열람 로그(package_link_viewed) → 구조화된 회신(package_reply)
```

- **지금(mock)**: `expertId`가 URL에 그대로 = 토큰. 열람·회신 evidence는 실제로 남는다.
- **백엔드 필요**: ① 서명·만료 토큰 발급/검증 + 이메일 OTP 재확인(민감 조회 전), ② tenant
  scope **서버** 강제(scope 밖 404 — 클라이언트 가드는 데모용), ③ 실 계정·재발급.

## 4. 화면 (구현: `src/features/expert/`)

| 화면 | 파일 | 내용 |
|---|---|---|
| 개인 대시보드 | `ExpertDashboardPage.tsx` | 브랜드 헤더 + "검토 대기 N건" + 회사별 그룹 리스트 → 패키지 뷰 |
| 패키지 뷰 | `ExpertPackagePage.tsx` | 브랜드 헤더(+대시보드 뒤로·소속 회사) + `DocumentPreview` + `StructuredReplyForm` 재사용 |
| 브랜드 헤더 | `ExpertBrandHeader.tsx` | 로고 이니셜(brandColor) + 행정사무소명 + "외고반장 제공" |

- 라우트: `/expert/:expertId`, `/expert/:expertId/package/:packageId` — 둘 다 Shell(로그인 앱
  챙) **바깥** 형제 라우트(무인증 `/link/:packageId`와 동일 관례).
- 콘텐츠(문서 미리보기·구조화된 회신)는 기존 것을 그대로 재사용 — `StructuredReplyForm`을
  `ExpertLinkPage`에서 별도 파일로 추출해 두 화면이 공유(중복 0).
- `/link/:packageId`(단발 링크)는 **유지**한다 — 화이트라벨은 그 위에 얹는 영속 계층이지 대체가 아니다.

## 5. tenant scope (스펙 §1 원칙 준수)

- 이 행정사에게 오지 않은 패키지(`recipient` 불일치)·없는 토큰은 **"링크를 찾을 수 없습니다"**
  로만 응답(존재 여부 비노출) + 열람 로그 남기지 않음. `ExpertPackagePage`의 `inScope` 검사.
- 데모는 클라이언트 가드지만, 실서비스는 §3의 서버 scope 강제로 승격해야 한다(클라이언트만으론 불충분).

## 6. 화이트라벨 범위 (결정: 통합 뷰 + 브랜딩)

- **통합 뷰**: 한 행정사가 여러 고객 회사의 검토 대기를 한 대시보드에서(스펙 §7 문구 그대로).
- **브랜딩**: 행정사무소 로고 이니셜·이름·브랜드색이 expert가 보는 전 화면 헤더에 노출 +
  "외고반장 제공" powered-by. (완전 도메인 화이트라벨·행정사 재판매 BM은 이 위의 후속.)

## 7. 미해결 → 후속

- 실 인증(서명 토큰·이메일 OTP)·tenant scope 서버 강제·실 계정 — 백엔드 접속점.
- 담당자→행정사 **초대 플로우**(지금은 시드로 연결 전제) — 설정 화면에 초대 UI 추가 시.
- 행정사 회신이 담당자 케이스 **타임라인에 실시간** 반영(현재 M8 전역 기록까지만, 7-3f와 동일 한계).
- 완전 도메인 화이트라벨(행정사 자체 도메인/앱 이름) — 재판매 BM 확정 후.
