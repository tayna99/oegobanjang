# 9단계 — "이게 에이전틱한가?" 비판적 검토와 보완 설계 (v1)

> 작성일: 2026-07-04
> 방법: 1~8단계 설계를 에이전틱 시스템의 6가지 기준(목표 지향 자율성 / 도구 루프 / 능동성 / 적응성 / 연속성·메모리 / 멀티 에이전트 협업)에 대조
> 결론 먼저: **현재 설계의 에이전틱한 부분은 M9 하나다. 나머지는 룰 엔진 + 정적 워크플로우에 에이전트라는 이름을 붙인 것에 가깝다.** 고칠 수 있고, 고치는 방향은 기존 안전 철학과 충돌하지 않는다.

---

## 1. 정직한 채점표

| 에이전틱 기준 | 현재 설계 | 판정 |
|---|---|---|
| 도구 호출 루프 (think→act→observe) | M9 에이전트 런에 존재. 가드레일도 도구 레벨 | **충족** — 단 사용자가 시켜야만 돈다 |
| 능동성 (먼저 움직임) | 아침 브리핑이 "먼저 말을 건다" — 그러나 실체는 **스케줄러+룰 엔진**. LLM은 문장만 다듬음 | **절반** — 능동적으로 보이지만 에이전트가 아니라 cron이 능동적임 |
| 목표 지향 자율성 | "메시지 보내줘" 수준의 단발 목표만. "Nguyen 체류연장을 끝까지 챙겨줘" 같은 **며칠짜리 목표**를 맡길 수 없음 | **미충족** |
| 적응성 (실패 시 재계획) | M9 error에 "대안 시도 or 멈춤" 한 줄뿐. 재계획 정책 없음 | **미충족** |
| 연속성 / 메모리 | 런은 1회성·세션 전제·동시 1개. 케이스는 여러 날 흐르는데 에이전트는 순간만 산다. 학습된 운영 컨텍스트 저장 없음 | **미충족** |
| 멀티 에이전트 협업 | 3 Mission/8 Sub 매핑은 문서 서사로만 존재. 런타임 핸드오프 스펙 없음 (목업에 있던 "Workforce Agent와 협업 중"이 새 스펙에서 오히려 사라짐) | **미충족** |

핵심 진단: **트리거의 주체가 전부 사용자다.** 초안 보기를 눌러야 초안이 있고, 응답 요약 보기를 눌러야 해석이 있다. 에이전틱 시스템의 정의는 반대다 — 사건이 트리거하고, 에이전트가 준비하고, 사람은 승인만 한다. 다행히 이 방향은 기존 원칙("자동 제출이 아니라 자동 준비")의 위반이 아니라 **완성**이다: 자동 준비의 범위를 넓히는 것이지 자동 실행이 아니기 때문.

## 2. 채워야 할 것 — 우선순위별 보완 설계

### P0-1. 프로액티브 런 (Proactive Run) — 가장 큰 구멍

이벤트가 에이전트 런을 자동 시작한다. 사람이 열기 전에 준비가 끝나 있다.

```
이벤트 발생 (D-30 진입 / 응답 도착 / 케이스 생성)
→ 시스템이 M9 런을 자동 실행 (actor_type: 'agent')
→ 도구 루프: 상태 확인 → 근거 검색 → 초안 생성 → 승인 요청 생성에서 정지
→ 산출물이 브리핑 카드에 "준비 완료" 상태로 부착
→ 사용자가 카드를 열면 이미 초안·근거·다음 행동이 있음
```

- 스펙 반영: M1 카드에 `preparedBy: 'agent' | 'rule'` 필드 추가. 에이전트가 준비한 카드는 "AI가 준비를 마쳤습니다 · 런 #4791 보기" 링크 노출 — 준비 과정 전체가 M9 뷰로 재생 가능
- 안전: 프로액티브 런의 도구 화이트리스트는 **읽기 + 초안 생성만**. 승인 요청 생성이 항상 종착점
- 지금 브리핑과의 관계: 룰 엔진은 감지(무엇이 위험한가), 프로액티브 런은 준비(그래서 무엇을 만들어둘 것인가)로 역할 분리

### P0-2. 케이스 오케스트레이터 — 며칠짜리 목표를 맡는 주체

케이스마다 수명주기 전체를 담당하는 장기 러닝 에이전트를 붙인다.

```
케이스 "Nguyen 체류연장" 오케스트레이터의 삶:
D-30  감지 → 서류요청 초안 준비 → 승인 요청          (프로액티브 런 #1)
D-30  승인됨 → 발송 경계 통과 → 응답 대기 상태로 전환
D-28  응답 없음 → 리마인드가 필요한지 판단 → 리마인드 초안 + 승인 요청   (런 #2)
D-27  응답 도착 → 해석 → 상태 갱신 제안 → 부분수신이면 나머지 요청 판단  (런 #3)
D-21  서류 확보 → 행정사 패키지 준비 → 전달 승인 요청                    (런 #4)
```

- 지금 설계에서 "2일 뒤 리마인드"는 scheduler 예약(고정값)이다. 오케스트레이터에서는 **에이전트의 판단**이 된다: 응답 패턴·기한 여유를 보고 리마인드 시점·톤을 결정하고, 그 이유를 RunStep으로 남긴다
- 데이터 모델 추가: `Case.orchestrator_state` (다음 깨어날 조건: 이벤트 or 시각), 런 이력은 케이스에 체이닝 (`run.parent_case_id`)
- M2 바텀시트에 "이 케이스의 에이전트 활동" 타임라인 블록 추가 — 런 #1~#4가 이어진 하나의 이야기로 보임

### P0-3. 응답 도착 자동 런

M6의 트리거를 사용자에서 이벤트로 뒤집는다. 현재: 알림 탭 → 화면 열림 → 해석 표시. 보완: 응답 수신 즉시 해석 런 자동 실행 → 알림에 이미 요약이 담김("여권은 내일, 계약서는 회사 보관 — 확인 필요 1건") → 화면을 열면 상태 갱신 제안과 **다음 초안까지** 대기 중. `is_final=false`(사람 확인 전 미확정) 원칙은 유지.

### P1-4. 재계획 정책 (Replanning)

| 실패 유형 | 에이전트 행동 |
|---|---|
| 도구 일시 실패 (RAG 타임아웃 등) | 1회 재시도 → 대체 도구(캐시된 근거) → 그래도 실패면 해당 단계 제외하고 진행 가능 여부 판단 |
| 필수 정보 부족 | QuestionCard로 사람에게 질문 (이미 스펙 있음 — 프로액티브 런에서는 질문을 브리핑 카드로 적재) |
| 가드레일 차단 | 우회 금지. 승인 요청 or 행정사 핸드오프로 전환하고 guardrail 스텝 기록 (기존 스펙 유지) |
| 판단 확신 부족 | confidence 낮음을 명시하고 사람 확인 요청 — 몰래 진행 금지 |

모든 재계획은 RunStep `{kind:'replan', reason, from, to}`으로 가시화. "AI가 조용히 딴 길로 갔다"가 없어야 신뢰 자산이 된다.

### P1-5. 서브 에이전트 핸드오프 가시화

8개 세부 에이전트를 런타임 개념으로 복원한다. RunStep에 추가:

```ts
| { kind:'handoff'; from:'Visa Document Agent'; to:'Multilingual Contact Agent';
    reason:'서류 요청 메시지 생성 필요'; }
```

M4/M9 헤더의 에이전트명이 핸드오프 시 전환되는 연출 (목업 화면_모바일1의 "Workforce Agent와 협업 중"을 스펙으로 승격). 발표 서사와 런타임이 일치하게 됨.

### P1-6. 자율성 사다리 (Autonomy Ladder) — 배지를 작동하게 만들기

"현재 자율성: Medium"이 표시가 아니라 정책이 되도록:

| 레벨 | 의미 | 승급 조건 (액션 유형별) |
|---|---|---|
| L1 감시 | 모든 산출물 승인 필요 (현재 기본) | — |
| L2 준비 | 프로액티브 런 허용, 승인 필요 유지 | 온보딩 완료 시 기본 |
| L3 조건부 자동 | **low risk 한정**(리마인드 재발송 등) 자동 실행 + 사후 보고 | 같은 유형 승인 20회 + 수정률 5% 미만 + **사용자가 명시적으로 승급 동의** |
| 상한 | mid/high risk는 영구히 L2 이하 (비자·계약·행정사 전달은 승급 불가) | 구조적 고정 |

- 승급·강등 모두 EvidenceEvent. 사용자가 언제든 강등 가능. 수정률 급등 시 자동 강등
- 이것이 기존 성공 지표(Action Approval Rate, Revision Rate)를 **에이전트 행동 개선으로 되먹이는** 유일한 경로가 됨 — 지금은 지표가 대시보드에서 끝난다

### P2-7. 에이전트 메모리 (운영 노트)

런에서 관찰된 사실을 구조화 저장: "Nguyen은 오후 응답률 높음", "이 회사 기한 관행 3일", "행정사 김사무소는 PDF 선호". 다음 런의 컨텍스트로 주입.

- 가드레일: 저장 대상은 운영 패턴만. 성실도·성격·이탈 추정성 메모는 스키마 차원에서 금지(기존 금지 원칙의 메모리 버전). 사용자가 열람·삭제 가능한 "AI가 기억하는 것" 화면 필요

### P2-8. 동적 제안 + 런 큐

- 커맨드 바 suggestions를 정적 3개에서 컨텍스트 생성으로: 응답 2건 도착 상태면 "도착한 응답 2건 정리해줘"가 뜬다
- 동시 런 1개 제한을 사용자 런 1개 + 백그라운드 프로액티브 런 N개(우선순위 큐)로 수정 — P0-1과 현행 제한이 충돌하므로 필수 수정

## 3. 에이전틱하게 만들지 말아야 할 것 (경계 재확인)

- 승인 그 자체 — 영원히 사람. 자율성 사다리의 상한이 이를 보증
- D-day·서류 충족 계산 — 결정적 룰 유지. 에이전트는 룰을 도구로 호출할 뿐
- Evidence 기록 — 에이전트가 기록을 "요약·생략"하지 않음. append-only 원문 이벤트
- 비자 판정·법률 자문·정부 제출 — 기존 금지 목록 그대로

## 4. 기존 문서 반영 사항 (개정 대상)

| 문서 | 개정 |
|---|---|
| 1단계 M1 | CaseCard에 `preparedBy`, "AI가 준비를 마쳤습니다" 상태 추가 |
| 1단계 M2 | "이 케이스의 에이전트 활동" 타임라인 블록 추가 |
| 1단계 M9 | 동시 런 규칙 수정(사용자 1 + 백그라운드 큐), RunStep에 handoff·replan kind 추가, 런 체이닝 |
| 2단계 알림 | N05 확장: 프로액티브 런 완료는 개별 푸시가 아니라 브리핑 카드 상태로 (알림 피로 방지) |
| 통합설계 D3 | "자유는 루프에, 경계는 도구에" + **"트리거는 이벤트에"** 추가 |
| 7단계 | L3 승급 동의를 owner 권한으로 명시 |

## 5. 로드맵 위치

P0 3개(프로액티브 런, 케이스 오케스트레이터, 응답 자동 런)는 **MVP 데모 차별화의 본체**다 — 이것이 있어야 "버튼 누르는 SaaS를 벗어났다"는 주장의 근거가 화면에 존재한다. P1은 파일럿 단계, P2는 운영 데이터가 쌓인 뒤. 구현 관점에서 P0는 새 화면이 아니라 **트리거 배선 변경**(이벤트→런 시작)이라 UI 추가 비용이 낮다 — M9 컴포넌트를 그대로 재생한다.
