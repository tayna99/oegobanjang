## 0. 결론 요약

- 화이트라벨은 새 데이터 모델이 아니다. v0가 이미 심어둔 `Tenant`/`ExpertAccount`/
  `ExpertMembership`(`src/types.ts:163-180`)을 실서비스 계약으로 승격하는 작업이다.
- legacy 백엔드에는 유사한 문제("한 사용자가 여러 회사를 회사별 역할로 봄")를 푸는 패턴이
  이미 있다 — `DailyBriefingUserCompanyAccess` 조인 테이블 + 서명 JWT의 `company_ids`
  클레임 + `_assert_company_scope` 계열(`daily_briefing_service.py:2716-2736`). 다만 이번
  검증에서 확인된 바, 이 패턴은 **문자 그대로 재사용(import)할 수 있는 자유함수가 아니라**
  daily-briefing 도메인 클래스·저장소에 강결합된 private 메서드다(§4.1 정정). v1은 이
  **패턴을 특수화해 패키지 도메인용으로 새로 구현**한다 — "재발명 금지"는 설계 사상을
  베낀다는 뜻이지, 코드를 import한다는 뜻이 아니다.
- 이번 v1이 새로 채워야 하는 공백은 원래 두 개로 좁혀 잡았으나(위탁 생애주기, 열람 감사
  로그), 적대적 검증 결과 **최소 세 개의 축**이 추가로 확정 필요 상태임이 드러났다:
  ③ 접근 패턴이 "단발 큐레이션 전달"에서 "상시 셀프서비스 조회"로 바뀌는 데 따른 PII
  재최소화 검토(§9 결정 A 보강), ④ 위탁의 법적 성격(위탁 §26 vs 제3자 제공 §17) 확정
  (§9 결정 C 보강, 법무 검토 필요), ⑤ 정보주체 권리(열람권 §35, 처리정지 요구권 §37)
  — 이번 v1도 **구현하지 않지만, 이번엔 §10에 명시적으로 등재**해 누락되지 않게 한다.

## 1. 범위 / 비범위

### 1.1 이번 v1이 커버하는 것

- 데이터 모델 프로덕션화: `ExpertOfficeMember`(신규) + `ExpertMembership → ExpertGrant`
  승격(PK·상태 enum 포함) + PII 노출 정책 테이블 + 자유 텍스트 PII 스캔 방어
- 인증: mock 매직링크 → 서명 토큰(JWT, `exp` 검증 신규 구현) + 이메일 OTP + 비밀번호
  재설정 전환의 mock→프로덕션 경계
- Tenant scope 서버 강제(패턴 특수화) — **scope 검사 + 승인 상태 게이트 + 사무소 일치
  검사**의 3중 체크로 패키지 조회 API 신설
- 위탁(Grant) 생애주기(명시적 status enum) + EvidenceType 확장 제안 + 위탁 근거의 법적
  성격(§26 vs §17) 확인 필요성 명시
- 열람 감사 로그(`PackageViewLog`) 신설 + 소비 화면(고객사·사무소 양쪽) + 보존기간·파기
  배치 설계
- 화면 반영(기존 3화면 확장 + 초대 관리 화면 + 사무소 직원 관리 화면 신설 필요성)
- 마이그레이션 경로(mock → v1, 단계별 표 — 필드 정규화 포함)
- 결정 A/B/C 문서화(근거·대안·뒤집는 법 + 법무 검토 필요 플래그)

### 1.2 비범위 (이번에도 안 함 — 그러나 전부 §10에 명시적으로 등재)

- **완전 도메인 화이트라벨**(행정사 자체 도메인/앱 이름) — v0 §7이 이미 후속으로 미뤘고,
  재판매 BM 확정 전까지는 이번에도 후속이다.
- 실제 백엔드 구현(코드) — 이 문서는 프로덕션 계약 설계이지, 구현 mission이 아니다.
  `backend/app/agent_runtime/` 등 구현 코드는 별도 mission에서 이 설계를 따른다.
- 7단계 스펙 §7의 다른 미해결 항목(viewer PII 마스킹 차등, 승인 정책 케이스 유형별
  세분화) — 이번 v1은 expert 화이트라벨에만 집중한다.
- 행정사 재판매 BM(가격·정산·수수료) — BM 검증 자체는 이 문서의 전제가 아니라 여전히
  선행 조건이다.
- `ExpertGrant.scope`의 세분화(문서 종류별/근로자 그룹별/**사무소 직원 개인별** 위탁 범위
  차등, §10에 개인별 축을 신규 추가) — v1은 `'package_review'` 단일 scope, 사무소 단위
  전원 접근으로 고정한다.
- **정보주체(근로자) 권리 구현**(열람권 §35, 처리정지 요구권 §37) — 검증 결과 이 항목이
  이전 초안에서 완전히 누락되어 있었다는 지적을 받아들여, 이번엔 "몰라서 빠진 것"이
  아니라 "알고도 이번 v1 범위 밖으로 둔 것"임을 §10에 명시한다. 근로자 개별 데이터
  주체 권리 UI는 이 화이트라벨 스펙과 별도로, 근로자 대상 제품(워커 포털 등) 트랙에서
  다뤄야 할 만큼 범위가 크다.
- 위탁계약 필수조항(제26조①)의 실질 검증(템플릿 법무 검수) — v1은 체크박스 확인만 하고,
  계약 내용 자체의 적법성 검증은 법무 결과에 따른 별도 작업이다.

#### 2.2a 정정 — "필드 확장"이 아니라 "필드 리네임 + 확장"

v1 초안은 `ExpertMembership → ExpertGrant`를 "필드 확장"이라 서술했으나, 기술 검증에서
지적된 대로 실제로는 `expertId`(`src/types.ts:178`, `src/mocks/expert.ts:16-19,38`에서
실사용 중인 필드명)를 `expertAccountId`로 **리네임**한다. `mocks/expert.ts`의
`packagesForExpert`(필터/맵 로직, 16-49행)가 이 필드명 변경에 맞춰 함께 수정돼야
한다는 것을 §8 마이그레이션 표에 명시적 항목으로 올린다(초안에서 빠졌던 부분).

### 2.3 신규 / 확장 타입

```ts
// 사무소 소속 개인 구성원 — CompanyMember(src/types.ts:139-143)의 대칭.
// PIPA 열람기록 요건(§6)의 최소 단위: "이 사람이 봤다"까지 추적 가능해야 한다(결정 B).
export interface ExpertOfficeMember {
  id: string;              // 'eom-001'
  expertAccountId: string; // FK → ExpertAccount.id
  name: string;            // '이 아무개' — 사무소 내부 개인 이름, 브랜드 표시명 아님
  email: string;           // 로그인 + 이메일 OTP 대상(§3)
  status: 'active' | 'suspended';
  isOfficeAdmin: boolean;  // §5.6(신규) — 이 사무소의 다른 ExpertOfficeMember를 등록/정지할 권한.
                           // 최초 초대 수락자가 기본값 true, 이후 사무소 자율로 위임 가능
}

// ExpertMembership의 승격 — 생애주기가 boolean 하나로 표현 불가능하다는 검증 지적(일관성 #4)을
// 반영해 status를 단일 진실 공급원으로 둔다. active: boolean 파생 필드는 두지 않는다
// (status === 'active'로 계산 — 저장 값 중복에 따른 드리프트 방지).
export type ExpertGrantStatus =
  | 'invited'             // 초대 발송, 사무소 미수락
  | 'company_authorized'  // 위탁계약 근거 확인 완료, 첫 로그인 대기
  | 'active'              // ExpertOfficeMember 최초 로그인 성공, 조회 가능
  | 'expired'             // until 도달 자동 종료
  | 'revoked';            // owner 수동 철회

export interface ExpertGrant {
  id: string;                    // PK(신규) — DailyBriefingUserCompanyAccess.id(daily_briefing.py:242)와
                                  // 대칭. 재초대 시 과거 grant와 새 grant를 구분하는 이력 추적에 필수
                                  // (자연키 (expertAccountId, tenantId)만으로는 재초대 이력이 사라짐 — 기술 검증)
  status: ExpertGrantStatus;
  expertAccountId: string;
  tenantId: string;
  scope: 'package_review';       // v1 고정값 — 세분화는 §10 후속(문서종류/근로자그룹/사무소 직원 개인 축 모두)
  grantedBy: string;              // CompanyMember.id — 초대한 owner/manager(결정 C)
  basis: 'processing_agreement'; // §5.1 동의근거 확인의 근거 종류 — 법무 확정 시 값 종류 추가(§9 결정 C)
  from: string;                  // 'YYYY-MM-DD'
  until: string;                 // 'YYYY-MM-DD' — 무기한 위탁 금지(결정 C)를 타입 레벨에서 강제하려는
                                  // 의도이나, 이는 TypeScript 컴파일타임 제약일 뿐이다. 실제 서버 측
                                  // 강제(DB NOT NULL + API 입력 검증, zod 등 런타임 스키마)는 별도
                                  // 구현이 필요하다 — "타입이 있으니 안전하다"고 오독하지 않는다(기술 검증 #8)
  reviewIntervalDays?: number;    // 미지정 시 365(연 1회 재확인) — 결정 C 근거
  revokedReason?: 'expired' | 'manual'; // status가 expired/revoked로 전이될 때 채움
}
```

¹ `DelegationConfig.until`은 내부 승인 위임(같은 회사 안, owner→manager)이라 무기한을
허용해도 리스크가 다르다. `ExpertGrant`는 외부자에게 상시 접근권을 주는 것이라 타입
레벨에서부터 다르게 취급한다 — 이 비대칭은 의도적이다. 단, 위 주석대로 이 강제는
컴파일타임 한정이므로 서버측 검증을 반드시 별도 구현한다(§8 항목).

### 2.4 PII 노출 정책 테이블 (결정 A의 구현)

결정 A의 대안 조항("마스킹 규칙을 역할×근거유형별 정책 테이블로 설계해 하드코딩하지
않는다")을 타입으로 명시한다.

```ts
export type PiiExposure = 'plain' | 'masked' | 'hidden';

// 역할×필드 정책 테이블 — 화면 컴포넌트에 exposure 분기를 하드코딩하지 않는다.
// 법무 검토로 특정 필드의 노출 수준이 바뀌면 이 표의 행 하나만 바꾼다.
export interface PiiFieldPolicy {
  field: string;                     // 'HandoffPackage.workerName' | 'HandoffPackage.alienRegistrationNumber' | ...
  roles: Array<Role | 'expert'>;     // 이 노출 수준이 적용되는 역할
  exposure: PiiExposure;
}
```

기준으로 정정**(일관성 검증 #5: 초안은 `CaseCard.workerRef`의 필드 경로를 인용했으나,

#### 2.4.1 자유 텍스트 PII 방어 (신규 — 법률/기술 검증 반영)

**왜 신규 요건인가**: `PiiFieldPolicy`(§2.4)는 **명명된 필드** 단위 정책이다. 그러나
legacy `_walk_forbidden`/`_validate_safe_text`(`handoff_persistence_service.py:462-486`)는
필드명과 무관하게 **모든 문자열 값**을 정규식(외국인등록번호·전화·여권 패턴,
동 파일:24-26)으로 재귀 스캔해, 자유 텍스트에 섞여 들어간 PII까지 차단하는 방어
깊이(defense in depth)를 가진다. v1의 `PackageDocSection.lines`(서버 템플릿 문장),
`StructuredReplyForm`의 회신 상세 내용, 향후 에이전트가 생성하는 케이스 요약 등은 전부
자유 텍스트라 명명된 필드 정책만으로는 방어되지 않는다 — 이는 결정 A의 방향(노출 수준
자체)을 바꾸는 게 아니라, **결정 A가 실제로 지켜지도록 만드는 구현 요건**이다.

- API 응답 생성 직전, `HandoffPackage`의 모든 문자열 필드(`sections[].lines`,
  `items[].note`, evidence `summary` 등)에 legacy와 동형의 정규식 스캔(외국인등록번호
  `\d{6}-[1-4]\d{6}`, 전화 `010-\d{4}-\d{4}`, 여권 `[A-Z]{1,2}\d{7,8}` 패턴 재사용)을
  적용해, 매치되면 요청을 500이 아니라 해당 필드를 마스킹 처리한 안전 응답으로 대체한다
  (legacy는 `ValueError`로 즉시 실패시키지만, expert 대상 API는 가용성을 우선해 마스킹
  폴백을 채택 — 단, 이 폴백 발생 자체를 내부 알림으로 남겨 원본 데이터 파이프라인의
  PII 혼입 버그를 조기 발견한다).
- `StructuredReplyForm`의 회신 상세 입력값(`detail`)도 evidence summary에 그대로
  들어가므로(`StructuredReplyForm.tsx:50`) 같은 스캔을 저장 시점에 적용한다.

### 2.5 세 갈래 비교 — 왜 결정 A인가 (실제 조사 인용)

| | legacy 백엔드 allowlist | 구 Expert Portal (legacy 프론트) | 지금 목업 → v1 채택 |
|---|---|---|---|
| worker_name / 국적 | 금지 — `_FORBIDDEN_KEYS`에 `worker_name`, `nationality` 포함(`legacy/backend/app/services/handoff_persistence_service.py:27-40`) | 평문 무제한 — `<PackageField label="표시명" value={worker?.name}/>`, `label="실명" value={worker?.full_name}`, `label="국적" value={worker?.nationality}`(`legacy/frontend/features/expert/ExpertPortalPage.tsx:443-446`) | 평문 |
| 외국인등록번호/여권/전화 | 금지 — 동일 목록(`passport_number`, `alien_registration_number`, `phone`) | 평문 무제한(마스킹 로직 자체가 코드에 없음) | 마스킹 |
| 회신/메시지 원문 | 금지 — `_FORBIDDEN_TEXT_MARKERS`(동 파일:41-54) | 원문 그대로 렌더 — `<span>{message.body_ko}</span>`(`ExpertPortalPage.tsx:382`) | 요약만 |

이 표가 세 소스의 **불일치**를 보여준다는 점이 핵심이다. §2.4는 이를 "3사 유일 공통
합의 지점"이라 서술했으나(초안 오류), 위 표 자체가 legacy 백엔드=금지·구 Portal=평문
무제한으로 **정반대** 값을 취했음을 보여주므로 "합의"는 부정확한 표현이었다 —
최종본에서 삭제했다(일관성 검증 #3). 정확한 서술은: **v1이 세 갈래 중 마스킹 쪽을
독자적으로 선택**했다(legacy 백엔드와 방향은 같지만 legacy는 필드 자체를 아예 금지한
반면 v1은 마스킹으로 완화).

또한 이 표의 legacy allowlist를 "단일 API 응답 설계였을 뿐, 제품 전체 의도는 아니었을
가능성"으로 격하 해석하는 것은 위험하다는 법률 검증 지적을 받아들인다 — 실제로
`HANDOFF_DRAFT_CREATED_SUMMARY`(`handoff_persistence_service.py:20`)가 "전문가 검토용"
문구를 명시하므로, 이 allowlist는 **바로 이 화이트라벨과 동일한 유스케이스**(회사→외부
행정사 전달)를 위해 설계된 통제다. v1이 그 allowlist보다 완화된 노출(평문 이름·국적)을
택한 것은 legacy의 의도가 달랐기 때문이 아니라, **운영 필요성(검토 업무엔 이름이
필요하다)을 근거로 한 독자적·의도적 재분류**로 정정 서술한다(§9 결정 A 본문에 반영).

### 2.6 GLOSSARY 불일치 — 용어표 수정 제안

`docs/GLOSSARY.md:32`는 "테넌트 = `companyId`"(필드) 관례를 쓰는데, v0는 `Tenant`라는
별도 엔티티 인터페이스를 만들었다(`src/types.ts:163-166`). 실질적으로 `Tenant.id`가 곧
`companyId` 값 공간을 쓰므로 개념 충돌은 아니지만(내부 앱은 필드로만 참조, 화이트라벨은
회사 자체를 리스트업해야 해서 엔티티로 승격했을 뿐), 이 차이를 문서에 명시하지 않으면
다음 작업자가 두 개념을 다른 것으로 오인할 수 있다. GLOSSARY 표에 다음 행 추가를
제안한다(이 문서 범위에서 실제 GLOSSARY.md는 수정하지 않는다 — CLAUDE.md "mission 범위
밖 파일 미수정" 원칙, §10에 별도 mission으로 등재):

| 한국어 | 코드 | 정의 |
|---|---|---|
| 테넌트 엔티티 | `Tenant` | `companyId` 값 공간의 레코드형. 내부 앱은 필드(`companyId`)로만 참조하지만, 화이트라벨은 회사 목록·브랜드 매칭이 필요해 엔티티로 승격했다(`src/mocks/expert.ts:6-9`) |

추가로, `GLOSSARY.md:33`은 `Role`을 "owner/manager/viewer/**expert**"로 서술하지만
실제 `Role` 유니온(`src/types.ts:21`)엔 `expert`가 없다(주석으로 "expert는 계정이
아니라 링크 토큰이라 이 유니온에 넣지 않는다"고 명시). 이 불일치는 v1이 만든 게 아니라
선행 문제이지만, GLOSSARY를 손보는 김에 §10에 함께 등재한다(일관성 검증 #7).

### 3.3 신규 — 비밀번호 재설정 · 세션 즉시 무효화 (UX 검증 반영)

초안은 "재발급 개념은 소멸, 비번 재설정으로 대체"라고만 선언하고 실제 흐름을 설계하지
않았다(UX 검증 #1). 최소 설계:

- **비밀번호 분실**: `ExpertOfficeMember.email`로 단발 재설정 토큰(서명, 30분 만료,
  1회용) 발급 → 이메일 발송 → 새 비밀번호 설정 화면. legacy에 전례가 없으므로 신규
  구현임을 명시한다.
- **정지(suspended) 처리와 세션 관계**: `ExpertOfficeMember.status`가 `suspended`로
  바뀌어도 이미 발급된 access token은 자체 만료(idle 30분 또는 access token TTL)까지
  유효할 수 있다는 한계를 인지한다. 이를 좁히기 위해 access token TTL을 짧게(예: 15분)
  두고, refresh 요청마다 서버가 `ExpertOfficeMember.status`/`ExpertGrant.status`를
  재조회해 `suspended`/`revoked`/`expired`면 refresh를 거부한다 — 완전한 즉시 차단은
  아니지만(최대 15분 창), stateless JWT 모델에서 실무적으로 감내 가능한 절충으로 설계한다.
  완전한 즉시 무효화(서버측 세션/토큰 deny-list)가 필요하다고 판단되면 별도 구현
  범위가 커지므로 §10 후속으로 등재한다.
- **다른 회사로 재초대**: 로그인 시점에 이메일로 기존 `ExpertOfficeMember`를 찾고,
  있으면 새 `ExpertGrant`만 추가한다(계정 중복 생성 금지, UX 검증 #8 반영).

## 4. Tenant scope 서버 강제

### 4.1 참고 대상 — "재사용"이 아니라 "패턴 특수화" (기술 검증 반영)

초안은 아래 legacy 요소들을 "그대로 승격/재사용"이라 표현했으나, 검증 결과 실제로는
문자 그대로 import 가능한 자유함수가 아니다. 정확한 관계를 다시 정리한다.

- **`resolve_daily_briefing_allowed_company_ids`**(`daily_briefing_service.py:3232-3256`,
  실제 시그니처 `(db: Session, *, user_id, header_company_id, authorization=None)`):
  ① JWT `company_ids`/`company_id` 클레임이 있으면 **즉시 그 값을 반환**하고 ②
  `DailyBriefingUserCompanyAccess` 조회는 실행되지 않는다. **expert 세션은 로그인
  시점에 이미 `company_ids`를 JWT에 구워 넣으므로, 요청마다 scope를 판정하는 실제
  런타임 경로에서 히트하는 건 항상 ①번(JWT 파싱)이다.** `DailyBriefingUserCompanyAccess`
  조인 테이블 구조(및 그 특수화인 `ExpertGrant` 테이블)는 **로그인 시 JWT 클레임을
  채우는 발급 단계에서만 조회**되지, 요청 단계의 scope 해석에는 관여하지 않는다 —
  이 구분을 명확히 하지 않으면 "조인 테이블이 매 요청 조회된다"는 오해를 낳는다(초안의
  모호했던 서술을 정정). 이 함수 자체는 순수 로직이라 **패턴은 그대로 가져오되, 패키지
  API용 함수로 새로 작성**한다(시그니처의 `db: Session` 의존 등은 그대로 특수화).
- **`_assert_company_scope` / `_assert_case_scope`**(`daily_briefing_service.py:2716-2736`):
  이 둘은 **`DailyBriefingService` 클래스의 private 인스턴스 메서드**이고, 특히
  `_assert_case_scope`는 `self.repository.cases.get(case_id)`(daily-briefing 전용
  인메모리 저장소)에 강결합돼 있다. `HandoffPackage`는 이 저장소에 대응 테이블이 없는
  별개 도메인(현재는 백엔드 API 자체가 없는 프론트 mock)이므로 **이 메서드를 import해서
  그대로 쓸 수 없다** — 동일한 판정 로직(scope 밖이면 `PermissionError`)을 패키지
  도메인용으로 **새로 구현**한다. "재사용 대상(그대로 승격, 재발명 금지)"이라는 초안의
  표현은 기술적으로 과장돼 있었다(기술 검증 #3) — 재발명 금지는 **판정 로직의 모양**을
  베낀다는 뜻이지, 코드를 그대로 갖다 쓴다는 뜻이 아니다.
- **`DailyBriefingUserCompanyAccess(user_id, company_id, role)`**
  (`legacy/backend/app/models/daily_briefing.py:238-252`, `id` PK 포함) — `ExpertGrant`가
  이 모델의 **PK 포함 형태**를 그대로 특수화 대상으로 삼는다(§2.3에서 `id` 필드를
  추가한 이유). 단, 위에서 정리했듯 이 테이블이 실제로 조회되는 시점은 **JWT 발급 시**
  뿐이다.
- 리스트 필터도 쿼리 레벨: `WHERE company_id IN (allowed_company_ids)`
  (`legacy/backend/app/api/v1/daily_briefings.py:707-708` 패턴) — 그린리스트 밖 행은
  애초에 응답에 실리지 않는다. 이 패턴은 그대로 패키지 리스트 쿼리에도 적용한다.

### 4.2 패키지 조회에 적용 — 3중 체크(scope + 승인 게이트 + 사무소 일치) + 404

초안의 pseudocode는 tenant scope만 검사했다. 법률 검증(1.4)에서 지적된 대로 **패키지가
실제로 승인 완료 상태인지에 대한 게이트가 없었고**, 기술 검증(#2)에서 지적된 대로
**어느 사무소 앞 패키지인지 검사가 없어** 같은 tenant 안의 다른 사무소 패키지까지
열람 가능해지는 회귀가 있었다. 최종 흐름:

```
GET /api/v1/expert/packages/:packageId
  1. JWT 검증(서명 + exp 만료, §3.1 신규 구현) — 실패 시 401
  2. allowed_company_ids = resolve_expert_allowed_company_ids(JWT)  (§4.1, 패턴 특수화)
  3. package = repository.find(packageId)
  4. package is None                                    → 404 "존재하지 않음"(원문 그대로)
  5. package.tenantId not in allowed_company_ids         → 404 (동일 응답)
  6. package.expertAccountId !== JWT.sub이 속한 ExpertOfficeMember.expertAccountId
                                                          → 404 (동일 응답) — §2.2 정규화된
     FK로 검사. 한 tenant에 사무소 A/B가 각각 grant를 가진 과도기(사무소 교체 등)에도
     A 소속 expert가 B 앞 패키지를 보지 못하게 한다(기술 검증 #2 수정)
  7. 연결된 케이스(caseId = packageId)의 CaseState가 'human_approved' 이상이 아니면
                                                          → 404 (동일 응답) — legacy Decision
     007(`DECISIONS.md:241`, 공용 approval API 원칙 "approve 이후에도 전문가 자동 전달은
     실행하지 않는다")과 Decision 009(`DECISIONS.md:335-338`, `auto_send_to_admin_scrivener`
     영구 차단)가 지키려는 "승인 없이 외부 노출 금지" 취지를 패키지 **조회** 경로에도
     적용한다. 이 게이트가 초안엔 없었다(법률 검증 1.4) — 새 CaseState 필드를 만들지
     않고 기존 `CaseState`(`src/types.ts:6-13`)를 그대로 참조한다(재발명 금지 원칙 유지)
```

- 위 인용에서 `DECISIONS.md:241`과 `:335-338`은 **서로 다른 결정**이다 — 241행은
  Decision 007(206행~, "공용 Approval API는 target resolver 기반 scope를 사용한다")
  소속이고, 335-338행은 Decision 009(309행~, "Approval 이후 실행은 outbox/checkpoint를
  먼저 기록") 소속이다. 초안은 이 둘을 "Decision 009" 하나로 묶어 인용했다(기술 검증
  5.1의 인용 오류) — 최종본은 각각을 정확히 구분해 인용한다. 내용 결론(전문가 자동
  전달 금지) 자체는 두 결정 모두에서 일관되므로 바뀌지 않는다.

² **여기서 403이 아니라 404**를 쓴다. 내부 API(owner/manager/viewer 대상,
`_assert_company_scope`)의 기존 관례는 403이지만(`daily_briefings.py:370-374` 확인,
실제로 `HTTPException(status_code=403, ...)`), expert 대상은 7단계 스펙 §1 "scope
밖은 404로 처리(존재 여부 자체 비노출)"과 v0 §5("링크를 찾을 수 없습니다"로만 응답,
`ExpertPackagePage.tsx:42-48`)가 이미 이 원칙을 지키고 있다.

  **일관성 검증에서 지적된 표현 문제(§1)를 인정한다**: 상위 스펙(`7단계_권한모델_승인위임_v1.md`
  §1)의 "**모든** 조회·행동은 tenant scope 검사를 먼저 통과한다 — scope 밖은 404"라는
  문구는 예외 없는 보편 원칙처럼 읽히는데, 이번 v1이 내부 API는 403을 유지하고 expert
  API만 404로 못박는 것은 그 문구와 표면적으로 어긋난다. 이 어긋남을 "의도적 결정이니
  괜찮다"고 넘기지 않는다 — 실제로는 상위 스펙 §1이 애초에 expert 링크(v0 §5의 "링크를
  찾을 수 없습니다")를 염두에 두고 쓰인 문구였을 가능성이 높지만, 내부 API 403과의
  관계를 상위 문서 자신이 명시적으로 정리한 적이 없다. 이 문서는 상위 스펙 파일을
  고치는 mission이 아니므로(CLAUDE.md 범위 원칙), **상위 스펙 §1의 "모든" 표현과 내부
  API 403 관행 사이의 정합을 맞추는 별도 문서 정리 작업을 §10에 후속으로 등재**한다.
  이번 v1 자체의 입장은 명확히 한다: expert 대상 신규 API는 404, 기존 내부 API는
  건드리지 않는다(403 유지) — 보안 리뷰 필요(§10, 기존 항목 유지).

- 프론트 클라이언트 가드(현재 `inScope` 체크, `src/features/expert/ExpertPackagePage.tsx:26`)는
  유지하되 **UX 보조용**으로 격하한다 — 실제 방어선은 항상 서버다.

## 5. 위탁(Grant) 생애주기

### 5.1 단계 (status enum 기준으로 재정의 — 일관성 검증 #4 반영)

```
invited → company_authorized → active → expired | revoked
                                   ↑
                         재확인 주기(reviewIntervalDays) 경과 시 알림 (상태 전이 아님, 배지만)
```

| 단계(`ExpertGrant.status`) | 트리거 | 근거 | evidence |
|---|---|---|---|
| `invited` | owner/manager가 설정에서 "행정사 초대" — 회사 + 행정사(사무소 검색/신규 등록, 담당자 이메일 필수 입력) + scope + 기간 지정 | 개별 근로자 동의가 아니라 **회사 단위 위탁 근거**(결정 C, 법무 검토 필요 — §5.5) | `expert_access_granted`(status: invited) |
| `company_authorized` | 위탁계약 문서 링크 첨부 또는 확인 체크박스(v1 최소구현 — 필수조항 실질 검증은 §10 후속) | 결정 C — "법무 확인 필요"임을 코드가 아니라 프로세스로 남긴다 | 위 이벤트 payload에 `basis: 'processing_agreement'` |
| `active` | `ExpertOfficeMember` 최초 로그인 성공(이메일 매칭으로 기존 사무소 구성원 재사용 또는 신규 생성) | — | `expert_access_granted`(status: active로 갱신 — 초안의 "또는 `member_invited`" 양자택일 표현을 삭제하고 단일 이벤트로 확정, 일관성 검증 #4) |
| `expired` | `until` 도달(자동, 무기한 금지 — 결정 C) | 서버 배치 조회 | `expert_access_revoked`(`reason: 'expired'`) |
| `revoked` | owner/manager가 즉시 해제 | — | `expert_access_revoked`(`reason: 'manual'`) |
| *(상태 아님, 배지)* | `reviewIntervalDays`(기본 365) 경과 시 owner/manager **및 사무소 측**(§5.3)에 "재확인 필요" 알림. 미확인이라고 즉시 자동 차단하지 않는다(silent 실패 금지 — 7단계 §3.2 "blocked 표면화 원칙"과 동일 사상). 단, 무기한 방치를 막을 최종 강제 장치는 없다는 한계를 §10에 명시한다(UX 검증 #18) | — | `expert_grant_review_due`(신규, 선택) |

### 5.2 신규 EvidenceType 제안

기존 `role_granted`/`role_changed`/`delegation_granted`/`delegation_revoked`
(7단계 §5, `src/types.ts:110-116`)와 동일한 관례로 추가한다:

```ts
export type EvidenceType =
  | /* ...기존 항목... */
  | 'expert_access_granted' // ExpertGrant 상태 전이(invited/company_authorized/active) — payload: { grantId, tenantId, expertAccountId, scope, from, until, grantedBy, basis, status }
  | 'expert_access_revoked'; // 만료 또는 철회 — payload: { grantId, reason: 'expired' | 'manual' }
```

`payload`에 `from`을 누락했던 초안 오류를 수정했다(일관성 검증 #8) — `until`만 있고
`from`이 없으면 위탁 개시 시점을 감사 로그에서 복원할 수 없다.

`EvidenceEvent.actor`엔 기존 관용구 형식을 그대로 따른다: `"김담당 (본인 확인 완료)"`
(승인 이력)과 대칭으로 `"김담당 → 김앤리 행정사무소 위탁 (회사 위탁계약 근거)"`.

### 5.3 초대 무응답 · 재발송 (UX 검증 반영)

초안은 `invited` 상태에서 가입 링크를 방치하는 경우를 다루지 않았다(UX 검증 #7). 최소
설계: 초대 이메일 링크는 7일 만료(기존 패키지 링크 관행과 동일 숫자, 재발명 금지),
`invited` 상태로 7일 경과 시 owner/manager 화면에 "미수락" 배지 + 재발송 버튼 노출.
재발송은 새 `ExpertGrant.id`를 만들지 않고 같은 grant의 초대 링크만 재발급한다(이력은
grant 하나로 유지).

### 5.5 §26 공개·감독 의무 훅 (신규 — 법률 검증 반영, 법무 검토 필요)

법률 검증(2.1~2.4)에서 지적된 대로, 이번 v1은 다음을 **결정하지 않고** 프로세스 훅만
설계한다 — 최종 법적 결론은 법무 검토가 필요하다:

- **위탁(§26) vs 제3자 제공(§17) 법적 성격 분류**: 행정사가 수행하는 업무가 "위탁자
  지시에 따른 자기 목적 처리"(위탁)인지 "행정사 자신의 전문적 판단이 개입되는 별개
  목적 처리"(제3자 제공)인지에 따라 필요한 법적 근거 자체가 달라진다(제3자 제공이면
  회사 단위 위탁계약만으로 개별 근로자 동의를 생략할 수 없을 가능성). **이 분류 자체가
  §9 결정 C의 전제이며, 법무가 가장 먼저 확정해야 할 항목**이다 — 이번 v1은 이 분류를
  "위탁"으로 잠정 가정하고 설계했을 뿐, 그 가정이 맞다고 주장하지 않는다.
- **§26② 공개 의무**: 위탁으로 확정될 경우, 위탁받는 자(행정사무소)와 위탁업무 내용을
  회사의 개인정보 처리방침에 공개해야 한다. v1은 §7.2 초대 관리 화면에 "이 사무소를
  처리방침에 반영 필요" 알림 훅을 추가한다(문구·타이밍의 법적 요건은 법무 검토 필요).
  신규 사무소 추가/철회 시 이 알림이 갱신 트리거로 뜬다.
- **§26④ 정기 감독**: `reviewIntervalDays`(§5.1)는 현재 "위탁 기간 갱신 클릭" UX일 뿐,
  수탁자(행정사무소)의 안전성 확보조치 이행 여부 점검이 아니다. v1은 재확인 시점에
  "수탁자 보안 자가진단 체크리스트" 항목을 추가로 노출하는 것을 제안하나(구현은 후속),
  이 체크리스트가 법정 "정기적 감독" 요건을 충족하는지는 법무 검토 필요로 남긴다.
- **§26① 필수조항**: 위탁계약에 목적 외 이용·재위탁 제한·안전성 확보조치·손해배상
  책임 등 법정 필수조항이 실제로 담겼는지는 v1의 체크박스 확인으로 검증되지 않는다.
  표준 위탁계약 템플릿을 법무가 확정하면 그 템플릿 첨부를 강제하는 것으로 강화 가능 —
  이번 v1 범위 밖(§10).

### 5.6 신규 — 사무소 내부 직원 관리 화면 (UX 검증 반영)

초안은 §7.2(초대 관리)를 고객사가 `ExpertGrant`(위탁)를 관리하는 화면으로만 설계했고,
**사무소가 자기 직원(`ExpertOfficeMember`)을 등록/정지하는 화면의 오너십이 없었다**
(UX 검증 #3). v1은 이를 명시적으로 분리한다:

- 고객사(owner/manager)가 관리하는 것: **어느 사무소에게, 어떤 기간·scope로 위탁했는가**
  (`ExpertGrant`, §7.2).
- 사무소(`ExpertOfficeMember.isOfficeAdmin: true`인 구성원)가 관리하는 것: **우리
  사무소의 어떤 직원이 로그인 가능한가**(`ExpertOfficeMember` 로스터) — 신규 직원 등록,
  퇴사자 `suspended` 처리. 최초 초대를 수락한 사람이 기본 관리자이며, 사무소 자율로
  다른 구성원에게 관리자 권한을 위임할 수 있다.
- 두 화면 다 §7.2에 반영한다(오너십이 다른 별개 화면으로).

## 6. 열람 감사 로그

### 6.1 왜 신규 요건인가

legacy Evidence Log는 "AI 행동·승인 결정"만 기록하고, "사람이 언제 누구의 PII를 열람했는지"는
명시적으로 기록하지 않는다 — "공용 approval API의 조회는 Evidence Log를 남기지 않는다"
(`legacy/docs/EVIDENCE_LOG_SCHEMA.md:250`). 내부 owner/manager는 조직 내부 감사(HR,
접근 로그 등)로 어느 정도 보완되지만, **상시 접근권을 가진 외부 행정사에게는 이 공백이
PIPA상 신규 필수 요건**이 된다 — 시스템이 유일한 증적이기 때문이다.

### 6.2 설계

```ts
export interface PackageViewLog {
  id: string;
  packageId: string;
  tenantId: string;
  expertOfficeMemberId: string; // 개인까지 추적(결정 B) — "사무소가 봤다"가 아니라 "이 아무개가 봤다"
  viewedAt: string;             // ISO
  ip?: string;                  // 선택 — 프론트 구현 범위 밖, 서버 접속 로그 연동 시 채움
}
```

- 기존 `package_link_viewed` EvidenceEvent(`src/features/expert/ExpertPackagePage.tsx:32-39`)는
  유지한다. 다만 지금은 `actor`가 `pkg.recipient`(사무소명 문자열)까지만 담을 수 있다 —
  v1은 여기에 `expertOfficeMemberId`를 별도 컬럼(`PackageViewLog`)으로 분리해, 검색·보존
  기간 정책을 Evidence Log(케이스 감사, 목적: 판단 근거 추적)와 다르게 가져간다
  (`PackageViewLog`, 목적: 접근 통제 증빙) — 목적이 다르므로 테이블도 분리한다.
- 응답 시점 PII redact 이중 방어 관례(`legacy/docs/EVIDENCE_LOG_SCHEMA.md:244` "API 응답
  단계에서도 PII redaction을 한 번 더 적용한다")를 `PackageViewLog`에도 적용한다 — 단
  이 로그엔 애초에 PII 필드가 없다(id 참조뿐)이므로 이중 방어가 자연히 성립한다.

### 6.3 보존기간·파기 (법률 검증 반영 — 배치 메커니즘 추가)

- 접근기록(누가 언제 봤는지)은 통상 PIPA 처리방침상 최소 1년 이상 보존 관행 — v1 제안:
  **3년**(행정사 위탁 관계 종료 후에도 분쟁 대응이 가능한 기간). **법무 확인 필요**.
- **실제 파기 메커니즘**(초안 누락, 법률 검증 3.1): 보존기간 "숫자"만으로는 제21조
  파기 의무를 충족하지 못한다 — 실제로 그 기간이 되면 삭제하는 배치가 필요하다. v1은
  `viewedAt` 기준 3년(또는 법무 확정 기간) 경과 행을 주기적으로 하드 삭제하는 배치
  잡을 설계 항목으로 명시한다(스케줄·소유권은 구현 mission에서 확정, 이 문서는 "배치가
  반드시 존재해야 한다"는 요건만 못박는다).
- **`ExpertOfficeMember` 개인정보(이름·이메일) 자체의 보존기간**(초안 누락, 법률 검증
  3.2): 위탁 종료(모든 관련 `ExpertGrant`가 `expired`/`revoked`) 후 이 개인 데이터를
  얼마나 보관할지 규정이 없었다. v1은 "모든 grant 종료 + N일 경과 시 `status:
  suspended` 유지만으로는 부족하고 익명화/삭제 대상"이라는 원칙만 세우고, N값과
  익명화 방식은 법무 확인 필요로 남긴다.
- Evidence Log(케이스 감사)는 기존 append-only 정책을 그대로 따른다 — 이 문서가 바꾸지 않는다.

### 6.4 신규 — 열람 로그 소비 화면 (UX 검증 반영)

초안은 로그를 "쌓는" 설계만 하고 **조회 주체를 전혀 정의하지 않았다**(UX 검증 #9, #10
— "PIPA상 신규 필수 요건"이라 자평하면서 정작 감사 주체가 볼 통로가 없었음). v1은 최소
두 소비 화면을 §7.2에 추가한다:

- **고객사 측**: 초대 관리 화면(§7.2)에 위탁별 "최근 열람 이력" 요약(누가·언제, 사무소
  구성원 이름까지)을 노출한다.
- **사무소 측**: §5.6의 사무소 직원 관리 화면에 "우리 사무소 구성원이 어떤 회사 자료를
  언제 봤는지" 열람 이력 조회를 추가한다(사무소 대표의 내부 통제 목적).
- 근로자 본인이 자신의 정보에 대한 열람 이력을 조회하는 경로는 이번 v1 범위 밖이다 —
  §10에 명시적으로 등재(법률 검증 3.3, 정보주체 열람권 §35).

말하는 화면이 배포될 위험이 있다(§8 마이그레이션 항목).

### 7.2 신설 필요 — 초대 관리 화면 (고객사 측)

v0 §7이 "설정 화면에 초대 UI 추가 시"로 미뤄둔 항목. v1 최소 요건:

- 담당자/대표 설정 내 "행정사 관리" 탭: 사무소 검색(고유 식별자 `businessRegistrationNo`
  포함, §2.2 — 동명 사무소 오초대 방지)/신규 등록 → **사무소 담당자 이메일 입력**(§3.2
  수정 — 초안에 빠졌던 필드) → scope(`'package_review'` 고정, §10 세분화는 후속) +
  기간(`until` **필수 입력, 무기한 선택 불가** — 결정 C) + 위탁계약 근거 확인(체크박스 +
  문서 링크 필드, §5.5 필수조항 검증 강화는 후속) + 처리방침 공개 반영 알림(§5.5)
- 목록: 사무소별 활성 grant, 만료일, 재확인 필요 배지(`reviewIntervalDays` 경과 시),
  **최근 열람 이력 요약**(§6.4), 미수락 배지 + 재발송 버튼(§5.3)
- 철회 버튼 → `expert_access_revoked` evidence 기록

7단계 §2 권한 매트릭스의 "구성원 초대/역할 변경" 행(owner: C·U, manager: 초대만)과
동일한 원칙을 외부 위탁 초대에도 적용한다: **owner만 철회 가능, manager는 초대까지만**.
내부 구성원 초대와 외부 위탁 초대는 개념이 다르므로(내부는 역할 부여, 외부는 회사
데이터 처리위탁) 설정 화면에서 별도 탭으로 분리한다.

### 7.3 신설 필요 — 사무소 직원 관리 화면 (사무소 측, §5.6)

- 사무소 관리자(`isOfficeAdmin: true`)가 접근: 소속 `ExpertOfficeMember` 로스터 조회,
  신규 직원 등록(이메일 입력 → 초대), 퇴사자 `suspended` 처리.
- 우리 사무소 구성원의 회사별 열람 이력 조회(§6.4).
- 이번 v1은 화면 요건만 정의하고 구현은 후속(§10) — 단, 오너십(고객사 vs 사무소)이
  불명확했던 초안 공백은 이 절로 메운다.

## 9. 결정 A / B / C

### 결정 A — PII 노출 수준

**결정(방향 유지)**: 화이트라벨 화면(대시보드 + 패키지 뷰)은 근로자 표시명·국적·팀은
평문으로, 외국인등록번호·여권번호·전화번호 등 숫자 식별자만 마스킹한다(§2.4 정책 테이블
그대로 채택).

**근거**:
1. 이미 이 앱 전체의 관례다 — `WorkerRef.maskLevel`은 `'masked'` 고정 리터럴이지만
   (`src/types.ts:47`), owner/manager/viewer 어느 내부 화면에도 이름·국적을 가리는
   곳이 없다(`WorkerDataWorkbench.tsx:69`, `CaseWorkbench.tsx:537`, `CsvUploadWorkbench.tsx:83`).
2. 이름 없이는 검토 업무 자체가 운영 불가능하다.
3. **정정(법률 검증 반영)**: legacy 백엔드의 "국적까지 금지" allowlist
   (`_FORBIDDEN_KEYS`, `handoff_persistence_service.py:27-40`)는 **바로 이 유스케이스
   (회사→외부 전문가 전달)를 위해 설계된 통제**였다(`HANDOFF_DRAFT_CREATED_SUMMARY`
   문구, 동 파일:20) — 초안처럼 "단일 API 응답 설계였을 뿐, 제품 전체 의도가
   아니었을 가능성"으로 격하하는 건 부정확하다. 구 Expert Portal이 그 allowlist를
   지키지 않고 이름·국적·회신 원문까지 평문으로 노출했다는 사실
   (`ExpertPortalPage.tsx:443-446,382`)은 legacy의 "의도가 달랐다"는 방증이 아니라
   **알려진 정책 위반 사례**다 — v1은 이 위반을 정당화 근거로 삼지 않고, 대신 위 1·2
   번 근거(내부 화면 관례, 운영 필요성)만으로 결정 A를 **독자적으로** 재확인한다.

**신규 반영 — 접근 패턴 변화에 따른 재검토(법률 검증 1.3)**: legacy 구조는 사람이
승인한 뒤 수동으로 큐레이션해 단발 전달하는 모델이었다(`transferred_at`이 항상
`null`로 유지, `handoff_persistence_service.py:155-156,253-254`). v1(§4.2)은 **상시
로그인 + 셀프서비스 조회 API**로 바꾸는데, 이는 노출의 빈도·범위·통제 지점이 질적으로
달라지는 변화다. "예전에도 이름·국적은 봤다"는 사실만으로 결정 A를 정당화하는 것은
불충분하다는 지적을 받아들여, **접근 패턴이 상시화되는 데 따른 재최소화 필요성 자체를
법무 검토 대상으로 명시**한다(임의로 "괜찮다"고 결론 내지 않는다).

**대안**: 법무 검토에서 "국적도 익명화 대상"이거나 "접근 패턴 상시화에 따라 추가
마스킹이 필요"라 판단되면 즉시 뒤집을 수 있어야 한다. 이를 위해 마스킹 규칙을 §2.4의
`PiiFieldPolicy` 정책 테이블로 설계했다 — 코드 분기 하드코딩이 아니므로, 정책 표만
바꾸면 화면 컴포넌트 수정 없이 반영돼야 한다(구현 시 이 전제가 실제로 지켜지는지가
코드 리뷰 포인트). 단, **명명된 필드 정책만으로는 불충분**하다는 점을 §2.4.1에서
보강했다 — 자유 텍스트(케이스 요약, 구조화된 회신 등)에 PII가 섞여 들어가는 경로는
정규식 스캔(legacy `_walk_forbidden` 패턴 재구현)으로 별도 방어한다. 이는 결정 A의
노출 수준 자체를 바꾸는 게 아니라, 그 수준이 실제로 지켜지도록 만드는 구현 요건이다.

**뒤집는 법**: `PiiFieldPolicy` 테이블에서 해당 필드의 `roles: ['expert']` 행 `exposure`를
`'plain'` → `'masked'`로 변경 → §2.4 표 갱신 → 화면은 정책 조회로 렌더링해야 하므로(코드
변경 없이) 정책 배포만으로 반영된다.

### 결정 B — 계정 단위

**결정(방향 유지)**: 화이트라벨 신원은 **사무소 단위(`ExpertAccount`, 브랜드·과금 경계)
+ 그 안의 개인 구성원(`ExpertOfficeMember`)** 2층 구조로 한다.

**근거**: PIPA 열람기록 요건상 "김앤리 행정사무소가 봤다"가 아니라 "이 아무개(김앤리
행정사무소 소속)가 봤다"까지 남아야 한다(§6) — 사무소 공유 로그인 1개로는 개인 책임
추적이 불가능하다. 이미 이 코드베이스에 `CompanyMember { id; name; role }`
(`src/types.ts:139-143`, 회사 구성원) 패턴이 있으므로, 행정사 쪽도 대칭으로
`ExpertOfficeMember { id; expertAccountId; name; email; status; isOfficeAdmin }`(§2.3)로
설계해 새 패턴을 발명하지 않는다.

**신규 반영(UX 검증)**: 2층 구조를 선택한 이상 "누가 이 사무소의 로스터를 관리하는가"도
결정 B의 일부로 확정해야 했으나 초안엔 없었다 — §5.6에서 "사무소 자체가 자기 로스터를
관리하고, 고객사는 위탁(grant)만 관리한다"로 오너십을 명시했다.

**대안**: 소규모 1인 사무소만 상대한다면 `ExpertAccount`와 `ExpertOfficeMember`를 1:1로
강제해 단순화할 수도 있으나, 실제 행정사무소는 대개 복수 직원이 사건을 나눠 처리하므로
1:N을 기본으로 둔다.

**뒤집는 법**: `ExpertOfficeMember` 개념을 없애고 `ExpertAccount`에 직접 `email`/
`passwordHash`를 붙이는 1층 구조로 되돌리려면, §6의 `PackageViewLog.expertOfficeMemberId`를
`expertAccountId`로 바꾸면 된다 — 단 이 경우 PIPA 개인 단위 추적 요건을 못 채우므로,
법무가 "사무소 단위 추적으로 충분하다"고 명시적으로 판단한 경우에만 되돌린다.

### 결정 C — 위탁 관계 성립 근거

**결정(방향 유지)**: 위탁은 개별 근로자 동의가 아니라 **회사(고객사) 단위의 위탁계약
(개인정보 처리위탁)을 근거**로 하고, 담당자/대표가 설정에서 명시적으로 초대(회사+행정사+
범위+기간)해야 성립한다. **무기한 grant는 금지** — 반드시 종료일(`until`) 또는 주기적
재확인(`reviewIntervalDays`, 기본 연 1회)을 요구한다.

**근거**: 실제 한국 행정사 위탁 관행은 고용주(회사)의 위탁 결정이지 근로자 개별 동의
사안이 아니다. 기존 `DelegationConfig { active; ownerId; delegateId; from; until? }`
(`src/types.ts:146-152`)와 동일 사상으로 `ExpertGrant`를 설계해(§2.3) 일관성을 유지하되,
`until`을 required로 두어 무기한 금지 의도를 표현한다(단 §2.3 각주에서 이것이 컴파일
타임 제약일 뿐이라는 한계를 명시했다).

**법률 검증 반영 — 이 결정의 최우선 전제가 확정되지 않았음을 명시(법무 검토 필요)**:
- **위탁(§26) vs 제3자 제공(§17) 분류가 미확정**이다(§5.5). 행정사가 "위탁자 지시에
  따른 처리"가 아니라 "자기 전문적 판단을 부가하는 처리"를 한다면 법적 성격이 제3자
  제공에 가까워지고, 그 경우 "회사 단위 위탁계약만으로 충분하다"는 이 결정의 전제
  자체가 무너진다. 이 문서는 위탁으로 **잠정 가정**하고 설계를 진행했을 뿐, 그 가정이
  맞다고 주장하지 않는다 — **법무가 가장 먼저 확정해야 할 질문은 "동의가 필요한가"가
  아니라 "이게 위탁인가 제3자 제공인가"임을 명시적으로 남긴다**(초안은 "동의 요건"만
  대안으로 다뤄 이 상위 질문 자체를 놓쳤다 — 법률 검증 2.1).
- **처리정지 요구권(§37)은 동의 여부와 무관하게 이미 존재하는 법정 권리**인데, 이번
  v1 설계엔 근로자 개인이 "내 정보를 이 행정사에게 보여주지 말라"고 요구할 경로가
  없다(`ExpertGrant`는 tenant 단위 on/off뿐). 이 항목은 §9의 "대안" 각주로 슬쩍
  덧붙이지 않고 **§10 최상위 후속 항목으로 별도 등재**한다(법률 검증 2.5 — "검토
  누락이 아니라 애초에 고려되지 않은 것으로 보인다"는 지적을 받아들여, 이번엔 고려는
  했으나 구현하지 않았다는 것을 분명히 한다).
- §26②(공개)·§26④(정기감독) 훅은 §5.5에 반영했다.

**대안**: 법무 검토에서 "이건 제3자 제공"이라 판단되면 §17에 따른 별도 법적 근거(동의
또는 계약상 필요성 등)가 필요해지고, `ExpertGrant`는 tenant 단위에서 tenant+worker
단위로 세분화돼야 하므로 스키마 변경 규모가 크다(§10에 재확인 필요 항목으로 등재).
"근로자 개별 동의도 필요하다"는 결론이 나오는 경우도 동일한 확장이 필요하다 —
`ExpertGrant`에 `workerConsentRequired: boolean` 필드와 근로자별 opt-out 목록 추가.

**뒤집는 법**: 법무 확정 결과에 따라 (a) 위탁으로 확정되고 근거 문구만 바뀌면 §5.1
"동의근거 확인" 단계의 `basis` 값 종류만 늘리고, (b) 제3자 제공 또는 근로자 개별 동의가
필요하다고 확정되면 위 확장을 별도 mission으로 착수한다.

## 10. 미해결 → 후속

7단계 스펙 §7과 동일 형식 — 이번에도 못 푸는 것을 명시한다. 적대적 검증에서 새로
드러난 항목은 **(신규)** 표기로 구분한다.

- **완전 도메인 화이트라벨**(행정사 자체 도메인/앱 이름) — v0 §7에서 이미 후속으로
  미뤘고, 재판매 BM 확정 후까지는 이번에도 후속이다.
- `ExpertGrant.scope` 세분화(현재 `'package_review'` 단일값 고정) — 문서 종류별/근로자
  그룹별 위탁 범위 차등은 실사용 데이터 축적 후. **(신규)** 여기에 **사무소 직원 개인별
  접근 제한 축**도 추가한다 — 현재 설계는 사무소 전체가 grant를 공유해, 고객사가 "이
  사무소의 특정 담당자만 우리 회사를 보게" 제한할 방법이 없다(UX 검증 #4, 초안엔 이
  축 자체가 등재돼 있지 않았다).
- 위탁 성립 근거의 **법무 확정**(결정 C) — **(신규 상위 항목으로 재분류)** "위탁(§26)
  vs 제3자 제공(§17) 분류"를 최우선 확인 대상으로 명시(§9). 근로자 개별 동의 요건이
  추가되면 `ExpertGrant` 스키마 자체가 tenant 단위 → tenant+worker 단위로 커지는 큰
  변경이 따른다.
- **(신규)** **정보주체 권리 — 처리정지 요구권(제37조)**: 근로자 개인이 특정 행정사의
  자기 정보 열람을 거부할 경로가 v1에 없다. 동의 요건 논의와 별개로 이미 존재하는
  법정 권리이므로 별도 트랙으로 검토 필요(법무 검토 필요 + 제품 설계 필요).
- **(신규)** **정보주체 권리 — 근로자 본인의 열람권(제35조)**: `PackageViewLog`가
  회사/사무소의 통제 증빙 목적으로만 설계돼, 근로자 본인이 "누가 내 정보를 봤는가"를
  조회하는 경로가 없다(§6.4).
- **(신규)** `PackageViewLog` 및 `ExpertOfficeMember` 개인정보의 **실제 파기 배치
  구현**(스케줄·소유권 확정) — §6.3이 요건만 못박고 구현은 후속.
- **(신규)** 위탁의 법적 성격에 따른 **§26② 공개(처리방침 갱신) 절차의 정확한 문구·
  타이밍** 확정(§5.5, 법무 검토 필요).
- **(신규)** 위탁계약 **필수조항(§26①) 실질 검증** — 체크박스를 넘어선 표준 계약
  템플릿 강제(§5.5).
- **(신규)** **§26④ 정기 감독**을 만족하는 수탁자 보안 자가진단 절차의 실제 구현(§5.5).
