# 외고반장 보안 관련 액션 정리

## 1. 문서 목적

이 문서는 외고반장 E-9 운영 리스크 MVP 고도화 과정에서 논의한 **보안 관련 액션**만 따로 정리한 문서다.

외고반장은 외국인 근로자의 체류 정보, 계약 정보, 서류 상태, 여권/외국인등록번호 등 민감정보를 다룰 수 있기 때문에, 단순히 기능이 잘 동작하는 것보다 **개인정보 보호, 승인 통제, 근거 검증, 회사별 접근 제한, 감사 가능성**이 중요하다.

보안 설계의 핵심은 다음이다.

```text
민감정보를 보호한다.
→ 회사별 데이터 접근을 제한한다.
→ AI가 자동 실행하지 못하게 막는다.
→ 모든 판단 근거와 승인 이력을 남긴다.
→ 실패하거나 불확실하면 실행하지 않는다.
```

---

## 2. 보안 설계의 핵심 원칙

### 2.1 AI는 최종 실행자가 아니다

외고반장은 메시지 발송, 행정사 전달, 정부 포털 제출 같은 외부 실행을 자동으로 하지 않는다.

Sprint 1에서는 다음을 명확히 금지한다.

- 실제 메시지 발송
- 실제 행정사 전달
- PDF/export
- 정부 포털 자동 제출
- 외부 API 실행
- 담당자 승인 없는 액션 실행

모든 실행 후보는 `NextAction`으로 생성되고, 기본 상태는 다음과 같다.

```text
approval_required=true
blocked_until_approved=true
status=pending_approval
```

즉, AI는 실행하지 않고 **승인 가능한 초안과 패키지만 만든다**.

---

### 2.2 개인정보는 조회용과 로그/LLM 입력용을 분리한다

Nguyen 같은 이름은 근로자 조회에 필요할 수 있다.  
하지만 LLM 입력이나 EvidenceEvent에는 원문 이름과 민감정보를 남기면 안 된다.

원칙은 다음과 같다.

```text
운영용 식별값
→ State Loader 전까지 worker lookup에 사용

LLM 입력 / EvidenceEvent / 로그
→ worker_id 또는 masked name만 사용
```

예시:

```text
사용자 입력:
"Nguyen 체류만료 확인해줘"

조회 단계:
Nguyen → worker_001로 매핑

LLM / EvidenceEvent:
worker_001 또는 [WORKER_NAME]만 사용
```

---

### 2.3 EvidenceEvent에는 원문 PII를 저장하지 않는다

EvidenceEvent는 판단 근거와 실행 이력을 남기는 감사 로그지만, 원문 개인정보를 저장하는 곳이 아니다.

EvidenceEvent에 저장하지 않는 것:

- 원문 이름
- 전화번호
- 여권번호
- 외국인등록번호
- worker reply 전문
- 메시지 본문 전문
- 문서 본문 전문
- 원본 서류 전문

EvidenceEvent에 저장하는 것:

- redacted summary
- citation ids
- sha256 hash
- optional payload_ref
- event_type
- trace_id
- case_id
- approval status

## 3. 보안 관련 구현 액션

## 3.1 Input Guard / PII 처리

### 목적

사용자 입력이나 내부 상태가 LLM, 로그, EvidenceEvent로 넘어가기 전에 민감정보를 통제한다.

### 구현 액션

- 사용자 입력에서 전화번호, 여권번호, 외국인등록번호 등 민감정보를 감지한다.
- worker lookup에 필요한 이름/식별 단서는 State Loader 전까지 임시 사용한다.
- LLM 입력에는 `worker_id` 또는 masked name만 전달한다.
- EvidenceEvent에는 원문 PII를 저장하지 않는다.
- PII redaction 실패 시 workflow를 차단한다.

### 차단 조건

```text
PII redaction failure
→ workflow blocked
→ final response / handoff preview 생성 금지
```

### 관련 에러 코드

```text
PII_REDACTION_FAILED
```

---

## 3.2 Tenant Scope / 회사별 접근 제한

### 목적

A 회사 담당자가 B 회사의 근로자 정보, 케이스, 승인 이력, EvidenceEvent를 볼 수 없게 한다.

### 구현 액션

- 모든 query는 `company_id` scope 안에서 실행한다.
- API 요청의 `company_id`가 사용자의 allowed company scope 안에 있는지 확인한다.
- worker 조회 시 `worker.company_id`와 요청 `company_id`가 일치하는지 확인한다.
- Case, Approval, EvidenceEvent 조회도 company scope 검증을 통과해야 한다.
- Sprint 1에서는 운영용 SSO 대신 seed user 또는 header 기반 mock auth를 사용한다.

### Sprint 1 MVP auth assumption

```text
Full login / SSO는 Sprint 1 범위 밖이다.

Sprint 1에서는 다음 중 하나를 사용한다.
- seed user
- X-Company-Id header
- X-User-Role header
```

### 차단 조건

```text
request company_id outside allowed company scope
→ TENANT_SCOPE_VIOLATION

worker.company_id != request company_id
→ TENANT_SCOPE_VIOLATION
```

### 관련 에러 코드

```text
TENANT_SCOPE_VIOLATION
UNAUTHORIZED_ROLE
```

---

## 3.3 Action-level Approval / 승인 기반 실행 차단

### 목적

AI가 만든 실행 후보가 담당자 승인 없이 외부 실행으로 이어지지 않게 한다.

### 구현 액션

- 모든 `NextAction`은 approval pending 상태로 생성한다.
- Sprint 1에서 구현할 액션은 `request_document`, `create_handoff` 두 개다.
- 각 `NextAction`은 하나의 `Approval`과 연결한다.
- Sprint 1에서는 `pending → approved`만 구현한다.
- 승인되어도 Sprint 1에서는 실제 발송, 실제 행정사 전달, export는 실행하지 않는다.

### 기본 상태

```text
NextAction.status=pending_approval
approval_required=true
blocked_until_approved=true
```

### Approve effect

```text
Approval.status=approved
NextAction.status=approved
approved_at 저장
approval_approved EvidenceEvent 저장
외부 실행 없음
```

### 권한 규칙

```text
manager/admin만 approve 가능
viewer/auditor는 approve 불가
approver_id 저장
```

### Sprint 2에서 추가할 것

- rejected
- revision_requested
- rejection_reason 저장
- revision_reason 저장
- approval 관련 EvidenceEvent 확장

---

## 3.4 Handoff Preview 보안

### 목적

행정사 검토용 패키지를 만들되, 실제 외부 전달이나 export로 이어지지 않게 한다.

### 구현 액션

- Sprint 1에서는 `create_handoff` action이 있을 때만 Handoff preview를 만든다.
- Handoff preview는 내부 검토용이다.
- 외부 전송, export, 실제 행정사 전달은 하지 않는다.
- PII redaction 실패 시 Handoff preview 생성을 차단한다.
- tenant scope violation이 있으면 Handoff preview를 생성하지 않는다.
- `missing_evidence=true`인 경우 preview는 생성하되 “근거 부족” warning을 표시한다.

### HandoffPreview 최소 구조

```text
preview_id: str
case_id: str
action_id: str
content_redacted: dict | str
citation_ids: list[str]
warning_flags: list[str]
created_at: str
```

### 차단 조건

```text
PII redaction failure
→ preview 생성 금지

tenant scope violation
→ preview 생성 금지
```

---

## 3.5 EvidenceEvent / 감사 로그 보안

### 목적

AI 판단, 위험도 계산, 승인 요청, handoff preview 생성 과정을 감사 가능하게 남기되, 원문 개인정보는 저장하지 않는다.

### Sprint 1 EvidenceEvent type

```text
input_received
state_loaded
risk_flagged
approval_requested
handoff_preview_generated
approval_approved
```

### EvidenceEvent v1 필드

```text
event_id: str
event_version: v1
trace_id: str
case_id: str | None
request_id: str | None
event_type: input_received | state_loaded | risk_flagged | approval_requested | handoff_preview_generated | approval_approved
actor_type: system | user | agent | approver
node_name: str
summary: str
citation_ids: list[str]
redacted_input_hash: str | None
redacted_output_hash: str | None
hash_algorithm: sha256
payload_ref: str | None
created_at: str
```

### 보안 규칙

- EvidenceEvent는 append-only audit event로 취급한다.
- 원문 PII를 저장하지 않는다.
- redacted summary와 hash를 저장한다.
- 필요한 경우 secure storage 참조값인 `payload_ref`만 저장한다.
- EvidenceEvent 조회에도 tenant scope 검사를 적용한다.

---

## 3.6 Source Snapshot Hash / PII 없는 재실행 검증

### 목적

같은 회사/날짜로 Daily Briefing을 다시 실행할 때, 원천 데이터가 바뀌었는지 확인하되 개인정보 원문을 hash input에 넣지 않는다.

### 구현 액션

- `source_snapshot_hash`를 생성한다.
- 같은 `company_id/date` 재실행 시 같은 `briefing_run_id`를 유지한다.
- `source_snapshot_hash`가 같으면 같은 결과를 반환한다.
- hash가 바뀌면 기존 briefing을 업데이트한다.
- pending approval action은 중복 생성하지 않는다.

### Hash input에 포함하는 것

- company_id
- date
- sorted worker ids
- visa_expiry_date
- contract_end_date
- document status rows including due_date
- citation ids
- citation ingest_at

### Hash input에 포함하지 않는 것

- raw names
- phone numbers
- passport numbers
- alien registration numbers
- message bodies
- document bodies

---

## 3.7 Repository Transaction / 부분 저장 방지

### 목적

Daily Briefing 생성 중 일부 데이터만 저장되고 실패하는 상태를 방지한다.

### 구현 액션

다음 저장 작업은 하나의 transaction으로 묶는다.

- Case
- NextAction
- Approval
- EvidenceEvent
- DailyBriefingResult

### 실패 시 처리

```text
Transaction failure
→ STATE_SAVE_FAILED 반환
→ partial row 남기지 않음
→ final response / handoff preview 생성 금지
```

### 관련 에러 코드

```text
STATE_SAVE_FAILED
```

---

## 3.8 Citation / 근거 검증 보안

### 목적

근거 없는 행정 안내나 허위 citation을 막는다.

### Sprint 1 최소 규칙

- `citation_id`는 seed fixture 또는 retrieved mock chunk에 존재해야 한다.
- API/UI는 citation title, source, ingest_at만 표시한다.
- `citation_summaries`는 DailyBriefing response에 embed한다.
- 상세 citation API는 Sprint 2로 미룬다.

### Sprint 2에서 강화할 것

- citation id가 실제 retrieved chunk에 존재하는지 검증
- synthetic-only evidence로 official conclusion 금지
- stale-only evidence 처리
- strict chunk version validation
- `GET /api/v1/citations/{citation_id}` 필요 시 추가

### 관련 에러 코드

```text
INVALID_CITATION
STALE_EVIDENCE_ONLY
```

---

## 3.9 Bounded Router / 위험 요청 차단

### 목적

Router가 과도한 추론이나 tool calling을 하지 않게 하고, 위험 요청은 초기에 차단한다.

### 구현 액션

Router는 bounded classifier로 유지한다.

Router가 하는 일:

- 채용 관련 요청 분류
- 비자/서류 관련 요청 분류
- 다국어 컨택 요청 분류
- 행정사 패키지 요청 분류
- 위험 요청 차단

Router가 하지 않는 일:

- ReAct reasoning
- tool loop
- 외부 실행
- 법적 판단
- 자동 제출 판단

### 차단할 요청

- 법률 확답
- 자동 제출
- 국적 선호
- 성실도/인성 판단
- 채용 가능 여부 확정
- 비자 가능 여부 확정

## 3.10 External Side Effect 차단

### 목적

Sprint 1에서 외부 시스템에 실제 영향을 주는 액션을 원천적으로 막는다.

### Sprint 1에서 하지 않는 것

- 실제 메시지 발송
- 실제 행정사 전달
- 정부 포털 제출
- 카카오/문자/이메일 발송
- PDF/export
- external tool calling
- scheduler 자동 실행
- Long-term LLM Memory

### 원칙

```text
Sprint 1의 approve는 내부 상태 승인이다.
외부 실행은 아니다.
```

## 5.2 테스트해야 할 보안 조건

### PII 보호

- EvidenceEvent에 raw name이 저장되지 않는다.
- EvidenceEvent에 phone number가 저장되지 않는다.
- EvidenceEvent에 passport number가 저장되지 않는다.
- EvidenceEvent에 alien registration number가 저장되지 않는다.
- message body 전문이 저장되지 않는다.
- document body 전문이 저장되지 않는다.

### Tenant scope

- 허용되지 않은 company_id 요청은 `TENANT_SCOPE_VIOLATION`.
- worker.company_id가 request company_id와 다르면 `TENANT_SCOPE_VIOLATION`.
- EvidenceEvent 조회도 company/case scope 검증을 통과해야 한다.

### Approval

- 모든 NextAction은 `pending_approval`으로 시작한다.
- approval 전에는 외부 side effect가 없다.
- approve 시 `approval_approved` EvidenceEvent가 저장된다.
- viewer/auditor는 approve할 수 없다.

### Transaction

- Daily Briefing 생성 중 DB 저장 실패 시 partial row가 남지 않는다.
- 실패 시 `STATE_SAVE_FAILED`를 반환한다.

### Idempotency

- 같은 `company_id/date` 재실행 시 pending approval action이 중복 생성되지 않는다.
- source data가 바뀌면 같은 `briefing_run_id`로 업데이트된다.

### Handoff Preview

- PII redaction 실패 시 preview 생성 금지.
- tenant scope violation 시 preview 생성 금지.
- missing_evidence=true면 warning 표시.

### Citation

- Sprint 1에서는 citation_id가 fixture/mock chunk에 존재해야 한다.
- Sprint 2에서는 invalid citation, synthetic-only evidence, stale-only evidence를 차단한다.

## 7. 보안 관점 최종 요약

외고반장 보안 고도화의 핵심은 다음이다.

```text
1. 개인정보 원문을 로그와 LLM 입력에 남기지 않는다.
2. 회사별 데이터 접근을 강하게 제한한다.
3. 모든 액션은 승인 전 실행되지 않는다.
4. 승인되어도 Sprint 1에서는 외부 실행하지 않는다.
5. 모든 판단과 승인 요청은 EvidenceEvent로 남긴다.
6. 같은 브리핑을 재실행해도 중복 action을 만들지 않는다.
7. DB 실패 시 partial row를 남기지 않는다.
8. 근거 없는 행정 안내를 방지하기 위해 citation을 관리한다.
9. Router는 안전한 분류기 역할만 한다.
10. 실패하거나 불확실하면 실행하지 않는다.
```

한 줄로 정리하면:

> 외고반장의 보안 설계는 “AI가 더 많이 하게 만드는 것”이 아니라, **AI가 민감정보를 흘리지 않고, 회사별 권한을 지키며, 승인 전에는 멈추고, 모든 판단 근거를 남기게 만드는 것**이다.
