# 외국인 고용 Agentic OS — RAG 설계 및 데이터 소스 수집 계획

> 목적: **외국인 고용 Agentic OS** MVP에서 사용할 RAG 구조, 데이터 소스, 인덱스 설계, chunking 전략, 에이전트별 검색 방식, 평가 데이터셋을 정리한다.  
> 핵심 원칙은 단순하다. **RAG는 판단을 대신하는 장치가 아니라, 에이전트가 만든 계획·메시지·패키지에 공식 근거를 붙이는 안전장치다.**

---

## 0. 최종 결론

이 제품의 RAG는 “자료를 많이 넣는 것”이 핵심이 아니다.  
핵심은 **각 에이전트가 어떤 근거를 찾아야 하는지 분리하는 것**이다.

외국인 고용 Agentic OS의 기본 방향은 다음과 같다.

```text
사용자가 목적을 말함
→ AI가 경로를 짬
→ 필요한 자료를 찾음
→ 문서와 메시지를 준비함
→ 사람 승인 후 다음 단계로 넘김
→ 모든 근거와 승인 기록을 저장함
```

따라서 RAG도 이 흐름에 맞춰 설계해야 한다.

```text
RAG = 공식 근거와 절차를 찾는 곳
SQL/DB = 현재 직원·후보자 상태를 저장하는 곳
Rule Base = 날짜 계산과 true/false 판단을 하는 곳
LLM = 자연어 구조화, 요약, 메시지 생성, 설명을 하는 곳
Human Approval = 발송·제출·전달 전 최종 승인 지점
```

가장 중요한 원칙은 아래와 같다.

> **법령·절차·서식·안전자료는 RAG에 넣고, 직원 상태·후보 상태·서류 보유 여부·D-day 계산은 DB/룰베이스로 빼야 한다.**

---

## 1. RAG 설계의 큰 원칙

이 제품에서 RAG가 해야 하는 일은 4가지다.

### 1.1 공식 근거 찾기

예시 질문:

```text
- E-9 고용 절차가 어떻게 되지?
- 고용허가서 발급 전에 뭘 준비해야 하지?
- 고용변동 신고는 어떤 경우에 해야 하지?
- 체류기간 연장 시 어떤 서류를 확인해야 하지?
```

RAG는 이런 질문에 대해 공식 문서 기반으로 답변 근거를 찾아야 한다.

사용할 수 있는 소스:

```text
- 국가법령정보센터
- EPS / 고용24
- 한국산업인력공단
- 정부24
- HiKorea
- 고용노동부
```

### 1.2 서류 체크리스트 만들기

예시 케이스:

```text
- 체류기간 연장
- 고용변동 신고
- 신규 E-9 고용허가 신청
- 후보자 온보딩
- 행정사 전달 패키지 생성
```

이때 RAG는 “어떤 서류가 필요한지”와 “그 서류가 왜 필요한지”의 근거를 찾아준다.  
하지만 실제 누락 여부는 RAG가 아니라 룰베이스가 판단한다.

```text
룰:
필요 서류 - 보유 서류 = 누락 서류

RAG:
왜 이 서류가 필요한지 공식 문서 기반으로 설명
```

### 1.3 업무 메시지 생성 근거 제공

다국어 컨택 에이전트는 단순 번역기가 아니다.

나쁜 방식:

```text
한국어 문장을 베트남어로 번역해줌
```

좋은 방식:

```text
서류 요청 목적
+ 제출 기한
+ 개인정보 사용 목적
+ 담당자 확인 포인트
+ 승인 후 발송 조건
까지 포함한 업무용 메시지를 생성
```

따라서 RAG에는 메시지 템플릿, 안전교육 안내 자료, 개인정보 사용 목적 문구, 기숙사·근무조건 안내 템플릿 등이 들어가야 한다.

### 1.4 행정사 전달 패키지의 근거 붙이기

행정사에게 넘길 때는 “AI가 이렇게 말했습니다”가 아니라, 아래처럼 상태와 근거를 묶어야 한다.

```text
직원: Nguyen Van A
비자: E-9
체류만료: D-76
누락 서류:
- 표준근로계약서
- 고용허가서 사본

관련 쟁점:
- 체류기간 연장 검토 필요
- 계약종료일과 체류만료일 확인 필요

주의:
AI는 최종 판단하지 않음.
행정사 검토 필요.
```

RAG는 이 패키지에 필요한 공식 절차·서식·관련 문서 근거를 붙인다.

## 2. 전체 구조

4–5주 MVP에서는 처음부터 복잡한 6개 인덱스를 모두 만들 필요는 없다.  
대신 **3개 RAG 인덱스 + 구조화 DB + 룰베이스**로 시작하는 것이 현실적이다.

```text
[User Goal]
“베트남 E-9 근로자 3명 추가 채용 준비해줘.
Nguyen 체류만료도 확인해줘.”

↓ Intent Router

[1] 인력 확보 Agent
→ RAG: E-9 고용절차, 허용업종, 신청 흐름
→ SQL/Mock: 회사 정보, 후보자 정보

[2] 다국어 컨택 Agent
→ RAG: 메시지 템플릿, 개인정보 안내 문구, 안전/생활 안내 자료
→ LLM: 한국어 원문 + 베트남어 번역
→ Rule: 승인 전 발송 금지

[3] 비자·서류 운영 Agent
→ SQL: 직원 상태, 체류만료일, 계약종료일, 보유 서류
→ Rule: D-day 계산, 누락 서류 비교
→ RAG: 관련 법령, 서식, 절차, 제출 서류
→ Output: 행정사 패키지

↓ Human Approval

↓ Evidence Log
```

## Index 2. Document & Case Template RAG

### 목적

서류 체크리스트·패키징 에이전트를 살리는 인덱스다.

법령 조문만 넣으면 실제 데모에서 약하다.  
사용자는 결국 아래 질문을 한다.

```text
그래서 지금 뭐가 빠졌는데?
행정사에게 뭘 넘겨야 하는데?
```

이 인덱스는 그 질문에 답하는 데 쓰인다.

### 사용하는 에이전트

```text
- 제출 서류 간소화 에이전트
- 비자·체류 관리 에이전트
- 사람 승인·전문가 전달 에이전트
- 감사 로그·근거 저장 에이전트
```

### 들어갈 데이터

```text
- 고용변동 등 신고서 PDF
- 고용사업장 정보변동 신고서
- 체류기간 연장 관련 제출 서류
- 고용허가 신청 관련 준비 서류
- 표준근로계약서 관련 안내
- 행정사 전달 패키지 템플릿
- synthetic 케이스 20~50개
```

### 합성 케이스 예시

```text
합성 케이스:
E-9 베트남 근로자 Nguyen Van A
체류만료 D-76
여권 있음 / 외국인등록증 있음 / 표준근로계약서 없음 / 고용허가서 사본 없음

정답:
- 누락 서류: 표준근로계약서, 고용허가서 사본
- 다음 행동: 베트남어 서류 요청 메시지 생성
- high-risk 여부: 행정사 검토 필요
```

### 주의

합성 데이터는 **근거가 아니라 데모/평가용**이다.

```text
근거로 사용 가능:
- 공식 문서
- 내부 승인 체크리스트
- 행정사 검수 템플릿

근거로 사용 금지:
- 합성 케이스
- 검증되지 않은 인터넷 글
- 개인 블로그
```

## 4. RAG에 넣으면 안 되는 데이터

아래 데이터는 RAG에 넣지 말고 SQL/DB로 관리해야 한다.

```text
- 직원명
- 외국인등록번호
- 여권번호
- 체류만료일
- 계약종료일
- 입국일
- 보유 서류 true/false
- 후보자 개인정보
- 담당자 연락처
- 발송 기록
- 승인 기록
```

이유는 간단하다.

RAG는 의미 검색에 강하지만, 정확한 현재 상태 관리에는 약하다.  
예를 들어 Nguyen의 체류만료일이 `2026-08-15`인지, 계약종료일이 `2026-07-31`인지, 표준근로계약서가 있는지 없는지는 검색으로 찾는 것이 아니라 DB에서 정확히 읽어야 한다.

따라서 구조는 이렇게 간다.

```text
Postgres / SQLite
- companies
- employees
- candidates
- documents
- cases
- approvals
- audit_logs

Vector DB / RAG
- laws
- procedures
- forms
- templates
- safety_guides
- approved_case_examples
```

4–5주 MVP에서는 아래 조합으로 충분하다.

```text
SQLite + CSV + Chroma/Qdrant
```

## 5. 데이터 소스 수집 계획

## 5.1 공식 법령/절차 데이터

### 1순위: 국가법령정보센터

#### 구할 것

```text
- 출입국관리법
- 출입국관리법 시행령
- 출입국관리법 시행규칙
- 외국인근로자의 고용 등에 관한 법률
- 외국인근로자의 고용 등에 관한 법률 시행령
- 외국인근로자의 고용 등에 관한 법률 시행규칙
- 관련 별지 서식
```

#### 사용처

```text
- 비자·체류 관리
- 고용변동 신고
- high-risk 판단
- 행정사 전달 패키지 근거
```

#### 주의

법령은 **조문 단위로 chunking** 해야 한다.

나쁜 방식:

```text
문서 전체를 1,000자 단위로 자르기
```

좋은 방식:

```text
출입국관리법 제18조
외국인근로자고용법 제17조
시행규칙 별지 제12호서식
```

문서 전체를 기계적으로 자르면 조문 citation이 흐려진다.

### 2순위: EPS / 고용24 / 한국산업인력공단

#### 구할 것

```text
- 고용허가제 소개
```

#### 사용처

```text
- 인재 요건·매칭 에이전트
- 후보 적합성 검토 에이전트
- 신규 인력 요청서 생성
- 송출회사/행정사 전달 문서 생성
```

#### 핵심 해석

EPS 구조에서는 AI가 “인재 추천”을 직접 하면 위험하다.  
따라서 이 영역은 아래처럼 제한해야 한다.

```text
나쁜 방향:
AI가 후보자를 추천함

좋은 방향:
AI가 후보군 확인 요청서와 요건 정리 문서를 만듦
```

```text
- 체류자격별 안내 매뉴얼
- 민원서식
- 방문예약/체류 관련 안내
- 출입국 관련 법령지침정보
```

#### 사용처

```text
- 체류만료 D-day 이후 준비 서류 안내
- 행정사 전달 패키지
- 외국인 근로자용 안내 메시지
```

---

## 5.2 시장/통계 데이터

이 데이터는 RAG 답변의 법적 근거가 아니다.  
**타깃팅과 수요 설명**에 쓴다.

### 구할 것

```text
- 지역별 E-9 근로자 수
- 국적별 E-9 근로자 수
- 업종별 E-9 근로자 수
- 외국인 고용 사업장 현황
```

### 소스

```text
- 공공데이터포털
- KOSIS
- 한국고용정보원 고용행정통계
```

### 사용처

```text
- 어느 지역을 먼저 공략할지
- 어떤 업종을 먼저 타깃할지
- 발표에서 시장 수요를 설명할 때
- 파일럿 사업장 후보를 좁힐 때
```

### 주의

```text
통계 데이터는 “시장 분석”용이다.
개별 직원의 법적 판단 근거로 쓰면 안 된다.
```

## 5.3 다국어/안전/생활 안내 데이터

### 구할 것

```text
- 다국어 안전표지
- 외국인 기초안전보건교육 자료
- 외국인 근로자 생활 안내
- 상담센터 안내
- 의료/은행/통신/교통 안내
```

### 소스

```text
- 산업안전보건공단 KOSHA
- 고용노동부 보도자료
- 중소기업중앙회 외국인력지원실
- 외국인력상담센터
- 지자체 외국인근로자지원센터
```

### 사용처

```text
- 다국어 컨택 화면
- 기숙사/근무조건 안내
- 안전교육 안내
- 문제 발생 시 공식 상담 채널 안내
```

## 5.4 케이스 데이터

여기가 가장 어렵다.

공개 데이터만으로는 실제 행정사 보완 요청 패턴이나 사업장별 누락 사례를 충분히 얻기 어렵다.  
그래서 MVP에서는 3단계로 간다.

### Phase 1: 합성 케이스

직접 만든다.

#### 예시

```text
케이스 1:
E-9 베트남 근로자 체류만료 D-76
누락: 표준근로계약서, 고용허가서 사본

케이스 2:
신규 베트남 근로자 3명 채용 요청
누락: 내국인 구인노력 확인, 기숙사 정보, 안전교육 자료

케이스 3:
외국인 근로자 답변
“여권은 있는데 사진은 내일 보낼 수 있어요.”
상태 업데이트: 여권 확인, 사진 제출 예정
```

#### 용도

```text
- 데모용
- 평가용
- 흐름 검증용
```

#### 주의

```text
합성 케이스는 법적 근거가 아니다.
공식 답변 근거로 사용하면 안 된다.
```

---

### Phase 2: 공개 상담 사례

수집 후보:

```text
- 외국인노동자지원센터
- 노동권익센터
- 지자체 상담사례집
- 외국인근로자지원센터 공개 자료
```

용도:

```text
- 케이스 유형 분류
- 예상 쟁점 파악
- 메시지 톤 참고
- 상담 흐름 참고
```

주의:

```text
개인정보·맥락이 섞일 수 있으므로 그대로 RAG 답변 근거로 쓰지 않는다.
```

### Phase 3: 행정사/노무사 인터뷰 기반 케이스

이게 진짜 자산이다.

#### 인터뷰 질문

```text
1. E-9 사업장에서 제일 자주 빠지는 서류는 뭔가요?

4. 행정사에게 넘길 때 어떤 정보가 있으면 일이 빨라지나요?
5. 반대로 어떤 정보가 없으면 다시 물어보게 되나요?
```

## 7. 메타데이터 설계

RAG 품질은 메타데이터가 거의 반이다.  
문서 chunk마다 최소 아래 정보를 붙인다.

```json
{
  "source_id": "eps_employer_process_001",
  "title": "사업주를 위한 고용허가제 안내 - 외국인근로자 고용절차",
  "publisher": "한국산업인력공단 EPS",
  "source_type": "official_procedure",
  "url": "https://eps.hrdkorea.or.kr/...",
  "retrieved_at": "2026-04-29",
  "effective_date": null,
  "doc_type": "procedure",
  "mission_agent": ["workforce_agent", "visa_document_agent"],
  "visa_type": ["E-9"],
  "country": ["ALL"],
  "industry": ["manufacturing", "construction", "agriculture", "fishery"],
  "risk_level": "medium",
  "evidence_grade": "official"
}
```

### 7.1 문서 유형

```text
official_law
official_procedure
official_form
official_statistics
safety_guide
message_template
synthetic_case
internal_checklist
```

### 7.2 Evidence Grade

```text
A: 법령/정부 공식 문서
B: 공공기관/공식 절차 안내
C: 공공데이터/통계
D: 센터 상담사례/참고자료
E: 내부 템플릿
F: 합성 데이터
```

답변 근거로 사용할 수 있는 등급:

```text
A: 가능
B: 가능
E: 가능, 단 내부 승인 템플릿일 때
```

제한적 사용:

```text
C: 시장/통계 설명용
D: 참고자료
F: 데모/평가용
```

## 8. Chunking 전략

일반적인 RAG처럼 500~1,000자씩 자르면 안 된다.  
이 도메인은 **조항 / 서식 / 절차 단계** 단위가 중요하다.

### 8.1 법령

조문 단위 chunking.

예시:

```text
출입국관리법 제18조
외국인근로자고용법 제17조
시행규칙 별지 제12호서식
```

### 8.2 절차 안내

단계 단위 chunking.

예시:

```text
1. 내국인 구인노력
2. 고용허가서 발급
3. 근로계약 체결
4. 사증발급인정서 신청
5. 입국 및 취업교육
```

### 8.3 서식

필드 단위 chunking.

예시:

```text
- 사업장 정보
- 외국인 인적사항
- 신고 사유
- 유의사항
- 제출기한
```

### 8.4 메시지 템플릿

목적 단위 chunking.

예시:

```text
- 여권 사본 요청
- 외국인등록증 요청
- 증명사진 요청
- 기숙사 안내
- 안전교육 안내
- 급여명세서 설명
```

---

## 9. 검색 전략

4–5주 MVP에서는 과도하게 복잡하게 가지 않는다.

### 9.1 1차 검색

```text
BM25 + Dense Hybrid
```

이유:

```text
- 법령/서식은 정확한 키워드가 중요하다.
- “고용변동 신고서”, “체류기간 연장”, “E-9”, “표준근로계약서” 같은 단어는 BM25가 잘 잡는다.
- 사용자가 “직원 그만두면 뭐 신고해?”처럼 자연어로 물으면 dense embedding이 필요하다.
```

### 9.2 2차 필터

예시:

```text
visa_type = E-9
doc_type = official_law / official_form / official_procedure
mission_agent = visa_document_agent
country = VN or ALL
```

### 9.3 3차 Rerank

MVP에서는 reranker가 없어도 된다.  
시간이 있으면 multilingual reranker를 붙인다.

### 9.4 답변 생성 조건

공식 근거가 없으면:

```text
공식 근거를 찾지 못했습니다. 행정사 검토가 필요합니다.
```

근거가 있으면:

```text
- 요약
- 필요한 행동
- 근거 문서
- 사람 승인 필요 여부
```

## 12. document_requirements.csv

이 파일은 RAG보다 중요할 수 있다.

RAG가 직접 “이 서류가 필요하다”고 판단하게 하지 말고,  
`document_requirements.csv`에서 케이스별 필수 서류를 구조화해둔다.

그 다음 RAG는 해당 서류가 왜 필요한지 근거를 설명한다.

### 12.1 예시

```csv
case_type,visa_type,required_doc,required,source_id,notes
stay_extension,E-9,passport,true,gov24_stay_extension,여권
stay_extension,E-9,arc,true,gov24_stay_extension,외국인등록증
```

## 16. 최종 판단

RAG를 “똑똑한 검색”으로만 보면 약하다.

이 프로젝트에서 RAG의 역할은 아래와 같다.

> **AI가 위험한 판단을 하지 않도록, 모든 제안 뒤에 공식 근거와 사람 승인 지점을 붙여주는 안전장치.**

따라서 데이터 소스도 아래처럼 나눠야 한다.

```text
공식 근거:
법령, EPS, 정부24, HiKorea, 고용노동부, KOSHA

업무 템플릿:
서류 요청, 온보딩, 안전교육, 행정사 패키지

상태 데이터:
직원 CSV, 후보자 CSV, 보유 서류, D-day

평가 데이터:
합성 케이스, 누락 서류 테스트, 메시지 테스트
```

최종적으로 이렇게 이해하면 된다.

> **RAG는 답을 만드는 엔진이 아니라, 에이전트가 만든 계획과 메시지에 근거를 붙이는 장치다.**  
> **실제 판단은 룰과 사람 승인으로 막고, 실제 상태는 DB가 관리한다.**
