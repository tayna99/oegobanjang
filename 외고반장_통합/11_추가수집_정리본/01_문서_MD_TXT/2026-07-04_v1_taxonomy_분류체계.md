# taxonomy.md — 분류 체계 통합 색인 (노션 원본)

> 출처: Notion "5-7월 프로젝트 > Decision Log > taxonomy.md"
> 버전 태그: v1 (2026-07-04 수집) — 기존 폴더에 이 수준의 통합 분류체계 문서 없음. 코드/스키마/DB 전반의 enum·분류값을 한 곳에 모은 레퍼런스 색인이라 원문 그대로 보존.
> 식품 관련 내용 없음.

이 문서는 WorkBridge/oegobanjang 프로젝트 전체에 흩어진 분류값을 한 곳에 모은 택소노미 색인이다. 기준 소스는 루트 지침, `docs/`, `missions/active/`, `backend/`, `data-pipeline/seed/`, `evals/datasets/`, `frontend/`이다. 실제 계약의 최종 권위는 각 mission 문서, schema 문서, 코드 타입 정의, DB migration에 있다. 새 분류값을 추가하면 관련 계약 문서와 이 파일을 함께 갱신한다.

## 1. Runtime Intent Taxonomy (런타임 의도 분류)

| Intent | 의미 | 현재 처리 기준 |
| --- | --- | --- |
| `HIRING` (신규 채용) | 신규 채용, 사업장 상태 확인, 후보자 intake, 계약 준비 초안 | `workforce_agent` / hiring flow |
| `VISA_CHECK` (비자/체류 점검) | 체류자격, 만료일, 갱신 리스크 점검 | `visa_document_agent` 또는 문서/비자 flow |
| `DOCUMENT_CHECK` (서류 점검) | 필요 서류, 누락 서류, 제출 상태 점검 | `visa_document_agent` |
| `CONTACT` (소통/연락) | 근로자/담당자/전문가 메시지 초안 또는 전달 요청 | 초안 가능, 발송은 승인 필요 |
| `BRIEFING` (상태 브리핑) | 현재 케이스 상태 요약, 담당자 브리핑 | final response / briefing placeholder |
| `UNSUPPORTED_VALUE_JUDGMENT` | 성실도·이탈가능성·국적선호·감시 등 가치판단 요청 | blocked |
| `UNSUPPORTED_LEGAL_JUDGMENT` | 비자 가능 확정, 법률/노무 자문 요청 | blocked |
| `UNSUPPORTED_AUTO_SUBMISSION` | 정부 포털 제출, 무승인 대외 발송 요청 | blocked |

복합 intent는 허용된다. (예: `HIRING` + `VISA_CHECK`)

## 2. Case Type Taxonomy (케이스 유형 분류)

| Case Type | 상태 | 의미/출처 |
| --- | --- | --- |
| `new_hiring` | current runtime | 신규 고용 흐름 (Mission 001, hiring agent 주력) |
| `workplace_change_intake` | current runtime | 사업장 변경 접수, 현재 최소 지원 |
| `stay_extension` | seed/DB | 체류 연장/갱신 서류 요구 케이스 |
| `employment_change` | seed/DB | 고용 변경/사업장 변경 관련 서류 요구 케이스 |

| `same_worker_rehire` | legacy/synthetic | 동일 근로자 재고용 관련 과거 synthetic 분류 |

정규화 권장: runtime canonical case type은 `new_hiring`, `workplace_change_intake`로 두고, seed/DB의 `stay_extension`, `employment_change`는 별도 도메인 케이스로 분리.

## 3. Workflow State Taxonomy (워크플로 상태 분류)

`draft`(초안) → `site_check`(사업장 확인) / `candidate_intake`(후보자 접수) / `site_check_and_intake` → `contract_prep`(계약 준비) → `risk_review`(리스크 검토) / `blocked`(차단) → `human_approved`(승인 완료) → `handoff_package`(전달 패키지). Seed/schema fixture에는 `approval_pending`, `completed`도 존재.

주요 전이:
- `new_hiring`: draft → site_check → candidate_intake → contract_prep → human_approved
- `workplace_change_intake`: draft → site_check_and_intake → contract_prep → human_approved
- 공통 리스크: any_state → risk_review / 공통 차단: any_state → blocked
- handoff: human_approved → handoff_package

## 4. Agent And Tool Taxonomy (에이전트와 도구 분류)

### Agents
- `workforce_agent`: 신규 고용, 사업장 확인, 후보자 intake, 계약 준비 초안
- `visa_document_agent`: 체류/비자/서류 누락 점검
- `multilingual_contact_agent`: 다국어 메시지 초안, 안전 안내, 응답 요약
- `communication_agent`: Mission 001/eval에서 쓰이는 contact placeholder 명칭
- `hiring_agent`: runtime 구현 파일 기준의 workforce agent 본체

정규화 이슈: RAG seed는 `multilingual_contact_agent`, Mission 001/eval/planner는 `communication_agent`를 씀 → 하나로 정리 필요.

### Tool Grades
`SAFE_READ`(읽기 전용) / `SAFE_CALCULATE`(결정론 계산) / `SAFE_DRAFT`(초안 생성) / `APPROVAL_REQUIRED`(승인 후 실행) / `FORBIDDEN`(구현/실행 금지)

### Tool Response Status
`SUCCESS` / `FAILED` / `NEEDS_APPROVAL` / `FORBIDDEN`

### Safe Read Tools
`get_company_profile`, `get_worker_profile`, `get_candidate_profile`, `get_document_status`, `search_policy_documents`, `search_message_templates`, `search_safety_guides`

### Safe Calculate Tools
`calculate_visa_d_day`, `calculate_contract_gap`, `calculate_missing_documents`, `calculate_candidate_readiness`, `classify_case_type`, `quota_tool`

### Safe Draft Tools
`generate_hiring_request_draft`, `generate_multilingual_message_draft`, `generate_expert_handoff_package_draft`, `generate_safety_notice_draft`

### Approval Required Tools
`send_worker_message`, `send_manager_notification`, `send_expert_package`, `update_case_status_completed`, `export_handoff_package`

### Forbidden Tools / Behaviors
`submit_government_portal`, `confirm_visa_eligibility`, `provide_legal_advice`, `predict_worker_absconding`, `monitor_worker_sns`, `score_worker_reliability`, `recommend_worker_by_nationality`

## 8. Risk And Readiness Taxonomy

### Risk Level
RAG metadata·seed chunks: `low/medium/high` (소문자) / DB·Evidence schema: `LOW/MEDIUM/HIGH` (대문자) — API/RAG는 소문자 권장, DB enum은 migration 기준.

### Evidence Grade

| Grade | 의미 | 답변 근거 사용 |
| --- | --- | --- |
| A | 공식 법령/정부 문서 | 가능 |
| B | 공식 절차 안내 | 가능 |
| C | 공공데이터/통계 | 시장/상황 분석용 |
| D | 상담사례/참고자료 | 참고용 |
| E | 내부 템플릿 | 승인 전 초안/메시지 템플릿용 |
| F | 합성/평가 데이터 | 공식 근거로 사용 금지 |

Mission 002 기준으로 A/B/E만 answer evidence로 사용 가능, F는 demo/eval 전용.

### Quota / Readiness Status
`needs_more_information` / `needs_human_review` / `ready_for_review`

### Runtime Risk Flags
`headcount_not_numeric`, `quota_insufficient`, `workplace_change_history_unverified`, `unsupported_case_type`, `missing_company_id`, `missing_headcount`, `missing_evidence_source`

## 15. Naming Conflicts And Normalization Notes

| 영역 | 현재 혼재 | 권장 정리 |
| --- | --- | --- |
| case type | runtime `workplace_change_intake`, seed/DB `employment_change`/`stay_extension` | runtime case와 document requirement case 분리 |
| risk level | RAG 소문자, DB/Evidence 대문자 | API/RAG는 소문자, DB enum은 migration 기준 |
| contact agent | `multilingual_contact_agent` vs `communication_agent` | 하나를 canonical로, alias map 유지 |
| approval flag | `approval_required`/`requires_human`/`approval.required` | 레이어별 필드 책임 문서화 |
| evidence naming | `event_type` vs `action_type` | event_type=canonical event, action_type=실행 동작 |
| legacy phase | `phase` | current_state/mission acceptance 기준 전환 |
| chunk metadata | `chunk_type` vs `doc_type` | active RAG contract는 doc_type, 필요시 chunk_type 재도입 |
| downloaded format | raw source `html`/`pdf` | source manifest의 실제 format 그대로 기록 |
