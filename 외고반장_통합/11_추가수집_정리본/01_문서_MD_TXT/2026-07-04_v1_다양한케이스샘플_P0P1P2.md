# 다양한 케이스 샘플 (P0/P1/P2) — 노션 원본 (Decision Log)

> 출처: Notion "5-7월 프로젝트 > Decision Log > 다양한 케이스 샘플"
> 버전 태그: v1 (2026-07-04 수집) — 기존 폴더에 없는 케이스 라이브러리 정리본 (9개 케이스 분류 P0/P1/P2).
> 식품 관련 내용 없음.

## 1. Daily Risk Briefing Case
사용자 요청 또는 매일 실행으로 생성되는 케이스. 예시: "이번 달 외국인 직원 중 급한 케이스만 정리해줘."

결과: HIGH/MEDIUM/LOW/CRITICAL 위험 목록, 체류만료 D-day, 계약종료 D-day, 누락서류, 고용변동 신고 기한, 추천 NextAction, 승인 대기 상태, EvidenceEvent 생성

→ 외고반장의 **대표 MVP 케이스**

## 2. Visa Expiry Case
기존 직원 체류만료 확인 케이스. 예시: "Nguyen 체류만료일 확인하고 필요한 조치 알려줘."

결과: 체류만료 D-day, expired 여부, days_overdue, risk_level, 필요한 서류 목록, 누락서류, citation_ids, next_actions, approval_required

→ P0 중에서도 제일 핵심. 외고반장이 "운영 리스크 관리"라는 걸 직관적으로 보여줌.

## 3. Contract-Visa Conflict Case
계약 종료일과 체류만료일 충돌 탐지. 예시: "이번 달 계약 끝나는 외국인 직원 중 체류만료랑 겹치는 사람 있어?"

결과: contract_end_date, visa_expiry_date, 두 날짜 간 차이, 충돌 여부, expert_review_required, handoff draft 필요 여부

단순 D-day 계산보다 한 단계 더 "운영 판단"에 가까움. 다만 확정 판단은 하지 말고 "검토 필요"로 둬야 함.

## 4. Document Gap Case
서류 누락 일괄 점검. 예시: "외국인 직원들 갱신 서류 빠진 거 한번 점검해줘."

결과: 직원별 누락서류, document_status(missing/submitted/verified/expired), 우선순위, 요청 메시지 초안, EvidenceEvent

→ P0. 서류 누락은 실무 담당자가 제일 많이 놓치는 반복 업무라 MVP 경쟁력이 있음.

## 5. Multilingual Contact Draft Action
"케이스"라기보다 **액션**에 가까움. 예시: "Tran에게 베트남어로 여권 사본이랑 표준근로계약서 요청 메시지 만들어줘."

결과: 베트남어 메시지 초안, 한국어 번역, 쉬운 표현 검증, 개인정보 사용 목적 문구, 발송 전 승인 상태, 실제 발송 없음

"다국어 메시지 생성 케이스"라고 하기보다, 기존 `Document Gap Case`나 `Visa Expiry Case`에서 파생되는 `NextAction`으로 보는 게 더 좋음:
```
Document Gap Case → NextAction: request_document → Multilingual Contact Draft 생성 → Approval pending
```

## 6. Handoff Draft Case
행정사 전달 패키지 생성. 예시: "Nguyen 갱신 건 행정사에게 전달할 패키지 만들어줘."

결과: worker summary, risk summary, 만료일, 계약 종료일, 누락서류, citation list, recommended questions, approval status, handoff preview

이름 주의: "행정사 전달 패키지"라고 하면 실제 전송까지 하는 것처럼 보일 수 있음. MVP에서는 **Handoff Draft** 또는 **Handoff Preview**가 더 안전.
> 추천 표현: 행정사에게 실제 전달하지 않고, 담당자가 검토할 수 있는 전달 초안 패키지를 생성한다.

## 7. Recruitment Readiness / Quota Review Case
신규 고용 준비상태 점검 케이스. 예시: "베트남 E-9 근로자 3명 추가 채용 준비해줘." (표현 주의 필요)

위험한 결과 표현(금지): 후보 추천, 채용 가능 여부 확정, 적합도 점수
안전한 결과 표현(권장): 쿼터 검토 필요 여부, 현재 외국인 근로자 수, quota_limit 대비 여유, 필요한 절차 체크리스트, 준비해야 할 서류 목록, 다국어 안내문 초안, 전문가 검토 필요 flag

## 8. Candidate Readiness Check Case
후보자 서류 준비상태 점검 케이스. 예시: "등록된 후보자들의 E-9 신청 준비 서류 상태를 점검해줘."

결과: 후보자별 제출 서류 상태, 누락서류, 확인 필요 항목, 체류자격/절차 검토 필요 flag, 전문가 확인 필요 여부

## 9. Evidence / Audit Review Case
감사 로그/근거 재현 케이스. 예시: "Nguyen 케이스에서 AI가 왜 high risk라고 했는지 근거 보여줘."

결과: trace_id, EvidenceEvent 목록, 사용한 rule, 사용한 DB snapshot, citation_ids, RAG retrieved chunk, risk_flagged 이유, approval history, PII masking 상태

업무 케이스라기보다 **조회/감사 케이스**로 분리하는 게 좋음.

## 최종 케이스 구조

```
P0 Core Cases
1. Daily Risk Briefing Case
2. Visa Expiry Case
3. Contract-Visa Conflict Case
4. Document Gap Case
5. Handoff Draft Case
6. Evidence / Audit Review Case

P1 Action / Support Cases
7. Multilingual Contact Draft Action
8. Recruitment Readiness / Quota Review Case

P2 or Guarded Case
9. Candidate Document Readiness Case
```

케이스와 액션 구분 필요:
```
Visa Expiry Case → NextAction: request_document → NextAction: create_handoff → NextAction: ask_admin_review
```

최종 구조화:
```
Case: visa_expiry / contract_expiry / contract_visa_conflict / missing_document / reporting_deadline / quota_review / audit_review / candidate_readiness
NextAction: request_document / create_handoff / ask_admin_review / request_revision / mark_resolved
```

## 사용자 요청 → 케이스 매핑 요약

1. 신규 외국인 근로자 채용 준비 → 신규 고용 준비상태 점검 / 쿼터 검토 케이스
2. 기존 직원 체류만료 확인 → 체류만료 리스크 케이스
3. 계약 종료일과 체류만료일 충돌 탐지 → 계약-체류 충돌 리스크 케이스
4. 서류 누락 일괄 점검 → 서류 누락 리스크 케이스
5. 다국어 서류 요청 메시지 생성 → 서류 요청 메시지 초안 액션
6. 행정사 전달 패키지 생성 → 행정사 검토용 Handoff Draft 케이스
7. 위험 케이스 우선순위 브리핑 → Daily Risk Briefing 케이스
8. 신규 후보자 요건 매칭 → 후보자 서류 준비상태 점검 케이스 (단, 매칭 점수/추천은 빼기)
9. 감사 로그/근거 재현 케이스 → Evidence/Audit Review 케이스
