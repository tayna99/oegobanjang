## 1. 프로젝트 개요 — 우리가 만들려는 것

이번 프로젝트의 중심 문제의식은 "외국인 고용"이라는 도메인이 갖는 복합성이다. 제조업 중소기업(특히 E-9 등 외국인 근로자를 30명 이상 고용하는 사업장)은 체류/고용변동/서류/다국어 소통/산업안전이 얽힌 일을 매일 운영해야 한다. 그런데 정부 시스템(EPS/하이코리아/정부24)은 '제출/조회' 기능을 제공하긴 하지만, 실제 현장의 운영 문제(기한을 놓치지 않기, 누락 서류를 미리 잡아내기, 다국어로 직원에게 정확히 안내하기, 행정사에게 바로 전달 가능한 패키지를 준비하기)는 충분히 해결하지 못한다.

이 프로젝트는 그 운영 문제를 "버튼 클릭형 SaaS"가 아니라, 사용자가 목적을 말하면 AI가 경로를 짜는 Agentic OS 형태로 풀어내려는 시도다. 핵심은 자동 제출이 아니라 자동 준비다.

### 1.1 프로젝트 정의(태나 보고서 기반): "외국인 고용 관리 AI 반장 서비스(Agentic OS)"

외고반장은 외국인 고용 업무에서 발생하는 채용, 쿼터 확인, 다국어 컨택, 비자 만료 관리, 서류 누락 확인, 행정사 전달 패키징 과정을 하나의 워크플로우로 통합하는 AI 기반 업무 운영 시스템이다.

- 발표 장표: [발표 자료 보러가기](https://drive.google.com/file/d/1ZR26LPQ45DSwjiPt4uPL8V_0BKuG0InR/view?usp=sharing)

### 1.2 핵심 평가 지표("운영 OS"로서 성공 조건)

- 정책 변경을 감지하고 체크리스트를 업데이트
- 행정사 반려/보완 요청 패턴을 학습해서 다음 케이스의 누락을 줄임
- 다국어 메시지 반려율과 재질문율을 낮춤
- 케이스 패키지 생성 시간을 줄이고, 전문가가 다시 물어보는 횟수를 줄임
- 고객사별 체류/서류 리스크를 매일 아침 자동 브리핑

구체 지표: 체류만료 D-day 알림 정확도 / 담당자 승인 전 자동 발송 0건 / 개인정보 마스킹 성공률 / 다국어 메시지 반려율 / 케이스 패키지 생성 시간 / 고객사별 월간 위험 케이스 감소 / 행정사 검토 준비시간 단축

## 2. 초기 킥오프와 리서치 — 무엇을 확인했고, 어떤 방향을 잡았나 (4/22~4/24)

### 2.0 문제를 어떻게 재정의했나: '파편화된 행정 리스크'

외국인 고용 업무는 단순 인사 관리가 아니라, 비자 제도/고용 쿼터/체류기간/계약 상태/필수 서류/외국어 소통/행정사 협업이 얽힌 복합 업무다. 기존 방식에서는 담당자가 법령 사이트, 엑셀, 메신저, 이메일, 행정사 연락망을 오가며 확인해야 해서 업무가 파편화되고 실수 가능성이 높다.

반복적으로 발생하는 리스크:
- 비자 규정과 고용 쿼터가 복잡하고 수시로 변동됨
- 언어 장벽으로 서류 취합이 지연됨
- 만료일/누락 서류를 놓치면 불법 고용/체류 문제/행정 리스크로 이어짐
- 행정사 전달 자료가 정리되지 않아 검토 시간이 길어짐
- AI가 주관적 판단/법률자문처럼 말하면 오히려 리스크가 커짐

초기 리서치 단계의 결론: "AI가 대신 판단"이 아니라 업무 경로를 정리하고 근거를 붙이고 사람이 승인할 수 있게 만드는 운영 시스템이 되어야 한다.

### 2.1 킥오프

프로젝트는 팀 킥오프 및 각자 리서치로 시작했다. 목적은 "우리가 해결하려는 문제를 정확히 정의하고, AI로 풀 수 있는 구조인지(검색→계산으로 전환 가능한지) 검증"하는 것이었다.

### 2.2 워크플로 정의/검증

단순히 기능 목록을 늘어놓기보다 어떤 요청이 들어오고 → 어떤 정보가 필요하고 → 어떤 단계를 거쳐 → 어떤 출력이 나가야 하는지를 워크플로 관점에서 정리하는 방향으로 전환했다.

## 3. 고도화 아이데이션과 프로젝트 선별 — "무엇에 집중할 것인가" 결정 (4/26~4/28)

### 3.0 "AI가 대신 판단"이 아니라 "사람이 승인할 수 있게 만드는 운영 시스템"

프로젝트의 기획 목표는 단순한 자동화가 아니라, 외국인 고용 업무의 복잡한 경로를 운영 체계로 바꾸는 것이다. 때문에 'AI가 최종 판단'하는 방향은 의도적으로 피하고, 아래를 목표로 둔다.

- 자연어 업무 요청을 구조화된 업무 태스크로 변환
- 채용/다국어 컨택/비자·서류 관리를 Agent로 분리
- RAG로 공식 근거 기반 안내·체크리스트 생성
- 판단 과정과 근거를 Evidence Log로 저장
- 승인 전에는 외부 발송·제출이 실행되지 않도록 HITL 설계
- 행정사/노무사가 검토하기 쉬운 패키지 생성

### 3.1 고도화 아이데이션: 옵션을 넓히되, 구현 가능한 구조로 수렴

4/26~4/27에는 고도화 아이디어를 확장하면서도, 구현 가능한 구조로 수렴시키려는 시도가 있었다. 이 단계의 특징은 "좋은 아이디어를 많이"보다, 나중에 구현 계약서(PRD)로 떨어지도록 데이터/메타데이터/룰 vs LLM 분리까지 같이 생각하기 시작했다는 점이다.

> 참고: 이 이데이션 단계에서 팀은 "외국인 고용" 외에 "식품 표시·광고 검증/입점 운영" 아이디어도 병행 검토했으나, 최종적으로 채택되지 않았다. 식품 관련 상세 내용은 본 정리본에서 의도적으로 제외했다.

### 3.2 프로젝트 선별 기준 확정: 멀티에이전트/데이터/차별화 기준으로 '외국인 고용' 선택

프로젝트 선택은 "멋있어 보이는 도메인"이 아니라, 멀티에이전트 설계와 데이터 접근성/전처리 가능성, 그리고 차별화 포인트가 실제로 구현 가능한지로 결정됐다. 핵심 결론은, 멀티에이전트에 더 적합하고(업무가 분해 가능), 공개 자료 기반 RAG가 가능하며(법령/가이드), 다국어/운영 자동화가 차별화가 될 수 있는 외국인 고용 운영으로 수렴하는 것이었다.

## 4. "외고반장"의 차별화와 역할 정의 — 정부/전문가와의 경계 설정 (5/1)

프로젝트가 외국인 고용으로 확정되면, 바로 다음 과제는 "정부 시스템/행정사와 무엇이 다르고, 어디까지를 제품이 책임질지"를 명확히 하는 것이다. 이 정의가 모호하면 안전/신뢰에서 바로 무너진다.

강하게 고정된 원칙:
- 정부 시스템이 제공하지 않는 '운영' 영역(알림, 일괄 모니터링, 누락서류 탐지, 다국어 메시지 생성, 응답 한국어 요약, 행정사 전달 패키지, 카톡/메신저 진입)이 외고반장의 영역
- 정부 포털 자동 제출은 하지 않음 (안전 원칙)
- 행정사는 "케이스가 도착하면 처리하는 전문가"이고, 외고반장은 "케이스를 미리 챙겨주는 운영 매니저"

즉, "전문가 대체"가 아니라 "전문가 앞단 운영 자동화"를 한다는 구조가 이 시점에 명문화됐다. (세부 비교표는 기존 `차별화포인트_비교.md` 참고 — 동일 내용)

## 5. 제품 구조의 중심 설정 — Agentic OS, 3 Mission Agent / 8 Sub-agent, 그리고 공통 엔진 (4/28~5/3)

### 5.0 사용자 워크플로(자연어 → 의도 분류 → 계획 → 에이전트 실행 → 승인 → 로그)

1. 사용자가 자연어로 업무를 요청한다. (예: "베트남 E-9 근로자 3명 추가 채용 준비해줘. Nguyen 체류만료도 확인해줘.")
2. Intent Router가 요청을 분류한다. (신규 채용 / 체류만료 확인 / 서류 누락 / 행정사 패키지 등)
3. Planner가 실행 계획을 세운다. (쿼터 확인, 근로자 상태 조회, 필수 서류 체크, 다국어 안내문 생성, 행정사 검토 자료 생성)
4. 각 Agent가 필요한 작업을 수행한다. (인력 확보 / 다국어 컨택 / 비자·서류 운영)
5. 시스템이 실행 결과를 관리자에게 보여준다.
6. 관리자가 최종 승인한다.
7. 승인 이력과 판단 근거가 Evidence Log에 저장된다.

### 5.1 Agentic OS 포지션 고정: "자동 준비"가 핵심

반복적으로 강조되는 결론은, 사용자의 자연어 목표를 다음과 같은 경로로 바꾸는 것이다.
- 신규 인력 요청서 생성
- 후보 요건 정리 및 제출 준비도 비교(사람 평가/추천 금지)
- 다국어 서류 요청 메시지 생성(승인 전 발송 금지)
- 기존 직원 체류만료 D-day 점검 + 누락서류 탐지
- 사람 승인 이후 행정사 검토용 패키지 생성 + 감사 로그 저장

#### 5.1-bis 고도화(5/8): "큰 Agentic OS" → "Daily Briefing MVP 운영 루프"로 수렴

고도화의 핵심은 기능을 무작정 늘리는 게 아니라, 매일 실제로 쓰고 검증할 수 있는 하나의 운영 루프(=Daily Briefing)로 MVP를 좁히고, 그 루프를 끝까지 닫기 위한 구성요소를 추가한 것이다.

고도화 이후 MVP 루프(요약):
```
위험 감지 (Rule)
→ 근거 연결 (Citation/RAG)
→ 승인 가능한 액션 생성 (NextAction)
→ Handoff Preview / Export
→ Evidence Log 저장
→ Scheduler + Admin + Metrics로 운영화
```

추가/강화된 요소:
- Risk Rule Engine: D-day/누락/기한을 LLM이 아니라 룰로 계산
- Case / NextAction / Approval: "발견"을 "업무 처리"로 전환하는 데이터 모델
- EvidenceEvent / Evidence Log: 왜 이렇게 판단했는지 재현 가능한 감사 이벤트(INSERT-only)
- Scheduler: 사용자가 묻기 전에 매일 브리핑을 자동 생성
- Admin + Metrics: 파일럿에서 승인률/근거부족/Export 등 운영 품질을 관리

### 5.2 3 Mission Agent + 8 Sub-agent 구조

외부 커뮤니케이션(사용자에게 보여주는 제품 단위)은 단순하게 3개로 제한한다.
- 인력 확보 에이전트
- 다국어 컨택 에이전트
- 비자·서류 운영 에이전트

내부 구현과 팀 분담은 8 Sub-agent로 더 세분화한다. 중요한 점은 "기능을 많이 갖춘다"가 아니라, 역할/책임을 분해해 각 에이전트가 어떤 입력을 받고 어떤 출력(계약)을 내야 하는지를 분명하게 하는 것이다.

### 5.3 세 에이전트 수렴 아키텍처: '만나는 지점'을 두 개로 고정

멀티에이전트는 쉽게 결합도가 폭발한다. 이를 막기 위해 '서로가 만나는 지점'을 고정했다. 핵심 설계는 아래 두 지점이다.
- `Aggregator` (각 에이전트 출력 합치기)
- `Human Approval` (외부 발송/전달 전 승인 게이트)

그 전 단계는 각 에이전트가 독립적으로 움직이고, 공통 schema/shared state로만 소통한다. "에이전트 간 직접 호출 금지" 같은 규칙은 실제 구현 안정성에 직결된다. (상세는 `2026-07-04_v1_세에이전트_수렴아키텍처.md` 참고)

## 6. PRD의 재정렬 — 구현 계약서로서의 PRD 1 (RAG 중심 재설계)

프로젝트가 단단해지는 핵심 장치가 PRD다. 이 프로젝트에서 PRD는 단순한 기획 요약이 아니라, RAG 구축 결정에 직접 영향을 주는 설계 사양(=구현 계약서)로 정의됐다.

### 6.1 PRD 1 (외국인 고용 운영 OS): "모든 어려운 문제를 검색으로 reframe"

핵심은, 어려운 문제를 판정/실행/예측/감시로 풀지 않고 "검색 문제"로 바꾸는 것이다.
- 판정 금지 → 후보 탐색
- 실행 금지 → 준비 상태 탐색
- 예측 금지 → 지원 필요 신호 탐색
- 감시 금지 → 자발적 상담 신호 탐색

이 재프레이밍이 있어야 안전 원칙(법률 판정 금지, 자동 제출 금지)을 지키면서도 실제 가치를 낼 수 있다. 또한 PRD 1은 시스템을 "6 인덱스 + 룰베이스 모듈"로 정리하면서, 어디는 vector가 맞고 어디는 SQL/lookup이 맞는지까지 명시한다. 특히 "Lookup Table은 vector가 아니라 SQL" 같은 결정은 구축 비용/정확도에 직접적인 영향을 준다.

## 7. 안전(SAFETY) 설계 — 데모를 프로덕션으로 옮기기 위한 '실패해도 안전하게 무너지기' (5/4)

### 7.0 Middleware Layer가 "보조 장치"가 아니라 "안전장치"인 이유

외고반장은 외국인등록번호/여권번호/연락처/체류 정보 등 민감한 데이터를 다루기 때문에, Agent의 답변 품질을 높이는 수준을 넘어 개인정보 보호·법적 리스크 제어·근거 기반 응답·사람 승인 흐름을 보장하는 미들웨어가 필요하다.

### 7.1 확장성/안정성 공통 원칙(재현성·감사 가능성을 엔지니어링 원칙으로 고정)

- Embedding Cache: chunk text hash 기준으로 embedding vector 캐싱 (비용/시간 절감 + 인덱싱 재현성)
- Idempotency: 동일 case_id 중복 실행에도 같은 결과가 나오도록 transition key 설계 (중복 요청/재시도/네트워크 오류 대응)
- Audit Log as Event Source: 단순 로그가 아니라 상태 재구성이 가능한 이벤트 소스로 저장 (사후 감사·사고 대응)
- Chunk Version Pinning: chunk_id에 hash 또는 ingest_at 포함 (과거 패키지 근거 재현)
- Retry / Backoff: 외부 API 실패 시 exponential backoff (운영 안정성)
- Schema Evolution: chunk_type/evidence_type 등 변경 시 마이그레이션 정책 (기존 케이스 손상 방지)
- Deterministic Seed: 평가 단계에서 temperature=0, seed 고정 (평가 재현성)
- Cost / Rate Guard: API 호출 횟수 제한 및 circuit breaker (무한루프/과금 폭주 방지)

대표 미들웨어 구성(설계 의도 중심):
- Pre-router: PII 마스킹(민감정보가 LLM/로그로 전파되는 것을 선제 차단)
- Pre-RAG: Query rewrite(행정 문서 검색 정확도 강화)
- Post-RAG: Evidence Grade Gate(근거 품질 낮으면 차단)
- Pre-state-machine: Schema validation(상태 전이/예외 케이스 방지)
- Pre-LLM: Prompt injection guard(제어권 탈취/무관 요청 차단)
- Post-LLM: Citation validator(허위 인용 방지)
- Pre-send: Human-in-the-loop(외부 발송/전달 전 승인 강제)
- Runtime: Summarization middleware(장기 케이스에서 컨텍스트 초과 방지)

핵심 5가지:
1) PII 마스킹: router/LLM 앞에서 마스킹하고 출력에서만 복원
2) Embedding cache: 중복 임베딩 비용과 불안정성 방지
3) Idempotency: 같은 요청이 두 번 와도 결과가 중복 생성되지 않도록 state machine에서 dedup
4) Audit log: UPDATE 없이 INSERT-only event source로 남겨 "왜 이렇게 했는지"를 재현 가능하게
5) Failure mode 명시: 빈 retrieval/메타 누락 등을 조용히 넘어가지 않고 `blocked/error`로 표면화

## 8. 케이스 설계와 기능 구현 관점 정리 — '케이스'와 '액션'의 분리 (5/6~5/8)

### 8.0 3개 핵심 Agent를 '업무 흐름' 기준으로 정리

- Intent Router: 자연어 업무 요청을 채용/비자/서류/컨택 등 업무 의도로 분류
- 인력 확보 Agent: E-9 추가 채용 요청을 분석하고 고용 쿼터/구인 노력 기간/신청 문서 초안 준비 (최종 가능 여부 단정 금지)
- 다국어 컨택 Agent: 서류 요청 메시지를 다국어로 생성하고, 외국어 응답을 요약해 업무 상태에 반영
- 비자·서류 운영 Agent: 체류 만료 D-day/계약 종료/누락 서류를 탐지하고 행정사 검토용 패키지 생성
- RAG 기반 정책 검색: 공식 정책/체류 안내/법정 서식 등 근거 문서를 검색해 체크리스트 생성
- Human Approval: 자동 발송/자동 제출 없이 승인 후 실행되도록 통제
- Evidence Log: 판단 근거/참조 문서/실행 계획/승인 이력 기록

### 8.1 P0 코어 케이스의 뼈대

- Daily Risk Briefing (오늘 급한 일/위험 케이스 브리핑)
- Visa Expiry (체류만료 D-day)
- Contract-Visa Conflict (계약종료와 체류만료 충돌 탐지)
- Document Gap (서류 누락 점검)
- Handoff Draft (행정사 검토용 패키지 초안)
- Evidence/Audit Review (근거/로그 재현)

중요한 설계는, "다국어 메시지 생성"을 단독 케이스로 두기보다 케이스에서 파생되는 NextAction으로 두는 방식이다. (상세 케이스 목록은 `2026-07-04_v1_다양한케이스샘플_P0P1P2.md` 참고)

### 8.2 안전한 표현 가이드

- "연장 가능합니다" 같은 단정은 금지
- "검토가 필요합니다 / 확인해야 합니다 / 다음 안전한 행동은 X입니다" 형태로 출력

## 9. 데모 보고서 — Nguyen 케이스로 End-to-End를 보여주는 이유 (5/11~)

### 9.0 데모 목표: '기능'이 아니라 '업무 거버넌스'가 보이게

- 자연어 요청 → 의도 분류 → 실행 계획 → 에이전트 실행 → 승인(HITL) → Evidence Log 저장
- 승인 전 자동 발송/제출이 '구조적으로 불가능'하다는 통제
- 근거/인용(어떤 문서로 판단했는지)이 남아 사후 검토가 가능하다는 신뢰

### 9.1 데모 고정값: Nguyen V. 체류기간 연장 서류 요청

- 체류만료 D-30 같은 "긴급성"이 직관적
- 누락 서류(표준근로계약서/여권 사본)라는 "반복 실무"가 들어감
- 과거 메시지 이력 확인 같은 "운영 맥락"이 들어감
- 다국어 메시지 초안 생성 + 사람 승인 게이트를 통해 "안전하게 자동화"가 드러남
- 최종적으로 "업무 기록(판단 기록)"이 남아 감사 가능성을 시연 가능

(전체 시연 흐름/JSON 스펙은 기존 `외고반장_데모2_시연흐름.md`에 동일 내용 있음)
