## 0. 한 줄 정의

**외고반장**은 외국인 고용 운영에서 발생하는 채용/서류/다국어/행정사 협업 업무를 "판정/자동제출"이 아니라 **검색·준비·승인·기록**으로 안전하게 굴리는 **Agentic OS**이다.

## 1. 배경과 문제

### 1.1 문제의식
외국인 고용 운영은 비자/체류/서류/다국어/행정사 협업이 얽힌 복합 업무이며, 정부 포털은 "제출/조회"는 제공하지만 운영(누락/기한/컨택/패키징)을 끝까지 닫아주지 못한다.

### 1.2 해결 방향(핵심 원칙)
- 자동 제출/법률 판단 금지
- 근거 기반(RAG) + 룰 기반(결정적 계산) + 사람 승인(HITL)
- 결과는 "답변"이 아니라 Next Action(버튼) + Evidence Log로 남아야 한다.

## 2. 목표 / 비목표

### 2.1 목표(제품)
- 사용자가 입력을 많이 하지 않아도 Daily/Risk Briefing으로 "오늘 해야 할 일"이 먼저 보인다.
- 케이스 단위로 누락/기한/리스크가 탐지되고, 다음 행동이 버튼(Next Action)으로 제안된다.
- 외부로 나가는 행동(메시지/전달)은 승인 전 차단된다.
- 모든 결정/생성/승인이 Evidence Log(감사 가능)로 저장된다.

### 2.2 비목표(명시)
- 정부 사이트 자동 제출/대리 신청
- 법률 자문/최종 가능 여부 단정
- 특정 개인의 평가/채용 추천(사람 평가)

## 3. 사용자 / 페르소나 / JTBD

### 3.1 페르소나
- 인력담당자(회사 내부 운영자): 채용/상태/병목/리드타임 관리
- 행정사(전문가): 제출요건/리스크/누락/패키지 검토
- 근로자(외국인 직원): 서류 준비/안내/질문 응답(쉬운 문장)

### 3.2 핵심 JTBD
- (운영자) "오늘 급한 케이스/해야 할 일을 한 화면에서 보고, 승인만 하면 다음 단계로 넘어가게 하고 싶다."
- (행정사) "검토에 필요한 자료가 누락 없이 정리된 패키지를 받고 싶다."
- (근로자) "무엇을 언제까지 어떻게 제출해야 하는지 모국어로 정확히 알고 싶다."

## 4. 핵심 사용자 시나리오(데모 기준)

### 4.1 대표 시나리오: Nguyen 체류기간 연장 서류 요청
1) 오늘 브리핑 화면에서 Nguyen 카드 노출 (D-30 / 누락 2건 / 승인 필요)
2) 케이스 상세 패널에서 왜 필요한지(D-day), 누락 서류, 이전 요청 이력, Next Action(승인 필요 배지 포함)
3) "초안 보기"에서 VN/KR 메시지 초안 확인
4) 승인하기 → 상태 변화 + 판단 기록(#4789) 생성/열람

## 5. 제품 범위(MVP v2)

### 5.1 MVP 루프(운영 루프)
Risk 감지(Rule) → 근거 연결(RAG/Citation) → Next Action 생성 → Approval → Evidence Log 저장

### 5.2 MVP에 반드시 포함
- Briefing 생성(버튼 또는 자동)
- 케이스 상세 패널
- Next Action 제안 + 승인 게이트
- Evidence Log(타임라인 형태라도)

### 5.3 MVP에서 다음 단계로 미룸
- 실제 메시지 발송 연동("초안 생성 + 승인"까지만)
- 자동 리마인드 예약의 완전 자동화(데모에서는 예정 표시로 충분)

## 6. 정보구조(IA) / 화면 목록

### 6.1 상위 네비게이션(권장)
오늘 브리핑 / 케이스 / 템플릿 / Evidence·로그 / 설정(Policy)

### 6.2 화면 목록(최소)
S1. 오늘 브리핑 / S2. 케이스 상세(패널/바텀시트) / S3. 초안 보기(메시지/문서) / S4. 승인 모달 / S5. 판단 기록(타임라인)

## 7. 화면 기획(UX/UI Spec)

> "UI 정리"와 "데모"에 맞춰 실제로 그릴 수 있는 단위로 정리한 스펙이다.

### 7.0 모바일 화면 원칙(별도 스펙)
- 모바일은 "관리 화면"이 아니라 승인/확인/발송(또는 발송 승인) 중심으로 최소 동선만 제공한다.
- 모바일의 기본 진입은 오늘 브리핑 카드이며, 상세는 바텀시트로 끝낸다. (페이지 이동 최소화)
- 모바일에서는 "설정/템플릿/로그 탐색"은 후순위. 승인 버튼과 초안 확인이 1순위다.

#### M1. (Mobile) 오늘 브리핑
**목표:** 대표/운영자가 이동 중에도 "승인 필요한 1~3건"만 빠르게 처리한다.
**필수 컴포넌트**
- `MobileBriefingHeader` — 타이틀: 오늘 브리핑 / (옵션) [새로고침]
- `MobileBriefingCardList` — 카드 1~3개

**카드(최소 필드)**
- 제목(케이스명) / 배지: D-day · 승인 필요 · 누락 개수 / CTA 2개 고정: [초안 보기] [승인하기](또는 [승인 요청])

#### M2. (Mobile) 케이스 바텀시트(상세)
**목표:** 승인 전에 필요한 정보만 "짧게" 보여준다.
**필수 섹션**
1) 요약(1문장) 2) 누락 서류(체크리스트) 3) 초안 미리보기(베트남어/한국어 토글) 4) 근거(Evidence) 링크(최소 1~2개) 5) [승인하기] / [반려]

#### M3. (Mobile) 승인 완료 결과
**목표:** "승인이 실제로 상태를 바꿨다"를 체감시키고, PC에서의 후속 작업으로 연결한다.
- 토스트/배너: "승인 완료 · 판단 기록 #xxxx 저장"
- 해당 카드 상태가 `승인 완료`로 변경
- (옵션) 다음 할 일 1줄 노출: "다음: 발송 실행 가능(PC에서)" 또는 "다음: 리마인드 예정"

#### M4. (Mobile) 표현 가이드(안전/신뢰)
- 모바일에도 고정 문구 1줄을 반드시 노출: "승인 전에는 외부 발송이 차단됩니다."
- 단정형 문장 금지(가능/불가능 판단). "확인 필요/검토 필요/다음 안전한 행동은 …" 형태 유지

### S1. 오늘 브리핑
**목표:** 입력 없이 '오늘 처리할 일 Top 3'를 보여준다.
**레이아웃(필수 영역)**
- Header — 타이틀: 오늘 브리핑 / CTA: [브리핑 생성] (옵션: 자동 진입)
- BriefingCardList (카드 3개 고정이 가장 데모 친화적)

**브리핑 카드(공통 스키마)**
- 제목 / 배지: D-day · 누락 · 승인 필요 / 핵심 버튼(2~3개): 초안 보기, 요청서 보기(채용 준비로 이동), 승인 요청/승인하기

**상태/표현 규칙**: 카드에는 "상태, 기한, 업무"만 1차 노출 (디테일은 S2 패널)

### S2. 케이스 상세 패널
**목표:** '왜/무엇/다음 행동'이 한 화면에서 끝난다.
**필수 블록(4개)**: 1) 왜 확인 필요? 2) 누락 서류 3) 이전 기록 4) 다음 할 일(Next Actions)

**Next Action 표현(중요)** — 각 액션은 라벨 / 상태(준비됨·대기·예정) / 승인 필요 배지(있으면 잠금) / 추천 이유 1줄(토글/작은 글씨)을 가진다.

### S3. 초안 보기
**목표:** 다국어 초안(VN/KR)을 동시에 보여주고, 발송은 승인 전 차단.
**필수 요소**: 언어 탭 또는 2단 컬럼(VN / KR) / 템플릿 출처(선택) / 경고 문구: "승인 전에는 외부 발송이 차단됩니다."

### S4. 승인 모달
**목표:** 외부 발송/전달 같은 위험 행동을 승인으로 통제.
**필수 요소**: 승인 대상(무엇을 실행하는지) / 요약(1문장) / 근거(Evidence) 링크(최소 1~2개) / 승인/반려 버튼

### S5. 판단 기록(타임라인)
**목표:** 감사 가능성(누가/언제/무엇을)을 10초 안에 전달.
**필수 요소**: 타이틀: 판단 기록 #xxxx / 타임라인 이벤트(최소): approval_requested, approval_completed

## 8. 데이터 모델(최소)

### 8.1 Core Entities
Company(고객사) / Worker(근로자) / Case(업무 케이스) / NextAction(다음 행동) / Approval(승인) / EvidenceEvent·EvidenceLog(감사 이벤트) / Template(메시지/체크리스트) / Policy(Routing/Approval Rule)

### 8.2 상태값(권장)
- Case.state: draft / risk_review / approval_pending / human_approved / completed / blocked
- NextAction.state: locked(approval_required) / ready / scheduled / waiting
- Approval.status: pending / approved / rejected / cancelled

## 9. 안전/컴플라이언스 설계(필수 요건)
- PII 마스킹(입력/로그/LLM 컨텍스트 전파 차단)
- 승인 전 외부 발송/전달 차단(구조적)
- 근거 품질 게이트(근거 없으면 "차단/확인 필요"로)
- 이벤트 로그는 INSERT-only(감사 재현)
- 실패 모드는 조용히 넘어가지 않고 blocked/error로 표면화

## 10. 품질 지표(초기)
- 승인 전 자동 발송 0건
- 근거 없는 단정/법률자문 표현 0건
- 브리핑에서 "처리 필요 케이스 Top 3" 적중률(정성/정량)
- 케이스 패키지 생성 시간 단축(전/후 비교)

## 11. 오픈 이슈 / 결정 필요
- 실제 MVP에서 '케이스'의 최소 단위는 무엇인가(근로자 단위 vs 요청 단위)
- "행정사 패키지" 산출물 포맷(이메일/PDF/링크)
- 다국어 컨택 채널 범위(Zalo/Kakao/SMS)와 로그 수집
