agent들을 한 파일로 합치지 않고, AgentOutput 표준 schema를 만든 다음 executor → aggregator → approval → final_response 흐름으로 수렴시킨다.

다음 구현 단위는 aggregator.py, agent_output.py, executor.py, workflow.py, approval_gate.py, final_response.py, 테스트/eval 순서.

실행 옵션은 두 가지:
1. Subagent-Driven: task별로 나눠 구현하고 중간 리뷰하면서 진행
2. Inline Execution: 이 세션에서 바로 순서대로 구현

## Layer 1 — Data (가장 깊은 공유점)

```
[Shared State (SQLite)]
  companies / employees / candidates / documents / audit_logs

[Shared RAG]
  data/processed/regulation_chunks.jsonl
  data/cache/embeddings.sqlite
```

세 agent 모두 같은 풀을 봄. 각자 메타필터(case_type, mission_agent)로 자기 영역만 가져옴.

## Layer 2 — Schema (계약)

같은 enum 공유:
- `case_type` 4종 (new_hiring / rehire_loyalty / workplace_change_intake / same_worker_rehire)
- `chunk_type` 6종, `evidence_grade` A~F
- 공통 Input/Output schema (기획문서 §11)

깨지면 agent 간 데이터 교환이 비뚤어짐.

## Layer 3 — Workflow (LangGraph)

```
[User: 자연어 목적 입력]
            ↓
    [Intent Router]    ← 단일 진입점
            ↓
       [Planner]        ← 어떤 agent를 어떤 순서로 부를지 결정
            ↓
    [State Loader]      ← 회사·직원·후보 DB 로드
            ↓
   ┌────────┼────────┐
   ↓        ↓        ↓
[인력 확보] [다국어] [비자·서류]   ← 병렬 또는 순차 실행
   └────────┼────────┘
            ↓
     [Aggregator]       ← ⭐ 세 agent 결과 합치는 지점
            ↓
    [Risk Classifier]   ← low/mid/high
            ↓
    [Human Approval]    ← ⭐ 외부 발송 전 필수 게이트
            ↓
[Handoff Package + Evidence Log]   ← 행정사 패키지
```

세 agent가 진짜 만나는 곳은 두 개: `Aggregator` + `Human Approval`. 그 전에는 각자 독립.

## Layer 4 — Output (감사·인용)

- 결과는 `case_id`로 그룹화된 single output
- 모든 결정에 `chunk_id` 인용 첨부
- 모든 state transition은 `audit_logs`에 INSERT-only

## 인력 확보 agent의 수렴 인터페이스

| 방향 | 파트너 | 내용 |
| --- | --- | --- |
| In | Intent Router | `{case_type: "new_hiring", company_id, headcount, country}` 같은 구조화 입력 |
| Read | Shared State | companies.csv (사업장 조건) + candidates.csv (있으면) |
| Read | Shared RAG | `case_type=new_hiring` 필터로 절차·허용업종·점수제 chunk |
| Out | Aggregator | 신규 인력 요청서 + 사업장 조건 체크리스트 + 질문 리스트 + risk_flags + evidence_sources |
| Handoff | 비자·서류 agent | "후보 매칭됐음, 이 후보의 D-day·누락 서류 확인 부탁" 같은 trigger |

## 핵심 디자인 원칙 (4가지)

1. **Agent 간 직접 호출 금지** — 모두 shared state로 소통. 직접 호출하면 결합도 폭발.
2. **Single source of truth** — DB·RAG·audit_logs. agent마다 자기 사본 갖지 않음.
3. **Standardized handoff format** — `case_id` 기반 case JSON. agent A의 출력이 agent B의 입력으로 들어갈 때 같은 형식.
4. **Human approval gate 통일** — 어느 agent가 외부 발송 시도해도 같은 큐로. low/mid/high risk별 분기는 단 한 곳에서만.

이걸 잡지 않으면 세 명이 각자 따로 노는 챗봇 3개가 됨.
