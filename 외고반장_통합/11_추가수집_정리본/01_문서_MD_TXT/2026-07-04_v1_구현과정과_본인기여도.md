## 구현 범위 한 줄 요약

"외국인 고용 운영" 업무를 **감지 → 분석 → 준비 → 승인(HITL) → 기록(Evidence Log)** 흐름으로 닫히게 만들기 위해, *Agent Chat*과 *Daily Briefing(Scheduler)* 두 진입점이 **공통의 Case/NextAction/Approval/Evidence 모델**로 수렴하도록 설계·구현했다.

## 1) 전체 구현 과정(Top-down)

### 1. 문제 재정의 → '운영 OS'로 범위 고정
- 단순 Q&A/자동 제출이 아니라, 현장에서 매일 반복되는 체류만료/서류누락/다국어 소통/행정사 전달 준비를 "운영" 관점에서 묶어서 다루는 방향으로 정했다.
- 중요한 경계: 법적 판정/자문처럼 보이는 출력 금지 / 승인 전 외부 발송/전달/제출 금지(구조적으로 막기)

### 2. 사용자 경험을 워크플로우로 고정(제품 → 구현 계약)
- 감지 → 분석 → 준비 → 승인 → 기록의 5단계로 고정하고, 각 단계에서 "무슨 일이 일어나야 하는지"를 시스템 구조에 그대로 반영했다.
- 데모 시나리오(Nguyen 케이스)를 기준으로 "끝까지 닫히는" E2E 플로우가 되도록 설계했다.

### 3. 아키텍처/데이터 모델 먼저 고정(안전·재현성 중심)
- 두 진입점: Scheduler(Daily Briefing으로 위험 감지) / Chat(자연어 요청으로 케이스 파고들기·신규 업무 요청)
- 두 진입점 모두 동일한 저장 단위로 수렴: Case / NextAction / Approval / EvidenceEvent(Evidence Log)
- 구현 원칙: Rule(결정론 계산) vs DB(현재 상태) vs RAG(공식 근거) vs LLM(초안 생성) 분리, trace_id/request_id 기반 추적, Evidence Log는 insert-only 이벤트 소스로 취급

### 4. 멀티 에이전트 구조를 '만나는 지점'으로 통제
- 외부 노출은 3 Mission Agent로 단순화(인력 확보 / 다국어 컨택 / 비자·서류 운영)
- 내부 구현은 더 잘게 쪼개되, 공통 만나는 지점(Aggregator, Human Approval)을 고정

### 5. 운영화 관점(비용/보안/품질)까지 같이 설계
- 비용: embedding cache, 호출 제한, top-k/컨텍스트 길이 관리, 재시도 정책
- 보안/프라이버시: PII 마스킹(LLM/로그 전파 차단), 권한 기반 조회
- 품질/신뢰: evidence grade gate, citation validator, source freshness/버전 관리
- 실패 모드: "근거 없음/불명확"을 조용히 덮지 않고 blocked로 표면화

## 2) 핵심 기능별 구현 포인트(What we built)

### A. Intent Router + Planner
자연어 요청을 "업무 의도"로 분해하고, 실행 계획(steps/required_data)을 구조화해 에이전트가 마구 실행하지 못하도록 제한. multi-intent를 배열 구조로 설계.

### B. Rule Engine(결정론 계산)
체류만료 D-day, 누락 서류, 기한/갭 탐지는 LLM이 아니라 Rule이 계산하도록 분리해 신뢰성과 재현성을 확보. LLM은 "판정"이 아니라 "설명/요약/초안" 역할로 제한.

### C. RAG(공식 근거 연결)
공식 절차/서식/가이드를 검색해 근거를 붙이는 구조로 만들고, chunking/메타데이터/버전 핀ning을 통해 나중에 근거를 다시 추적할 수 있는 형태로 설계.

### D. Approval Gate(HITL)
메시지 발송/전달/제출처럼 외부에 영향을 주는 행동은 승인 없이는 실행 경로가 열리지 않게 설계. 승인 이후에도 "외부 실행"을 곧바로 하지 않고 outbox(준비 상태)로 분리할 수 있는 방향성.

### E. Evidence Log(감사·재현)
어떤 근거를 썼는지, 어떤 계획으로 실행했는지, 어떤 단계에서 승인/차단되었는지를 이벤트로 남겨 사후 검토 가능한 운영 시스템으로 만들었다.

## 3) 본인 기여도(내가 맡은 핵심)

1. "운영 OS" 방향성 고정 및 워크플로우 설계 — 감지→분석→준비→승인→기록의 워크플로우를 제품/데모/구현 구조에 일관되게 반영
2. 아키텍처/데이터 모델의 기준점 제시(수렴 구조) — Chat과 Daily Briefing을 분리하면서도 Case/NextAction/Approval/Evidence로 통일, Rule vs DB vs RAG vs LLM 책임 분리 명문화
3. 안전·운영화(Guardrail) 관점 설계 — 승인 전 외부 실행 차단, PII 마스킹, 근거 품질 게이트, 추적 가능성(trace/request id) 체크리스트화
4. 데모 시나리오(E2E) 관통 정리 — Nguyen 케이스를 통해 "업무 거버넌스(승인/근거/기록)"가 보이게 구성
