## 0) 한 줄 아키텍처
Trigger(버튼) → Router(설정 기반 + 규칙/LLM 혼합) → Orchestrator(에이전트 실행/집계) → Briefing UI → Action 실행(Approval 게이트) → Evidence/Log 기록

## 1) 표준 이벤트(Envelope): 버튼이 만드는 입력 규격

브리핑 버튼은 "명령어"가 아니라 컨텍스트 스냅샷 이벤트를 만든다.

| 필드 | 예시 | 메모 |
| --- | --- | --- |
| trigger_type | briefing_button | 입력 텍스트 없어도 OK |
| persona | 근로자 / 행정사 / 인력담당자 | 라우팅 1순위 feature |
| policy | Daily / Risk / Submission / Outreach | 버튼 옵션(기본값 Daily) |
| case_url | (현재 보고 있는 케이스) | 항상 포함(핵심 컨텍스트) |
| org_url | (회사/고객) | 템플릿/권한 스코프 |
| language | ko / vi / en | 다국어 에이전트 입력 |
| constraints | needs_approval=true | 외부 발송/제출은 승인 필수 |

## 2) 라우팅 설계: 하드코딩을 "설정(Policy/Rules)"로 빼기

### 2.1 Router의 결과물(표준)
- 이번 브리핑에서 호출할 에이전트 목록: `agents_to_run[]`
- 브리핑 카드 출력 템플릿: `briefing_template`
- 다음 액션 후보: `next_actions[]`
- 승인 게이트 규칙: `approval_ruleset`

### 2.2 라우팅 우선순위(권장)
1) Rule layer(결정적): 금지/필수/우선순위/승인 조건
2) LLM layer(보조): 상황 요약, Top 3 선택, 문구 구성
3) Fallback: Rule만으로도 최소 브리핑 가능

### 2.3 Policy 모드 정의(초기 4개)
- Daily Briefing: 오늘 해야 할 일 / 막힌 점 / Top 3 링크
- Risk Briefing: 리스크 / 기한 임박 / 미결정 / 컴플라이언스
- Submission Prep: 제출 패키지 구성(체크리스트/문서 묶기)
- Outreach: 컨택 메시지 생성(다국어 중심, 발송은 승인)

## 3) 화면 흐름(UX): 2-Step이 가장 안정적

### Step A. Briefing (0~10초)
[브리핑 생성] 버튼 → 라우팅/실행 → Briefing 카드 1장. 카드 구성: 요약(1문장) / 오늘·이번 주 Top 3 / 리스크·막힌 점(최대 3) / 추천 Next Action 버튼들 / (토글) 근거(Evidence) 보기

### Step B. Execute (Approval 포함)
Next Action 클릭 → Plan 생성 → (필요 시 승인) → 실행 → Evidence/Log 기록

## 4) 필요한 UI(최소 6개 컴포넌트)
1) 브리핑 버튼(persona/policy 기본값 기억) 2) Briefing 카드 뷰(출처 에이전트 태그 포함) 3) Next Action 패널(승인 필요 배지) 4) Approval 모달(발송/제출/법적 문구) 5) 템플릿 선택기(회사 기본 + 케이스 특화) 6) Evidence/Log 탭(재현성/디버깅)

## 5) 페르소나별 출력 가이드(같은 버튼, 다른 정보)
- 근로자: 해야 할 일/서류 준비/방문 일정/주의사항(쉬운 문장)
- 행정사: 제출요건/누락/리스크/근거 메모(전문가용)
- 인력담당자: 파이프라인/컨택 현황/병목/비용/리드타임

## 6) 3개 에이전트 공통 실행 규격(표준 I/O)

**Input**: CaseContext + Persona + Policy + Language + Constraints
**Output**: `briefing_items[]`(카드 단위), `next_actions[]`(버튼/플로우 연결), `artifacts[]`(문서/메시지/체크리스트 산출물), `evidence[]`(근거 링크/데이터 포인트)

## 7) Notion에 잡을 DB/페이지 템플릿(권장 최소 구성)

하드코딩을 설정으로 빼는 핵심 저장소 4개:
- **Routing Policies (DB)**: Policy 이름, Persona, 기본 Agents, 금지/필수, 우선순위, 승인 조건
- **Templates (DB)**: 메시지/체크리스트/문서 템플릿, 언어, 적용 조건
- **Agent Actions Catalog (DB)**: 액션 이름, 담당 에이전트, 입력 요구사항, 승인 필요 여부
- **Evidence Log (DB)**: 케이스, 실행 액션, 생성 산출물 링크, 근거, trace_id, 실행자

### 7.1 DB 스키마 힌트 (taxonomy.md 기반, 재사용 가능한 속성)

- **Intent/Case/State**: Intent(HIRING/VISA_CHECK/DOCUMENT_CHECK/CONTACT/BRIEFING/UNSUPPORTED_*), Case Type(new_hiring/workplace_change_intake/stay_extension/employment_change), Workflow State(draft~completed), Approval Status(PENDING/APPROVED/REJECTED/CANCELLED), Evidence Event Type(intent_classified~block) — 자세한 값은 `2026-07-04_v1_taxonomy_분류체계.md` 참고
- **RAG Source/Chunk 메타데이터(추천 속성)**: source_id, title, publisher, source_type, doc_type, evidence_grade(A~F), retrieved_at, effective_date, risk_level, mission_agent, case_type, visa_type, industry/region/country, output_usage → Templates DB / RAG Sources DB 공통 적용 가능
- **Actions Catalog 필드**: action_id, label, agent, intent, tool_grade(SAFE_READ/SAFE_CALCULATE/SAFE_DRAFT/APPROVAL_REQUIRED/FORBIDDEN), requires_human_approval, blocked_actions, output_schema_version

### 7.2 페이지 템플릿 초안

**Template 1) Routing Policy**: Policy 이름 / 적용 Persona / 기본 Intent / agents_to_run(우선순위) / 금지 Intent·행동 / Approval ruleset / Evidence ruleset(A/B/E만 허용 등) / 기본 Briefing 카드 섹션

**Template 2) Evidence Event**: event_type / case_url / intent(s) / agent·tool / inputs(PII 제거 요약) / outputs(artifact 링크) / rag_sources(source_id 리스트) / evidence_grade / approval_status / trace_id

## 8) MVP 범위(2주 컷 제안)
- 브리핑 버튼 1개 + Policy 2개(Daily, Outreach)
- 페르소나 2개(행정사, 인력담당자)부터
- Evidence Log는 반드시(나중에 후회 방지)
- 외부 발송은 "초안 생성 + 승인"까지만(실발송은 다음 단계)

### 10) 데모 패키지 점검: 현재까지 확인된 '빠진 것' 체크리스트

- **(1순위) 근로자 수신 단계(인박스) + 응답 1회**: 승인 후 근로자 인박스에서 메시지 도착 화면, 근로자 응답 1회(예: "보낼게요 / 어떻게 보내요? / 조금만 기다려주세요"), 관리자 화면에 '응답 도착'으로 반영
- **(필수) Next Action '추천 근거' 1줄**: "D-30이고 누락 2건이라 오늘 요청이 필요", "3일 전 요청 이력이 있어 응답 지연 가능성"
- **(권장) Next Action이 '조건 충족'으로 갱신되는 장면 1회**: 근로자 응답 도착 → 다음 행동이 갱신(예: "서류 제출 방법 안내", "행정사 패키지 생성 활성화")
- **(권장) 판단 기록에 '근거' 흔적 최소 1~2개**: D-30 계산 근거(만료일), 누락 서류 판정 근거(필수서류 매칭), 3일 전 요청 이력, (선택) source_id 1개
- **(권장) 페르소나별 화면 1컷(스냅샷)**: 근로자/행정사/인력담당자 화면을 10초 스냅샷으로 비교
- **(권장) 채널 배지 1회**: 근로자 인박스 상단에 채널 배지/문구로 현실감 부여
