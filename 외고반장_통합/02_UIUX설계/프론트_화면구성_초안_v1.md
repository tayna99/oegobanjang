# 프론트 화면 구성

# 1. 전체 화면 구조

## 1.1 기본 원칙

- 역할에 따라 다르게.
    
    관리자/인사팀/행정 담당자는 PC 대시보드 중심.
    
    사업자/대표자는 모바일 브리핑 챗 중심.
    
    행정사는 검토 자료 확인 중심.
    
- 모든 업무: 오늘 할 일 큐로 돌아오기
    
    예를 들면 체류기간 임박, 계약 종료 임박, 누락 서류, 승인 대기, 행정사 검토 준비, 메시지 응답 도착 같은 것들.
    
- 에이전트가 어떤 다음 액션을 준비했는지 보여줌
    
    각 업무를 누르면 AI가 어떤 액션을 준비했는지 보여줘야 해.
    
    예를 들면:
    
    “Nguyen V. 체류기간 D-30”
    → Visa Document Agent가 필요 서류 확인
    → Multilingual Contact Agent가 베트남어 요청 메시지 초안 생성
    → Workforce Agent가 케이스 상태 업데이트
    → 승인 요청 생성
    
- 판단 기록: 왜 이 판단을 했는지 기록이 남아야 함

## 1.2 화면 연결

1. PC 대시보드에서 오늘 할 일 확인
“체류기간 임박 4명, 서류 보완 필요 7건이 있습니다.”

2. Nguyen 케이스 클릭
오른쪽 상세 패널 열림.

3. AI가 준비한 액션 확인
“베트남어 서류 요청 메시지 초안 생성 완료.”

4. 관리자 버튼 클릭
“대표 승인 요청 보내기.”

5. 모바일 화면 전환
대표자에게 브리핑 도착.

6. 대표자가 초안 확인
베트남어 원문과 한국어 번역 확인.

7. 대표자 승인
“이 메시지로 컨택할까요?” 클릭.

8. 결과 저장
“업무 기록 #4789에 저장되었습니다.”

## 2.1 탭 구성

- **오늘 할 일**
    - 관리자가 매일 들어와서 보는 메인 화면.
- **근로자**
    - 외국인 근로자 목록, 체류자격, 체류만료, 계약종료일 확인.
- **케이스**
    - 체류연장, 신규채용, 계약종료, 서류보완 같은 사건 단위 관리.
- 메시지 관
    - 근로자에게 보낸 메시지, 응답, 다국어 초안 관리.
- **행정사 검토**
    - 행정사에게 넘길 자료, 검토 패키지, 승인 상태 관리.
- 판단 기록
    - AI 판단 근거, 사람 승인, 발송 이력 확인.

## 2.2 관리자 메인 화면 구성

- 예를 들면 서류 보완 필요 7건을 누르면 아래 테이블이 서류 보완 업무만 보이게 필터링 되는 구조로

```jsx
예시.

체류기간 임박 4명
서류 보완 필요 7건
신규 채용 준비 1건
컨택 대기 4건
응답 도착 2건
승인 대기 5건
행정사 검토 준비 2건
```

# 3. 사업자용 모바일 클라이언트: 단순화가 포인트

- 모바일 클라이언트의 목적: 오늘 뭘 승인하면 되는지 알려주고, 승인 버튼을 누르게 한느 것.
- 승인할 내용이 먼저 보여야 함.
- 주요 구성: 오늘 브리핑, 모바일 챗, 승인 요청 카드, 위험 알림, 메시지 초안 확인, 간단한 “승인/수정/보류” 버튼

```jsx

│ 한별제조 · 승인 필요 3건 │
├────────────────────┤
│ AI 반장 메시지        │
│ 대표님, 오늘 확인이 필요한 업무가 3건 있습니다. │
├────────────────────┤
│ 승인 카드 1           │
│ Nguyen V. 체류기간 연장 서류 요청 │
│ D-30 / 누락 서류 2건 / 승인 필요 │
│ [초안 보기] [승인하기] │
├────────────────────┤
│ 승인 카드 2           │
│ Candidate A 입국 전 서류 패키지 │
│ [검토 자료 보기] [승인하기] │
```

```
1. 근로자 정보 확인 완료
2. 이전 대화 기록 확인 중
3. 메시지 초안 생성 완료
4. 발송 전 승인 대기
```

1. 근로자 정보 확인

2. 이전 기록 확인

### 예상 시나리오

- 긍정 응답 시
    - 서류 수신 후 행정사 검토 자료에 자동 반영

- 응답 지연 시
    - 2일 뒤 리마인드 메시지 제안

# 변경 사항

- **채용 준비 / 인력 확보 탭이 없음.**
    
    현재 데이터에는 `WORKERS`, `CASES`, `ACTIONS`가 있고, `ACTIONS` 안에 `request_document`, `create_handoff`, `review` 정도가 들어 있어. 그런데 **인력 확보/채용 준비를 보여주는 데이터 모델이 없어.**
    
    지금 MD에는 업무 큐 예시로 “신규 베트남 E-9 3명 채용 요청 / 송출회사 요청서 / 화성 1공장 / 준비 중 / 이번 주 / 요청서 보기”가 들어가 있어.  이건 사실 Workforce Agent의 핵심 시나리오야.
    
    그런데 zip에는 아래 같은 데이터가 별도로 없어.
    
    ```
    RECRUITMENT_REQUESTS
    CANDIDATES
    WORKFORCE_READINESS
    QUOTA_STATUS
    HIRING_REQUIREMENTS
    RECRUITMENT_ACTIONS
    ```
    
    그래서 현재 화면에서 “신규 채용 준비 1건”이라는 카드가 있더라도, 클릭했을 때 진짜 인력 확보 화면으로 들어가는 구조가 약해.
    
    인력 확보 쪽은 이렇게 추가돼야 해.
    
    ```
    채용 준비 탭
    - 신규 채용 요청 목록
    - 사업장/라인별 필요 인원
    - 체류자격/E-9 가능 여부
    - 쿼터 상태
    - 구인노력/서류 준비 상태
    - 송출회사 요청서
    - 후보자 입국 전 서류 패키지
    - 행정사 검토 전 확인 필요
    ```
    
    즉 “채용 추천 AI”가 아니라, **신규 고용 준비 상태 점검 화면**으로 가야 해. 후보자를 점수화하거나 추천하는 방향은 위험하고, 데모에서는 “준비 상태를 체크해준다”가 맞아.
    
- **메시지 관리 / 다국어 컨택 탭이 없음.**
    
    현재 `data.jsx` 안에 `DOC_REQUEST_DRAFT`가 있어서 한국어/베트남어 서류 요청 초안은 들어 있음. 그리고 `mobile-chat.jsx`에서도 “Nguyen V.님께 보낼 서류 요청 초안”을 띄우는 bottom sheet는 있음.
    
    하지만 이건 **다국어 컨택 기능의 한 조각**이지, “Multilingual Contact Agent 화면”이라고 보기엔 부족함.
    
     “초안 상세, 베트남어 메시지, 한국어 번역, 예상 응답 시나리오, 수정 요청/보내기 승인”까지 포함해야 함  또 Live 화면은 메인 화면이 아니라 승인 직전 화면으로 쓰고, 4단계 진행 구조를 가져가야 한다고 되어 있음. 
    
    현재 zip에 없는 것:
    
    ```
    메시지 관리 탭
    다국어 컨택 목록
    근로자별 메시지 스레드
    채널 표시: Zalo / SMS / Kakao / WhatsApp 등
    VN/KR 동시 미리보기
    예상 응답 시나리오 카드
    LiveAgentProgress 컴포넌트
    판단 기록 #4789 연결
    승인 후 상태 변화
    ```
    
    현재는 “서류 요청 초안 보기” 정도에서 멈춰 있음. 데모에서 보여주려면 **다국어 컨택이 별도 화면처럼 보여야 하고**, 모바일에서는 “초안 보기 → Live Agent 처리 화면 → 보내기 승인”이 연결돼야 함
    
- **모바일 브리핑 카드형 승인 화면이 없음.(Live Agent 처리 화면이 실제 컴포넌트로 없음)**
    
    
    현재 모바일은 아래처럼 바꿔야 함. 
    
    현재:
    
    ```
    채팅 메시지 자동 출력
    위험도 요약 타일
    케이스 카드 일부
    서류 요청 bottom sheet
    입력창 있음
    ```
    
    바꿀 방향:
    
    ```
    오늘 브리핑 홈
    승인 필요 3건
    Nguyen 카드
    Candidate A 카드
    Tran T.H. 카드

    Live Agent 처리 화면
    보내기 승인
    판단 기록 저장 완료
    ```
    
- **Evidence Log라는 영어 표현이 아직 일부 남아 있음.**
    
    `Evidence Log #4789`를 `판단 기록`으로 바꾸기로 했고, 라이브 화면에서도 그렇게 정리했어.
    
    그런데 zip 안에는 아직 `Evidence Log`라는 표현이 일부 남아 있어.
    
    특히 수정할 곳:
    
    ```

    uploads/*.md
    ```

    ```

    EvidenceEvent → 판단 기록 이벤트 / 처리 기록
    Citation → 판단 근거 / 참고 자료
    Handoff → 검토 자료 / 전달 자료
    Approval → 승인 관리 / 확인 요청
    ```
    
    다만 코드 내부 변수명은 `EvidenceLogView`로 남겨도 돼. 급하면 UI 표시 텍스트만 먼저 바꾸면 돼.
