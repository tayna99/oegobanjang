# 프론트 화면 구성 — 외고반장 데모2 맞춤 개선본

## 0. 이번 개선의 목적

이 문서는 기존 `프론트 화면 구성.md`를 외고반장 데모2 흐름에 맞춰 다시 정리한 버전이다.

기존 문서의 핵심인 **PC 대시보드 → 업무 큐 → 오른쪽 상세 패널 → 대표 승인 요청 → 모바일 브리핑 → 승인 → 판단 기록 저장** 흐름은 유지한다. 특히 오른쪽 상세 패널은 데모 흐름을 끊지 않는 핵심 구조이므로 새 페이지 이동 방식으로 바꾸지 않는다.

다만 데모2에서 강조하는 세 가지 축, 즉 **인력 확보 Agent / 다국어 컨택 Agent / 비자·서류 운영 Agent**가 화면에서 더 분명히 보이도록 PC 탭, 업무 큐, 모바일 승인 화면, Live Agent 화면을 보강한다.

## 1. 제품 데모의 한 줄 구조

외고반장은 외국인 고용 업무를 사람이 매번 찾아서 처리하는 도구가 아니라, **오늘 처리해야 할 채용·컨택·비자서류 업무를 먼저 큐로 올리고, AI가 초안과 검토 자료를 준비한 뒤, 사람이 승인하는 운영 OS**다.

데모의 기본 메시지는 아래처럼 잡는다.

```text
외국인 고용 업무,
놓치기 전에 먼저 챙깁니다.

채용 · 컨택 · 비자서류를
경로로 짜주는 AI 반장
```

## 2. 전체 화면 원칙

### 2.1 역할별 화면

| 사용자 | 주 화면 | 목적 |
| --- | --- | --- |
| 관리자 / 인사팀 / 행정 담당자 | PC 대시보드 | 오늘 처리할 업무 확인, 상세 패널 검토, 대표 승인 요청, 행정사 전달 자료 생성 |
| 사업장 대표 / 의사결정자 | 모바일 브리핑 | 승인 필요한 일만 빠르게 확인하고 승인 / 수정 요청 / 보류 |
| 행정사 / 외부 전문가 | 검토 자료 화면 | AI가 준비한 서류 패키지와 판단 기록 확인 |

### 2.2 모든 업무는 “오늘 할 일 큐”로 돌아온다

외고반장의 메인 화면은 기능별 메뉴가 아니라 **오늘 처리해야 할 업무 큐**다.

예시 업무는 아래와 같다.

```text
체류기간 임박
서류 보완 필요
신규 채용 준비
컨택 대기
응답 도착
승인 대기
행정사 검토 준비
```

사용자는 “근로자 관리”, “메시지 관리”, “행정사 검토”를 따로 찾아 들어가지 않아도 된다. 오늘 문제가 되는 업무가 먼저 떠야 한다.

### 2.3 AI는 자동 실행자가 아니라 준비자다

외고반장의 AI는 메시지를 마음대로 보내거나 서류를 자동 제출하지 않는다.

AI가 하는 일은 다음까지다.

```text
1. 상황 감지
2. 관련 데이터 확인
3. 필요한 서류 / 메시지 / 검토 자료 준비
4. 위험도와 판단 근거 표시
5. 사람에게 승인 요청
```

최종 발송, 외부 전달, 행정 절차 시작은 반드시 사람의 승인 뒤에만 진행된다.

### 2.4 화면 용어

데모 화면에서는 영어를 최소화한다.

| 기존 표현 | 화면 표시 권장 |
| --- | --- |
| Evidence Log | 판단 기록 |
| Evidence Event | 판단 기록 이벤트 / 처리 기록 |
| Citation | 판단 근거 / 참고 자료 |
| Handoff | 검토 자료 / 전달 자료 |
| Approval | 승인 관리 / 확인 요청 |
| Live Agent | 실시간 처리 / 승인 직전 처리 |

코드 내부 변수명은 `EvidenceLogView`, `handoff`, `approval`처럼 남아 있어도 되지만, 데모 화면에 직접 노출되는 텍스트는 한국어로 정리한다.

## 3. 데모2 기준 사용자 시나리오

데모2의 핵심 사용자는 “베트남 E-9 근로자 3명 추가 채용 준비해줘. Nguyen 체류만료도 확인해줘.”라고 요청한다.

이 요청은 하나의 기능이 아니라 세 개의 업무 축으로 분기된다.

```text
사용자 요청
→ Intent Router
→ 인력 확보 Agent
→ 다국어 컨택 Agent
→ 비자·서류 운영 Agent
→ Human Approval
→ 판단 기록 저장
```

### 3.1 시나리오 A — 인력 확보 및 쿼터 판별

사용자 요청:

```text
베트남 E-9 근로자 3명 추가 채용 준비해줘.
```

AI 처리:

```text
1. 고용 쿼터 확인
2. 채용 의도 분석
3. 신청 서류 초안 작성
4. 내국인 구인노력 기간 산정
5. 채용 요청서 초안 생성
```

화면 결과:

```text
신규 베트남 E-9 3명 채용 요청
현재 준비 상태 72%
잔여 쿼터: 5명까지 추가 고용 가능성을 검토할 수 있는 상태
단, 최종 가능 여부는 고용센터 / 행정사 확인 필요
```

주의: 이 화면은 “좋은 근로자 추천” 화면이 아니다. 성실도, 이탈 가능성, 국적별 우열처럼 차별적이거나 주관적인 판단은 하지 않는다. 데모에서는 **채용 준비 상태 점검 / 쿼터 확인 / 서류 준비** 화면으로 보여준다.

### 3.2 시나리오 B — 다국어 컨택 및 응답 해석

사용자 요청:

```text
Nguyen에게 비자 연장 서류 요청 메시지 보내줘.
```

AI 처리:

```text
1. 근로자 프로필 확인
2. 이전 대화 기록 확인
3. 베트남어 요청 메시지 생성
4. 한국어 번역본 생성
5. 예상 응답 시나리오 생성
6. 발송 전 승인 대기
```

화면 결과:

```text
VN / KR 메시지 초안 동시 표시
예상 응답 시나리오 표시
수정 요청 / 보내기 승인 버튼 표시
판단 기록 #4789 연결
```

### 3.3 시나리오 C — 비자 리스크 및 전문가 전달

상황:

```text
Nguyen 근로자의 E-9 체류기간이 30일 후 만료됩니다.
표준근로계약서 사본과 여권 사본이 누락되어 있습니다.
```

AI 처리:

```text
1. 체류만료 D-day 확인
2. 계약종료일 충돌 여부 확인
3. 누락 서류 체크
4. 행정사 검토용 전달 자료 준비
5. 대표 / 관리자 승인 요청 생성
```

화면 결과:

```text
체류기간 연장 서류 요청
D-30 / 우선 확인 / 승인 전
누락 서류 2건
행정사 검토 자료 준비 가능
```

## 4. 전체 화면 연결 흐름

기존 상세 패널 흐름은 유지하되, 데모2에 맞춰 채용·컨택·비자서류가 모두 보이도록 연결한다.

```text
1. PC 대시보드 진입
   “오늘 처리할 외국인 고용 업무 7건이 있습니다.”

2. 업무 큐 확인
   - Nguyen V. 체류기간 연장 서류 요청
   - 신규 베트남 E-9 3명 채용 준비
   - Candidate A 입국 전 서류 패키지
   - Tran T.H. 계약 종료 응답 확인

3. Nguyen 케이스 클릭
   오른쪽 상세 패널 열림

4. 상세 패널에서 AI가 준비한 일 확인
   - 누락 서류 확인
   - 베트남어 메시지 초안 생성
   - 행정사 검토 자료 준비
   - 판단 근거 표시

5. 관리자 액션
   “대표 승인 요청 보내기” 클릭

6. 대표 모바일 브리핑 도착
   승인 필요 3건 카드 표시

7. 대표가 Nguyen 카드에서 초안 확인
   VN/KR 메시지와 예상 응답 시나리오 확인

8. Live Agent 승인 직전 화면
   근로자 프로필 확인 → 이전 기록 확인 → 메시지 초안 생성 → 발송 전 승인 대기

9. 대표 승인
   “이 메시지로 컨택하기” 클릭

10. 결과 저장
   판단 기록 #4789에 저장
   PC 대시보드 상태 업데이트
```

## 5. PC 관리자 대시보드

### 5.1 /dashboard 기본 방향

`/dashboard`는 PC-first 데모의 메인 진입점이다.

권장 구조:

```text
/dashboard
→ DashboardShell
→ DailyBriefingPanel
→ 오늘 할 일 큐 + 오른쪽 상세 패널
```

PcShell이 예쁜 정적 화면이라면, DailyBriefingPanel은 승인, 수정 요청, 문서 초안, handoff 미리보기, 모바일 미리보기, AI 챗 훅까지 데모 연결성이 더 강하다. 따라서 PC-first 데모에서는 `/dashboard = DailyBriefingPanel`로 고정하는 편이 낫다.

### 5.2 상단 바

```jsx
외고반장
한별제조 · 경기 화성 · 자동차부품
근로자, 케이스, 서류, 메시지 검색
[AI 반장에게 물어보기]
알림 12
김대리 / 인사팀
```

필수 요소:

```text
로고
사업장 선택
검색
AI 반장에게 물어보기
알림
사용자 / 역할 표시
```

### 5.3 탭 메뉴

기존 탭에 `채용 준비`와 `메시지 관리`를 명확히 추가한다.

| 탭 | 역할 |
| --- | --- |
| 오늘 할 일 | 매일 들어와서 보는 메인 업무 큐 |
| 근로자 | 외국인 근로자 목록, 체류자격, 체류만료, 계약종료일 확인 |
| 케이스 | 체류연장, 신규채용, 계약종료, 서류보완 같은 사건 단위 관리 |
| 채용 준비 | 신규 E-9 채용 요청, 쿼터, 구인노력, 신청 서류 준비 상태 |
| 메시지 관리 | 다국어 컨택, 근로자별 대화, 초안, 응답 해석 |
| 행정사 검토 | 행정사에게 넘길 검토 자료, 전달 패키지, 승인 상태 |
| 판단 기록 | AI 판단 근거, 사람 승인, 발송 이력, 처리 로그 |

### 5.4 오늘 처리할 업무 카드

카드는 단순 숫자 표시가 아니라 아래 테이블을 필터링하는 버튼 역할을 한다.

```jsx
체류기간 임박 4명
서류 보완 필요 7건
신규 채용 준비 1건
컨택 대기 4건
응답 도착 2건
승인 대기 5건
행정사 검토 준비 2건
```

예를 들어 `서류 보완 필요 7건`을 클릭하면 업무 큐 테이블은 누락 서류가 있는 케이스만 보여준다.

### 5.5 업무 큐 테이블

| 업무 | 대상 | 사업장/라인 | 상태 | 기한 | 위험도 | 다음 처리 | 담당자 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| 체류기간 연장 서류 요청 | Nguyen V. | 화성 2공장 조립라인 | 우선 확인 | D-30 | 높음 | 초안 보기 | 김대리 |
| 신규 베트남 E-9 3명 채용 요청 | 송출회사 요청서 | 화성 1공장 조립라인 | 준비 중 | 이번 주 | 중간 | 요청서 보기 | 박대리 |
| 후보자 입국 전 서류 패키지 | Candidate A | 도장라인 | 승인 대기 | 5/20 | 중간 | 승인 요청 | 김대리 |
| 계약 종료 확인 | Tran T.H. | 포장라인 | 응답 도착 | 5/12 | 낮음 | 응답 요약 | 김대리 |

버튼 예시:

```text
처리하기
초안 보기
요청서 보기
승인 요청
응답 요약
검토 자료 만들기
```

## 6. 오른쪽 상세 패널 — 기존 구조 유지

오른쪽 상세 패널은 그대로 둔다. 이유는 데모에서 새 페이지로 이동하면 흐름이 끊기기 때문이다.

### 6.1 기본 구성

```text
제목
상태 배지
대상 정보
왜 확인이 필요한가요?
AI가 준비한 일
누락 서류
다음 할 일
판단 근거 / 업무 기록
대표 승인 요청 버튼
```

### 6.2 Nguyen 케이스 예시

```jsx
제목: Nguyen V. 체류기간 연장 서류 요청
상태: D-30 / 우선 확인 / 승인 전
대상: Nguyen Van A
국적: 베트남
체류자격: E-9
소속: 화성 2공장 조립라인
체류만료일: 2026.06.20
계약종료일: 2026.07.01
```

#### 왜 확인이 필요한가요?

```text
체류기간 만료까지 30일 남았고,
표준근로계약서 사본과 여권 사본이 아직 확인되지 않았습니다.
승인 없이 외부 발송은 진행되지 않습니다.
```

#### AI가 준비한 일

```text
베트남어 서류 요청 메시지 초안 생성
한국어 번역본 생성
행정사 검토용 자료 패키지 준비
이전 대화 기록 확인
예상 응답 시나리오 생성
```

#### 누락 서류

```text
표준근로계약서 사본
여권 사본
```

#### 다음 할 일

```text
Nguyen에게 서류 요청 메시지 발송 승인
응답 없을 경우 2일 뒤 리마인드 제안
서류 수신 후 행정사 검토 자료 생성
```

#### 판단 근거 / 업무 기록

```text
체류만료일: 2026.06.20
최근 서류 상태: 표준근로계약서 사본 누락, 여권 사본 누락
이전 메시지: 3일 전 서류 요청 기록 있음
승인 전 외부 발송 제한 적용
판단 기록: #4789
```

#### 하단 버튼

```text
[대표 승인 요청 보내기]
[행정사 검토 자료 만들기]
[판단 기록 보기]
```

## 7. 채용 준비 / 인력 확보 화면

이 화면은 데모2의 `인력 확보 Agent`를 보여주기 위해 필요하다.

### 7.1 화면 목적

`채용 준비` 탭은 근로자를 추천하는 화면이 아니다. 사업장이 신규 E-9 근로자를 뽑기 전에 확인해야 할 **쿼터, 구인노력, 신청 서류, 송출회사 요청서, 후보자 입국 전 서류 패키지**를 점검하는 화면이다.

### 7.2 화면 구성

```jsx
채용 준비
한별제조 · 화성 1공장 조립라인

[신규 베트남 E-9 3명 채용 요청]
현재 준비 상태 72%
잔여 쿼터: 5명까지 추가 고용 가능성을 검토할 수 있는 상태
최종 가능 여부: 고용센터 / 행정사 확인 필요

남은 작업
- 구인노력 기간 확인
- 고용허가 신청서 준비
- 송출회사 요청서 확인
- 후보자 입국 전 서류 패키지 준비

다음 처리
[요청서 보기]
[준비 체크리스트 보기]
[행정사 검토 요청]
```

### 7.3 체크리스트

| 항목 | 상태 | 설명 |
| --- | --- | --- |
| 고용 쿼터 확인 | 완료 | 현재 잔여 5명까지 검토 가능 |
| 내국인 구인노력 기간 | 진행 중 | 7일 기준 자동 산정 |
| 고용허가 신청서 | 초안 준비 | 관리자 검토 필요 |
| 송출회사 요청서 | 확인 필요 | 요청서 보기 필요 |
| 후보자 입국 전 서류 | 대기 | Candidate A 패키지 승인 필요 |

### 7.4 금지해야 할 표현

```text
좋은 후보 추천
성실한 근로자 예측
이탈 가능성 낮은 근로자 추천
국적별 선호도
성향 기반 채용 추천
```

대신 아래처럼 말한다.

```text
제도 요건 충족 여부
서류 준비 상태
쿼터 가능성 검토
행정 절차 준비도
승인 전 확인 필요
```

## 8. 메시지 관리 / 다국어 컨택 화면

이 화면은 데모2의 `다국어 컨택 Agent`를 보여주기 위해 필요하다.

### 8.1 화면 목적

`메시지 관리` 탭은 단순 채팅함이 아니다. 근로자에게 보낼 행정 안내문을 다국어로 준비하고, 근로자의 외국어 응답을 요약해서 업무 상태에 반영하는 화면이다.

### 8.2 화면 구성

```jsx
메시지 관리

왼쪽: 근로자별 대화 목록
- Nguyen V. / Zalo / 베트남어 / 초안 승인 대기
- Tran T.H. / Zalo / 베트남어 / 응답 도착
- Candidate A / SMS / 베트남어 / 입국 전 서류 요청 예정

오른쪽: 선택 스레드 상세
- 근로자 정보
- 채널
- 언어
- 최근 요청 이력
- VN/KR 메시지 초안
- 예상 응답 시나리오
- 판단 기록 연결
- 승인 후 상태 변화
```

### 8.3 Nguyen 메시지 초안 예시

```text
KR
안녕하세요 Nguyen 씨,
체류기간 연장 준비를 위해 표준근로계약서 사본과 여권 사본을 보내주세요.
감사합니다.

VN
Xin chào Nguyen,
vui lòng gửi bản sao hợp đồng lao động tiêu chuẩn và hộ chiếu để chuẩn bị gia hạn thời gian lưu trú.
Cảm ơn bạn.
```

### 8.4 응답 해석 예시

근로자 응답:

```text
Tôi đã gửi ảnh hộ chiếu qua Zalo rồi nhé.
```

AI 요약:

```text
Nguyen이 여권 사진을 Zalo로 이미 보냈다고 응답했습니다.
서류 상태를 ‘여권 사본 수신 확인 필요’로 업데이트합니다.
표준근로계약서 사본은 아직 미수신 상태입니다.
```

상태 변화:

```text
이전: 누락 서류 2건
변경: 여권 사본 수신 확인 필요 / 표준근로계약서 사본 누락
다음: 표준근로계약서 사본 리마인드 제안
```

## 9. 행정사 검토 화면

이 화면은 `비자·서류 운영 Agent`의 결과물을 외부 전문가에게 넘기기 전 확인하는 곳이다.

### 9.1 화면 목적

행정사 검토 화면은 AI가 법적 판단을 확정하는 화면이 아니다. 행정사가 빠르게 검토할 수 있도록 **기초 자료, 누락 서류 상태, 판단 기록, 메시지 이력**을 패키지로 묶는 화면이다.

### 9.2 화면 구성

```jsx
행정사 검토

검토 대기 패키지
- Nguyen V. 체류기간 연장 서류 패키지
- Candidate A 입국 전 서류 패키지
- Tran T.H. 계약 종료 검토 패키지

패키지 상세
- 근로자 기본 정보
- 체류자격 / 체류만료일 / 계약종료일
- 제출 가능 서류 목록
- 누락 서류 목록
- 근로자 컨택 이력
- 판단 기록
- 관리자 승인 상태

액션
[PDF 미리보기]
[행정사 전달]
[보완 요청]
```

### 9.3 주의 문구

```text
AI는 행정 절차의 최종 판단이나 서류 제출을 대행하지 않습니다.
본 자료는 행정사 / 담당자 검토를 위한 기초 패키지입니다.
```

모바일은 챗봇형보다 카드형 브리핑이 먼저 나와야 한다. 대표자는 긴 대화를 하고 싶은 사람이 아니라, 오늘 승인할 일을 빠르게 보고 결정하려는 사람이다.

```jsx

│ Candidate A 입국 전 서류 패키지 │
│ 행정사 검토 전 확인 필요 │
```

## 12. Live Agent 승인 직전 화면

Live Agent 화면은 메인 화면이 아니라 **승인 직전 화면**이다.


```jsx
Multilingual Contact Agent

   Nguyen Van A · 베트남 · E-9 · Zalo

   베트남어 원문 + 한국어 번역 생성

   대표님 승인 후 근로자에게 발송됩니다.
```

## 16. 구현 우선순위

### 1순위 — PC 탭에 `채용 준비`, `메시지 관리` 추가

이 두 탭이 없으면 데모2의 세 축 중 `인력 확보 Agent`와 `다국어 컨택 Agent`가 화면에서 보이지 않는다. 현재 외고반장이 단순 비자/서류 알림툴처럼 보일 수 있으므로 가장 먼저 추가한다.

### 2순위 — 모바일을 카드형 브리핑으로 변경

대표자는 챗봇과 대화하는 사람이 아니라 승인할 사람이다. 모바일 첫 화면은 채팅 입력창보다 **오늘 브리핑 + 승인 카드 3개**가 먼저 보여야 한다.

### 3순위 — Nguyen 카드 클릭 → Live Agent 화면 연결

데모의 와우 포인트다. 초안 보기만 보여주면 평범한 번역/메시지 도구처럼 보인다. Live Agent 화면에서 “AI가 무엇을 확인했고, 왜 승인 대기인지”가 보여야 agentic workflow처럼 보인다.

### 4순위 — Evidence Log → 판단 기록 UI 텍스트 정리

한국어 제품 톤에서는 Evidence Log가 튄다. 코드명은 남겨도 화면 텍스트는 `판단 기록`으로 바꾼다.

### 5순위 — 신규 채용 요청 데이터 추가

`신규 베트남 E-9 3명 채용 요청`, `Candidate A 입국 전 서류 패키지`, `송출회사 요청서` 샘플 데이터가 있어야 채용 준비 화면이 설득력을 갖는다.
