# 2단계 — 푸시 알림 카탈로그 + 딥링크 맵 (v1)

> 작성일: 2026-07-04
> 전제: D5(앱 푸시 + 카카오 알림톡 병행), D1(담당자=운영, 대표=승인), 1단계 화면 스펙 M1~M9의 진입 경로와 1:1 대응
> 원칙: 알림은 모바일 퍼스트의 실질적 진입점이다. 그러나 **알림 피로 = 제품 이탈**이므로, 실시간으로 쏘는 것과 아침 브리핑에 묶는 것을 엄격히 구분한다.

---

## 1. 채널·수신자 라우팅 원칙

| 수신자 | 기본 채널 | 폴백 | 근거 |
|---|---|---|---|
| 담당자(manager) | 앱 푸시 | 푸시 미등록/미도달 24h → 알림톡 | 매일 쓰는 사용자, 앱 설치 전제 |
| 대표(owner) | 앱 푸시 (설치 시) | 미설치 → 알림톡 기본 | 승인 루프가 끊기면 안 됨 |
| 행정사 | 이메일 (패키지 링크) | — | 앱 사용자 아님 (수신자) |
| 근로자 | Zalo/카카오 등 컨택 채널 | — | 본 카탈로그 범위 밖 (컨택 에이전트 발송, 승인 게이트 통과 후) |

채널별 성격: **푸시 = 딥링크 진입용 트리거** (본문 짧게), **알림톡 = 앱 미설치자용 요약 + 웹 링크** (심사 통과 가능한 정보성 문구, 버튼 최대 2개).

## 2. 발송 등급 (Priority Class)

| 등급 | 정책 | 해당 이벤트 |
|---|---|---|
| P1 즉시 | 발생 즉시 발송. 야간 제한의 예외는 CRITICAL만 | 승인 요청, 응답 도착, CRITICAL 리스크, 에이전트 질문 대기 |
| P2 다이제스트 | 실시간 발송 금지. 다음 아침 브리핑 알림에 합산 | D-day 임계 진입(HIGH 이하), 누락서류 감지, 리마인드 제안 |
| P3 주간 | 주 1회 묶음 (월요일 아침) | citation 노후화, 온보딩 미완료 리마인드 |

핵심 규칙: **P2를 실시간으로 쏘지 않는 것**이 이 설계의 절반이다. D-30 진입은 새벽에 발생해도 "아침 브리핑 3건" 안에 포함되면 충분하다.

## 3. 딥링크 스킴 맵

```
앱:      oegobanjang://{path}
알림톡:  https://app.oegobanjang.kr/m/{path}?nk={notification_key}   ← 웹 진입 후 앱 유도 배너
```

| path | 목적지 화면 | 파라미터 |
|---|---|---|
| `briefing` | M1 오늘 브리핑 | `date?` |
| `case/{caseId}` | M2 케이스 바텀시트 (M1 위에 오버레이) | — |
| `case/{caseId}/draft` | M3 초안 상세 | `lang?` |
| `case/{caseId}/approve` | M4 승인 직전 (M3 경유 스킵) | 승인 권한자만, 권한 없으면 M2로 강등 |
| `response/{threadId}` | M6 응답 도착 | — |
| `cases?filter={preset}` | M7 케이스 목록 (필터 적용) | `severity`, `type` |
| `evidence/{eventId}` | M8 해당 이벤트 하이라이트 | — |
| `run/{runId}` | M9 에이전트 런 복귀 | — |
| `onboarding/workers` | 근로자 등록 (3단계 설계 예정) | — |

공통 규칙: 딥링크 진입 시 백스택은 항상 `M1 → 목적지`로 구성 (뒤로가기가 브리핑으로 떨어지게). 만료된 대상(이미 승인됨/삭제됨)이면 M1 + 토스트 "이미 처리된 업무입니다".

## 4. 알림 카탈로그 (이벤트 → 수신자 → 채널 → 문구 → 딥링크)

### 4.1 P1 즉시 발송

| # | 이벤트 (트리거) | 수신자 | 푸시 문구 (제목/본문) | 알림톡 문구 | 딥링크 |
|---|---|---|---|---|---|
| N01 | `approval_requested` — 승인 필요 액션 생성 | 승인 권한자(대표 또는 담당자) | **승인 요청 1건** / Nguyen V. 서류 요청 메시지 · D-30 · 승인 전 발송 차단 중 | [외고반장] 승인 요청이 있습니다.\n대상: Nguyen V. 서류 요청\n체류만료 D-30 · 누락 서류 2건\n승인 전에는 외부로 발송되지 않습니다.\n▶ 내용 확인 | `case/{id}/approve` |
| N02 | `worker_replied` — 근로자 응답 수신 | 담당자 | **응답 도착** / Nguyen V.님이 답했습니다 · 요약 확인 필요 | [외고반장] Nguyen V.님의 응답이 도착했습니다.\nAI 요약을 확인하고 상태를 업데이트해 주세요.\n▶ 응답 확인 | `response/{threadId}` |
| N03 | `risk_flagged(CRITICAL)` — 체류/계약/신고기한 경과 | 담당자 + 대표 | **즉시 확인 필요** / Tran T. 체류기간이 만료되었습니다 (D+2) | [외고반장] 즉시 확인이 필요합니다.\nTran T. 체류기간 경과 (D+2)\n행정사 검토 준비가 필요합니다.\n▶ 케이스 확인 | `case/{id}` |
| N04 | `agent_question` — 에이전트 런 interrupt | 런 시작자 | **AI가 확인을 기다립니다** / "어느 사업장 기준으로 정리할까요?" | (알림톡 미발송 — 런은 앱 세션 전제) | `run/{runId}` |
| N05 | `run_completed` — **사용자가 시작한** 백그라운드 런 완료 | 런 시작자 | **요청 처리 완료** / 결과 3건 · 승인 필요 1건 | (미발송) | `run/{runId}` |
| N05b | `proactive_run_completed` — 이벤트 트리거 프로액티브 런 완료 (v1.2) | — | **개별 푸시 발송 금지.** 산출물은 브리핑 카드의 "AI가 준비를 마쳤습니다" 상태로만 노출 — 알림 피로 방지. 단, 런이 승인 요청을 생성하면 그 승인 요청이 N01로 발송됨 | — | (카드 경유) |
| N06 | `approval_decided` — 승인/반려 결정됨 | 요청 생성 담당자 | **승인 완료** / 대표님이 Nguyen 서류 요청을 승인했습니다 (반려 시: 반려 사유 확인) | (미발송 — 담당자는 푸시 전제) | `evidence/{eventId}` |
| N07 | `handoff_ready` — 행정사 패키지 준비 완료 | 담당자 | **검토 자료 준비 완료** / Nguyen 체류연장 패키지 · 전달 전 확인 필요 | [외고반장] 행정사 검토 자료가 준비되었습니다.\n전달 전 담당자 확인이 필요합니다.\n▶ 자료 확인 | `case/{id}` |

### 4.2 P2 아침 다이제스트 (스케줄러 발송, 기본 08:30)

| # | 이벤트 | 수신자 | 문구 | 딥링크 |
|---|---|---|---|---|
| N10 | `daily_briefing` — 아침 브리핑 생성 (P2 전체를 합산하는 컨테이너) | 담당자, 대표 | **오늘 브리핑** / 승인 필요 2건 · 우선 확인 1건 · 응답 대기 1건 — 알림톡: [외고반장] {회사명} 오늘 브리핑\n승인 필요 2건, 우선 확인 1건이 있습니다.\n▶ 브리핑 보기 | `briefing` |
| N11 | `risk_flagged(HIGH)` — D-30 진입 등 | (N10에 합산) | 브리핑 본문 카드로만 노출, 개별 알림 없음 | — |
| N12 | `missing_document` — 누락서류 감지 | (N10에 합산) | 〃 | — |
| N13 | `reminder_proposed` — 응답 지연 D+2 리마인드 제안 | (N10에 합산) | 브리핑 카드 "리마인드 제안 1건 · 승인 필요" | `case/{id}` |
| N14 | `contract_visa_conflict` — 충돌 감지 | (N10에 합산, 단 severity가 CRITICAL로 계산되면 N03으로 승격) | 〃 | — |

다이제스트 규칙: 합산 건수 0이면 **알림 자체를 발송하지 않는다** (빈 브리핑 알림 금지). 대표에게는 승인 필요 건수만 있는 축약 문구.

## 5. 발송 시간·빈도 룰

### 5.1 시간 창

| 룰 | 값 | 예외 |
|---|---|---|
| 야간 금지 | 21:00 ~ 익일 08:00 발송 보류 → 08:30 다이제스트에 합류 | CRITICAL(N03)만 22:00까지 허용, 그 이후는 익일 08:00 최우선 발송 |
| 휴일·주말 | P1 중 N01(승인)·N02(응답)는 정상 발송, 나머지는 다음 영업일로 이월 | CRITICAL은 예외 없이 발송 |
| 아침 다이제스트 | 기본 08:30, 회사 timezone 기준, 회사별 설정 가능 (Daily Briefing PRD의 scheduler 준용) | — |

### 5.2 빈도·중복 제어

| 룰 | 정의 |
|---|---|
| 이벤트 idempotency | 같은 case + 같은 event_type + 같은 임계값은 1회만. D-30 알림 후 D-29에 재발송 금지 (다음 임계 D-7에서만 재발송) |
| 리마인드 쿨다운 | 같은 케이스의 재촉성 알림(N01 미응답 재알림 포함)은 **24시간 내 재발송 금지** |
| 승인 미응답 재알림 | N01 후 48시간 미응답 → 1회 재발송 ("승인 대기 중 · 2일 경과"). 이후는 다이제스트에만 표시. 에스컬레이션(대리 승인)은 7단계에서 설계 |
| 일일 상한 | 수신자당 P1 최대 5건/일. 초과분은 6번째부터 묶음 "그 외 승인 요청 N건" 1건으로 병합 |
| 채널 중복 금지 | 푸시 도달 확인 시 같은 이벤트의 알림톡 미발송. 푸시 미도달(미등록 포함) 판정 시에만 알림톡 폴백 |

### 5.3 문구 안전 규칙 (전 알림 공통)

- 단정 표현 금지: "만료됩니다 → 조치하세요"(X) / "확인이 필요합니다"(O)
- 알림 본문에 PII 금지: 외국인등록번호·여권번호·연락처 노출 불가. 이름은 표시명(마스킹 규칙 준용)만
- 승인 관련 알림에는 안전 문구 포함: "승인 전에는 외부로 발송되지 않습니다" (알림톡 필수, 푸시는 본문 여유 시)
- 모든 발송은 EvidenceEvent(`notification_sent`)로 기록: 수신자·채널·이벤트·도달 여부

## 6. 알림 설정 화면 요구사항 (설정 > 알림)

| 항목 | 기본값 | 비고 |
|---|---|---|
| 아침 브리핑 시각 | 08:30 | 회사 단위 설정 (개인 아님) |
| 승인 요청 즉시 알림 | ON | 승인 권한자만 노출, OFF 불가(강제) — 승인 루프는 꺼질 수 없음 |
| 응답 도착 즉시 알림 | ON | OFF 시 다이제스트로 강등 |
| CRITICAL 야간 알림 | ON | OFF 가능하되 경고 문구 표시 |
| 주간 요약 | OFF | 옵트인 |
| 채널 우선순위 | 푸시 우선 | 알림톡 폴백 여부 표시 (읽기 전용) |

## 7. 이벤트 소스 매핑 (구현 참조)

| 알림 | 발생 소스 | 기존 문서 근거 |
|---|---|---|
| N01, N06 | Approval.status 전이 (pending 생성 / approved·rejected) | PRD §11.3 NextAction, M4/M5 스펙 |
| N02 | 컨택 스레드 inbound 수신 → 응답 해석 에이전트 큐 | M6 스펙, 화면구성 v2 §8 |
| N03, N11, N14 | Risk Rule Engine 일일 계산 + 이벤트 감지 | Daily Briefing PRD §9 severity 표 |
| N04, N05 | Agent Run 상태 전이 (interrupt / completed) | M9 스펙, D3 결정 |
| N10 | Scheduler (회사별 cron) | PRD P0 Scheduler |
| N13 | NextAction.state=scheduled 도래 (리마인드 쿨다운 검사 후) | 룰베이스 메모 §3 발송 채널/시간 룰 |
| N20 | citation validation worker | PRD P1 citation refresh queue |

## 8. 미해결 → 다음 단계로

- 알림톡 템플릿 심사용 최종 문구 확정 (변수 자리 `#{이름}` 포맷으로 재작성 필요 — 카카오 심사 규격)
- 대표 다수(공동대표) 시 N01 수신자 정책 → 7단계 권한 모델과 함께 결정
- 근로자 0명 계정의 알림 전면 mute 여부 → 3단계 온보딩과 함께 결정
- 웹 진입(`/m/{path}`) 후 앱 설치 유도 배너의 강도 → 5단계 프로토타입에서 검증
