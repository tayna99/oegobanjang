## 1. 기존 계획에서 확정된 Agentic OS 핵심 (변하지 않는 뼈대)

여러 문서에 반복적으로 등장하며 이미 합의된 것들. 모바일 퍼스트로 가도 이 뼈대는 유지한다.

### 1.1 운영 루프 (제품의 심장)

```
Risk 감지(Rule Engine)
→ 근거 연결(RAG / Citation)
→ Next Action 생성(버튼)
→ Human Approval(승인 게이트)
→ Evidence Log(감사 기록)
```

- 출처: Daily Briefing PRD, 전체 PRD v2, MVP v2 종합본
- 핵심 철학: **자동 제출이 아니라 자동 준비.** AI는 경로를 짜고 초안을 만들고, 사람 승인 후 다음 단계로 넘긴다.

### 1.2 에이전트 구조

- 외부 노출: **3개 Mission Agent** — 인력 확보 / 다국어 컨택 / 비자·서류 운영
- 내부 구현: **8개 세부 에이전트** (요건매칭, 후보적합성, 컨택온보딩, 응답해석, 비자체류, 서류패키징, 승인전달, 감사로그)
- 공통 엔진: Intent Router → Planner → Tool Executor → Human Approval → Evidence Log
- 역할 분리: RAG=공식 근거, DB=운영 상태, Rule=날짜/충돌 계산, LLM=구조화·요약·초안, Human=최종 승인

### 1.3 안전 가드레일 (전 화면 공통)

- 승인 전 외부 발송/전달 구조적 차단 + 고정 문구 "승인 전에는 외부 발송이 차단됩니다"
- 단정 표현 금지 (가능/불가능 → "검토 필요 / 다음 안전한 행동은…")
- 금지 UI: 비자 확정 버튼, 성실도/이탈 점수, 국적별 정렬, PII 원문 표시, 자동 발송 버튼
- Evidence Log는 INSERT-only, PII 마스킹, trace_id 기반 재현

### 1.4 이미 그려져 있던 모바일 설계 (재사용 자산)

- 전체 PRD v2의 **M1~M4 스펙**: 오늘 브리핑 / 케이스 바텀시트 / 승인 완료 / 표현 가이드
- 화면구성 v2의 **모바일 5단 흐름**: 오늘 브리핑 → 상세 확인 → AI 처리 과정(Live Agent) → 메시지 초안(VN/KR) → 승인 완료
- 목업에서 확정된 요소: 하단 탭(브리핑/케이스/메시지/기록), D-day·누락·승인필요 배지, Live Agent 단계 타임라인, 자율성 표시("현재 자율성: Medium — 승인 필요"), 판단 기록 #번호
- 모바일 컴포넌트 리스트: MobileBriefingShell, MobileApprovalTaskCard, MobileDraftDetail, LiveAgentProgressScreen, ScenarioCards, ApprovalActionBar, ApprovalCompleteState
- 디자인 시스템: ~~Pretendard Variable + WDS 컬러 토큰~~ → **v1.1 개정: 사용자 제공 DESIGN.md(SOCAR 스타일) 채택.** 무채색 크롬 + 브랜드 블루 #0066FF는 primary CTA 전용, radius 12/14/16 사다리. 위험도 색상·D-day 뱃지 규칙은 기능색으로 유지 (4단계 토큰 문서 참조)

## 2. 모바일 퍼스트 재설계 — 통합 설계

기존 문서는 "PC=담당자 작업대, 모바일=사장님 승인 채널"의 보조 관계였다.
모바일 퍼스트 전환의 핵심 선언은 다음과 같다.

> **모바일이 제품의 본체다. 에이전트가 일하고, 사람은 모바일에서 확인·승인한다. PC는 대량 데이터 관리와 정밀 검토를 위한 확장 화면이다.**

### 2.1 설계 원칙 5가지

1. **Zero-Input First** — 홈은 검색창/메뉴가 아니라 에이전트가 이미 만들어 둔 오늘 브리핑이다. 사용자는 입력하지 않아도 오늘 할 일 Top 3를 받는다.
2. **카드 → 바텀시트 → 승인, 3뎁스 이내** — 페이지 이동 최소화. 모든 상세는 바텀시트로 끝내고, 처리 후 브리핑 홈으로 복귀한다.
3. **에이전트 가시성(Agent Visibility)** — AI가 무엇을 확인했고(1~4단계 타임라인), 어디서 멈춰 있는지(승인 대기)를 항상 보여준다. Live Agent 화면은 메인이 아니라 **승인 직전 화면**이다.
4. **승인이 곧 인터랙션 문법** — 모바일의 1차 행동은 탐색이 아니라 [초안 보기]→[승인/반려/수정 요청]. 모든 카드 CTA는 2개로 고정.
5. **표현 안전** — 발송/완료/확정 대신 초안/검토/준비/승인 대기. 모든 승인 지점에 근거(citation) 1~2개 노출.

### 2.2 정보 구조 (모바일 IA)

```
외고반장 (모바일 본체)
├── ① 브리핑 (홈)          ← 오늘 승인 필요 1~3건 + 요약 통계
├── ② 케이스               ← 근로자/채용 케이스 목록 (위험도·D-day 필터)
├── ③ 메시지               ← 다국어 컨택 스레드 + 응답 해석
├── ④ 기록                 ← 판단 기록(Evidence) 타임라인
└── [플로팅] AI에게 요청     ← 자연어 커맨드 바 (Intent Router 진입점)

PC 확장 (세컨드 스크린)
├── 업무 큐 테이블 · CSV 업로드 · 행정사 검토 패키지 편집 · Admin/메트릭
```

목업(태나ver, GPT 0513)에서 이미 두 가지 탭 구성이 존재했다: `브리핑/케이스/메시지/기록` vs `브리핑/승인/업무기록/설정`. 통합안은 **브리핑/케이스/메시지/기록 4탭 + 설정은 프로필 진입**으로 정리한다. 승인은 별도 탭이 아니라 브리핑 홈의 1차 콘텐츠이므로 탭에서 제외.

### 2.3 화면 세트 (M-시리즈 확장)

| 화면 | 목적 | 핵심 요소 | 출처 |
|---|---|---|---|
| M1. 오늘 브리핑 (홈) | 승인 필요 1~3건 즉시 처리 | 인사말 배너, 승인 카드(제목/D-day/누락/승인필요 배지 + CTA 2개), 요약 통계 카드, 안전 고정 문구 | PRD v2 M1, 목업 확정 |
| M2. 케이스 바텀시트 | 승인 전 판단 정보만 짧게 | 요약 1문장, 누락 서류 체크리스트, AI가 확인한 내용, 근거 링크 1~2개, [초안 보기][승인하기] | PRD v2 M2 |
| M3. 초안 상세 | VN/KR 초안 검토 | 언어 토글, AI 확인 내용, 예상 응답 시나리오 3종(긍정/추가질문/지연), [수정 요청][보내기 승인] | 화면구성 v2 §11, 프로토타입 |
| M4. Live Agent 승인 직전 | 에이전트 작업 과정 투명화 | 단계 타임라인(프로필→이력→초안→승인대기), 자율성 표시, 판단기록 #, [최종 승인] | 화면구성 v2 §12, 화면_모바일1 |
| M5. 승인 완료 | 상태 변화 체감 | 토스트 "승인 완료 · 판단 기록 #xxxx 저장", 다음 할 일 1줄, 브리핑 복귀 | PRD v2 M3 |
| M6. 응답 도착 | 응답 해석 → 상태 업데이트 | 원문+한국어 요약, 상태 업데이트 후보, 새 추천 액션(추천 이유 포함) | 프로토타입 Claude_00 |
| M7. 케이스 목록 | 브리핑 밖 전체 현황 | 위험도/유형 필터, 케이스 카드(D-day, citation 등급 배지) | UIUX 기획정리 4-5 신규 모바일화 |
| M8. 판단 기록 | 감사 가능성 10초 전달 | 이벤트 타임라인(risk_flagged → approval_granted…), 마스킹 식별자 | PRD v2 S5 모바일화 |

### 2.4 에이전트 상태 모델 (UI에 그대로 노출)

```
Case.state:       draft → risk_review → approval_pending → human_approved → completed / blocked
NextAction.state: locked(승인 필요) / ready / scheduled / waiting
Approval.status:  pending / approved / rejected / cancelled
자율성(autonomy):  low risk → 승인 후 발송 / mid → 담당자 검토 필수 / high → 행정사 전달 강제
```

모바일에서 이 상태는 배지 색으로 직결: 승인대기=노랑, 승인완료=초록, 차단/CRITICAL=빨강, HIGH=주황, 예정=파랑.

### 2.5 알림 → 딥링크 진입 (모바일 퍼스트의 실제 진입점)

모바일 퍼스트에서 진짜 홈은 앱 아이콘이 아니라 **푸시 알림**이다.

```
아침 스케줄러: "오늘 승인 필요 3건" → 탭 → M1 브리핑
이벤트 발생:  "Nguyen D-30 진입" → 탭 → M2 해당 케이스 바텀시트
응답 도착:    "Nguyen 응답 도착" → 탭 → M6 응답 해석
```

### 2.6 역할별 모드 (한 앱, 두 프로필)

| | 대표(사장) 모드 | 담당자 모드 |
|---|---|---|
| 홈 | 승인 필요 카드만 (1~3건) | 승인 + 서류/컨택 진행 현황 |
| 케이스 탭 | 요약 카드 위주 | 필터·검색 포함 전체 목록 |
| 권한 | 승인/반려/수정요청 | 초안 편집, 케이스 생성, 행정사 패키지 준비 |
| PC 필요성 | 없음 | CSV 업로드·패키지 편집 시 PC 권장 |

행정사는 앱 사용자가 아니라 **수신자**: 승인된 패키지를 링크/PDF로 받는다 (Handoff Preview).

## 3. 설계 누락 항목 (기존 문서 어디에도 없거나 미완인 것)

우선순위 순.

**P0 — 이것 없이는 모바일 퍼스트가 성립 안 됨**

1. **푸시 알림 설계 전체** — 알림 종류/문구/빈도/딥링크 맵이 어느 문서에도 없다. UIUX 기획정리에서도 "알림/푸시 설정 화면 미구현"으로만 표시.
2. **온보딩 + 최초 데이터 입력의 모바일 경로** — CSV 업로드는 PC 전제. 모바일에서 근로자 등록(수기 입력, 서류 사진 촬영→OCR 등) 경로가 미설계. 첫 브리핑이 비어 있으면 제품 가치가 전달 안 된다.
3. **로그인/인증** — auth API만 있고 화면 미구현. 모바일이면 생체 인증, 승인 행위의 본인 확인(승인=법적 책임 이력) 설계 필요.
4. **빈 상태/에러/오프라인 상태** — "오늘 승인할 것 없음" 화면, 네트워크 불안정 시 동작(문서엔 데모 배지만 존재). 현장 사업장의 네트워크 제약은 사용자 환경 제약에 이미 명시돼 있으나 대응 설계가 없다.

**P1 — 초기 버전에서 곧 부딪힘**

5. **자연어 커맨드(Intent Router)의 모바일 위치** — GPT 목업엔 "AI에게 요청하기" 입력창이 있으나 M1~M4 스펙에는 없음. 커맨드 바 vs 채팅 탭 vs 제외, 결정 필요.
6. **승인 위임/부재 처리** — 대표가 승인 못 하는 상황(휴가, 미응답 N일)의 대리 승인·에스컬레이션 규칙.
7. **리마인드/예약된 액션의 가시성** — "2일 뒤 리마인드 예정"이 어디서 확인·취소되는지 미정의 (NextAction.state=scheduled의 UI).
8. **역할 전환 UX** — 사장/담당자 권한 모델은 P2 RBAC로 미뤄져 있으나, 모바일 퍼스트에서는 초기부터 최소 2역할 분기가 필요.
9. **케이스 최소 단위 미결정** — PRD v2 오픈 이슈 그대로: 근로자 단위 vs 요청 단위. 모바일 카드 설계의 전제 조건.

**P2 — 확장 시점**

10. 행정사 패키지 산출물 포맷(이메일/PDF/링크) — 오픈 이슈 미해결.
11. 컨택 채널 확정(Zalo/Kakao/SMS)과 실발송 연동 시점.
12. 접근성 — 40~60대 사장 타깃인데 폰트 스케일/터치 타깃 기준 없음.
13. 근로자 접점 — 근로자는 앱을 안 쓰지만, 서류 제출용 웹뷰 링크(모바일 업로드) 같은 최소 접점 검토 여지.
14. 반응형 breakpoint 가이드 — "PC/모바일 분기만 있음" 상태 그대로.

## 4. 앞으로 나(사용자)가 Claude에게 물어봐야 할 설계 질문 로드맵

### 4.1 먼저 결정해야 할 제품 질문 (답이 설계를 가른다)

1. **1순위 사용자는 누구인가?** 대표(승인만) vs 담당자(운영까지). → 홈 화면과 탭 구성이 달라짐. 권장: 담당자 겸용 단일 앱 + 역할별 홈 분기.
2. **케이스의 최소 단위는?** 근로자 단위 vs 요청 단위. → 카드/바텀시트/딥링크 설계의 전제.
3. **자연어 입력을 모바일 1급 기능으로 둘 것인가?** 커맨드 바(보조) vs 채팅 탭(1급) vs 제외(승인 전용).
4. **PC는 반응형 웹 하나로 갈 것인가, 별도 콘솔인가?** → 컴포넌트 시스템 분리 여부 결정.
5. **알림 채널은?** 앱 푸시 vs 카카오 알림톡 vs 병행. (사장님이 앱을 설치 안 하면 알림톡이 현실적)
6. **첫 데이터 입력 방식은?** CSV(PC) / 모바일 수기 / 서류 사진 OCR / 행정사가 대신 세팅.

### 4.2 결정 후 Claude에게 순서대로 요청할 작업

```
1단계  "M1~M8 화면별 상태 스펙 만들어줘"
        (기본/빈/로딩/에러/오프라인 상태 + 컴포넌트 props 정의)
2단계  "푸시 알림 카탈로그와 딥링크 맵 설계해줘"
        (이벤트→알림 문구→진입 화면 매핑, 야간 발송 제한 룰 포함)
3단계  "온보딩 플로우 설계해줘"
        (가입→사업장 등록→근로자 최초 입력→첫 브리핑 도달까지)
4단계  "모바일 디자인 토큰 정리해줘"
        (기존 WDS/Pretendard 기반, spacing/타이포 스케일/터치 타깃 추가)
5단계  "인터랙티브 HTML 프로토타입 만들어줘"
        (M1→M2→M3→M4→M5 승인 플로우 1개 완주, mock 데이터는 화면구성 v2 §14 재사용)
6단계  "기존 React 코드(oegobanjang-ui)와의 갭 분석해줘"
        (재사용 가능 컴포넌트 vs 새로 만들 것)
7단계  "역할별 권한 모델과 승인 위임 규칙 설계해줘"
8단계  "데모 시나리오를 모바일 퍼스트 버전으로 다시 써줘"
        (기존 데모2 시연흐름 → 알림 진입 시작으로 재구성)
```

### 4.3 각 단계에서 Claude가 되물을 기준 (미리 생각해두면 빠름)

- 브리핑 카드는 몇 건까지? (기존 합의: 1~3건 고정이 데모 친화적)
- Live Agent 화면을 실서비스에도 유지? (데모 와우 포인트 vs 실사용 시 단계 생략 옵션)
- 승인 완료 후 실제 발송 시점 (즉시 vs 담당자 2차 확인 — 기존 문서는 "발송은 MVP 범위 밖, 승인까지만")
- Evidence 기록을 사용자에게 얼마나 노출? (기록 탭 전체 공개 vs 케이스 내 요약만)

### 5.1 D3 상세 — 자연어 입력을 "진짜 에이전트"로 만드는 구조

요구: LLM이 스스로 판단해 질문 → 생각 → 도구 호출 → 결과까지 진행해야 한다.

**결론: "채팅 vs 커맨드 바"는 입력 표면의 문제일 뿐이다. 에이전트다움은 그 아래 실행 구조에서 나온다. 권장안은 하이브리드 — 커맨드 바로 진입하고, 실행은 에이전트 런 스레드로 펼친다.**

```
[입력]   브리핑 홈 하단 커맨드 바 (상시 노출, Zero-Input 원칙 유지)
   ↓ 제출
[실행]   에이전트 런(Agent Run) 풀스크린 스레드
         = 기존 LiveAgentProgressScreen을 범용화한 화면
         - 단계가 실시간 스트리밍: 생각 → 도구 호출(이름+요약) → 관찰 → 다음 행동
         - 에이전트가 정보가 부족하면 [질문 카드]로 사용자에게 되물음 (interrupt)
   ↓ 완료
[결과]   Next Action 카드 (승인 필요 배지 포함) + 판단 기록 # 저장
         → 승인하면 기존 M4→M5 승인 플로우로 합류
```

기술 구조 (기존 문서 자산과의 연결):

1. **런타임**: LangGraph `createAgent` 기반 tool-calling loop. `03_체크리스트/LangChain전환_04_createAgent전환_체크리스트`와 `도구_미들웨어_에이전트_패턴` 문서가 그대로 기반이 된다.
2. **도구(Tools)**: 에이전트에게 주는 손발. 예 — `get_worker_status`, `calc_dday`, `search_regulation(RAG)`, `check_document_gap`, `generate_message_draft`, `create_case`, `request_approval`, `build_handoff_package`.
3. **가드레일은 모델이 아니라 도구에 건다**: 루프는 자유롭게 돌되, 부작용이 있는 도구가 경계를 강제한다.
   - `send_message` 같은 외부 발송 도구는 존재하지 않거나, 호출 시 무조건 `pending_approval`을 반환 (승인 전 발송 구조적 차단 유지)
   - high-risk 의도는 미들웨어가 `request_expert_handoff`로 강제 라우팅
   - 모든 도구 호출에 tenant scope + PII 마스킹 미들웨어, 결과는 EvidenceEvent로 append
4. **기존 PRD와의 정합**: Daily Briefing PRD는 "LLM이 도구를 마음대로 실행하지 않는다(Bounded Planner)"였다. 이 결정은 그 철학을 폐기하는 게 아니라 이동시키는 것 — **자유는 루프에, 경계는 도구에.** 결정적 계산(D-day, 서류 비교)은 여전히 rule 기반 도구가 수행하고, LLM은 어떤 도구를 언제 쓸지만 판단한다.
   **(v1.2 추가) 그리고 트리거는 이벤트에.** 런은 사용자만 시작하는 게 아니라 사건(D-30 진입, 응답 도착)이 시작한다. 프로액티브 런은 읽기+초안 생성까지만 허용되고 종착점은 항상 승인 요청 — "자동 준비의 확대, 자동 실행의 금지". 상세는 9단계 문서.
5. **UI 재사용**: 목업의 Live Agent 화면(단계 타임라인 + 자율성 표시 + 판단기록 #)이 이미 에이전트 런 화면의 정답 형태다. 승인 직전 전용에서 "모든 에이전트 실행의 공통 화면"으로 승격하면 된다.

채팅 탭을 따로 두지 않는 이유: 상시 채팅은 "빈 입력창에서 시작하는 챗봇" 인상을 주고 Zero-Input First와 충돌한다. 대화의 연속성이 필요한 경우는 에이전트 런 스레드 안에서 후속 질문으로 처리한다 (런 = 1회성 대화 스레드).

## 6. 한 줄 결론

> 외고반장 모바일 퍼스트 재설계 = **"아침 푸시 → 오늘 브리핑 카드 → 바텀시트 확인 → Live Agent 투명화 → 승인 → 판단 기록"** 루프를 모바일 본체로 삼고, 기존 Agentic OS 뼈대(3 Mission/8 Sub Agent, Rule+RAG+HITL, Evidence Log)를 그 아래에 그대로 배치하는 것.

기존 자산 중 M1~M4 스펙, 모바일 5단 흐름 목업, 컴포넌트 리스트, mock 데이터 구조는 그대로 재사용 가능하며, 새로 설계할 것은 **알림, 온보딩, 인증, 빈/에러 상태, 역할 분기** 다섯 가지다.
