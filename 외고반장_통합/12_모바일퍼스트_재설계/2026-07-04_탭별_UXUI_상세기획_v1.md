# 탭별 UX·UI 상세 기획 — 브리핑 / 케이스 / 메시지 / 기록 (v1)

> 작성일: 2026-07-04
> 전제: 1단계 스펙 v1.2(상태·컴포넌트), 4단계 토큰(SOCAR), 프로토타입 v2 반영본 기준
> 이 문서의 역할: 1단계가 "무엇이 있는가(상태·데이터)"라면, 이 문서는 "어떻게 보이고 움직이는가(위계·간격·제스처·마이크로카피)"를 정의한다.

## 0. 탭 공통 문법

### 0.1 탭의 성격 구분 — 앱의 멘탈 모델

| 탭 | 성격 | 사용자의 질문 | 체류 시간 |
|---|---|---|---|
| 브리핑 | **받은 편지함** (오늘, 처리) | "지금 뭘 해야 하지?" | 짧고 반복 — 하루 2~5회 |
| 케이스 | **서랍** (전체, 탐색) | "전체 상황이 어떻지?" | 필요할 때만 |
| 메시지 | **대화 기록** (컨택, 흐름) | "근로자와 어디까지 왔지?" | 응답 이벤트 시 |
| 기록 | **장부** (감사, 신뢰) | "왜/누가 이렇게 했지?" | 드물게, 그러나 결정적 |

설계 결과: 브리핑만 "처리하는 탭"이고 나머지 셋은 "확인하는 탭"이다. 따라서 **primary CTA(파랑)는 브리핑과 그 하위 플로우에만 존재**하고, 케이스·메시지·기록 탭 자체에는 파랑 CTA가 없다 — 확인 탭에서의 행동은 항상 브리핑 플로우(시트→승인)로 합류시킨다. SOCAR의 "achromatic chrome, branded action"을 탭 구조로 번역한 것.

### 0.2 공통 UI 규격

| 요소 | 규격 |
|---|---|
| 앱바 | 높이 68(상태바 제외), 타이틀 20/700 좌측, 우측 보조 정보 13/500 muted. 하단 보더 없음 — 여백이 구분 (SOCAR 헤더 원칙) |
| 탭바 | 62px, 아이콘 22px 모노라인 1.8px stroke, 라벨 11/600. 활성=primary, 비활성=`--faint`. 배지는 빨강 7px 도트만(숫자 금지 — 불안 유발 방지) |
| 좌우 가터 | 20px 고정 |
| 섹션 라벨 | 13/600 muted, 위 26 아래 10 |
| 리스트 행 | 상하 15px, 구분선 `--surface`(hairline보다 한 단계 연함 — 행 구분은 최대한 조용하게) |
| 당겨서 새로고침 | 4개 탭 모두 지원. 인디케이터는 무채색 |
| 스크롤 위치 | 탭 전환 시 각 탭의 스크롤 위치 보존, 같은 탭 재탭 시 최상단 복귀 |

## 1. 브리핑 탭 (M1) — 받은 편지함

### 1.1 화면 해부 (위→아래 시각 위계)

```
① 앱바        [심볼+워드마크]              [날짜 칩]
② 인사 문장   23/700 2줄 — 화면에서 가장 큰 활자. 건수만 primary 색
③ 승인 카드   hero 1장 + compact 2장 (최대 3)
④ 현황        섹션 라벨 + 무채색 stat 3타일
⑤ 안전 고지   surface 필 1줄 (고정 문구)
⑥ 커맨드 바   입력 필 + 전송 버튼
⑦ 탭바
```

위계 원칙: **한 화면에 "가장 눈에 띄는 것"은 딱 2개** — 인사 문장의 건수(파랑 숫자)와 hero 카드의 primary CTA. 나머지는 전부 무채색으로 물러난다.

### 1.2 hero 카드 내부 위계 (핵심 컴포넌트)

```
제목 16/600 (1~2줄, 말줄임 없음 — 업무명은 자르지 않는다)
배지 행: [D-day] [누락 N건] [승인 필요]  ← 최대 3개, 4개째부터 시트에서만
프로액티브 행: [✦] AI가 준비를 마쳤습니다     런 #4788 보기 →
추천 이유 13/400 muted 1줄
CTA 행: [초안 보기(line)] [보내기 승인(pri)]
```

- hero만 **보더 없이 그림자**(shadow-as-border), compact는 hairline 보더 — 시각적 1순위를 그림자 깊이로 표현
- 프로액티브 행은 surface 필 12px radius. 스파크 아이콘은 primary — 화면에서 파랑이 허용된 세 번째이자 마지막 자리 (에이전트의 존재 표시이므로 브랜드색 자격 있음)
- 카드 본문 탭=시트, CTA 탭=직행. 히트 영역 분리: CTA는 `stopPropagation`, 카드 탭 영역과 버튼 사이 간격 16px 이상

### 1.3 인터랙션·모션

| 트리거 | 반응 |
|---|---|
| 카드 press | 그림자 4→8px lift (fast). translate·scale 금지 (SOCAR: 모션은 available이지 playful이 아님) |
| 승인 완료 복귀 | 해당 카드 배지 교체 + CTA가 "응답 대기 중"(sec 1개)으로 축소 + 리스트 하단 이동. 애니메이션 없이 즉시 — 상태 변화는 명확하게, 화려하지 않게 |
| 프로액티브 행 탭 | 런 재생 뷰 push (std) |
| 커맨드 바 포커스 | 필 → 흰 배경 + primary 2px 아웃라인 |
| 카드 4건 이상 | 3건 + "케이스에서 N건 더 보기" 텍스트 링크 (케이스 탭 승인대기 필터 딥링크) |

### 1.4 마이크로카피

- 인사: "{호칭}, 오늘 확인이 필요한 업무가 N건 있습니다." / 0건: "오늘 승인할 업무가 없습니다."
- 추천 이유는 항상 "~해서 ~이 필요합니다" 구조 — 근거(조건)와 행동(필요)을 한 문장에
- 커맨드 placeholder: "AI에게 요청하기" (예시문은 suggestions 칩으로, placeholder에 길게 넣지 않음)

## 2. 케이스 탭 (M7) — 서랍

### 2.1 화면 해부

```
① 앱바     "케이스"            [사업장 · 총 건수]
② 필터 칩  [전체] [즉시 확인] [우선 확인] [확인 필요]   ← 가로 스크롤
③ 그룹 섹션  승인 대기 → 즉시 확인 → 확인 필요 → 예정 → 완료(접힘)
④ 케이스 카드 (compact)
⑤ 탭바
```

### 2.2 UI 규격

- 필터 칩: 높이 38, radius 14, 선택=`--ink` 배경+흰 글자 (primary 아님 — 필터는 행동이 아니라 상태). 선택 전환 fast
- 그룹 라벨: "승인 대기 · 2" — 라벨과 건수를 같은 줄, 건수는 tabular-nums
- 케이스 카드: 제목 16/600 + 배지 행 + 설명 14/400 muted 1줄. **CTA 버튼 없음** — 탭하면 시트(M2)가 열리고 행동은 거기서. 카드가 버튼 무더기가 되는 것을 구조적으로 방지
- 완료 그룹: 기본 접힘, 헤더 탭으로 펼침(아코디언, std). 완료 카드는 배지·설명 muted 처리

### 2.3 UX 규칙

| 상황 | 처리 |
|---|---|
| 딥링크 진입(stat 타일 등) | 해당 필터 칩 선택 상태 + 상단에 "적용됨: 승인 대기 [해제]" 표시 |
| 필터 결과 0건 | "조건에 맞는 케이스가 없습니다" + [필터 해제] 텍스트 버튼 — SOCAR: 원인 1줄+행동 1개 |
| 검색 | 앱바 우측 아이콘 → 인라인 검색 바 확장(제목·근로자명). owner 모드에서는 검색·필터 숨김, 요약 카드만 |
| 정렬 | 사용자 정렬 옵션 제공하지 않음 — 그룹→severity→D-day 고정 (deterministic 원칙. 정렬을 고민하게 하지 않는다) |
| 프로액티브 준비 카드 | 브리핑과 동일한 "AI 준비 완료" 표시를 설명 줄에 축약: "· AI 준비 완료" |

## 3. 메시지 탭 — 대화 기록

### 3.1 화면 해부

```
① 앱바     "메시지"           [컨택 N건]
② 스레드 리스트 (근로자 단위)
   [아바타] 이름 + 국적·채널      [시각]
           마지막 상태 요약 1줄    [상태 배지]
③ 하단 캡션 "모든 메시지는 승인 후에만 발송됩니다"
④ 탭바
```

### 3.2 UI 규격

- 아바타: 42px, radius 14(원형 금지 — radius 사다리 준수), surface 배경 + 이니셜 15/700 muted. 사진 없음(PII 최소화)
- 이름 줄: 15/600 + 국적·채널 11/600 faint 인라인 — 국적은 정보로만, 색·국기 강조 금지 (가드레일)
- 상태 배지 우선순위: 응답 도착(info) > 승인 대기(pend) > 발송됨(succ) > 초안(neut)
- 미리보기 줄: 13/400 muted 1줄 말줄임. **근로자 원문은 미리보기에 노출하지 않고** 상태 요약으로 대체 ("응답 도착 · 해석 준비됨") — 원문은 스레드 내부에서만 (PII·오해 방지)

### 3.3 스레드 상세 (탭하면)

- 승인 대기 초안 → M3 직행 (별도 스레드 뷰를 거치지 않음 — 뎁스 절약)
- 응답 도착 → M6 응답 해석 화면
- 완료 스레드 → 대화 타임라인 뷰: 보낸 메시지(우측, surface 필)·받은 응답(좌측, 흰+hairline)·시스템 이벤트(중앙 캡션 "승인 후 발송됨 · 판단 기록 #4789"). 채팅 UI지만 **입력창이 없다** — 회신은 반드시 초안 생성(에이전트) 경유. 하단 고정 버튼: [새 메시지 초안 만들기(sec)]

### 3.4 UX 규칙

- 응답 도착 스레드는 리스트 최상단 고정 + 탭바 도트
- 리마인드 예정 스레드: 시각 자리에 "리마인드 7.6 예정" (info 색 텍스트) — 예정된 에이전트 행동의 가시성
- 빈 상태: "아직 컨택 이력이 없습니다" + [케이스에서 시작하기] — 메시지는 케이스에서 파생된다는 모델 학습

## 4. 기록 탭 (M8) — 장부

### 4.1 화면 해부

```
① 앱바     "판단 기록"         [기간 · 사업장]
② 필터 칩  [전체] [위험 감지] [초안 생성] [승인]
③ 세로 타임라인 (시간 역순)
   ●─ 시각 12/400 faint
   │  요약 14.5/600
   │  액터 · 이벤트 타입 12.5/400 muted
④ 하단 캡션 "기록은 수정·삭제할 수 없습니다"
⑤ 탭바
```

### 4.2 UI 규격

- 타임라인 노드: 10px 원, 기본=hairline 보더 흰 원. **승인 이벤트만 primary 채움** — 장부에서 파랑이 허용된 유일한 자리(사람의 결정 지점). 시스템 이벤트는 전부 무채색
- 연결선 1.5px hairline. 이벤트 간 간격 18px — 밀도보다 판독성
- 액터 표기: "시스템 · Visa Document Agent" / "김담당 (본인 확인 완료)" — 사람 이벤트에는 본인확인 여부 병기 (7단계 인증 규칙의 UI 표면)
- 날짜 경계: sticky 날짜 라벨 (오늘 / 어제 / 7.2)

### 4.3 이벤트 상세 시트 (탭하면)

```
요약 문장 (17/700)
[액터] [이벤트 타입] [케이스 링크] 3열 정보
근거 citation 카드 (있는 경우)
입력 스냅샷: 해시만 표시 "sha256:a3f…" + ⓘ "원문은 저장되지 않습니다"
승인 체인 (승인 이벤트): 요청 시각 → 결정 시각 → 결정자 → 본인/대리 여부
[관련 케이스 열기] (sec 버튼 1개)
```

### 4.4 UX 규칙

- 딥링크(#4789) 진입: 해당 이벤트로 자동 스크롤 + 노드·행 하이라이트 2초 후 잔잔히 해제
- 런 단위 접기: 같은 runRef의 연속 이벤트는 "런 #4788 · 6개 이벤트" 한 행으로 접고 탭하면 펼침 — 프로액티브 런 도입 후 이벤트 폭증 대응
- 검색·내보내기: v1 제외, 앱바에 두지 않음 (장부는 조용해야 한다). 감사 내보내기는 PC 확장 화면 담당
- **신뢰 카피 두 곳 고정**: 하단 "기록은 수정·삭제할 수 없습니다", 오류 시 "기록은 유실되지 않습니다" — 장부 탭의 존재 이유를 문장으로

## 5. 탭 간 이동 규칙 (요약 맵)

```
브리핑 카드 ──탭──▶ M2 시트 ──▶ M3 초안 ──▶ M4 승인 ──▶ M5 완료 ──▶ 브리핑(상태 반영)
브리핑 stat ──▶ 케이스(필터 적용)          브리핑 Tran 카드 ──▶ 메시지
케이스 카드 ──▶ M2 시트 (같은 시트 재사용)   메시지 초안 행 ──▶ M3
M5 "기록 #4789" ──▶ 기록(하이라이트)        시트 "런 #4788" ──▶ 런 재생 뷰
커맨드 바 ──▶ M9 런 ──결과 카드──▶ M2 시트
```

원칙: **어느 탭에서 출발해도 "행동"은 같은 시트·같은 승인 플로우로 합류한다.** 탭은 진입각일 뿐, 처리 경로는 하나다 — 이것이 학습 비용을 최소화하고, Evidence 기록 경로도 단일화한다.
