# 키노트 정리

## 0. Why(왜 이 목차인가)

- **왜 지금 이 문제를 이 방식으로 풀어야 하는지**를 납득시키는 것입니다.
- 그래서 흐름을 **(1) 문제의 절박함 → (2) 우리가 겨냥한 고객/빈틈 → (3) 해결책과 실제 동작(데모) → (4) 반복 가능한 운영/신뢰(워크플로우·아키텍처·테크 신뢰) → (5) 검증/시장/사업화/요청(ask)** 순으로 설계
- 각 섹션의 why
    - Cover: **한 문장으로 기억시키기 위해**
    - Problem/Customer/Gap: **이 문제가 우리 고객에게 얼마나 자주/크게 터지는지 증명하기 위해**
    - Solution/Demo/Workflow: **해결이 개념이 아니라 업무 흐름으로 이어짐을 보여주기 위해**
    - Architecture/Technical Trust: **그래서 믿고 맡겨도 된다를 기술/안전/테스트 관점에서 보장하기 위해**
    - Validation/Market/GTM/Roadmap: **이게 사업으로 성장하고, 다음 실행이 무엇인지(Ask) 합의하기 위해**

## 1. Cover

- 제품명: **외고반장**
- 태그라인: **외국인 고용 업무, 놓치기 전에 먼저 챙깁니다**
- 한 문장 요약: Daily Briefing + Approval + Evidence 기반 **외국인 고용 업무 관리 시스템**
- (선택) 핵심 약속 3가지:
    - D-day 리스크를 먼저 보여준다
    - 다음 행동(Next Action)을 준비해준다
    - 승인/증빙으로 안전하게 실행하고 남긴다

## 2. Problem

### 무엇이 문제인가(현장 언어)

- 외국인 고용 업무는 **엑셀/메신저/정부 시스템/행정사/서류 폴더**로 분산돼 오늘 챙겨야 할 것이 한 화면에 모이지 않습니다.
- 결과적으로 운영은 **뒷북 대응**이 되고, 리스크가 시스템이 아니라 사람의 기억에 의존합니다.

### 왜 아픈가(운영 리스크)

- 체류만료·서류누락·기한초과 → 생산 공백/벌금·제재/행정사 재요청(리드타임↑)/내부 신뢰 하락

### 오늘의 현실(원인)

- 정부 시스템은 **신청** 중심, 행정사는 **검토/대행** 중심이라 기업 내부의 매일 업무 단계가 비어 있습니다.

### Why

- 외국인 고용은 정보를 아는 것보다 기한/누락을 놓치지 않는 운영이 핵심
- 문제를 **운영 리스크(기한·누락·벌금·생산공백)**로 정의

**핵심 메시지: 외국인 고용의 리스크는 정보 부족보다 기한·누락을 놓치는 운영 공백에서 발생한다.**

## 3. Customer

### 핵심 타겟(초기)

- E-9 외국인 근로자 10~100명 규모 제조업체의 인사/총무 담당자
- 특징:
    - 케이스가 한 번이 아니라 **동시에 여러 개** 돌아갑니다.
    - 서류/증빙/승인/기록은 **반복 업무**입니다.
    - 현장 커뮤니케이션(다국어, 모바일 승인)이 자주 필요합니다.

### 보조 고객(확장)

- 행정사/노무사: 여러 사업장을 운영하는 B2B2B 운영 주체 (White-label 가능)

### Why

- 이 도메인은 케이스가 동시다발/반복적으로 돌아가며(10~100명 규모), 다국어·모바일·증빙·행정사 협업이 필수인 집단이 있어 **요구되는 시스템 형태(OS/브리핑/승인/증빙)**가 결정되기 때문.

**핵심 메시지: 10~100명 규모 제조업체는 반복 케이스가 많아 한 곳에서 관리할 필요가 있다.**

## 4. Gap

### 기존 플레이어가 채우지 못하는 빈 공간

- 정부: 신청/민원 · 행정사: 검토/대행 · ERP/HR: 근태/급여 · 메신저: 소통 · 엑셀/폴더: 저장

### 내부 운영단이 빈약하다

- **오늘 누구를 챙겨야 하지?** / **지금 뭘 준비해야 하지?** / **누가 승인해야 하지?** / **무엇을 증빙으로 남겨야 하지?**

### Why

- 정부/행정사/ERP/메신저는 각각 제출/대행/관리/소통은 하지만, 기업 내부의 **매일 업무 단계(브리핑·누락탐지·승인·증빙)**가 비어 있어 그 공백을 채우는 제품이 필요하기 때문.

## 5. Solution

### 핵심 컨셉

- 외고반장 = 신청 시스템이 아니라 **사업장 내부의 외국인 고용 업무 관리 시스템**
- 운영을 표준화하는 3요소:
    
    1) **Daily Briefing**: 오늘/이번 주 리스크·해야 할 일 선제 제시  
    
    2) **Approval**: 중요한 실행을 승인으로 통제  
    
    3) **Evidence**: 실행 결과를 증빙으로 축적(인수인계/감사/재현)
    

### 사용자가 얻는 경험(제품 언어)

- 검색이 아니라 **Next Action 제안**으로 업무가 진행됩니다.
- 예시: D-30 감지 → 누락 서류 확인 → 요청 메시지/패키지 준비 → 승인 후 실행 → Evidence 자동 기록

### Why

- 안전 원칙상 자동 제출/법적 판정은 금지해야 하므로, 가치는 **자동 준비(근거 연결→액션 초안→승인→증빙 저장)**로 만들어야 하고, 이를 반복 가능한 데이터 모델(Case/NextAction/Approval/Evidence)로 닫아야 하기 때문.

**핵심 메시지: 외고반장은 답변형 챗봇이 아니라 브리핑부터 승인, 기록까지 한 번에 이어지는 처리 방식이다.**

## 6. Demo

### Nguyen 케이스(고정값)

- Nguyen Van A / 베트남 / E-9 / 화성 2공장 조립라인
- 체류만료: 2026-06-20 (**D-30**)
- 누락: **표준근로계약서 사본**, **여권 사본**
- 과거 이력: 3일 전 서류 요청 기록 있음
- 정책: **대표 승인 전 외부 발송 불가**

### Demo Scene 1: 오늘의 브리핑 (전환점)

- Nguyen V. 체류만료 **D-30** 감지
- 누락 서류 2건(예: 여권 사본/표준근로계약서)
- 다음 할 일(Next Action) 추천
- (모바일 브리핑 카드) **[초안 보기] [수정 요청] [보내기 승인]**

**핵심 메시지: 오늘 놓치면 사고 나는 것을 브리핑이 먼저 알려주고, 곧바로 처리 화면으로 이어진다.**

### Demo Scene 2: 케이스 상세와 메시지 초안

- 왜 확인이 필요한지(리스크/근거)
- 누락 서류 체크 + 이전 기록 확인
- 베트남어 요청 메시지 초안 + 한국어 번역/요약
- 예상 응답 시나리오(긍정/질문/지연) → 다음 행동 제안

**핵심 메시지: 답변이 아니라 실행 가능한 초안을 만든다(다국어/현장 맥락 포함).**

### Demo Scene 3: 승인과 업무 기록(Evidence)

- Live Agent(4단계 진행):
    - 근로자 정보 확인
    - 이전 대화/업무 기록 확인
    - 메시지 초안 생성
    - **발송 전 승인 대기**
- 승인 전에는 실행 불가(Human Approval Gate)
- 승인 후 발송/전달 자료(패키지) 생성
- Evidence Log + 업무 기록 저장(예: 업무 기록 #4789)

**핵심 메시지: 실행은 승인으로 통제되고, 결과는 기록으로 남아 다음에도 같은 방식으로 처리할 수 있다.**

### Scene ↔ 화면 매핑(PC/모바일)

- Scene 1(브리핑)
    - PC: 대시보드 요약 카드 + 업무 큐(오늘 할 일)
    - Mobile: 오늘 브리핑 카드(체류만료 D-30 / 누락 2건 / 승인 후 발송)
- Scene 2(상세/초안)
    - PC: 케이스 상세 패널(근거/누락/다음 할 일) → 초안 보기(VN/KR)
    - Mobile: Nguyen 카드 상세 → 메시지 초안 확인
- Scene 3(승인/기록)
    - Mobile: Live Agent 처리 화면 → 보내기 승인
    - PC: 승인 반영 → 업무 기록/Evidence 확인

### Why

- 이 시스템이 필요한 이유(놓치면 사고)와 안전장치(승인/근거/증빙)가 동시에 드러나는 대표 반복 업무가 체류만료 D-30 + 누락서류 + 다국어 커뮤니케이션이어서, Nguyen 케이스가 **업무 관리 시스템의 본질**을 가장 압축해서 보여주기 때문.

## 7. Workflow

### 업무 처리 흐름(제품/업무 공통)

- 감지 → 분석 → 준비 → 승인 → 기록

### 설계 원칙(핵심만)

- 감지: D-day/필수조건/누락·변경
- 분석: 리스크 레벨 + 이유 + 근거
- 준비: 실행 가능한 패키지(메시지/서류/체크리스트)
- 승인: 승인 전 실행 불가(안전)
- 기록: Evidence는 **append-only**에 가깝게 축적

### Why

- 이 제품은 챗봇이 아니라 업무 관리 시스템이라, 모든 케이스가 **감지→분석→준비→승인→기록**의 동일한 상태머신으로 수렴해야 확장/인수인계/감사가 가능하기 때문.

## 8. Architecture (현재 Production 기준)

### 핵심 설계 6가지(발표에서 말할 버전)

#### 1. 여러 진입점이 있어도 안전 기준은 하나다

**모든 요청은 PII 마스킹, tenant scope, rate limit을 거친 뒤 Agent Runtime으로 들어갑니다.**

#### 2. 3개의 Mission Agent로 업무를 나눈다

- M1 Workforce Agent: 신규 채용 · 쿼터 · 후보/요건 검토
- M2 Visa Document Agent: 체류만료 · 계약충돌 · 누락서류 · 행정사 패키지
- M3 Multilingual Contact Agent: 다국어 메시지 · 응답 해석 · 다음 행동

**외부에는 3개의 Mission Agent가 보이고, 내부에서는 각 Agent가 세부 작업을 순차적으로 처리합니다.**

#### 3. 병렬보다 순차 실행을 택했다

**병렬 실행보다 순차 실행을 선택해 상태 충돌, 중복 호출, 감사 로그 누락을 방지했습니다.**

#### 4. 판단은 Rule/DB/RAG가 하고, LLM은 정리와 생성에 집중한다

**위험 판단은 deterministic rule과 DB/RAG 근거를 사용하고, LLM은 설명·초안·요약을 담당합니다.**

#### 5. Approval Gate는 실제 외부 실행을 막는다

**외부 발송과 제출은 자동 실행하지 않고, pending-first 승인 구조와 outbox 기록으로만 관리합니다.**  

**AI는 준비하고, 사람은 승인하며, 실제 외부 실행은 통제됩니다.**

#### 6. Evidence Log와 Metrics로 추적 가능하다

**모든 판단 근거, 승인 이력, 실행 상태는 Evidence Log와 runtime metrics로 남기며, 원문 개인정보는 저장하지 않습니다.**

---

### Architecture 한 장(발표용 다이어그램)

**외고반장은 AI가 자유롭게 실행하는 챗봇이 아니라, 통제된 Agentic Workflow입니다.**

```
Entry (User Chat · Daily Briefing Scheduler · Admin Upload)
  ↓ Input Guard (PII Masking · Tenant Scope · Rate Limit)
  ↓ Bounded Router (업무 의도 분류)
  ↓ 3 Mission Agents (Workforce / Visa Document / Multilingual Contact)
  ↓ Rule · RAG · DB Tools (D-day · 누락서류 · 공식 근거 · 내부 데이터 조회)
  ↓ Approval Gate (외부 발송 전 사람 승인, pending-first)
  ↓ Evidence Log / Metrics (판단 근거 · 승인 이력 · 실행 기록 저장)
  ↓ UI (Dashboard · Mobile Approval · Admin · Evidence Viewer)
```

**LLM은 업무를 정리하고 초안을 만들지만, 위험 판단·외부 실행·기록 관리는 Rule, Approval, Evidence 구조로 통제합니다.**

- Data: Chroma 단일 RAG 경로 + context_data_service 기반 DB 조회
- Approval: `approval_actions`, `delivery_outbox`, `agent_checkpoints`
- Observability: Evidence Log는 INSERT-only, trace_id 기반, 원문 PII 저장 없음 / Metrics는 latency, token usage, retrieval count, approval rate, forbidden block count

## 9. Technical Trust

### 왜 안전한가

- Rule Engine(필수조건/금지/리스크) + Human Approval(중요 실행 통제) + PII Masking

### 왜 테스트 가능한가

- 라우팅/룰/RAG/승인/idempotency를 분해해 자동 테스트 가능

### 왜 실무에서 믿을 수 있는가

- Evidence Log + trace_id를 남겨서, 나중에 “왜 이렇게 처리했는지”를 다시 따라갈 수 있게 합니다.
- 그래서 누가 봐도 설명이 되고, 문제가 생기면 원인을 찾을 수 있고, 인수인계도 훨씬 쉬워집니다.

### Why

- 이 도메인은 PII/대외 발송/법령 인용이 얽혀 그럴듯함이 가장 위험하므로, **룰(결정론)·근거 게이트·승인 게이트·증빙/트레이스**로 실패해도 안전하게 무너지도록 설계해야 하기 때문.

## 10. Validation (Evaluation)

### 무엇을 검증하는가(핵심)

- Route correctness / Risk rule correctness / Citation correctness / Approval correctness / Idempotency

### 측정 지표 예시

- 누락률/오탐률 · 승인 지연 · E2E 처리시간 · 증빙 누락률

### Why

- 업무 관리 시스템의 품질은 데모가 아니라 반복 사용에서 결정되므로, Route/Risk/Citation/Approval/Idempotency를 **테스트 가능한 지표/하네스**로 정의해 회귀를 막아야 하기 때문.

# 엔지니어링 파트

## 0) 큰 그림: 무엇을 보장하려는가

- 외고반장의 엔지니어링은 **검색 품질(정확한 근거)**, **비용(운영 가능한 단가)**, **보안(PII/권한/감사)**, **평가(지표로 검증)** 4가지를 동시에 만족해야 합니다.
- 아래 부록은 구성 요소 나열이 아니라, 이 4가지 보장을 위해 각 컴포넌트(API/DB/RAG/라우팅/미들웨어/옵저버빌리티)가 어떻게 연결되는지에 초점을 둡니다.

### Why

- 운영 제품에서 기술의 목적은 구현이 아니라 **신뢰(정확·안전·운영비·검증가능성)**를 보장하는 것이어서, 엔지니어링 항목을 그 보장 축으로 묶어야 하기 때문.

## 1) 검색 품질 (Search Quality)

### 1-1. RAG 파이프라인(정확도/근거성)

- 문서 소스 → 정규화/청킹 → 인덱싱 → 검색 → 인용(citation) → 답변 생성
- 근거가 없으면 강한 결론을 내리지 않는다를 정책으로 둔다 (보류/재질문/승인 요청으로 전환)
- 문서 신뢰도/버전 관리:
    - 어떤 문서를 정답 근거로 인정할지(출처/버전/유효기간)
    - stale evidence(구버전 인용) 탐지/차단

#### Why

- 외고반장의 답변은 안내문이 아니라 **업무 실행을 좌우하는 근거**가 되므로, 검색 파이프라인이 근거/버전까지 보장하지 못하면 곧바로 안전/신뢰 문제가 된다.

### 1-2. 라우팅 품질(검색으로 가야 할 질문 vs 액션으로 가야 할 질문)

- Bounded intent → 구조화된 plan
    - Why: 의도가 경계(bound)되지 않으면 계획이 해야 할 일 리스트가 아니라 **추측/환각의 서사**가 되기 쉽다. 먼저 intent를 제한해야 필요한 데이터/툴/승인만 포함한 plan을 만들 수 있다.
- Rule vs LLM 책임 분리:
    - Rule: 필수 조건/리스크/금지 행동/승인 필요 여부
    - LLM: 요약/작성/설명/플랜
    - Why: 리스크/금지/승인 같은 통제 로직을 LLM에 맡기면 재현성이 깨지고 사고가 난다. Rule로 **결정론적 가드레일**을 세우고, LLM은 표현/초안에만 쓰는 게 운영 안정성의 전제다.
- 불확실성 처리:
    - low_confidence면 재질문(추가 정보 수집) 또는 안전한 fallback
    - unknown intent면 조회 모드로 제한하고 실행 액션은 만들지 않는다
    - Why: 불확실성을 숨기면 그럴듯한 실행으로 이어진다. 불확실성은 제품의 실패 모드이자 안전장치이므로, 재질문/차단/조회 모드로 **안전하게 무너지도록** 설계해야 한다.

#### Why

- 검색으로 가야 할 질문을 액션으로 보내면 허위 확신이나 불필요한 실행이 생기고, 반대로 실행해야 할 일을 검색으로만 처리하면 전체 흐름이 느려집니다. 그래서 라우팅은 비용/안전/UX를 동시에 결정합니다.

## 2) 비용 (Cost)

### 2-1. 비용이 터지는 지점(운영 관점)

- 문서 인덱싱/검색 비용: 전체 문서를 매번이 아니라, 변경분 중심으로 업데이트
- LLM 호출 비용: 매 질문마다 풀 생성이 아니라, 라우팅/룰로 좁히고 필요한 단계만 호출
- 대량 이벤트(스케줄러): 사업장/근로자 수가 늘수록 브리핑/리마인드/체크가 폭증 → 배치/큐잉/캐싱 전략 필요

#### Why

- 업무 관리 시스템은 사용자가 하루에 한 번만 쓰는 형태가 아니라, 스케줄러나 브리핑처럼 자동으로 도는 작업이 늘어납니다. 그래서 비용이 어디서 터지는지부터 잡아두지 않으면, 운영이 커질수록 과금이 쉽게 폭발합니다.

### 2-2. 비용을 낮추는 설계 원칙

- Embedding Cache (chunk text hash 기반)
    - Why: 임베딩 재생성은 중복 chunk에서 가장 많이 새므로, hash 기반 캐시가 있어야 인덱싱 비용/시간이 안정화된다.
- Rate/Cost Guard (호출 횟수 제한, circuit breaker)
    - Why: 무한 재시도/루프/폭주를 구조적으로 막지 않으면, 장애가 곧 과금 폭발로 이어진다(운영 안전장치).
- Idempotency 키(같은 case_id 중복 실행 방지)
    - Why: 재시도/중복 트리거는 운영에서 기본값이다. 중복 실행을 막아야 동일 업무를 여러 번 생성/계산하는 비용과 혼란을 줄인다.
- 요약 미들웨어(장기 케이스 컨텍스트 절약)
    - Why: 장기 케이스는 컨텍스트가 길어져 토큰이 늘고 품질도 흔들린다. 중간 요약을 상태로 저장해 호출당 토큰을 일정하게 유지한다.
- temperature=0/seed 고정(평가/재현성 + 불필요한 재시도 감소)
    - Why: 재현성이 없으면 디버깅이 어려워지고 다시 돌려보기가 비용이 된다. seed 고정은 평가/회귀 테스트에서 비용을 안정화한다.
- 캐시 가능한 것은 캐시한다: 동일 근거/동일 질문 패턴, 동일 브리핑 템플릿
    - Why: 업무 관리 시스템는 같은 패턴이 반복되므로 캐시가 곧 단가다. 캐시가 없으면 사용자 증가가 곧바로 비용 폭증으로 이어진다.
- 작은 모델/규칙으로 걸러낸 뒤 큰 모델을 쓴다(계단식 추론)
    - Why: 대부분의 요청은 룰/소형 모델로 분류·정리만 해도 충분하다. 큰 모델은 **정말 생성이 필요한 단계**에만 써야 비용/지연/리스크가 줄어든다.
- 승인 전 단계에서는 초안을 만들고, 승인 이후에만 확정/발송/패키지 생성을 수행해 불필요한 실행 비용을 줄인다
    - Why: 승인 전에는 결과가 뒤집힐 수 있어, 확정 생성/발송까지 해버리면 재작업 비용과 리스크가 커진다. 초안↔확정 경계를 분리하면 비용과 안전을 동시에 얻는다.
    
    #### Why
    
    - 비용 절감은 최적화가 아니라 **제품이 유지되느냐**의 문제다. 특히 승인 전/후를 분리하면 초안 생성까지만으로도 가치를 주면서, 불필요한 실행 비용과 리스크를 동시에 줄일 수 있다.

## 3) 보안 (Security)

### 3-1. 데이터 경계와 권한

- tenant 단위 데이터 격리(사업장 간 데이터가 섞이면 즉시 치명적)
    - Why: 한 번의 cross-tenant leak은 곧 사고/신뢰 붕괴다. 격리는 기능이 아니라 제품이 존재할 수 있는 전제조건이다.
- 사용자 역할/권한: 작성자(담당) / 승인자(결재) / 열람자(감사/관리) 구분
    - Why: 승인/감사 흐름이 제품 가치의 중심이라, 권한 모델이 없으면 승인/증빙이 그럴듯한 UI로 끝난다. 역할 구분이 있어야 책임·감사가 작동한다.
- 커넥터/미들웨어에서 권한/레이트리밋/리트라이/감사 로그를 통제(툴 호출이 곧 외부 영향)
    - Why: 에이전트의 툴 호출은 외부 세계에 영향을 주므로, 앱 코드 곳곳이 아니라 중앙 미들웨어에서 통제해야 일관된 안전/감사/운영이 가능하다.
    
    #### Why
    
    - 한 번의 데이터 섞임은 바로 치명적 사고라, 테넌트 격리/권한 모델이 제품 기능보다 상위의 전제조건이다. 또한 외부 영향(툴 호출)은 곧 책임이므로, 미들웨어에서 통제해야 한다.

### 3-2. PII/민감정보 취급

- PII 최소화: 모델 입력에서 불필요한 PII 제거/마스킹
    - Why: LLM 호출/로그는 복구가 어렵다. 입력 단계에서 PII를 줄이면 유출면적 자체가 줄어든다(최소권한/최소노출).
- 로그/트레이스에는 원문을 남기지 않거나(또는) 민감 필드를 별도 보관(권한 기반)
    - Why: 운영을 하려면 로그는 필수지만, 원문을 남기면 운영이 곧 유출 위험이 된다. 원문 분리/권한 기반이 운영 가능성을 만든다.
- 승인 전에는 외부 발송/제출이 불가하도록 강제(Human Approval Gate)
    - Why: 대외 발송/제출은 실수 한 번의 비용이 크다. 승인 전 구조적 차단이 있어야 제품이 도입될 수 있다.
    
    #### Why
    
    - 이 도메인은 PII가 업무의 핵심 데이터라 조심하자가 아니라 **구조적으로 새지 않게** 만들어야 한다. 특히 로그/트레이스는 운영에 필수라서, 설계 단계에서 원문 비저장/권한 기반을 박아야 한다.

## 4) 평가 (Evaluation)

### 4-1. 무엇을 테스트하는가(데모가 아니라 신뢰성)

- Route correctness / Risk rule correctness / Citation correctness / Approval correctness / Idempotency[[7]](https://www.notion.so/Evaluation-Test-35d4b61242238072904af5c513385a79?pvs=21)

#### Why

- 모델이 똑똑한가보다 운영에서 사고가 나는가가 더 중요하다. 그래서 평가 항목도 자연어 품질이 아니라 **경로/룰/근거/승인/중복실행**처럼 사고로 이어지는 축으로 잡는다.

### 4-2. 지표/테스트 케이스 예시

- Risk Rule Test: D-30이면 HIGH가 나오는가, 필수서류 누락을 잡는가, 계약/체류 충돌 탐지
    - Why: 리스크 계산이 흔들리면 브리핑/액션이 전부 무너진다. 가장 먼저 결정론 로직의 회귀를 잡아야 한다.
- RAG Test: Hit@3, citation accuracy, hallucinated citation rate, stale evidence detection
    - Why: 근거 품질이 무너지면 답변이 아니라 사고가 된다. 특히 stale evidence는 맞는 내용이지만 지금은 틀린 실패라 운영에서 치명적이다.
- Approval Test: 승인 전 실행 불가, 승인 후 dispatch/mock dispatch 생성, 중복 승인 방지
    - Why: 승인 위반은 신뢰 붕괴로 직결된다. 승인 관련 테스트는 기능 테스트가 아니라 안전 테스트다.
- Idempotency Test: 스케줄러가 두 번 돌아도 Case 중복 생성이 없는가[[7]](https://www.notion.so/Evaluation-Test-35d4b61242238072904af5c513385a79?pvs=21)
    - Why: 운영에서는 중복 실행이 자주 생긴다(재시도/네트워크/스케줄러). 중복 케이스는 곧 업무 혼란/비용 폭증이므로 테스트로 먼저 잠근다.
    
    #### Why
    
    - 이 테스트 케이스들은 실패했을 때 비용이 큰 부분을 겨냥한다. 특히 승인/아이템포턴시는 한 번만 터져도 신뢰가 무너지므로, 기능 완성도보다 먼저 자동화 테스트로 박아야 한다.

## 5) 시스템 구성(참고 링크/스펙)

### 5-1. API Contract(입출력 스키마)

- 원칙: 제품의 모든 행동은 계약된 입력/출력으로 설명 가능해야 함[[10]](https://www.notion.so/API-35e4b6124223800fb853e5fb56d24067?pvs=21)
    - Why: 계약이 없으면 에이전트가 매번 다른 형태로 말하고, UI/로그/평가가 다 깨진다. 설명 가능은 곧 운영 가능이다.
- 예시: Agent Chat 요청/응답, Daily Briefing payload[[10]](https://www.notion.so/API-35e4b6124223800fb853e5fb56d24067?pvs=21)
    - Why: 챗과 브리핑이 같은 상태모델로 수렴해야, 채널이 달라도 동일한 케이스/액션/승인/증빙이 유지된다.
- 최소 포함 필드(권장):
    - tenantId, userId, currentScreen, selectedCaseId, workerId
    - intent, caseId, riskLevel, nextActions, evidenceEventIds
    - Why: 이 필드들이 있어야 누가/어떤 맥락에서/무슨 리스크로/어떤 액션을/무슨 근거로를 한 번에 추적할 수 있다(trace/evidence의 최소 조건).

#### Why

- 스키마가 없으면 에이전트 출력이 텍스트로 흩어져서 케이스/승인/증빙이 연결되지 않는다. 계약을 고정해야 **로그·평가·재현·프론트 UI**가 동시에 안정해진다.

### 5-2. DB Schema(상태 모델)

- 핵심 엔티티: Case / NextAction / Approval / Evidence
    - Why: 케이스 처리의 단위를 엔티티로 고정해야, 시스템이 텍스트가 아니라 상태로 움직이고 자동화/확장/감사가 가능해진다.
- 설계 포인트:
    - Case: 상태의 중심(업무 묶음)
    - NextAction: 실행 단위(메시지/요청/제출/패키지 생성 등)
    - Approval: 통제 단위(누가 무엇을 승인하는지)
    - Evidence: 감사/설명 단위(append-only 성격 권장)
    - Why: (1) 상태 중심(Case) (2) 실행 단위(NextAction) (3) 통제(Approval) (4) 재현(Evidence)가 분리돼야, 한 요소만 바뀌어도 전체가 안정적으로 진화한다.
    
    #### Why
    
    - 문제(케이스)와 행동(액션)을 분리하고 승인/증빙을 붙여야, 운영이 개인의 기억이 아니라 **데이터 모델**로 돌아간다. 이게 곧 확장성과 인수인계 가능성을 만든다.

### 5-3. Trace / Observability(운영 가능성)

- trace_id 기반 end-to-end 추적이 운영 가능한 제품의 최소 조건[[8]](https://www.notion.so/35d4b612422380868425c113b48da08b?pvs=21)
    - Why: 자동화가 늘수록 왜 이렇게 됐지?가 운영의 핵심 질문이 된다. trace가 없으면 문제를 해결할 수 없고 결국 자동화를 꺼야 한다.
- 최소 로그(예시):
    - trace_id, tenant_id, user_id, intent, route
    - retrieved_docs_count, risk_rule_result
    - case_id, next_action_ids, approval_id, evidence_event_ids
    - latency_ms, model_name, token_usage, fallback_reason
    - Why: 이 필드들이 있어야 품질(근거/룰), 안전(승인/차단), 비용(토큰/지연), 운영(케이스/액션)을 한 요청 단위로 묶어 원인분석이 가능하다.
- 장애/감사 대응 관점:
    - 왜 이 답이 나왔는지를 나중에 설명할 수 있어야 함
    - Why: 감사/분쟁/사고 대응에서 설명 가능성은 선택이 아니라 필수다. Evidence와 trace는 그 설명의 근간이다.
    
    #### Why
    
    - 운영에서 문제는 에러가 났다가 아니라 왜 이렇게 됐는지 모른다이다. trace/metric이 있어야 재현/디버깅/감사 대응이 가능하고, 그게 없으면 자동화는 결국 꺼야 한다.
