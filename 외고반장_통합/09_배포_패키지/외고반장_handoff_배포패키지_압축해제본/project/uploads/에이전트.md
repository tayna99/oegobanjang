# 에이전트 설명

<aside>
🗺️

아래 설명은 첨부된 3개 SVG(인력확보 / 다국어컨택 / 비자서류)의 “전체 아키텍처” 다이어그램을 글로 풀어쓴 것입니다. 공통적으로 **데이터→RAG/룰→툴→에이전트 그래프→미들웨어→승인(HITL)→출력/근거(Evidence)** 흐름으로 설계되어 있습니다.

</aside>

## 공통 전제: 3개 에이전트가 공유하는 설계 원칙

- **Tool 위험도 등급화**
    - SAFE_READ: 읽기 전용(부작용 없음)
    - SAFE_CALCULATE: 계산/판정(외부 영향 없음)
    - SAFE_DRAFT: 승인 전 “초안 생성”
    - APPROVAL_REQUIRED: 외부 전달/상태 확정 등 **사람 승인 필수(🔒)**
- **Middleware Stack(공통)**
    - Call limit: ModelCallLimit / ToolCallLimit
    - PII Middleware: 여권번호/전화번호 등 민감정보 redact
    - Safety Validation: 금지 키워드·금지 행위 차단
    - SQLite Checkpointer/State: 상태 저장·복원(중단/재시작 대비)
- **Evidence Logger (감사·추적성)**
    - node_name, actor_type(LLM/툴/사람), hash, citation_ids 등을 이벤트로 남겨 “왜 이런 결과가 나왔는지” 역추적 가능하게 함
- **Human-in-the-loop (Approval Gate)**
    - 외부 발송/전달/제출/케이스 완료 처리는 기본적으로 승인 게이트를 통과해야 실행됨
    - 승인 전에는 blocked_until_approved=true 같은 플래그로 “외부 실행”이 차단됨

## 1) 인력 확보 에이전트 (인력확보에이전트_태나.svg)

### 1.1 레이어별 구성

**(A) DATA SOURCES**

- 공식 정책 문서(RAG — Grade A/B)
    - EPS 사업주 고용절차, 고용허가 신청 구비서류, E-9 허용업종, 표준근로계약서 기준, 사증발급인정서 절차 등
- 내부 템플릿·DB(Grade E / rule)
    - 신규 인력 요청서 템플릿, 사업주 체크리스트, 후보 제출 준비도 체크리스트, companies.csv / candidates.csv
- “현재 없는 데이터”
    - 고용조정/임금체불/보험가입/구인노력 진행 상태 등 → 외부 연동 필요 영역으로 명시

**(B) RAG PIPELINE**

- [chunking.py](http://chunking.py): 문서 유형별 max 길이(법 조문/절차/서식/안전 섹션 등)로 청킹
- [embeddings.py](http://embeddings.py): text-embedding-3-small(테스트용 deterministic 경로도 별도)
- Chroma(workforce): official(A/B) + templates(E) + 메타데이터(visa_type/case_type/sub_agent/output_usage)
- [retriever.py](http://retriever.py): similarity_score ≥ 0.4 게이트 + visa_type/evidence_grade 필터 + 버킷 분리 반환(official/allowed_industry/template)

**(C) TOOL LAYER**

- SAFE_READ
    - get_company_profile / get_candidate_profile / search_policy_documents / get_document_requirements 등
- SAFE_CALCULATE
    - calculate_candidate_readiness / calculate_missing_documents / calculate_contract_gap / calculate_visa_d_day / assess_visa_risk / quota_tool
- SAFE_DRAFT
    - generate_hiring_request_draft (요청서 초안 payload 생성)
    - generate_expert_handoff_package (행정사 전달용 패키지 초안)
- APPROVAL_REQUIRED
    - send_expert_package(행정사 전달), update_case_status_completed(케이스 완료) 등

**(D) AGENT GRAPH**

- WorkforceAgent(Orchestrator)가 ForeignHiringState를 입력으로 받음
- 서브 에이전트 2개 순차 실행
    - WorkforceRequirementAgent
        - companies.csv 로드 → 필수필드 검증 → visa_type=E-9 고정
        - WORKFORCE_CHECKS 5개 자동 포함 + handoff_questions 5개 생성
        - approval_required = true (항상)
    - CandidateReadinessAgent
        - candidates.csv 조회 → 7개 항목을 **rule-based 체크**
            - 여권/증명사진/건강검진/근무가능일/희망직무/숙소안내/근무조건이해
        - ready vs missing_required_info 분류
        - 점수/순위/성실도 “판단” 금지(가드레일)

**(E) MIDDLEWARE**

- ModelCallLimit(10) / ToolCallLimit(20)
- PII redact / Safety Validation / SQLite Checkpointer

**(F) OUTPUT & HITL**

- 출력물
    - 신규 인력 요청서 초안(필드 + missing_inputs + citations)
    - 제도 체크리스트 5개(업종/내국인구인노력/고용허가신청/표준계약/숙소·교대 등)
    - 후보 준비도 테이블(ready/missing_required_info + risk_flag)
- HITL
    - 승인 전 외부 전달 차단
    - CRITICAL 시 행정사 핸드오프 초안 생성/추가

### 1.2 이 다이어그램의 포인트(요약)

- “인력확보”는 **요청서/핸드오프 문서화**와 **후보 준비도(7항목 rule check)**가 핵심 산출물.
- 외부 실행(send_expert_package 등)은 APPROVAL_REQUIRED로 분리되어 있고, 나머지는 SAFE_*로 안전하게 분해되어 있음.

## 2) 다국어 컨택 에이전트 (다국어컨택에이전트_현희.svg)

### 2.1 레이어별 구성

**(A) DATA ACQUISITION**

- EPS/Law/Gov24/HiKorea 크롤러로 공식 절차/법령 수집
- 다국어 템플릿 CSV(message_templates.csv): 베트남어/인도네시아어
- 안전·생활 가이드(PDF/HTML): vi/id
- 내부 케이스(case_record): (grade F)로 표시되어 “근거로 쓰기 어려움”을 구분

**(B) PREPROCESSING & CHUNKING**

- raw_[ingest.py](http://ingest.py)
    - txt/md/html/pdf/jsonl ingest
    - HTML 구조 텍스트 변환
    - 최소 10자 품질 게이트
    - source_hash 생성
- domain_[splitters.py](http://splitters.py)(비자 문서 전용 분류기)
    - law_article / procedure_step / form_section / safety_section
    - regex 기반 + confidence(high/low)
- rag_hyunhee/[chunking.py](http://chunking.py)(다국어 전용)
    - max_chars=1000, overlap=150
    - 문장 경계 보존(KO/EN/VI)
    - contextual_text 헤더 + chunk_id 해시 생성

**(C) EMBEDDING & VECTOR DB**

- [embeddings.py](http://embeddings.py): text-embedding-3-small(+테스트용 deterministic)
- Chroma 분리
    - workforce 컬렉션(공통)
    - multilingual_contact 컬렉션(언어 vi/id + doc_type counseling/safety/life/notice)

**(D) RAG RETRIEVER (다국어 특화)**

- rag_hyunhee/[retriever.py](http://retriever.py)
    - infer_intent(): 상담/안전/생활/공지 분류
    - infer_language_code(): vi/id 감지
    - 스코어 부스트: intent -0.25 + lang -0.6 등(의도/언어 맞춤 검색 강화)
    - 3단계 fallback(특정→관련→전체 doc_type)
    - _is_safe_context(): F등급·금지 마커 필터
- [citation.py](http://citation.py)
    - ANSWER_EVIDENCE: A/B/E만 허용
    - 중복 chunk 제거(best score)
    - citation_id ↔ chunk_id 매핑 + EvidenceEvent 기록

**(E) LANGCHAIN AGENT GRAPH**

- agent_[factory.py](http://factory.py): ChatOpenAI + system_prompt + SQLite Checkpointer
- Tool Registry: SAFE_READ / SAFE_CALCULATE / SAFE_DRAFT / APPROVAL_REQUIRED 정리
- contact_[subagents.py](http://subagents.py)(다국어 컨택 서브에이전트 2개)
    - Onboarding Subagent
        - CSV 템플릿 로드
        - search_multilingual_contact_rag_tool
        - LLM 번역 품질 검증
        - KO + VI/ID 초안 + citations 생성
    - Reply Interpreter Subagent
        - 근로자 답장 번역 → 한국어 요약
        - 상태 업데이트 후보(rule-based) 제안

**(F) OUTPUT & HITL**

- 메시지 초안(한국어 원문 + 베트남어/인니어 번역 + citation_ids)
- 예상 응답 시나리오(긍정/질문/지연)
- Human Approval 후 ExternalDeliveryJob(Zalo/이메일/수동 발송 등)로 이어짐
- EvidenceEvent: RAG_RETRIEVED / APPROVAL_REQUESTED / trace_id / redacted_hash 등 기록

### 2.2 이 다이어그램의 포인트(요약)

- “다국어 컨택”은 **의도(infer_intent) + 언어(infer_language_code)**를 먼저 결정하고, 그에 맞춘 RAG를 태워 **안전한 근거(A/B/E만)**로 메시지를 만들도록 설계됨.
- 답장 해석(Reply Interpreter)은 “번역+요약+상태 업데이트 후보”까지 가지만, 실제 외부 발송/확정은 승인 게이트로 분리됨.

## 3) 비자 서류 에이전트 (비자서류에이전트_현욱.svg)

### 3.1 전체 구조(레인 기반)

이 다이어그램은 위 2개보다 “런타임 레인(입력→그래프→에이전트→툴→출력)”이 더 선명하게 나뉩니다.

**(A) INPUT**

- DB/사업장: 근로자·비자·서류·계약 데이터
- RAG 벡터스토어: 출입국법·고용허가·서식
- 사용자 요청: 자연어 또는 스케줄 트리거
- Middleware: PII Filter / Call Limiter / Summarizer

**(B) GRAPH RUNTIME (LangGraph)**

- Intent Router → Planner → State Loader → Executor → Approval Gate
- Aggregator: 서브에이전트 결과 병합
- Evidence Logger: 실행 노드별 근거/해시/인용 기록
- Final Response: NextAction + 근거 반환

**(C) VISA DOCUMENT AGENT (오케스트레이터 + 서브에이전트 2)**

- 오케스트레이터: run_visa_agent()
    - CRITICAL 감지 시 handoff(전달 패키지)로 분기
- Sub-agent 1: 비자 위험도 분석
    - _run_visa_risk_sub_agent(), ReAct LLM 루프
    - D-day / 체류유형 / 계약 교차 분석
- Sub-agent 2: 서류 우선순위 분석
    - _run_document_priority_sub_agent(), ReAct LLM 루프
    - 누락서류 / CRITICAL vs SUPPLEMENTARY 분류

**(D) TOOLS (Tool Contract)**

- SAFE_READ
    - get_worker_profile / get_visa_status / get_document_status / search_policy_documents
- SAFE_CALCULATE
    - calculate_visa_d_day / calculate_missing_documents / assess_visa_risk / assess_document_priority
- SAFE_DRAFT
    - generate_multilingual_message / generate_expert_handoff / _package_draft
- APPROVAL_REQUIRED 🔒
    - send_worker_message / send_expert_package / update_case_status_completed

**(E) OUTPUT / HUMAN-IN-THE-LOOP**

- DailyBriefingItem: 리스크 요약 + D-day + citation_ids
- NextAction: request_document / create_handoff 등
- EvidenceEvent: 감사로그(hash, node_name, actor_type)
- Human Approval(모바일 승인/수정 요청) → ExternalDeliveryJob 생성

### 3.2 이 다이어그램의 포인트(요약)

- “비자 서류”는 **LangGraph 런타임(라우터/플래너/로더/실행/승인/병합/증거)** 구조를 전면에 두고,
- 서브에이전트 2개가 각각 “위험도”와 “서류 우선순위”를 맡은 뒤 Aggregator가 병합하는 패턴이 명확합니다.

## 4) 3개 다이어그램을 한 줄로 비교하면

- **인력 확보**: “요청서/핸드오프 문서화 + 후보 준비도(7항목 rule 체크)” 중심
- **다국어 컨택**: “의도+언어 판정 → RAG 안전 필터(A/B/E) → 메시지/답장 해석” 중심
- **비자 서류**: “LangGraph 런타임 + 위험도/우선순위 2서브에이전트 병렬 사고 → 승인 후 실행” 중심
