## 1) 전체 아키텍처 흐름 (채널 → 라우팅 → 규칙 → 상태 → 미션 → 승인 → 근거)

핵심은 **LLM이 시스템 밖에서 임의 실행하지 못하게** 하고, 모든 업무가 **Case / NextAction / Approval / Evidence** 상태 모델 안에서만 진행되도록 만드는 것입니다.

- 입력 채널
    - Daily Briefing (Scheduler) / User Chat
- 라우팅
    - Intent Router 또는 Scheduled Job
- 결정론 엔진
    - Risk Rule Engine (정책/임계값/우선순위)
- 업무 단위(상태) 생성
    - Case / NextAction 생성
- 생성/준비(LLM)
    - Mission Agent가 초안/계획/메시지/정리 생성
- 실행 통제
    - Approval Gate (승인 전 외부 발송/제출 금지)
- 추적/책임
    - Evidence Log (판단 근거 + 실행 이력)

## 2) 에이전트 패턴: Router → Mission Agent(+서브에이전트) + 공통 게이트

### 2.1 Intent Router

- 채널(Chat/Briefing) 입력을 받아 **bounded intent**로 정규화
- 케이스 컨텍스트/규칙 결과를 바탕으로 **어떤 미션으로 갈지** 결정
- 원칙: Agent는 자유롭게 외부 행동을 하지 않고, **Case/Action/Approval/Evidence 모델 안에서만** 실행

### 2.2 Mission Agent (업무별 전문 에이전트)

예시(개념):

- Workforce Agent
- Visa Document Agent
- Multilingual Contact Agent

공통 역할

- 입력: 케이스(근로자/서류/계약/이전 기록 등) + 룰 엔진 결과(리스크/우선순위/필수 체크)
- 출력: 사람이 바로 승인/수정 가능한 형태의 “준비물”
    - 다국어 메시지 초안
    - 다음 행동(NextAction) 초안
    - 행정사/내부 전달문(핸드오프 패키지)
    - 사용자 질문에 대한 자연어 답변(단, 정책 범위 내)

### 2.3 공통 게이트(모든 미션 공통)

- Approval Gate: 외부 커뮤니케이션/제출/발송 전에 반드시 사람 승인
- Evidence/Audit: 왜 이런 초안/결론이 나왔는지 근거와 이력 저장

## 3) Rule vs LLM 책임 분리 (미들웨어 설계의 요지)

### 3.1 Rule-base가 담당하는 것 (결정론/정책/리스크)

- D-7, D-30 등 임계값 기반 경보/우선순위
- 필수 서류 누락 감지
- 계약-체류일 불일치 등 충돌 후보 탐지
- 목적
    - “무엇을 해야 하는지/위험한지”를 **확정**
    - LLM이 잘못 건드리면 큰 문제가 되는 영역을 **선제 차단**
    - 케이스 생성/중복 방지/idempotency의 기준 제공

### 3.2 LLM이 담당하는 것 (생성/설명/요약/커뮤니케이션)

- 설명 생성(사용자 이해용)
- 다국어 메시지 초안
- 응답 요약
- 행정사 전달문 정리
- 사용자 질문 자연어 응답(단, 정책/가드레일 준수)

## 4) 상태 모델 (Agent의 “작업공간”)

최소 객체(요지):

- Worker / Document / Contract / Case / NextAction / Approval / EvidenceEvent
- (확장) HandoffPackage / DispatchLog / Tenant / User/Role

특히 핵심 4개:

- **Case**: 위험 또는 처리해야 할 업무 단위
- **NextAction**: 케이스에서 파생된 다음 행동
- **Approval**: 외부 발송/전달 전 승인 상태
- **EvidenceEvent**: 판단 근거 + 실행 이력 (감사/추적 관점)

엔지니어링 관점에서 자주 나오는 질문(설계 체크리스트)

- Case 중복 생성은 어떻게 막나? (idempotency 키 / 스케줄러 중복 실행 방지)
- Action 상태 전이는 어떻게 되나?
- Approval 취소/수정은 가능한가?
- EvidenceEvent는 append-only인가? (수정 가능 범위/버전 관리)
- 근거 문서 버전은 어떻게 저장하나?

## 5) Safety / Guardrail = 공통 미들웨어 체크

정책 예시(요지):

- AI 단독 법률 판단 금지
- 비자 발급 가능 여부 “확정” 금지
- 가치 판단(국적/성실도/이탈 가능성 등) 금지
- 대표 승인 전 외부 발송 금지
- 정부 포털 자동 제출 금지
- PII 원문 로그 저장 금지
- tenant scope 체크, RBAC, idempotency

공통으로 동작해야 하는 레일(단계별):

- State Update
- Risk Check (Rule/Policy)
- Approval Gate
- Evidence/Audit Log
- Cost Guard

---

## 6) RAG 설계(검색) = “정답 생성”보다 “근거 추적”

RAG의 품질은 “검색해서 말은 그럴듯하게”가 아니라 다음을 추적할 수 있어야 함:

- 근거가 언제 수집됐는지
- 공식 문서인지/내부 문서인지
- 문서가 오래되었을 때 어떻게 처리하는지
- 어떤 답변에 어떤 근거가 쓰였는지(답변-근거 연결)
- RAG 실패 시 fallback/승인 플로우는 무엇인지

---

## 7) 한 줄 요약

**Router가 미션을 고르고, Rule Engine이 케이스/리스크를 확정하고, Mission Agent(LLM)는 승인 가능한 초안을 만들고, Approval Gate를 통과한 뒤에만 실행되며, 모든 판단/검색/행동은 Evidence로 남는다.**
