RAG 파이프라인 + 라우팅 품질

RAG 파이프라인

① 문서 소스
사내 문서 / PDF / Wiki / DB
② 정규화 / 청킹
chunk text hash 기반 임베딩 캐시
③ 인덱싱
변경분 중심 업데이트 · 버전/유효기간 관리
④ 검색 → 인용(Citation)
stale evidence 탐지·차단
⑤ 답변 생성
근거 없으면 강한 결론 금지 → 보류 / 재질문 / 승인 요청
RAG 품질 = "정답 생성"이 아닌 "근거 추적"
근거가 언제 수집됐는지
공식 문서인지 / 내부 문서인지 (A/B/E vs F등급)
문서가 오래됐을 때 처리 방식 (stale evidence 탐지)
어떤 답변에 어떤 근거가 쓰였는지 (답변-근거 연결)
RAG 실패 시 fallback / 승인 플로우

라우팅 품질

Bounded Intent → 구조화된 Plan
의도를 제한해야 필요한 데이터·툴·승인만 포함한 plan 생성 가능
Rule vs LLM 책임 분리
Rule: 필수조건 · 리스크 · 금지행동 · 승인 필요 여부
LLM: 요약 · 작성 · 설명 · 플랜
불확실성 처리
low_confidence → 재질문 or 안전한 fallback
unknown intent → 조회 모드 제한, 실행 액션 미생성
