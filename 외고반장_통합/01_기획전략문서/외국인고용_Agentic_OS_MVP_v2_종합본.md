# 외국인 고용 Agentic OS — 최종 MVP 정리 v2

> **버튼을 누르는 외국인 고용 SaaS가 아니라, 사용자가 목적을 말하면 AI가 신규 인력 요청 → 다국어 컨택 → 비자·서류 점검 → 사람 승인 → 행정사 전달 패키지까지 경로를 짜주는 외국인 고용 에이전트.**

---

## 0. 최종 결론

이번 MVP는 “외국인 고용 관리 SaaS”가 아니라 **외국인 고용 Agentic OS**로 잡는다.

기존 SaaS는 사용자가 메뉴를 찾아 들어가서 하나씩 클릭해야 한다.

```text
직원 관리 클릭
→ 비자 관리 클릭
→ 서류 업로드
→ 메시지 작성
→ 발송
→ 로그 확인
```

우리가 만들 MVP는 이 흐름을 바꾼다.

```text
사용자:
“베트남 E-9 근로자 3명 추가 채용 준비해줘.
그리고 기존 직원 Nguyen 체류만료도 확인해줘.”

AI:
1. 신규 인력 요청서 생성
2. 후보 요건 매칭
3. 다국어 서류 요청 메시지 생성
4. 기존 직원 체류만료 D-day 확인
5. 누락 서류 탐지
6. 담당자 승인 요청
7. 행정사 전달 패키지 생성
8. 감사 로그 저장
```

핵심은 **자동 제출**이 아니라 **자동 준비**다.

AI가 비자를 판정하거나 법률 자문을 하거나 정부 포털에 직접 제출하지 않는다.  
AI는 경로를 짜고, 필요한 자료를 찾고, 문서를 만들고, 사람 승인 후 다음 단계로 넘긴다.

---

## 1. 제품 한 줄 정의

**외국인 고용 Agentic OS**

E-9 외국인 근로자를 고용하거나 채용하려는 제조업 사업장이 자연어로 목적을 말하면, AI가 신규 인력 요청, 다국어 컨택, 비자·서류 점검, 행정사 전달까지 필요한 흐름을 연결해주는 AI 케이스 매니저.

---

## 2. 최종 제품 포지션

이 제품은 **외국인 채용 추천 AI**가 아니다.

또한 **비자 자동 신청 서비스**도 아니다.

최종 포지션은 아래와 같다.

> 외국인 고용 사업장이 놓치면 큰일 나는 채용 준비, 다국어 소통, 체류·서류 업무를 AI가 케이스 단위로 정리하고, 사람 승인 후 안전하게 다음 단계로 넘기는 운영 에이전트.

제품 카피는 아래 방향이 좋다.

```text
나쁜 카피:
AI가 외국인 채용부터 비자까지 자동으로 처리합니다.

좋은 카피:
AI가 외국인 고용 업무의 경로를 짜고,
필요한 서류와 메시지를 준비한 뒤,
사람 승인 후 다음 단계로 넘깁니다.
```

---

## 3. MVP에서 보여줄 3개 Mission Agent

셋 다 비슷한 볼륨과 중요도를 갖도록 설계한다.

### 3.1 인력 확보 에이전트

목적:

> “몇 명 더 뽑고 싶다.”

결과:

> 신규 인력 요청서 + 후보 요건 매칭 + 송출회사/행정사 전달용 문서

내부 구성:

- 인재 요건·매칭 에이전트
- 후보 적합성 검토 에이전트

주의:

- “좋은 사람 추천” 금지
- 성실도, 이탈 가능성, 국적별 선호 판단 금지
- 사람 평가가 아니라 **요건 충족도**만 다룸

---

### 3.2 다국어 컨택 에이전트

목적:

> “외국인 근로자나 후보자와 소통해야 한다.”

결과:

> 다국어 메시지 + 응답 요약 + 상태 업데이트

내부 구성:

- 다국어 컨택·온보딩 에이전트
- 응답 해석·다음 행동 에이전트

주의:

- 자동 발송 금지
- 담당자 승인 전 발송 불가
- 압박성 표현, 차별적 표현, 감정 상태 단정 금지

---

### 3.3 비자·서류 운영 에이전트

목적:

> “놓치면 안 되는 체류·서류 업무를 처리해야 한다.”

결과:

> D-day 탐지 + 누락 서류 + 다국어 요청 메시지 + 행정사 전달 패키지

내부 구성:

- 비자·체류 관리 에이전트
- 서류 체크리스트·패키징 에이전트
- 사람 승인·전문가 전달 에이전트
- 감사 로그·근거 저장 에이전트

주의:

- 비자 가능/불가능 최종 판정 금지
- 정부 포털 자동 제출 금지
- 행정사·노무사 대체 금지

---

## 4. 공통 Agentic Engine 구조

겉으로는 3개 Mission Agent가 보이지만, 내부에는 공통 엔진이 있어야 한다.

```text
[Intent Router]
사용자 목적 분류

↓
[Planner]
해야 할 일 순서 생성

↓
[Tool Executor]
DB 조회 / RAG 검색 / 메시지 생성 / 서류 체크

↓
[Human Approval]
발송·전달 전 승인

↓
[Evidence Log]
근거와 실행 기록 저장
```

이 공통 엔진이 있어야 단순 챗봇이 아니라 agentic AI처럼 보인다.

# 5. 세부 에이전트 정의

MVP 외부 설명은 3개 Mission Agent로 단순화한다.  
하지만 실제 구현과 팀 분배에서는 아래 8개 세부 에이전트를 기준으로 잡는다.

---

## 5.1 인재 요건·매칭 에이전트

### 역할

원래의 “추천 인재 에이전트”에 해당한다.

다만 이름은 바꾸는 것이 좋다.

“추천 인재 에이전트”라고 하면 아래처럼 들릴 수 있다.

> AI가 외국인 근로자를 평가해서 추천한다.

그러면 위험하다.

그래서 MVP에서는 아래처럼 정의한다.

> 사업장의 채용 조건, 업종, 직무, 숙소, 언어, 근무시간, 기존 외국인 직원 구성, 필요 서류를 바탕으로 **채용 가능 조건**과 **후보군 매칭 기준**을 정리하는 에이전트.

즉 사람을 평가하는 에이전트가 아니라, **사업장 조건과 제도 요건을 정리하는 에이전트**다.

### 입력값

```text
- 업종: 자동차부품 제조
- 지역: 충북 음성
- 필요 인원: 5명
- 희망 언어: 베트남어
- 숙소 제공 여부: 제공
- 근무 형태: 주야 2교대
- 기존 외국인 직원 수: 35명
```

### 출력값

```text
- E-9 고용 가능성 체크
- 필요한 사업장 정보
- 후보군 필터 조건
- 송출회사/행정사 전달용 요청서
- 채용 전 확인해야 할 리스크
```

### 출력 예시

```text
신규 인력 요청서 초안

사업장: 충북 음성 자동차부품 제조업
필요 인원: E-9 근로자 5명
희망 언어권: 베트남어
숙소: 제공
근무 형태: 주야 2교대

확인 필요 항목:
1. 현재 외국인 고용 가능 인원 잔여분
2. 내국인 구인 노력 여부
3. 표준근로계약서 준비 여부
4. 숙소 정보 제공 가능 여부
5. 안전교육 다국어 자료 보유 여부

다음 행동:
송출회사 또는 행정사에게 위 조건을 전달하고 가능 후보군 확인 요청
```

### AI 판단 vs 룰 베이스

| 구분 | 내용 |
|---|---|
| 룰 베이스 | 필수 입력값 누락 체크, 인원·지역·업종 필드 검증, 요청서 템플릿 생성 |
| AI 판단 | 자연어 입력 구조화, 채용 조건 요약, 리스크 문장화, 전달용 요청서 작성 |
| RAG | E-9 고용 가능 업종, 고용허가 절차, 필요 서류 관련 공식 문서 검색 |
| 금지 | 국적별 선호 추천, 성실도 예측, 이탈 가능성 점수화, 개인 평가 |

### 성공 기준

```text
- 사용자가 자연어로 말한 채용 요청이 구조화된 요청서로 바뀐다.
- 필수 확인 항목이 빠지지 않는다.
- 송출회사/행정사에게 바로 보낼 수 있는 문서가 나온다.
- 특정 국적이나 개인을 차별적으로 평가하지 않는다.
```

## 5.2 후보 적합성 검토 에이전트

### 역할

이 에이전트는 5.1 인재 요건·매칭 에이전트와 붙어 있지만, 따로 있어야 한다.

이유는 명확하다.

> “요건 정리”와 “후보 비교”는 다른 일이다.

단, 후보 비교도 조심해야 한다.

AI가 하면 안 되는 것:

```text
- 성실할 것 같다
- 오래 일할 것 같다
- 이탈 가능성이 낮다
- 특정 국적이 더 낫다
- 성격이 맞다
```

AI가 해야 하는 것:

```text
- 서류가 준비되어 있는가
- 비자 유형이 맞는가
- 근무 가능 시점이 맞는가
- 언어 지원이 가능한가
- 직무 경험이 요구 조건과 맞는가
- 기숙사/근무시간 조건을 사전에 이해했는가
```

즉 이건 사람의 가치 판단이 아니라 **요건 충족 여부 체크**다.

### 입력값

```text
- 후보자 정보
- 희망 직무
- 근무 가능 시점
- 보유 서류
- 비자 유형
- 언어
- 기숙사/근무조건 이해 여부
- 사업장 요구 조건
```

### 출력값

```text
- 후보별 제출 준비도
- 후보별 요건 충족 여부
- 추가 확인 필요 항목
- 온보딩 필요 항목
- 송출회사/행정사에게 확인할 질문
```

### 후보 비교 예시

```text
후보 A
- 국적: 베트남
- 희망 직무: 제조 조립
- 근무 가능 시점: 6월
- 필요 서류: 여권 있음 / 건강검진 미확인 / 표준계약서 미작성
- 상태: 추가 확인 필요

후보 B
- 국적: 네팔
- 희망 직무: 용접 보조
- 근무 가능 시점: 7월
- 필요 서류: 여권 있음 / 계약서 초안 있음 / 숙소 안내 미전달
- 상태: 온보딩 안내 필요
```

추천 문구는 이렇게 써야 한다.

```text
나쁜 문구:
후보 B가 더 좋은 사람입니다.

좋은 문구:
후보 B는 현재 제출 준비도가 더 높습니다.
```

이 차이가 중요하다.

### AI 판단 vs 룰 베이스

| 구분 | 내용 |
|---|---|
| 룰 베이스 | 필수 서류 보유 여부, 근무 가능 시점, 비자 유형, 입력값 누락 확인 |
| AI 판단 | 후보별 상태 요약, 추가 확인 질문 생성, 담당자 친화적 비교 설명 |
| RAG | 필요 서류, 고용허가 절차, 제도 요건 검색 |
| 금지 | 성격 평가, 성실도 예측, 이탈 가능성 예측, 국적별 우열 판단 |

### 성공 기준

```text
- 후보자를 평가하지 않고 준비 상태를 비교한다.
- 후보별 추가 확인 항목이 명확히 나온다.
- 담당자가 송출회사/행정사에게 무엇을 물어봐야 하는지 알 수 있다.
- “좋은 사람”이 아니라 “준비도 높은 후보”로 표현한다.
```

## 5.3 다국어 컨택·온보딩 에이전트

### 역할

이건 원래의 “인재 컨택 에이전트”에 해당한다.

다만 그냥 “컨택”으로 만들면 위험하다.  
외국인 근로자에게 영업하듯 메시지를 보내는 구조가 되면 안 된다.

MVP에서는 이렇게 잡아야 한다.

> 채용 예정자 또는 기존 외국인 근로자에게 필요한 정보를 다국어로 정확히 안내하고, 답변을 받아 다음 행정 단계로 넘기는 에이전트.

### 하는 일

```text
- 서류 요청 메시지 작성
- 입국 전 안내 메시지 작성
- 기숙사 안내 메시지 작성
- 근무조건 확인 메시지 작성
- 안전교육 안내 메시지 작성
- 급여명세서 설명 메시지 작성
- 응답을 요약해서 담당자에게 전달
```

### 핵심 흐름

핵심은 **자동 발송**이 아니라 **승인 후 발송**이다.

```text
메시지 초안 생성
↓
한국어 원문 표시
↓
외국어 번역 표시
↓
담당자 확인
↓
승인
↓
발송
↓
로그 저장
```

### 출력 예시

```text
[한국어 원문]
안녕하세요. 입국 전 서류 확인을 위해 여권 사본과 증명사진 1장을 이번 주 금요일까지 보내주세요.
제출된 서류는 고용 및 체류 관련 행정 절차에만 사용됩니다.

[베트남어 번역]
...

[담당자 확인 포인트]
- 요청 서류: 여권 사본, 증명사진 1장
- 제출 기한: 금요일
- 개인정보 사용 목적: 고용 및 체류 관련 행정 절차
- 발송 전 승인 필요
```

이 에이전트는 데모에서 매우 좋다.  
사용자가 바로 이해하기 때문이다.

> 사장님이 한국어로 쓰면 외국인 직원에게 맞는 언어로 정리해서 보내주는구나.

### AI 판단 vs 룰 베이스

| 구분 | 내용 |
|---|---|
| 룰 베이스 | 서류명, 제출 기한, 담당자, 수신자, 언어 필드 확인 |
| AI 판단 | 한국어 메시지 작성, 외국어 번역, 쉬운 표현으로 재작성 |
| RAG | 법령·행정 관련 설명이 필요한 경우 근거 검색 |
| 금지 | 자동 발송, 압박성 표현, 차별적 표현, 감정 상태 단정 |

### 성공 기준

```text
- 담당자가 한국어로 입력한 요청이 외국어 안내문으로 변환된다.
- 한국어 원문과 외국어 번역이 함께 보인다.
- 개인정보 사용 목적이 포함된다.
- 담당자 승인 없이는 발송되지 않는다.
```

---

## 5.4 응답 해석·다음 행동 에이전트

### 역할

컨택 에이전트만 있으면 반쪽짜리다.

외국인 근로자가 답장을 했을 때 그걸 해석해야 진짜 에이전트가 된다.

예를 들어 근로자가 베트남어로 이렇게 답했다고 하자.

```text
여권은 있는데 외국인등록증은 아직 없어요. 사진은 내일 보낼 수 있어요.
```

그러면 이 에이전트가 담당자에게 이렇게 정리해야 한다.

```text
응답 요약:
- 여권: 보유
- 외국인등록증: 아직 없음
- 사진: 내일 제출 가능

다음 행동:
1. 외국인등록증이 아직 없는 사유 확인
2. 사진 제출 예정일 내일로 상태 업데이트
3. 서류 패키지 상태를 '부분 완료'로 변경
```

이게 있어야 컨택이 단순 메시지 작성이 아니라 **상태 업데이트**로 이어진다.

### 입력값

```text
- 외국인 근로자 답변
- 원문 언어
- 기존 요청 메시지
- 직원 상태
- 현재 서류 상태
```

### 출력값

```text
- 한국어 요약
- 확보된 정보
- 아직 부족한 정보
- 상태 업데이트 항목
- 다음 행동 후보
- 리마인드 필요 여부
```

### AI 판단 vs 룰 베이스

| 구분 | 내용 |
|---|---|
| 룰 베이스 | 제출 여부 상태값 업데이트, 제출 예정일 리마인드 생성, 필수 필드 누락 확인 |
| AI 판단 | 외국어 답변 요약, 의미 해석, 다음 행동 후보 생성 |
| RAG | 행정·서류 의미 설명이 필요한 경우 관련 문서 검색 |
| 금지 | 감정 상태 단정, 불만 예측, 이탈 가능성 판단 |

### 성공 기준

```text
- 외국어 답변이 한국어 업무 요약으로 바뀐다.
- 답변 내용이 서류 상태 업데이트로 이어진다.
- 담당자가 다음에 무엇을 해야 하는지 알 수 있다.
- 단순 번역기가 아니라 업무 상태를 갱신하는 에이전트처럼 보인다.
```

## 5.5 비자·체류 관리 에이전트

### 역할

이건 MVP 코어다.

이 에이전트가 없으면 외국인 고용 OS가 아니라 그냥 다국어 HR 툴이 된다.

### 하는 일

```text
- 직원별 입국일 저장
- 체류만료일 저장
- 계약종료일 저장
- D-90 / D-60 / D-30 / D-14 / D-7 알림
- 체류만료일과 계약종료일 충돌 탐지
- 고용변동 이벤트 탐지
- 행정사 검토 필요 시점 표시
```

### 출력 예시

```text
직원: Nguyen Van A
비자: E-9
체류만료일: 2026-08-15
현재 상태: D-76

탐지된 이슈:
1. 체류만료 90일 이내 진입
2. 표준근로계약서 최신본 없음
3. 고용허가서 사본 미등록

다음 안전 행동:
- 갱신 가능 여부는 행정사 검토 필요
- 우선 누락 서류 요청 메시지 생성 가능
```

### 중요한 표현 원칙

AI가 하면 안 되는 말:

```text
연장 가능합니다.
```

AI가 해야 하는 말:

```text
연장 검토가 필요한 상태입니다.
아래 서류와 쟁점을 확인한 뒤 행정사 검토로 넘기는 것이 안전합니다.
```

이 차이가 제품의 생존선이다.

### AI 판단 vs 룰 베이스

| 구분 | 내용 |
|---|---|
| 룰 베이스 | D-day 계산, 계약종료일 비교, 체류만료 90일 이내 감지, 상태값 업데이트 |
| AI 판단 | 케이스 요약, 다음 안전 행동 설명, 담당자 친화적 문장 생성 |
| RAG | 관련 규정, 공식 안내, 케이스별 확인 항목 검색 |
| 금지 | 비자 연장 가능/불가능 최종 판정, 법률 자문, 정부 포털 제출 |

### 성공 기준

```text
- 직원 CSV 업로드 후 D-day 상태가 자동 계산된다.
- 체류만료 90일 이내 직원이 자동 탐지된다.
- 계약종료일과 체류만료일 충돌 가능성이 표시된다.
- 행정사 검토 필요 시점이 명확히 표시된다.
```

## 5.6 제출 서류 간소화 에이전트

### 역할

원래의 “제출 서류 간소화 에이전트”에 해당한다.

다만 이름은 더 정확히 하면 아래와 같다.

> 서류 체크리스트·패키징 에이전트

이 에이전트는 MVP의 핵심이다.

### 하는 일

```text
- 케이스 유형별 필요 서류 검색
- 현재 확보 서류와 비교
- OCR로 문서 읽기
- 빠진 서류 탐지
- 빠진 필드 탐지
- 서류 요청 메시지 생성
- 행정사 전달 패키지 생성
```

### 출력 예시

```text
케이스: E-9 체류만료 D-76

필요 서류:
- 여권 사본
- 외국인등록증
- 표준근로계약서
- 고용허가서 사본
- 사업자등록증
- 재직 관련 확인 자료

현재 확보:
- 여권 사본: 있음
- 외국인등록증: 있음
- 표준근로계약서: 없음
- 고용허가서 사본: 없음
- 사업자등록증: 있음

제출 준비율: 60%

누락:
1. 표준근로계약서
2. 고용허가서 사본

다음 행동:
베트남어 서류 요청 메시지 생성 → 담당자 승인 → 발송
```

이건 MVP 발표에서 가장 시각적으로 강하다.

> AI가 실제 일을 줄여준다.

이 메시지가 바로 보인다.

### AI 판단 vs 룰 베이스

| 구분 | 내용 |
|---|---|
| 룰 베이스 | 보유 서류 true/false 비교, 필수 서류 누락 확인, 제출 준비율 계산 |
| AI 판단 | 누락 사유 설명, 요청 메시지 생성, 행정사 전달문 작성 |
| RAG | 케이스 유형별 필요 서류, 공식 안내, 규정 근거 검색 |
| OCR | 업로드 파일에서 이름·날짜·문서 유형·필드 추출 |
| 금지 | 제출 가능성 최종 판정, 법률 판단, 정부 포털 자동 제출 |

### 성공 기준

```text
- 케이스별 필요 서류가 나온다.
- 현재 확보 서류와 비교해 누락 서류가 표시된다.
- 제출 준비율이 시각적으로 보인다.
- 누락 서류 요청 메시지로 이어진다.
- 행정사 패키지 생성으로 이어진다.
```

---

## 5.7 사람 승인·전문가 전달 에이전트

### 역할

이건 제품 안전장치다.

멀티 에이전트에서 제일 중요한 건 **자동화**가 아니라 **어디서 멈출지**다.

### 해야 할 일

```text
- low risk: 단순 안내 메시지 → 담당자 승인 후 발송
- mid risk: 담당자 검토 필수
- high risk: 행정사/노무사 전달
```

### high risk 예시

```text
- 비자 신청 제출
- 고용변동 신고
- 사업장 변경
- 근로계약 종료
- 산재 신고
- 임금체불 관련 상담
```

이런 건 AI가 처리하면 안 된다.  
반드시 사람에게 넘겨야 한다.

### 행정사 전달 패키지 예시

```text
행정사 검토 요청 패키지

직원 정보:
- 이름: Nguyen Van A
- 국적: VN
- 비자: E-9
- 체류만료일: 2026-08-15

현재 상태:
- D-76
- 표준근로계약서 누락
- 고용허가서 사본 누락

관련 쟁점:
- 체류기간 연장 검토 필요
- 계약종료일과 체류만료일 확인 필요

첨부:
- 여권 사본
- 외국인등록증
- 사업자등록증

누락:
- 표준근로계약서
- 고용허가서 사본
```

### AI 판단 vs 룰 베이스

| 구분 | 내용 |
|---|---|
| 룰 베이스 | high-risk 조건 강제 handoff, 승인 전 발송 차단 |
| AI 판단 | 승인 요청 요약, 행정사 전달 요약, 쟁점 정리 |
| RAG | 관련 규정·공식 안내 근거 검색 |
| 금지 | high-risk 업무 자동 처리, 담당자 승인 없는 발송, 행정사 검토 없는 제출 |

### 성공 기준

```text
- 위험도에 따라 어디서 멈출지 결정된다.
- high-risk 케이스는 반드시 행정사/노무사 전달로 넘어간다.
- 담당자는 승인할 것과 전문가에게 넘길 것을 구분할 수 있다.
- AI가 책임지는 범위와 사람이 책임지는 범위가 분리된다.
```

## 5.8 감사 로그·근거 저장 에이전트

### 역할

이건 겉으로는 안 보이지만 반드시 있어야 한다.

왜냐면 이 제품은 법령, 비자, 개인정보, 외국어 메시지가 얽혀 있다.  
나중에 문제가 생기면 이렇게 물어볼 수밖에 없다.

```text
왜 이 메시지를 보냈나요?
누가 승인했나요?
어떤 근거로 이 서류를 요청했나요?
어떤 버전의 규정을 참고했나요?
```

그래서 모든 행동을 저장해야 한다.

### 저장해야 할 것

```text
- 검색한 규정
- 사용한 문서 버전
- 생성한 메시지
- 번역 원문
- 담당자 승인 여부
- 발송 시간
- 수신자
- 행정사 전달 여부
- 누락 서류 판단 근거
```

이게 있어야 이 제품은 “AI 챗봇”이 아니라 **감사 가능한 운영 시스템**이 된다.

### AI 판단 vs 룰 베이스

| 구분 | 내용 |
|---|---|
| 룰 베이스 | 로그 저장, 승인 여부 기록, 발송 시점 기록, 문서 버전 기록 |
| AI 판단 | 로그 요약, 감사용 설명문 생성 |
| RAG | 사용된 근거 문서와 citation 연결 |
| 금지 | 로그 누락, 승인 기록 없는 발송, 근거 없는 판단 저장 |

### 성공 기준

```text
- 모든 메시지·패키지·승인·발송 이력이 남는다.
- 나중에 왜 이 행동을 했는지 추적 가능하다.
- 담당자, 행정사, 내부 운영자가 같은 기록을 보고 확인할 수 있다.
```

| 인재 요건·매칭 | 사장, 인사담당자, 송출회사, 행정사 | 업종, 지역, 필요 인원, 언어, 숙소, 근무 형태 | 신규 인력 요청서, 확인 필요 항목 | 자연어 구조화, 리스크 문장화 | 필수 입력값 누락 체크 | 요청서 생성 완료, 금지 표현 0건 |
| 후보 적합성 검토 | 사장, 인사담당자, 송출회사 | 후보 정보, 서류, 가능 시점, 직무 | 제출 준비도, 추가 확인 항목 | 후보 상태 요약 | 서류 보유 여부 비교 | 사람 평가 없이 준비도 비교 |
| 다국어 컨택·온보딩 | 사업장 담당자, 외국인 근로자 | 대상자, 요청 서류, 기한, 언어 | 한국어 원문, 외국어 번역, 확인 포인트 | 메시지 작성·번역 | 필수 필드 확인 | 승인 전 발송 0건 |
| 응답 해석·다음 행동 | 사업장 담당자 | 외국어 답변, 기존 요청, 직원 상태 | 응답 요약, 상태 업데이트, 다음 행동 | 의미 해석, 요약 | 제출 상태 업데이트 | 답변이 업무 상태로 반영 |
| 비자·체류 관리 | 사장, 인사담당자, 행정사 | 입국일, 체류만료일, 계약종료일 | D-day, 충돌 탐지, 검토 필요 시점 | 케이스 요약 | D-day 계산 | D-day 정확도 100% |
| 서류 체크리스트·패키징 | 사장, 인사담당자, 행정사 | 보유 서류, 업로드 파일, 케이스 유형 | 필요 서류, 누락 서류, 제출 준비율 | 누락 설명, 요청문 작성 | 서류 true/false 비교 | 누락 탐지 정확도 90% 이상 |
| 사람 승인·전문가 전달 | 담당자, 행정사, 노무사 | 생성 메시지, 패키지, 위험도 | 승인 요청, 행정사 패키지 | 쟁점 요약 | high-risk handoff | high-risk 자동처리 0건 |
| 감사 로그·근거 저장 | 담당자, 행정사, 내부 운영자 | 생성물, 근거, 승인 여부 | 로그, 근거, 발송 기록 | 감사용 설명 | 기록 저장 | 추적 가능성 확보 |

# 9. 3명 역할 분배

기능별로 완전히 나누면 위험하다.

나쁜 분배:

```text
A: 추천 인재 에이전트
B: 컨택 에이전트
C: 비자 에이전트
```

이렇게 하면 4주 뒤에 서로 따로 노는 챗봇 3개가 나올 가능성이 높다.

권장 분배는 **Mission Agent 책임 + 공통 구조 합의** 방식이다.

# 12. 4–5주 구현 일정

## Week 1 — 공통 구조 + 각 에이전트 목업

목표:

> 세 에이전트가 같은 입력/출력 구조로 움직이게 만들기.

산출물:

```text
- 공통 schema
- Intent Router mock
- 세 Mission Agent별 mock response
- 8개 세부 에이전트 역할 정의
- 화면 와이어프레임
- 샘플 사업장 데이터
- 직원 CSV
- 후보자 CSV
- 서류 체크리스트 초안
```

역할별 할 일:

| 담당 | 할 일 |
|---|---|
| PM/인력확보 | 케이스 3개 정의, 신규 인력 요청서 템플릿, 후보자 CSV, 후보 적합성 기준 |
| 다국어 컨택 | 메시지 템플릿, 개인정보 사용 목적 문구, 베트남어 출력 형식, 응답 요약 기준 |

## Week 3 — Agentic Flow 연결

목표:

> 목적 하나를 입력하면 3개 Mission Agent와 8개 세부 에이전트가 이어진다.

예시 입력:

```text
베트남 근로자 3명 추가 채용 준비해줘.
그리고 기존 직원 Nguyen 체류만료도 확인해줘.
```

결과:

```text
1. 인력 요청서 생성
2. 후보별 제출 준비도 비교
3. 다국어 서류 요청 메시지 생성
4. 외국인 근로자 응답 요약 및 상태 업데이트
5. 기존 직원 비자/서류 리스크 점검
6. 행정사 패키지 생성
7. 승인 대기 상태 표시
8. 감사 로그 저장
```

역할별 할 일:

| 담당 | 할 일 |
|---|---|
| PM/인력확보 | 출력 문구 정리, 후보 비교 표현 검수, 발표용 시나리오 정리 |
| 다국어 컨택 | 응답 요약/상태 업데이트 연결 |
| 비자·서류 | Router → Planner → Agent 연결, Evidence Log, Human Handoff |

## Week 4 — 데모 완성 + 수요 검증

목표:

> 기능 추가 금지. 발표와 검증에 집중.

산출물:

```text
- 랜딩/데모 화면
- 3개 Mission Agent 실행 로그
- 8개 세부 에이전트 결과 카드
- 수요 검증 인터뷰 5개
- 리스크 정리
- 발표 스크립트
```

수요 검증 질문:

```text
1. 지금 외국인 직원 비자 만료일을 어떻게 관리하세요?
2. 한 번 놓쳤을 때 실제로 어떤 문제가 생기나요?
3. 서류 누락은 누가 확인하나요?
4. 행정사에게 넘기기 전 자료 정리에 얼마나 걸리나요?
5. 외국인 직원에게 서류 요청할 때 언어 문제는 누가 해결하나요?
6. 이 시스템이 있으면 누가 쓰나요? 사장님, 인사담당자, 행정사?
7. 월구독보다 건당 과금이면 쓸 의향이 있나요?
8. 행정사/송출회사 이름으로 제공되면 신뢰가 올라가나요?
9. 지금 이 일을 누구에게 돈 주고 맡기고 있나요?
10. 그 비용이 얼마인가요?
```

```text
7. SNS/단톡방/커뮤니티 감시
8. 출입국 직원 개인 성향 분석
9. 브로커 색출
10. 승인 없는 메시지 자동 발송
11. 후보자의 성격·성실도·장기근속 가능성 평가
```

# 16. 수요 판단

## 16.1 수요는 있는가?

수요는 있다.

다만 수요는 “외국인 채용 추천 AI”에 있는 것이 아니라, **외국인 고용 운영 사고를 줄이는 AI**에 있다.

강한 수요가 예상되는 영역:

```text
- 비자 만료 알림
- 제출 서류 누락 점검
- 행정사 전달 패키지 생성
- 다국어 서류 요청 메시지
- 신규 인력 요청서 정리
```

상대적으로 약하거나 후순위인 영역:

```text
- AI가 좋은 인재를 추천
- AI가 후보자를 자동 컨택
- 이탈 가능성 예측
- 감정 신호 탐지
- 기숙사 생활 리스크 분석
```

---

## 16.2 수요 검증의 핵심 질문

시장 규모보다 먼저 검증해야 할 질문은 이것이다.

```text
사업장 또는 행정사가
비자/서류/다국어 소통을 정리해주는 AI 케이스 패키지에
돈을 낼 것인가?
```

따라서 인터뷰에서는 “좋아 보이세요?”라고 묻지 않는다.

반드시 아래를 확인한다.

```text
1. 지금 이 일을 누가 하고 있는가?
2. 얼마나 자주 발생하는가?
3. 한 번 놓치면 어떤 손실이 생기는가?
4. 지금 돈을 내고 해결하고 있는가?
5. 어떤 형태면 돈을 낼 수 있는가?
6. 직접 SaaS 결제보다 행정사/송출회사 통해 제공되면 신뢰가 올라가는가?
```

# 20. 최종 요약

이 MVP는 외부적으로 아래 3개 에이전트를 같은 비중으로 보여준다.

```text
1. 인력 확보 에이전트
   “몇 명 뽑고 싶다”
   → 신규 인력 요청서 + 후보 요건 매칭

2. 다국어 컨택 에이전트
   “서류 받아줘 / 안내해줘”
   → 다국어 메시지 + 응답 요약 + 상태 업데이트

3. 비자·서류 운영 에이전트
   “이번 달 위험한 직원 찾아줘”
   → D-day 탐지 + 누락 서류 + 행정사 패키지
```

그리고 내부적으로는 아래 8개 세부 에이전트를 사용한다.

```text
1. 인재 요건·매칭 에이전트
2. 후보 적합성 검토 에이전트
3. 다국어 컨택·온보딩 에이전트
4. 응답 해석·다음 행동 에이전트
5. 비자·체류 관리 에이전트
6. 서류 체크리스트·패키징 에이전트
7. 사람 승인·전문가 전달 에이전트
8. 감사 로그·근거 저장 에이전트
```

공통 구조는 아래와 같다.

```text
Intent Router
→ Planner
→ Tool Executor
→ Human Approval
→ Evidence Log
```

최종 방향:

> 버튼을 누르는 외국인 고용 SaaS가 아니라, 목적을 말하면 채용·컨택·비자·서류 흐름을 스스로 연결해주는 외국인 고용 에이전트.

이 방향은 4–5주, 3명 팀이 만들 수 있는 범위 안에서 agentic AI 트렌드와 도메인 실효성을 동시에 잡을 수 있는 구조다.
