# 외고반장 고도화의 의미와 필요성

## 1. 한 줄 요약

외고반장의 고도화는 단순히 기능을 많이 붙인 것이 아니라, 기존의 넓고 추상적인 **Agentic OS 기획**을 실제로 매일 사용할 수 있는 **E-9 운영 리스크 Daily Briefing 중심의 MVP 운영 루프**로 바꾼 것이다.

즉, 제품의 중심이 아래처럼 바뀌었다.

> 매일 아침 회사별 E-9 운영 리스크를 브리핑한다.  
> 체류만료, 누락서류, 계약-비자 충돌, 신고기한, 승인 대기 액션을 보여주고, 근거와 로그를 남긴다.

---

## 2. 고도화 전후의 차이

### 고도화 전

기존 방향은 다음에 가까웠다.

> 사용자가 외국인 고용 관련 목적을 말하면 AI가 업무 흐름을 짜준다.

이 방향은 확장성은 크지만, MVP로는 너무 넓다.

인력 확보, 다국어 컨택, 비자, 서류, 행정사 전달, 승인, RAG, Evidence Log까지 모두 포함되면 제품의 중심이 흐려진다. 사용자가 매일 왜 들어와야 하는지도 약해지고, 개발자 입장에서도 어디까지 만들면 MVP인지 불명확해진다.

### 고도화 후

고도화 후에는 제품의 중심축이 다음처럼 좁혀졌다.

> 매일 아침 회사별 E-9 운영 리스크를 브리핑한다.  
> 체류만료, 누락서류, 계약-비자 충돌, 신고기한, 승인 대기 액션을 보여주고, 근거와 로그를 남긴다.

이렇게 되면 제품이 훨씬 선명해진다.

- 사용자가 매일 들어오는 이유가 생긴다.
- 개발자는 어떤 API, DB, UI를 만들어야 하는지 알 수 있다.
- RAG는 단순 검색 기능이 아니라 “왜 이 위험 판단이 나왔는지 보여주는 근거 시스템”이 된다.
- AI는 모든 것을 자동 처리하는 존재가 아니라, 리스크를 감지하고 근거를 붙이고 승인 가능한 액션으로 정리하는 운영 보조자가 된다.

---

## 3. 고도화가 필요했던 이유

고도화가 필요했던 가장 큰 이유는 **큰 그림만으로는 제품이 닫히지 않기 때문**이다.

Agentic OS라는 말은 멋있지만, 실제 MVP 단계에서는 다음 질문들에 답해야 한다.

- 그래서 첫 화면에 무엇이 뜨는가?
- 사용자는 매일 무엇을 하는가?
- AI가 판단한 결과는 어디 저장되는가?
- AI가 틀리면 누가 확인하는가?
- 근거 문서는 어떻게 보여주는가?
- 승인 전까지 실제 발송은 막히는가?
- 서버를 재시작해도 이력이 남는가?
- 같은 브리핑을 두 번 돌리면 중복 케이스가 생기는가?
- RAG 문서가 오래되면 어떻게 갱신하는가?

이 질문들에 답하기 위해 들어간 것이 Daily Briefing, Risk Rule Engine, Case/Action/Approval/Evidence 모델, DB 저장, Citation 운영, Scheduler/Admin, 외부 전송 안전 경계, 자연어 Workflow 제한, Frontend와 검증 체계다.

즉, 고도화는 기능 욕심이 아니라 **제품 신뢰도를 만들기 위한 구조화 작업**이다.

---

## 4. 핵심 고도화 요소

## 4.1 Daily Briefing

Daily Briefing은 외고반장의 대표 사용 흐름이다.

고도화 전에는 사용자가 무엇인가를 물어봐야 시스템이 반응했다. 그런데 실무자는 매일 바쁘고, 무엇을 물어봐야 하는지도 모를 수 있다. 특히 체류만료, 신고기한, 누락서류 같은 것은 “궁금할 때 검색하는 정보”가 아니라 “놓치면 문제가 되는 업무”다.

그래서 Daily Briefing은 제품의 성격을 바꾼다.

| 구분 | 의미 |
|---|---|
| 이전 | 사용자가 물어보면 답하는 AI |
| 이후 | 시스템이 먼저 오늘의 리스크를 정리해주는 운영 도구 |

B2B SaaS에서 매일 쓰이는 제품은 보통 사용자가 “오늘 무엇을 해야 하는지”를 알려준다. 외고반장에서 그 역할을 하는 것이 Daily Briefing이다.

예를 들면 다음과 같은 항목들이 매일 보인다.

- 오늘 체류만료 임박자 몇 명
- 서류 보완 필요 몇 건
- 승인 대기 몇 건
- 행정사 검토 준비 몇 건

따라서 Daily Briefing은 단순 화면이 아니라 **제품의 사용 습관을 만드는 장치**다.

---

## 4.2 Risk Rule Engine

Risk Rule Engine은 고도화에서 반드시 필요한 요소다.

체류만료 D-day, 누락서류, 신고기한, 계약-비자 충돌 같은 것은 LLM 감으로 판단하면 안 된다. 오늘은 HIGH, 내일은 MEDIUM, 어떤 날은 “괜찮아 보입니다”처럼 판단이 달라지면 실무자는 시스템을 신뢰할 수 없다.

그래서 위험도 판단은 deterministic rule, 즉 규칙 기반으로 빼야 한다.

예시는 다음과 같다.

```text
체류만료 D-7 이내      → CRITICAL
체류만료 D-30 이내     → HIGH
이미 만료됨            → expired
필수 서류가 없음       → missing_document
계약 종료일과 비자 만료일 충돌 → contract_visa_conflict
```

여기서 AI의 역할은 위험 판단 자체가 아니다.

AI는 이미 계산된 위험 결과를 사람이 이해하기 쉽게 설명하거나, 메시지 초안과 행정사 전달 패키지 초안을 만드는 쪽에 쓰는 것이 적절하다.

핵심은 다음과 같다.

> 판단 기준은 rule로 고정하고, 표현과 정리는 LLM이 맡는다.

이렇게 해야 테스트도 가능해진다.

예를 들어 “Nguyen 체류만료일이 20일 남았으면 HIGH가 나와야 한다” 같은 테스트를 만들 수 있다.

---

## 4.3 Case / NextAction / Approval / Evidence 모델

이 모델은 “AI가 위험을 발견했다”에서 끝나지 않게 하기 위해 필요하다.

단순히 브리핑에 다음처럼 표시되는 것만으로는 부족하다.

```text
Nguyen 씨 체류만료 임박.
서류 누락 있음.
```

이렇게만 나오면 담당자는 다시 다음 행동을 생각해야 한다.

고도화 후에는 이 위험을 실제 업무 단위로 쪼갠다.

| 모델 | 의미 | 예시 |
|---|---|---|
| Case | 위험 케이스 | Nguyen 체류만료 D-27 + 여권 사본 누락 |
| NextAction | 해야 할 일 | 여권 사본 요청 메시지 작성, 행정사 검토 패키지 생성, 계약 종료일 확인 |
| Approval | 승인 상태 | 메시지 발송 전 담당자 승인 대기, 행정사 전달 승인 대기 |
| EvidenceEvent | 판단 근거 기록 | 체류만료일은 worker DB 기준, 필수서류 목록은 공식 문서 기준, D-day 계산은 회사 timezone 기준 |

이 구조가 필요한 이유는 실무가 단순 정보 확인으로 끝나지 않기 때문이다.

실무는 보통 다음 흐름으로 움직인다.

```text
문제 발견
→ 해야 할 일 생성
→ 담당자 승인
→ 처리
→ 처리 이력 저장
```

따라서 Case / Action / Approval / Evidence는 외고반장을 단순 AI 답변기에서 **업무 처리 시스템**으로 바꾸는 핵심 모델이다.

---

## 4.4 DB-backed Persistent State

DB-backed persistent state는 상태를 DB에 저장하는 구조다.

초기 데모에서는 메모리나 mock 데이터로도 보여줄 수 있다. 하지만 제품처럼 보이려면 이력이 반드시 남아야 한다.

예를 들어 오늘 브리핑에서 Nguyen 씨가 HIGH로 떴고, 담당자가 메시지 초안을 만들고 승인 대기 상태로 놨다고 하자. 그런데 서버를 재시작했더니 이 정보가 사라진다면, 그것은 제품이 아니라 데모다.

그래서 다음 이력들이 DB에 저장되어야 한다.

- Briefing
- Case
- Action
- Approval
- Evidence
- Handoff
- Export
- Delivery

특히 외국인 고용 운영에서는 나중에 다음 질문에 답할 수 있어야 한다.

- 언제 누가 확인했는가?
- 어떤 근거로 위험 판단을 했는가?
- 어떤 액션이 생성되었는가?
- 누가 승인했는가?
- 실제 외부 전송은 되었는가, 아니면 mock dispatch였는가?

이것은 단순 편의 기능이 아니라 **책임 소재와 감사 가능성**의 문제다.

---

## 4.5 Citation / RAG 운영 기능

여기서 중요한 것은 RAG를 단순한 문서 검색 기능으로 보지 않았다는 점이다.

초기 RAG는 보통 다음 정도로 설명된다.

> 문서를 청킹하고 임베딩해서 질문하면 관련 문서를 찾아온다.

하지만 운영 제품에서는 이것만으로 부족하다.

실무자는 다음을 확인하고 싶어 한다.

- 이 근거가 진짜 공식 문서인가?
- 언제 기준 문서인가?
- 오래된 문서는 아닌가?
- 이 citation이 실제 답변과 연결되어 있는가?
- 합성 데이터가 섞인 것은 아닌가?
- 근거 문서가 바뀌면 재색인되는가?

그래서 고도화에서 다음 기능이 필요해진다.

- citation viewer
- citation validation
- stale / synthetic / missing evidence 표시
- refresh queue
- official source fetch
- PDF / HTML parser
- RAG chunk reindex
- Chroma upsert

이것은 RAG를 “검색 기능”에서 **근거 품질 관리 시스템**으로 올리는 작업이다.

특히 외고반장처럼 행정, 비자, 서류를 다루는 도메인은 근거가 틀리면 치명적이다. 답변이 그럴듯해도 근거가 오래됐거나 잘못된 문서라면 제품 신뢰가 무너진다.

따라서 Citation 운영 기능은 고도화에서 필수다.

## 4.6 Scheduler와 Admin 운영 기능

Daily Briefing이 제품의 중심이라면, 사용자가 버튼을 누를 때만 브리핑이 생성되는 것으로는 부족하다. 매일 자동으로 돌아가야 한다.

그래서 Scheduler가 필요하다.

예를 들면 다음 기능이 필요하다.

- 매일 오전 9시에 회사별 브리핑 생성
- 실행 성공/실패 기록
- 마지막 실행 시각
- 브리핑 히스토리
- source summary
- pilot metrics

Admin 화면도 같은 이유로 필요하다.

개발자만 보는 로그가 아니라 운영자가 볼 수 있는 상태판이 필요하다.

운영자는 다음을 확인해야 한다.

- 오늘 브리핑이 생성되었는가?
- 근거 문서가 오래되었는가?
- 승인 대기 액션이 몇 개 있는가?
- CRITICAL / HIGH가 몇 건인가?
- 어떤 회사 데이터가 들어와 있는가?

따라서 Scheduler/Admin은 멋진 부가 기능이 아니라 **파일럿 운영을 가능하게 하는 최소 장치**다.

---

## 4.7 외부 전송 안전 경계

외부 전송 안전 경계는 매우 중요하다.

MVP에서는 다음을 하지 않는다.

- 실제 카카오 전송
- 실제 이메일 전송
- 실제 행정사 전달
- 정부 포털 제출

대신 mock dispatch 기록만 남긴다.

이것은 후퇴가 아니라 좋은 판단이다.

외고반장에서 가장 위험한 것은 AI가 실제 외부 행동을 해버리는 것이다. 예를 들어 AI가 잘못된 베트남어 메시지를 근로자에게 보내거나, 아직 검토하지 않은 패키지를 행정사에게 전달하거나, 정부 포털 제출을 잘못하면 사고가 된다.

그래서 MVP에서는 실제 전송을 하지 않고, mock dispatch로 “전송 경로가 어떻게 기록될지”만 검증한다.

데모에서도 표현을 조심해야 한다.

| 나쁜 표현 | 좋은 표현 |
|---|---|
| 메시지 전송 완료 | 전송 요청 생성, 담당자 승인 필요, mock dispatch 기록 완료 |

전자는 위험한 자동화처럼 보이고, 후자는 안전한 운영 시스템처럼 보인다.

---

## 4.8 자연어 Workflow의 안전한 제한

자연어 요청은 필요하다. 사용자가 매번 버튼과 필터만 누르는 것보다 “오늘 위험한 케이스 정리해줘”라고 말하는 것이 편하다.

하지만 자연어를 받는다고 해서 LLM이 자유롭게 tool을 호출하면 위험하다.

그래서 고도화에서는 다음 구조가 필요하다.

```text
자연어 요청
→ bounded intent
→ structured plan
→ 기존 Daily Briefing service 실행
```

즉, LLM이 자유롭게 “내가 알아서 하겠습니다”라고 움직이는 것이 아니라, 자연어를 정해진 intent로 바꾸고, 실행은 이미 검증된 rule/service가 맡는다.

핵심은 다음과 같다.

> 자연어 UX는 열어주되, 실행 경로는 닫아둔다.

이렇게 해야 사용자는 편하고, 시스템은 안전하다.

## 4.9 Frontend와 검증 체계

`/dashboard`, `/admin`, `/evidence`가 생겼다는 것은 단순히 화면이 생겼다는 의미가 아니다.

각 화면은 제품 안에서 역할이 다르다.

| 화면 | 사용자 | 역할 |
|---|---|---|
| `/dashboard` | 실무 담당자 | 오늘의 위험, 승인 대기, 브리핑, 케이스 확인 |
| `/admin` | 운영자/관리자 | 스케줄러, 데이터 소스, citation refresh, metrics 관리 |
| `/evidence` | 실무자/관리자/검토자 | AI가 왜 그렇게 판단했는지 근거 확인 |

여기에 Backend tests 255 passed, Frontend build passed 같은 검증이 붙었다는 것은 “말로만 기획한 것”이 아니라 최소한 빌드와 테스트를 통과하는 구조로 갔다는 의미다.

팀 협업에서는 이것이 중요하다.

기획자는 “이런 기능이 있으면 좋겠다”고 말할 수 있지만, 개발자는 다음을 알아야 한다.

- 어떤 스키마로 저장하는가?
- 어떤 API가 필요한가?
- 어떤 상태값이 있는가?
- 어떤 테스트로 완료를 판단하는가?
- 어떤 화면에서 보여주는가?

이 문서는 그 간극을 줄인다.

## 5. 고도화의 전체 의미

전체 의미를 압축하면 다음과 같다.

> 외고반장을 “AI가 외국인 고용을 도와주는 서비스”에서  
> “매일 운영 리스크를 감지하고, 근거를 연결하고, 승인 가능한 액션으로 바꾸고, 이력을 남기는 업무 시스템”으로 바꾼 것.

이게 고도화의 핵심이다.

---

## 6. 고도화가 필요한 3가지 이유

## 6.1 제품 포지셔닝을 좁히기 위해

너무 넓은 Agentic OS는 팔기 어렵고 만들기도 어렵다.

E-9 운영 리스크 Daily Briefing으로 좁히면 고객의 지불 이유가 분명해진다.

> 놓치면 과태료, 행정 리스크, 운영 사고가 나는 것들을 매일 잡아준다.

이 메시지는 훨씬 구체적이다.

---

## 6.2 MVP를 실제 구현 가능한 Thin Slice로 만들기 위해

인력 확보, 컨택, 비자, 서류, 행정사 전달을 모두 같은 깊이로 만들면 MVP가 무거워진다.

Daily Briefing을 중심축으로 두면 다음 한 바퀴가 닫힌다.

```text
데이터 입력
→ 위험 판단
→ 근거 연결
→ 액션 생성
→ 승인 대기
→ 로그 저장
```

이것이 구현 가능한 MVP thin slice다.

---

## 6.3 AI 제품의 신뢰와 안전을 확보하기 위해

LLM이 마음대로 판단하고 전송하는 구조는 위험하다.

실무에서 신뢰받으려면 다음 장치들이 필요하다.

- rule 기반 판단
- citation 근거
- approval gate
- Evidence Log
- mock dispatch
- tenant scope
- RBAC
- idempotency
- refresh queue

이 장치들은 AI를 더 화려하게 만들기 위한 것이 아니라, AI가 함부로 하지 못하게 막고, 사람이 책임 있게 검토할 수 있게 만드는 구조다.

---

## 7. 결론

조금 냉정하게 말하면, 이 고도화는 “더 멋진 AI”를 만든 것이 아니다.

오히려 반대다.

> AI가 함부로 못 하게 막고, 제품이 반복 가능한 업무 루프로 돌아가게 만든 것.

B2B 운영 도구에서 중요한 것은 “AI가 똑똑해 보이는가”가 아니다.

중요한 것은 다음이다.

- 매일 같은 기준으로 위험을 잡는가?
- 사람이 승인할 수 있는 형태로 정리되는가?
- 나중에 왜 그렇게 판단했는지 설명 가능한가?
- 실수했을 때 추적 가능한가?
- 실제 외부 행동은 안전하게 막혀 있는가?

그래서 이 문서의 진짜 가치는 기능 수가 아니라 구조다.

최종 루프는 다음과 같다.

```text
리스크 감지
→ 근거 연결
→ 해야 할 일 생성
→ 승인 대기
→ 초안/패키지 생성
→ mock dispatch
→ Evidence Log
→ 관리자 운영
→ 테스트 검증
```

이 루프가 닫혔다는 것.

그것이 외고반장 고도화의 의미다.
