# 데모

## 1. 데모 케이스 고정값

**Nguyen V. 체류기간 연장 서류 요청**

상황

Nguyen Van A는 베트남 국적 E-9 근로자이고, 화성 2공장 조립라인에서 일하고 있음. 체류만료일은 2026년 6월 20일이고, 현재 D-30 상태. 체류기간 연장을 준비해야 하는데 **표준근로계약서 사본**과 **여권 사본**이 아직 누락되어 있음. AI는 이전 대화 기록을 확인했고, 3일 전에 서류 요청 이력이 있다는 것도 확인함. 그래서 베트남어 요청 메시지와 한국어 번역본을 만들고, 대표 승인 후 Zalo 또는 메시지 채널로 발송하는 흐름. Nguyen 케이스의 상세 패널 요소로 체류만료일, 계약종료일, 누락 서류, 이전 메시지, 승인 전 외부 발송 제한

**2. 전체 시연 흐름**

| 단계 | 화면 | 유저가 하는 일 | AI가 해주는 일 | 출력값 |
| --- | --- | --- | --- | --- |
| 1 | PC 대시보드 | 오늘 할 일 확인 | 업무 큐 정렬 | “체류기간 임박 4명”, “승인 대기 5건” |
| 2 | PC 업무 큐 | Nguyen 행 클릭 | 케이스 상세 불러오기 | 오른쪽 상세 패널 |
| 3 | PC 상세 패널 | 왜 필요한지 확인 | 체류만료, 누락서류, 이전 기록 분석 | 확인 이유, 누락 서류, 다음 할 일 |

| 8 | 모바일 초안 확인 | VN/KR 메시지 확인 | 예상 응답 시나리오 제안 | 긍정/질문/지연 시나리오 |

| 10 | PC 업무 기록 | 결과 확인 | 업무 기록 저장 | 업무 기록 #4789 |

## 3. 입력값과 출력값으로 나누기

### A. 사용자가 직접 입력하거나 선택하는 값

유저가 처음부터 많은 정보를 입력하는 구조는 별로야. 외고반장은 “업무툴”이니까, 대부분은 이미 시스템에 있는 정보를 AI가 끌어와야 함. 

유저 입력값은 최소화해야 해.

| 입력 주체 | 입력값 | 예시 | 왜 필요한가 |
| --- | --- | --- | --- |
| 관리자/인사 담당자 | 사업장 선택 | 한별제조 / 화성 2공장 | 해당 사업장 업무  필대터링 |
| 관리자/인사 담당자 | 업무 선택 | Nguyen V. 체류기간 연장 서류 요청 클릭 | 케이스 상세 열기 |
| 관리자/인사 담당자 | 액션 선택 | 초안 보기 / 대표 승인 요청 | AI가 준비한 작업을 다음 단계로 넘김 |
| 대표자/사업자 | 승인 여부 | 보내기 승인 / 수정 요청 / 보류 | 외부 발송 전 최종 사람 승인 |
| 대표자/사업자 | 수정 요청 내용 | “조금 더 정중하게 바꿔줘” | AI가 초안을 다시 생성 |

### B. 시스템에 이미 저장되어 있어야 하는 입력값

이건 사용자가 매번 입력하는 게 아니라, DB나 샘플 JSON에 들어 있어야 하는 값

| 데이터 영역 | 입력값 | Nguyen 예시 |
| --- | --- | --- |
| 근로자 정보 | 이름 | Nguyen Van A |
| 근로자 정보 | 국적 | 베트남 |
| 근로자 정보 | 체류자격 | E-9 |
| 근로자 정보 | 언어 | 베트남어 |
| 근로자 정보 | 연락 채널 | Zalo |
| 사업장 정보 | 소속 공장 | 화성 2공장 |
| 사업장 정보 | 라인 | 조립라인 |
| 체류 정보 | 체류만료일 | 2026.06.20 |
| 계약 정보 | 계약종료일 | 2026.07.01 |
| 서류 정보 | 누락 서류 | 표준근로계약서 사본, 여권 사본 |
| 메시지 이력 | 이전 요청 기록 | 3일 전 서류 요청 이력 있음 |
| 정책/운영 규칙 | 외부 발송 제한 | 대표 승인 전 발송 불가 |

## 7. 모바일 화면별 입력값/출력값

### 모바일 화면 1. 오늘 브리핑

대표자가 모바일을 열면 처음 보는 화면.

**유저 입력값**

없음. 그냥 앱/링크를 열면 됨.

**AI 출력값**

```
대표님, 오늘 확인이 필요한 외국인 고용 업무가 3건 있습니다.
```

카드 예시:

```
Nguyen V. 체류기간 연장 서류 요청

체류만료 D-30
누락 서류 2건
AI가 베트남어 요청 메시지를 준비했습니다.
승인 후 근로자에게 발송됩니다.

[초안 보기] [수정 요청] [보내기 승인]
```

여기서 모바일은 사장님 기준이라 “판단 근거”를 너무 길게 보여주면 안 돼.

대신 카드 안에 **왜 지금 봐야 하는지**가 한 줄로 보여야 해.

> 체류만료 D-30 / 누락 서류 2건 / 승인 후 발송
> 

이 정도가 딱 좋아.

### 모바일 화면 3. Live Agent 처리 화면

이 화면은 데모에서 꽤 중요해.

왜냐면 그냥 초안이 뜨는 것보다, AI가 실제로 “업무 단계를 밟았다”는 느낌이 나거든.

노션에도 Live 화면은 메인 화면이 아니라 **승인 직전 화면**으로 쓰고, 4단계 정도면 충분하다고 잡혀 있어.

**유저 입력값**

| 사용자가 하는 일 | 값 |
| --- | --- |
| 초안 보기 클릭 | Live Agent 처리 시작 |

**AI 출력값**

```
1. 근로자 정보 확인 완료
Nguyen Van A · 베트남 · E-9 · Zalo

2. 이전 대화 기록 확인 완료
3일 전 표준근로계약서 요청 이력 있음

3. 메시지 초안 생성 완료
베트남어 원문 + 한국어 번역 생성

4. 발송 전 승인 대기
대표님 승인 후 발송됩니다
```

이 화면은 “AI가 뭘 했는지”를 보여주는 장면이야.

그래서 디자인은 너무 화려하면 안 되고, 체크리스트처럼 조용하게 지나가는 게 좋아.

## 8. 유저 vs AI 역할 분리

이걸 발표에서 설명할 때는 아래 표가 제일 좋아.

| 구분 | 유저가 하는 일 | AI가 하는 일 |
| --- | --- | --- |
| 업무 발견 | 오늘 할 일 확인 | 체류만료/누락서류/승인대기 업무 큐 생성 |
| 케이스 확인 | Nguyen 케이스 클릭 | 근로자 정보, 체류정보, 서류 상태 불러오기 |
| 이유 확인 | 왜 필요한지 읽음 | D-day, 누락 서류, 이전 기록을 근거로 요약 |
| 메시지 준비 | 초안 확인 | 베트남어 요청문 + 한국어 번역 생성 |
| 리스크 통제 | 승인/수정/보류 선택 | 승인 전 외부 발송 차단 |
| 후속 처리 | 승인 클릭 | 업무 기록 저장, 상태 업데이트, 리마인드 예정 생성 |
| 기록 확인 | 나중에 업무 기록 확인 | 누가, 언제, 무엇을 승인했는지 로그화 |

**유저는 최종 책임과 승인을 맡고, AI는 반복 확인·초안 작성·상태 업데이트·기록 저장을 맡는다.**

중요한 건 완전 자유형 챗봇처럼 보이게 하지 않는 것이야. 추천 질문 버튼을 함께 두면 데모가 안정적이고, 사용자가 뭘 물어봐야 할지 고민하지 않아도 된다.

데모에서 쓰기 좋은 질문 5개
오늘 비자·서류 관련해서 먼저 볼 일 있어?
왜 Nguyen이 우선 확인이야?
무슨 서류가 빠졌어?
이 메시지 조금 더 정중하게 바꿔줘.
Nguyen이 2일 동안 답이 없으면 다음에 뭐 해?

이 5개면 충분하다. 너무 복잡한 질문을 넣으면 데모가 흔들릴 수 있으니, 현재 구현 기준으로 안정적으로 답할 수 있는 질문만 쓰는 게 좋다.
