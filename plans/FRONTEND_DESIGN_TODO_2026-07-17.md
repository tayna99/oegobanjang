## 플랫폼 관례 (전 항목 공통 — 브리프마다 반드시 표기)

이 앱은 반응형 웹 단일(통합설계 D4)이고, 기존 화면은 셋 중 하나의 패턴을 따른다. 아래 A·B절
각 항목에 "플랫폼:" 줄로 어느 패턴인지 명시했다 — 브리프·목업에도 그대로 옮긴다.

1. **단일 반응형**: 분기 없이 한 레이아웃이 모바일~PC를 커버(중앙 `max-w` 컨테이너).
   설정 화면군(`SettingsHubPage`·`MembersPage`·`DelegationPage` — `useIsDesktop` 없음),
   행정사 화면군(`ExpertLinkPage`·`ExpertDashboardPage` — `max-w-screen-md mx-auto`)이 선례.
2. **모바일/PC 분기**: 같은 데이터층 위에 폼팩터별 별도 프레젠테이션(`useIsDesktop`).
   홈(브리핑/컨트롤타워), 케이스(상세/워크벤치), 메시지(스레드/워크벤치), 거버넌스가 선례.
   **분기 화면을 고칠 땐 양쪽 프레젠테이션을 모두 확인** — PC 분기 누락 전례 있음
   (2026-07-16 docUpdates 오버레이가 모바일에만 반영됐던 버그, HANDOFF 기록).
3. **PC 전용 + 모바일 안내**: PC 워크벤치 전용 화면, 모바일 진입 시 `PcOnlyNotice` 렌더.
   CSV 업로드·근로자 데이터·발송 실행 큐가 선례.

## A. 신규 화면 — claude.ai/design 목업 신규 생성 대상

### A-1. 근로자 응답 링크 페이지 (R3.2 — SMS 인바운드의 프론트 절반)

- **근거**: `docs/MESSAGING_CHANNELS.md` §3(수신 파이프라인)·§5 ② — "만료형 토큰 응답
  페이지(모국어 버튼+자유입력) → 인바운드 정규화 → N02 → M6 해석 큐".
  `reference/specs/2단계_알림카탈로그_딥링크맵_v1.md` §4.1(N02 = worker_replied).
- **성격**: 근로자 휴대폰에서 열리는 **무인증 모바일 웹 단독 페이지**(앱 Shell 크롬 없음) —
  기존 `/link/:packageId`(`ExpertLinkPage`)와 같은 부류의 형제 라우트.
- **플랫폼**: 패턴① 단일 반응형 — `ExpertLinkPage`와 동일한 `max-w-screen-md` 중앙 카드
  레이아웃. 모바일 우선이되 PC 브라우저에서 열려도 같은 레이아웃이 그대로 동작(근로자가
  PC로 열 수 있음). 목업은 모바일 캔버스 1벌이면 충분 — 별도 PC 목업 불필요.
- **화면·상태 구성** (전 상태를 목업에 포함할 것):
  1. **정상**: 담당자가 보낸 메시지 원문(근로자 모국어) + 응답 수단 2종 — ① 모국어 버튼형
     선택지(예: "확인했습니다 / 다시 보내주세요 / 질문이 있습니다"류, 메시지 유형별 프리셋)
     ② 자유 텍스트 입력(선택 사항). 하단 제출 버튼 1개.
  2. **제출 완료**: 확인 문구 + 재제출 불가 잠금(`ExpertLinkPage` 회신 폼의 전송 후 잠금
     패턴과 동일 관례).
  3. **만료됨**: 링크 유효기간 경과 안내(만료형 토큰).
  4. **이미 응답함**: 기존 응답이 접수됐다는 안내(내용 재노출 없음).
  5. **링크 없음/무효**: "링크를 찾을 수 없습니다"만 — 존재 자체 비노출(`ExpertPackagePage`
     inScope 검사와 동일 원칙).
- **다국어**: UI 문구 자체가 근로자 모국어(`Message.lang` — vi/mn/bn…)로 나와야 한다.
  목업은 vi 기준 1벌 + 한국어 병기 여부를 디자인 결정으로 명시.
- **가드레일**(브리프에 반드시 포함):
  - 근로자 응답 원문은 이 페이지와 스레드 내부에서만 — 목록 미리보기·Evidence 요약 노출
    금지(GOTCHAS §3).
  - 응답이 케이스 상태를 직접 바꾸지 않는다 — M6 해석 큐로만 들어가고 담당자 확인 후 반영
    (`isFinal:false` 계약). 페이지에 "담당자 확인 후 반영됩니다"류 안내 문구.
  - 이 페이지에 외국인등록번호 등 PII 표기 금지. 단정형·이모지·느낌표 금지.
- **재사용**: Montage 모바일 토큰, Button/Chip. 백엔드 토큰 검증은 R3.2 동반 작업 —
  목업은 상태 5종의 표현만 확정하면 된다.

### A-2. 서류 스캔 업로드 · 자동분류 확인 플로우 (R5.2 — OCR)

- **근거**: `docs/DESIGN_SYNC_AUDIT_2026-07-13.md` §3 델타표 4b — 원 목업은 "확인 대기까지만
  그림"(OCR 파이프라인 없음, 자리표시 카드). 현재 코드도 정적 카드뿐:
  `WorkerDataWorkbench.tsx:94-100` "준비 중 — 이번 릴리스에는 포함되지 않습니다".
- **성격**: PC 워크벤치(`/cases/workers` 우측 레일)의 자리표시 카드를 실제 플로우로 교체.
  **원 목업(4b)이 안 그린 뒷단계부터가 신규 디자인 범위다.**
- **플랫폼**: 패턴③ PC 전용(1560px 워크벤치 캔버스) — 모바일 진입 시 기존 `PcOnlyNotice`
  재사용(CSV 업로드·발송 큐와 동일 처리, 별도 모바일 목업 불필요). 모바일 카메라 촬영
  업로드는 이번 범위 밖 — 필요해지면 별도 항목으로 분리.
- **화면·상태 구성**:
  1. 드롭존(여권·계약서·증명서, 지원 형식 안내) → 업로드·분류 진행 중(스켈레톤 — 수치는 `--`).
  2. **분류 결과 미리보기**(신규 핵심): 파일별 행 — 매칭된 근로자 · 서류 유형 · 분류 신뢰도.
     상태 칩 3종: 정상 매칭(positive) / 저신뢰·확인 필요(high) / 미매칭(critical).
     CSV 업로드(검증 미리보기 테이블)와 동일 구조·동일 Chip 색표 재사용.
  3. **수동 보정**: 미매칭·오분류 행의 근로자/서류 유형을 드롭다운으로 교정.
  4. **확인 대기 반영**: "확인 대기로 올리기" 확정 → 케이스 서류 상태에 "확인 대기" 항목으로
     연결(자동으로 '수령 완료' 처리하지 않음).
- **가드레일**: 자동 분류 결과를 사람 확인 없이 확정하지 않는다("확인 대기"가 종착).
  스캔 원본은 LLM/RAG에 전달되지 않는다는 문구(현 카드의 "개인정보는 DB에만 저장되고,
  LLM/RAG에 전달되지 않습니다"와 통일). 문서 미리보기에 외국인등록번호 원문이 보이면 안
  된다(썸네일 마스킹 처리 방침을 목업에서 결정).
- **재사용**: 워크벤치 크롬(52px nav·좌측 패널·우측 레일), CSV 업로드의 3단계 구조·검증
  테이블·요약 스탯 로우.

### A-3. 알림 설정 화면 (설정 > 알림 — 스펙만 있고 화면·목업 모두 없음)

- **근거**: `reference/specs/2단계_알림카탈로그_딥링크맵_v1.md` **§6이 요구사항 표까지 확정**해
  뒀으나 화면이 없다. 현 설정 허브(`SettingsHubPage`)는 구성원·위임·승인 정책만 있음.
  R5.4(알림·푸시 실발송)의 프론트 선행 조각.
- **화면 구성** (§6 표 그대로):
  | 항목 | 기본값 | 디자인 포인트 |
  |---|---|---|
  | 아침 브리핑 시각 | 08:30 | 회사 단위 설정(개인 아님) 명시 |
  | 승인 요청 즉시 알림 | ON | **OFF 불가(강제)** — "끌 수 없는 항목"의 UI 표현이 핵심 결정. 승인 권한자에게만 노출 |
  | 응답 도착 즉시 알림 | ON | OFF 시 "다이제스트로 강등" 안내 |
  | CRITICAL 야간 알림 | ON | OFF 가능하되 경고 문구 표시 |
  | 주간 요약 | OFF | 옵트인 |
  | 채널 우선순위 | 푸시 우선 | 읽기 전용(알림톡 폴백 여부 표시) |
- **플랫폼**: 패턴① 단일 반응형 — 기존 설정 화면군과 동일하게 `useIsDesktop` 분기 없이
  한 레이아웃(모바일 캔버스로 목업 1벌, PC에서도 같은 화면 그대로).
- **재사용**: `SettingsHubPage`의 행 목록·세그먼트 버튼 패턴, 기존 설정 화면 3종과 같은 크롬.
- **가드레일**: "승인 요청 즉시 알림"은 토글 자체가 비활성(승인 루프는 꺼질 수 없음) —
  비활성 사유를 1줄로 표기. 단정형·이모지 금지.

### A-4. 행정사 화이트라벨 v1 화면군 (R5.1)

- **근거**: `reference/specs/7-1_행정사_화이트라벨_v1.md` — §7이 화면 요건을 이미 정의.
- **선행 조건(디자인 착수 전)**: 위탁(§26) vs 제3자 제공(§17) **법무 분류 확인**(문서
  §5.5·§9·§10). 뒤집히면 `ExpertGrant`가 tenant → tenant+worker 단위로 커져 (a) 화면의
  구조가 바뀐다.
- **화면 목록**:
  - **(a) 초대(위탁) 관리 — 고객사 측 신설**(§7.2): 설정 내 "행정사 관리" 탭 — 사무소
    검색(사업자번호 포함, 동명 오초대 방지)/신규 등록 → 담당자 이메일 → scope(고정) +
    기간(**종료일 필수, 무기한 선택 불가**) + 위탁계약 근거 체크 → 목록(활성 grant·만료일·
    재확인 필요 배지·**최근 열람 이력 요약**(§6.4)·미수락 배지+재발송·철회 버튼).
    권한: owner만 철회, manager는 초대까지(내부 구성원 관리와 별도 탭으로 분리).
  - **(b) 사무소 직원 관리 — 행정사 측 신설**(§7.3): 사무소 관리자 전용 — 로스터 조회·이메일
    초대·퇴사자 suspended 처리 + 구성원별 회사 자료 열람 이력(§6.4).
  - **(c) 행정사 로그인·본인확인 화면군**(§3.2·§3.3): 이메일 로그인, 패키지 최초 진입 시
    이메일 OTP 모달, 비밀번호 재설정. v0의 "URL=토큰" 모델을 세션 모델로 교체하는 진입부.
  - **(d) 기존 3화면 확장**(§7.1 — 신규 화면 아님, 목업 델타로 처리): 대시보드 카피
    교체("계정 없이 안전 링크로" → 세션 모델 카피), 브랜드 헤더에 tenant별 저채도 accent
    배지(다중 회사 오조작 방지), 회신 폼에 회사명 전달 표시.
- **플랫폼**: (a) 고객사 측 = 패턴① 설정 화면 관례(단일 반응형 — 모바일에서 owner가
  철회하는 시나리오 포함). (b)(c) 행정사 측 = 패턴① expert 화면 관례(`max-w-screen-md`
  단일 반응형 — 행정사는 주로 PC지만 모바일에서도 동일 레이아웃 동작). (d)는 기존 화면
  플랫폼 그대로. 전부 별도 PC/모바일 이중 목업 불필요 — 캔버스 1벌씩.
- **가드레일**: PII 노출은 §9 결정 A(이름·국적 평문, 식별번호만 마스킹) — 고객사 앱의
  `maskId()` 전체 마스킹과 **의도적으로 다른 정책**임을 브리프에 명시(감사 때 결함으로
  오판하지 않도록). 열람 이력은 개인(사무소 구성원) 단위 표기.

### A-5. 앱 내 알림함 (R5.4 — 범위 자체가 미확정, 결정 선행)

- **확인 결과**: 2단계 스펙은 OS 푸시 + 딥링크 맵 + 설정 화면(§6, → A-3)까지만 정의하고,
  **앱 내 알림함(수신 이력 리스트)은 어느 스펙에도 없다.** R5.4의 "수신부만"이 OS 푸시
  수신을 뜻하는지 인앱 알림함까지인지가 미정.
- **진행**: "알림함 UI가 필요한가"부터 결정 → 필요 판정 시에만 브리프 작성(그때 플랫폼도

## B. 기존 화면 확장 — 부분 목업 또는 디자인 결정 필요 (신규 화면 아님)

### B-1. PC 케이스 테이블 프리셋 필터 5종 — 엔지니어링 브리프 (2026-07-17 코드 대조 확정)

- **재확인 결과 — "공유 여부"는 이미 결정돼 있었다.** `CaseWorkbench.tsx`가 이미 모바일
  `CASE_FILTERS`를 그대로 import·렌더 중(`import { CASE_FILTERS, ... } from '@/lib/cases'`,
  필터 칩 렌더 루프). 즉 미결정 상태가 아니라 **암묵적으로 "공유"가 이미 구현돼 있고,
  4a 목업 전용 프리셋 3종(이번달 D-30·서류준비율 80%↓·행정사 전달예정)만 미구현**인
  상태 — ROADMAP M4.5 4.5a가 "미채택"으로 기록한 그대로. **디자인 몫 없음(목업 존재
  확인됨), 아래는 순수 구현 브리프.**
- **결정 대상**: 목업 전용 3프리셋을 실제로 만들 것인가, 아니면 현 상태(모바일 필터 재사용)를
  최종 스코프로 굳힐 것인가.
- **만들 경우 난이도가 프리셋마다 다르다** — `matchesFilter(card, preset)`는 지금
  `CaseCard` 한 개만 받는다:

  | 프리셋 | 판정 로직 | 구현 난이도 |
  |---|---|---|
  | 행정사 전달예정 | `card.primaryAction.kind === 'package'` 또는 `secondaryAction.kind === 'package'` | `CaseCard` 필드만으로 즉시 가능(조인 불필요) |
  | 이번달 D-30 | `card.dDay`가 임계 이내(정확한 의미 — "이번 달" 캘린더 기준 vs 단순 "D-30 이내" — 는 제품 결정 필요) | `CaseCard.dDay`만으로 즉시 가능 |
  | 서류준비율 80%↓ | 현재 `docsFraction`은 `CaseWorkbench.tsx`가 `CASE_SHEETS`를 별도 조인해 그 자리에서 계산(`sheet?.docs` 기반) — `CaseCard`엔 이 값이 없다 | `matchesFilter`가 `CaseCard`만 받는 현재 시그니처로는 **불가능** — 아래 결정 필요 |

- **서류준비율만 진짜 결정 지점**: (A) `filterCases`/`matchesFilter` 시그니처에 `CaseSheet`
  조회를 추가로 넘긴다(함수 변경 파급 큼) vs (B) `CaseCard`에 `docsCompleteness?: number`
  (0~100) 필드를 추가한다 — **이미 존재하는 `evidenceCompleteness?: number`(근거 완성도,
  `types.ts:61`)와 정확히 같은 패턴**이라 최소 diff, `matchesFilter` 시그니처 불변.
  **(B) 권고.**
- **반영 지점**(만들기로 할 경우): `types.ts`(`CaseCard.docsCompleteness` 추가) ·
  `lib/cases.ts`(`PC_CASE_PRESETS` 3종 + `matchesFilter` 확장 + `normalizeCaseFilter`가
  폼팩터별 키 집합을 받도록 확장, `?filter=` 딥링크는 합집합으로 받되 폼팩터에 없는 키는
  '전체' 폴백 — M7 딥링크 계약 유지) · `CaseWorkbench.tsx`(필터 칩을 `CASE_FILTERS`→
  `PC_CASE_PRESETS`로 교체, 기존 로컬 `docsFraction` 계산을 `card.docsCompleteness`로
  대체) · 카드 빌드 지점(`mocks/fixtures.ts`, 실 배선 후엔 `lib/api/cases.ts` 응답 매핑).

### B-2. PC 나비게이션 IA — 확정 (2026-07-17 결정, R5.5 종결)

- **확인 결과**: 목업 탭 높이는 전 화면 52px로 일관(64px 혼재는 목업이 아니라 코드 PC 헤더

  메시지 탭 active 그대로).
- **메시지 PC 강등 보완(필수)**: 컨트롤타워에 "응답 도착 N건" 지표 카드 + `/messages`

1. **즉시 브리프 작성 가능(스펙·참고 패턴 확보됨)**: A-1(응답 링크) · A-2(스캔 분류) ·
   A-3(알림 설정). 셋 다 기존 브리프 2종과 같은 형식으로 개별 파일을
   `reference/design-system/design-briefs/`에 추가하면 된다. **모든 브리프에 "플랫폼
   관례" 절의 패턴 번호(①/②/③)와 반대 폼팩터 처리(단일 반응형 / 양쪽 프레젠테이션 /
   PcOnlyNotice)를 명시** — 목업 캔버스(모바일 375px vs PC 1560px)도 그에 따라 지정한다.
2. **A-4(화이트라벨 v1)**: 법무 분류 확인이 선행 — 확인 전에 목업을 뽑으면 (a) 화면 구조가
   통째로 바뀔 수 있다.
3. **B-1·B-2**: 목업 재수입 전에 IA 결정(모바일 필터 공유 여부, 탭 라벨·높이)이 먼저.
4. **A-5(알림함)**: 범위 결정("필요한가")이 먼저 — 결정 전 브리프 금지.
5. **디자인 이후 절차(관례 고정)**: `.dc.html`을 `reference/design-system/`에 고정 +
   `README.md` 내용물 표 갱신 → `docs/DESIGN_SYNC_AUDIT_{날짜}.md`로 토큰 히스토그램 대조
   감사(임의 보정 금지, 결함은 기록으로) → `plans/ROADMAP.md`에 태스크 승격(레벨·스펙·DoD)
   → 구현. 목업의 부분 마스킹류가 재발하면 2026-07-13 선례대로 "디자인 내부 결함"으로
   기록하고 구현에는 반영하지 않는다.
